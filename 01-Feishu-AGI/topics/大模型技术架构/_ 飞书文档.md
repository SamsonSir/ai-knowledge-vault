### 📋 核心分析

**战略价值**: Harness Engineering 标志着 AI 工程范式从"人直接干预"到"人设计环境"的根本跃迁，是 2025-2026 年 AI 工程领域最重要的概念创新。

**核心逻辑**:
- **范式跃迁**: Prompt/Context Engineering 仍是"人给输入、AI 给输出"；Harness Engineering 是"人设计赛道，AI 自主奔跑"
- **核心公式**: Harness Engineering ≠ 优化模型 → 优化模型运行的"环境"
- **角色重构**: 工程师从"代码编写者"转变为"环境建筑师"，80% 时间投入构建约束、反馈与控制系统

---

### 🎯 关键洞察

**Agent 的结构性缺陷与"信任债务"**
Anthropic 观察到所有 Agent 的共性规律：面对复杂项目，第一反应是在单会话内完成全部功能，导致 context window 耗尽、半成品代码、零记录遗留。Cassie Kozyrkov 提出的"信任债务"概念揭示：AI 会填补指令空白进行"自信的即兴发挥"，未审计的假设当前看似合理，未来某刻集中爆发，逆向工程代价极高。

**约束换自主的悖论**
Birgitta Böckeler 的核心洞见：为了获得更高的 AI 自主性，运行时必须受到更严格的约束。层级依赖规则写成 Linter 而非文档、架构约束由确定性检查强制执行——规矩越明确，Agent 独立做的事越多。这与"LLM 可以生成任何语言、任何模式"的早期炒作直接矛盾。

**Harness 的指数价值曲线**
Latent Space 的"Big Model vs Big Harness"辩论并非非此即彼。短期简单任务，模型能力主导；长期复杂项目中，Harness 价值随任务复杂度和持续时间指数增长——熵增、上下文丢失、模式漂移、信任债务累积，使缰绳成为必需。

**LangChain 的定量验证**
同一模型（gpt-5.2-codex），仅改变 Harness，SWE-bench 排名从 30+ 跃升至 Top 5。关键改动：Plan-Build-Verify-Fix 强制闭环、环境上下文注入、死循环检测、推理三明治策略（规划 xhigh → 执行 high → 验证 xhigh）。

**Böckeler 的批判性平衡**
OpenAI 叙事存在明显缺失：文章聚焦内部质量和可维护性，缺少功能和行为的验证讨论；5 个月建设周期表明 Harness 非一蹴而就；OpenAI 存在让人相信"AI 可维护代码"的既得利益。

### 📦 可执行模块

| 模块 | 内容描述 | 适用场景 |
|------|---------|---------|
| **AGENTS.md 地图化** | 目录式指引而非百科全书，指向 architecture/、domains/、plans/、specs/、runbooks/ | 任何启动 Agent 协作的项目 |
| **机械化架构约束** | 层级依赖规则编码为 Linter，错误信息即教学材料，CI 直接拒绝违规 | 多模块、长期维护的代码库 |
| **进度文件跨会话记忆** | claude-progress.txt 作为 Agent 外部记忆，新会话首步读取+git log 断点续作 | 超 context window 的复杂任务 |
| **全标失败策略** | 功能初始状态标记"失败"，Agent 仅能修改状态字段，禁止删除/编辑测试用例 | 防止 Agent"降低标准"完成任务 |
| **Agent 互审机制** | Agent A 写代码 → Agent B 审代码 → 迭代至通过，人类仅介入架构重大决策 | 高频 PR 场景（日百级别） |
| **推理三明治** | 规划 xhigh → 执行 high → 验证 xhigh，有限推理资源的阶段化配置 | 成本敏感且质量要求高的任务 |
| **垃圾回收机制** | 后台清洁 Agent 定期扫描文档-代码不一致、架构违规、模式漂移 | 大规模、长周期代码库维护 |
