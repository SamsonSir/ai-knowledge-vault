### 📋 核心分析

**战略价值**: 用单一 WebSocket Gateway 作为控制平面，把 CLI/WebChat/移动端/浏览器自动化/消息通道统一成一个可调度、状态一致、可治理的 AI 系统，而非多套互不相认的 AI 分身。

**核心逻辑**:

- **多入口 ≠ 多套系统**：每多一个端，就多一套协议、状态、上下文同步逻辑。CLI 跑到一半的任务 Web 端看不到；手机端授权了定位桌面端还在反复询问；浏览器自动化执行到第 7 步消息通道只剩"处理中"——问题根源在架构，不在模型。
- **控制平面 vs 数据平面必须分离**：控制平面负责会话/状态管理、路由调度、权限策略、事件分发；数据平面负责模型推理、工具执行、多媒体传输。Gateway 只做调度与一致性，不承担计算与执行，避免变成塞满业务逻辑的巨无霸服务。
- **WebSocket 是控制平面的唯一合理选择**：AI 交互是全双工协作（模型持续吐 token、工具持续汇报进度、用户随时插话），HTTP+REST+轮询天然别扭。WebSocket 提供全双工、低延迟长连接、跨平台一致（CLI/浏览器/移动端/桌面端均可稳定作为 WS Client）。本地模式统一接入 `ws://127.0.0.1:18789`。
- **Hub & Spoke 星形拓扑，Gateway 是唯一事实源（Single Source of Truth）**：会话状态、任务执行进度、事件广播全部在 Gateway 收敛。客户端越做越薄，只负责输入与渲染，不各自维护复杂状态机。扩展新入口（如 Telegram bot）= 多一个节点，而不是多一套系统。
- **协议层用 JSON-RPC 2.0 + 命名空间**：结构规整、可演进，按能力域分组：`chat.*`（消息/会话/typing）、`agent.*`（编排/任务启停/工具触发）、`node.*`（注册/能力上报/心跳）、`browser.*`（自动化控制/DOM/截图回传）、`system.*`/`debug.*`（系统诊断）。
- **设备抽象为 Node + 能力声明（capabilities）**：iOS 节点声明 `["camera", "location", "notification"]`，CLI 节点声明 `["exec", "file_read"]`。Gateway 维护能力感知路由表，Agent 需要"定位"时精准路由到具备该能力的节点，而非盲目广播。
- **消息通道统一骨架，通道差异压到边缘层**：WhatsApp/Telegram/Slack 的 webhook/回调/签名/格式差异不污染核心逻辑，统一消息结构为 `{id, type, channel, payload, metadata}`，再做版本协商向后兼容、通道隔离互通、ACK+超时重试+离线持久化+历史回放。
- **浏览器自动化支持人机共存于同一 Session**：用户手动点击后 Agent 接管，或 Agent 跑流程时用户从手机发"暂停"，均通过统一状态协调器处理，避免"到底谁在控制"的混乱。
- **安全模型分层**：local-only / token / tailscale 多模式，配合细粒度权限（admin/read/write/pairing）。
- **可观测性是演进前提，不是锦上添花**：全链路 tracing、实时指标、统一日志——多端协作出问题，没有这套根本查不清。

### 🎯 关键洞察

**为什么"控制平面"这个抽象是关键**：

传统做法是"能力 + 入口"捆绑——每个端自己维护 Agent 逻辑、自己管状态、自己做推送。这在单端时没问题，但一旦多端协作，状态漂移是必然结果（网络切换、后台挂起、多标签页、并发操作）。

OpenClaw 的反直觉之处在于：**先不做更漂亮的 UI，先把"调度大脑"抽出来**。这个顺序决定了系统能否真正统一。

原因链：
> 控制平面统一 → 状态在 Gateway 生产与分发，客户端只是"显示器" → 扩展新端不需要重写 Agent 逻辑 → 系统整体性出现

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| Gateway 本地接入地址 | `ws://127.0.0.1:18789` | 所有节点统一接入点 | 生产环境需配合 token/tailscale 鉴权 |
| JSON-RPC 消息格式 | `{"jsonrpc":"2.0","method":"chat.sendMessage","params":{"peer":"tg:12345","text":"Hello"}}` | 统一协议面，可演进 | method 命名空间必须规范，避免后期混乱 |
| iOS 节点能力声明 | `{"capabilities":["camera","location","notification"]}` | Gateway 精准路由到该节点 | 能力声明要与实际权限一致，否则路由到位但执行失败 |
| CLI 节点能力声明 | `{"capabilities":["exec","file_read"]}` | Agent 需要执行命令时路由到 CLI | exec 能力需严格权限控制 |
| 统一消息骨架 | `{"id":"unique-msg-id","type":"request\|response\|event","channel":"cli\|webchat\|mobile\|browser","payload":{},"metadata":{}}` | 通道差异压到边缘，核心逻辑不感知平台 | id 必须全局唯一，用于 ACK 和去重 |
| 消息路由类型 | 单播 / 多播 / 广播 | 一个事件可同步多端 | 优先级队列：中断/取消 > 普通消息，否则取消指令可能被堵塞 |
| 安全模式 | local-only / token / tailscale | 按部署场景选择 | 权限粒度：admin/read/write/pairing，pairing 建议用 6 位验证码 |
| 移动端连接稳定性 | 自适应重连 + 离线消息队列 + 设备配对（6位验证码） | 应对网络切换/后台挂起 | iOS 系统限制长连接，需心跳保活策略 |

### 🛠️ 操作流程

**案例一：CLI 发起任务，多端同步**

1. **CLI 输入 prompt** → 包装成 `chat.sendMessage` 控制消息发送至 Gateway
2. **Gateway 路由** → 转发至 Agent 或触发 Agent 编排
3. **模型流式返回** → Gateway 以事件形式把 token/进度广播到同一 Session 的所有订阅者
4. **WebChat + Mobile 同步展示** → 同一条流式回复，状态在 Gateway 统一生产，客户端只渲染

**案例二：WebChat 下发浏览器自动化**

1. **用户在 WebChat 输入**："帮我登录并截图订单页"
2. **Gateway 判断需要 `browser.*` 能力** → 调度 Browser Node（Puppeteer/Playwright）
3. **Browser Node 执行** → 持续上报步骤事件、截图、DOM 变化、失败原因
4. **Gateway 推送截图/进度** → 所有相关端（包括手机端）同步看到同一张截图、同一进度条
5. **可中断**：手机端发"停止" → Gateway 接收后下发中断指令，Browser Node 停止执行

### 📝 避坑指南

- ⚠️ **不要把控制平面做成"又一个业务服务"**：Gateway 只做调度与一致性，一旦开始在里面写业务逻辑，它就会变成新的巨无霸，失去统一控制平面的意义。
- ⚠️ **能力声明必须与实际权限对齐**：节点声明了 `location` 但系统权限未授予，Gateway 路由成功但执行失败，排查成本极高。
- ⚠️ **移动端长连接不稳定是必然，不是 bug**：Wi-Fi/4G/5G 切换、后台挂起都会断连，必须在设计阶段就加入自适应重连 + 离线消息队列，而不是上线后打补丁。
- ⚠️ **优先级队列不做会导致取消指令被堵塞**：中断/取消消息必须比普通消息有更高优先级，否则任务跑飞了取消指令还在队列里排队。
- ⚠️ **可观测性不是后期再加的事**：多端协作出问题时，没有全链路 tracing 和统一日志，根本无法定位是哪个节点、哪个事件、哪一步出了问题。
- ⚠️ **消息 id 必须全局唯一**：用于 ACK 确认和去重，重复消息在多端广播场景下会造成状态错乱。
- ⚠️ **协议命名空间要从第一天就规范**：`chat.*`/`agent.*`/`node.*` 等分组一旦混乱，后续扩展新端时会产生大量历史包袱。
