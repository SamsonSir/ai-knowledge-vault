### 📋 核心分析

**战略价值**: DeepSeek 提出 DualPath 双路径 KV-Cache 加载架构，通过激活解码引擎闲置存储网卡，在零硬件成本下将 Agentic LLM 推理吞吐量提升 1.87×。

**核心逻辑**:

- **Agentic LLM 的本质是多轮长上下文交互**：不是单轮问答，而是代码助手、任务 Agent 等与环境（编辑器、浏览器、Python 解释器）反复交互，平均 157 轮对话，平均上下文长度 32.7k tokens，每轮仅追加 429 tokens 新内容。
- **KV-Cache 命中率高达 98.7%**：计算公式 `(32700 - 429) / 32700 = 98.7%`，意味着每轮推理中 98.7% 的内容是重复加载的历史数据，只有 1.3% 是新计算内容。
- **推理已是 I/O-bound，不是 Compute-bound**：Cache-Compute 比率（要读的数据量 / 要算的计算量）：Qwen2.5-32B 高达 117–267 GB/PFLOP，GPT-OSS-120B 为 47–95，DeepSeek-V3.2 660B 最优也有 13–36。即使最优模型，I/O 需求也是计算需求的 13–36 倍。
- **传统 PD 分离架构的结构性缺陷**：预填充引擎（PE）负责从分布式 SSD 存储加载全部 KV-Cache，其存储网卡（SNIC）带宽上限约 400 Gbps，高并发时迅速饱和；而解码引擎（DE）的 SNIC 在解码阶段几乎完全闲置（KV-Cache 已在 HBM 中，无需再读存储）。
- **DualPath 核心思路：让闲置网卡一起搬砖**：新增路径 B：`存储 → 解码引擎 SNIC → RDMA 计算网络 → 预填充引擎`，与原有路径 A（`存储 → 预填充引擎 SNIC`）并行运行，聚合两侧 SNIC 带宽。
- **全局调度器动态分流**：不能让所有请求都走路径 B（否则 DE 的 SNIC 又饱和），调度器根据存储 NIC 队列长度、GPU 负载、请求特征动态决定每个请求走哪条路径。
- **CNIC-centric 流量隔离防止干扰**：计算网络（CNIC）用于模型推理通信（AllReduce、AllToAll），KV-Cache 传输复用该网络时，通过虚拟 lanes（VLs）隔离：模型推理优先级高（99% 带宽），KV-Cache 传输低优先级（1% 带宽），不影响推理延迟。
- **层式传输（Layer-wise Transfer）隐藏传输延迟**：KV-Cache 按 Transformer 层组织，计算第 N 层时同步加载第 N+1 层的 KV-Cache，计算与数据传输完全重叠，消除串行等待。
- **大规模线性扩展验证**：从 2P4D（2K agents）扩展到 48P96D（48K agents），JCT 仅从 3,167s 增至 3,201s，实现近线性加速；在线服务 44P88D 配置下吞吐量达 8.8 APS vs 基线 0.4 APS，提升 22 倍。
- **零硬件成本**：DualPath 是纯软件/架构层优化，不需要加网卡、换 GPU、升级存储，只改变数据流动方式。

### 🎯 关键洞察

**为什么 I/O 成为主要矛盾**：

- 原因：Agentic 场景下，每轮推理的新计算量极小（429 tokens），但需要重新加载 32.7k tokens 的历史 KV-Cache。随着 Agent 轮数增加，KV-Cache 体积线性增长，而每轮新增计算量几乎不变，I/O/Compute 比率持续恶化。
- 动作：传统系统只优化计算（更快的 GPU、更好的算子），对 I/O 路径没有重新设计。
- 结果：GPU 算力大量空转等待数据，系统吞吐量被存储带宽卡死。

**为什么解码引擎 SNIC 是闲置的**：

- 解码阶段逐 token 生成，KV-Cache 已经在 HBM 中（由预填充阶段加载完毕），解码引擎不需要再访问存储，其 SNIC 在整个解码过程中无事可做。
- DualPath 的洞察：这段闲置时间恰好可以用来预加载下一轮请求的 KV-Cache，时间窗口天然对齐。

**CNIC 带宽为何可以复用**：

- 模型推理的 AllReduce/AllToAll 通信是亚毫秒级突发，两次突发之间存在大量空闲窗口。
- KV-Cache 传输可以填充这些空隙，通过 VL 优先级机制确保推理通信永远优先，KV-Cache 传输只用剩余带宽。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/参数 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| 路径 A（传统路径） | 存储 SNIC → PE，带宽上限 ~400 Gbps | 处理低并发请求 | 高并发时 SNIC 饱和，成为瓶颈 |
| 路径 B（新增路径） | 存储 SNIC → DE → RDMA CNIC → PE | 激活 DE 闲置带宽 | 需全局调度器协调，防止 DE SNIC 过载 |
| 全局调度器 | 输入：SNIC 队列长度 + GPU 负载 + 请求特征；输出：路径选择 | 动态均衡两条路径负载 | 调度算法复杂度直接影响分流效果 |
| CNIC 流量隔离 | VL 优先级：推理通信 99% 带宽，KV-Cache 传输 1% 带宽 | 推理延迟不受影响 | VL 配置错误会导致推理通信被抢占 |
| 层式传输 | 计算第 N 层时异步加载第 N+1 层 KV-Cache | 隐藏传输延迟，计算与 I/O 重叠 | 需要精确的层级同步机制，否则出现数据竞争 |
| 扩展配置（离线） | 2P4D → 48P96D（最大 1,152 GPU） | 近线性 JCT 扩展（3,167s vs 3,201s） | 超大规模下调度器本身可能成为新瓶颈 |
| 扩展配置（在线） | 44P88D | 吞吐量 8.8 APS，TTFT 显著降低，TBT 稳定 | 在线场景需额外关注尾延迟（P99） |

### 🛠️ 操作流程

1. **准备阶段：识别系统瓶颈**
   - 测量当前系统的 Cache-Compute 比率（KV-Cache 读取量 GB / 实际计算量 PFLOP）
   - 监控预填充引擎 SNIC 利用率与解码引擎 SNIC 利用率，确认是否存在"PE 饱和 / DE 闲置"的不均衡
   - 统计 Agentic workload 的平均轮数、平均上下文长度、每轮追加 token 数，计算 KV-Cache 命中率

2. **核心执行：部署 DualPath**
   - 在现有 PD 分离架构基础上，新增路径 B 的数据通路：DE SNIC → RDMA → PE
   - 部署全局调度器，接入 SNIC 队列长度、GPU 负载、请求特征三类信号，实现动态路径分配
   - 配置 CNIC 虚拟 lanes（VLs）：推理通信 VL 设为高优先级（99% 带宽），KV-Cache 传输 VL 设为低优先级（1% 带宽）
   - 实现层式传输逻辑：在计算第 N 层时，异步触发第 N+1 层 KV-Cache 的加载请求

3. **验证与优化**
   - 对比 DualPath 启用前后的吞吐量（APS）、TTFT、TBT 指标
   - 验证 CNIC 上推理通信延迟未受影响（重点检查 AllReduce/AllToAll 的 P99 延迟）
   - 在不同并发规模（2P4D → 48P96D）下验证近线性扩展性
   - 调优调度器的分流策略，避免 DE SNIC 成为新瓶颈

### 📝 避坑指南

- ⚠️ **DE SNIC 过载风险**：若调度器将过多请求路由至路径 B，解码引擎的 SNIC 会成为新瓶颈，替代原来的 PE SNIC 瓶颈。调度器必须实时监控 DE SNIC 队列长度并动态限流。
- ⚠️ **CNIC 优先级配置错误**：若 VL 优先级设置不当，KV-Cache 传输流量会抢占推理通信带宽，导致 AllReduce/AllToAll 延迟上升，直接影响模型推理质量和延迟。
- ⚠️ **层式传输的同步问题**：计算第 N 层时异步加载第 N+1 层，若同步机制不精确，可能出现计算已到第 N+1 层但数据尚未就绪的情况，引发阻塞或数据竞争。
- ⚠️ **非 Agentic 场景收益有限**：DualPath 的收益来源于高 KV-Cache 命中率（98.7%）。对于单轮对话或短上下文场景，KV-Cache 命中率低，I/O 不是瓶颈，DualPath 的提升效果会大幅缩水。
- ⚠️ **调度器本身的扩展性**：超大规模（48P96D，48K agents）下，全局调度器需要处理海量请求的路径决策，调度器本身可能成为新的中心化瓶颈，需要分布式调度设计。
