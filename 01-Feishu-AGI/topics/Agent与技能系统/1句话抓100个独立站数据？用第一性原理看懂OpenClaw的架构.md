### 📋 核心分析

**战略价值**: 通过"抓取100个独立站"这个极端场景，从第一性原理推演出 OpenClaw 的五层单向数据流架构，以及其纯文本认知系统的底层逻辑，让你能看懂报错、能指挥 AI 改代码、能手搓复刻。

**核心逻辑**:

- **大模型本地运行的三要素**：输入端（接收文字指令）+ 本地代码运行环境 + 本地文件读写存储。三者缺一不可，OpenClaw 本体就是一个后台运行的 TypeScript CLI 进程。
- **第一层 Channel Adapters（通道适配器）**：飞书/Telegram 发来的是带鉴权、富文本标签的加密包，大模型无法直接解析。适配器负责将所有异构平台数据统一解析转换为标准 JSON 格式对象，再向下传递。
- **第二层 Gateway Server（网关服务器）+ Lane Queue（车道队列）**：并发请求会导致状态污染——大模型正在操纵本地浏览器时，另一条指令插入会直接崩溃任务。网关强制执行串行规则：同一会话内所有指令严格排队，前一任务未完成，后续指令绝对不许插队。
- **第三层 Agent Runner（智能体运行器）**：大模型开始推理前，该模块读取本地 browser-use 技能库说明，注入 System Prompt，同时监控 Token 用量，防止 100 个网站的 HTML 数据把上下文撑爆。
- **第四层 Agentic Loop（代理执行循环）**：核心死循环机制。大模型输出代码指令 → 本地执行 → 返回结果 → 大模型再次思考，循环直到任务结束。抓取 100 个网站共经历约 102 轮循环（1轮读CSV + 100轮抓取 + 1轮Python汇总）。
- **反爬核心突破**：browser-use 调用时在参数中附加本地真实浏览器的缓存和状态文件路径。无头浏览器启动后，电商平台检测到的是拥有完整历史记录、处于已登录状态的合法用户，绕过登录墙和指纹检测。
- **第五层 Response Path（响应路径）**：任务完成后以 Streaming 流式传输方式将报告推回飞书通道适配器，发到手机。同时将 100 多轮循环中每次网页访问、每次脚本重试，追加写入本地 `.jsonl` 日志文件，实现 100% 可追溯。
- **纯文本认知系统（workspace 文件夹）**：彻底抛弃向量数据库和 RAG，用本地 Markdown 文件实现持久记忆。每次新指令进入 Agent Runner 时，系统强制读取所有 MD 文件全文，硬拼接在 System Prompt 最前面。
- **heartbeat.md + Cron 定时器**：记录需要周期性执行的任务，配合系统 Cron 定时器，让 Agent 能主动醒来，从第一层发起自我执行流，实现无人值守自动化。
- **nano-claw 验证了架构的极简本质**：GitHub 上不到 150 行代码复刻了 OpenClaw 核心逻辑：极简 Agent 循环 + 基础工具集 + 文本状态机。掌握"执行权限注入"和"本地文件闭环"两个抓手，几分钟内可构建具备物理操作能力的 Agent。

### 🎯 关键洞察

**为什么五层架构是必然而非设计**：每一层都是被真实报错逼出来的——不是过度设计，是最小必要解。Channel Adapters 解决数据格式不兼容，Gateway 解决并发状态污染，Agent Runner 解决工具定义缺失，Agentic Loop 解决单次推理能力上限，Response Path 解决结果触达和审计。去掉任何一层，系统在真实业务场景下必然崩溃。

**为什么纯文本比向量数据库更适合这个场景**：向量数据库解决的是"从海量文档中检索相关片段"的问题。但 OpenClaw 的记忆文件（soul.md、user.md、agents.md、heartbeat.md）总量极小，全文注入的成本远低于检索的工程复杂度。更重要的是，全文注入保证了 100% 的信息完整性，不存在检索遗漏的风险。

**大模型自我更新偏好的闭环**：当你通过飞书告诉系统"以后报告都要加粗标题"，大模型会在 Agentic Loop 内主动调用文件覆写工具，将这个偏好写入 `user.md`。下次任务启动时，Agent Runner 读取更新后的 `user.md` 并注入 System Prompt，偏好在物理层面永久生效。这是一个不依赖任何外部服务的自我进化闭环。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|---|---|---|---|
| Channel Adapters | 将飞书/Telegram 异构数据解析为标准 JSON 对象 | 大模型可直接读取指令 | 各平台鉴权方式不同，适配器需分别处理 |
| Gateway Server Lane Queue | 同一会话内强制串行，前任务未完不允许插队 | 防止并发状态污染导致浏览器崩溃 | 串行会降低吞吐量，多任务场景需规划好队列顺序 |
| Agent Runner | 读取 browser-use 技能库说明注入 System Prompt，监控 Token 用量 | 大模型知道如何调用本地工具 | Token 监控必须开启，100个网站 HTML 极易撑爆上下文 |
| Agentic Loop | 死循环：输出代码 → 本地执行 → 返回结果 → 再推理 | 完成多步骤复杂任务 | Bug 多，报错时需明确告诉 AI 是哪一层出错，否则 AI 会乱改 |
| browser-use 反爬 | 调用时附加本地真实浏览器配置文件路径参数 | 绕过登录墙和指纹检测 | 依赖本地浏览器已登录状态，换机器需重新配置 |
| Response Path | Streaming 流式推回飞书，追加写入本地 `.jsonl` 日志 | 结果实时触达 + 全程可追溯 | `.jsonl` 日志长期运行会膨胀，需定期清理 |
| soul.md | 纯文本写死大模型基础设定，如"你是严谨的跨境电商数据分析师" | 构成系统安全底线，不可逾越 | 不要轻易修改，改错会影响所有任务的基础行为 |
| user.md | 记录个人偏好，大模型可在 Loop 内自动覆写更新 | 持久化用户习惯 | 大模型自动写入时可能覆盖你手动设置的内容，注意备份 |
| agents.md | 规定特定任务（如电商抓取）的拆解步骤和 SOP | 标准化任务执行流程 | SOP 写得越细，执行越稳定，模糊描述会导致 Loop 乱跑 |
| heartbeat.md | 记录周期性任务，配合 Cron 定时器触发自我执行流 | Agent 主动醒来无人值守执行 | Cron 表达式配置错误会导致任务不触发或频繁触发 |

### 🛠️ 操作流程

1. **准备阶段**:
   - 确认本地已安装 Node.js 环境（OpenClaw 本体为 TypeScript CLI 进程）
   - 在本地浏览器中提前登录所有 100 个电商平台后台，确保 Cookie/Session 有效
   - 准备包含 100 个店铺网址的 CSV 文件，放入 workspace 目录
   - 配置 workspace 文件夹下四个核心 MD 文件：`soul.md`（基础设定）、`user.md`（个人偏好）、`agents.md`（任务 SOP）、`heartbeat.md`（定时任务）

2. **核心执行**:
   - 通过飞书发送指令："打开我100个跨境电商平台的店铺后台抓取今天的销售数据，整理成综合数据报告发给我"
   - Channel Adapters 解析飞书富文本包 → 转为标准 JSON
   - Gateway Server 鉴权 → Lane Queue 排队
   - Agent Runner 读取 browser-use 说明 + 四个 MD 文件 → 拼接 System Prompt → 监控 Token
   - Agentic Loop 启动死循环：
     - 第1轮：读取本地 CSV 文件获取 100 个网址
     - 第2-101轮：调用 browser-use，附加本地浏览器配置文件路径，逐一访问各平台后台，提取 HTML 结构化数据
     - 第102轮：调用 Python 环境，写脚本对数据排序求和，生成综合报告

3. **验证与优化**:
   - Response Path 以 Streaming 方式将报告推回飞书
   - 检查本地 `.jsonl` 日志，核查每轮循环的访问记录和重试情况
   - 如需调整报告格式，直接通过飞书告知系统，大模型会自动更新 `user.md`
   - 如需定时自动执行，在 `heartbeat.md` 中写入任务描述和触发频率，配置系统 Cron

### 📝 避坑指南

- ⚠️ **报错时不要让 AI 自由发挥**：必须先判断是哪一层出的问题（适配器/网关/Runner/Loop/响应路径），再告诉 AI 具体改哪一层，否则 AI 会乱改导致更多 bug。
- ⚠️ **Token 监控不能省**：100 个网站的 HTML 数据量极大，Agent Runner 的 Token 监控必须开启，否则上下文溢出后任务静默失败。
- ⚠️ **Lane Queue 串行会阻塞**：任务执行期间不要发送无关指令（如查天气），会进入队列等待，不会立即响应。
- ⚠️ **本地浏览器登录状态依赖机器**：browser-use 挂载的是本机浏览器缓存，换机器或清除浏览器数据后需重新登录所有平台。
- ⚠️ **agents.md SOP 必须写细**：任务拆解步骤越模糊，Agentic Loop 越容易跑偏，建议参考标准 SOP 格式逐步细化。
- ⚠️ **`.jsonl` 日志长期膨胀**：长期运行后日志文件会持续增大，需定期归档或清理，否则影响磁盘空间。
- ⚠️ **soul.md 轻易不要动**：它是系统行为的安全底线，改错会影响所有任务的基础人格和边界设定。
