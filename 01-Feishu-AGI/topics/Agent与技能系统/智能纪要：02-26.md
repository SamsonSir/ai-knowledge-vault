### 📋 核心分析

**战略价值**: Selfware 协议将文件从静态数据容器升级为自包含、可进化的软件实体，彻底解决 AI agent 时代跨平台上下文流转的摩擦问题。

**核心逻辑**:

- **痛点根源**：传统文件与特定软件强绑定（PSD→Photoshop、DOCX→Word），格式兼容性差，agent 之间传递上下文时数据丢失、逻辑断裂，无法无损流转。
- **核心公理**：`a file is an APP, everything is a file`——传统软件三层结构（数据层/逻辑层/交互层）在 Selfware 中通过协议指定视图动态生成，AI 抹平视图与逻辑层差异。
- **文件结构**：`.self` 文件 = `selfware.md`（协议引用文件）+ `manifest`（定义启动、打包、安全范围、权限）+ `content/` 目录（所有写入操作只能由 agent 执行）。
- **多视图能力**：同一数据源可按需渲染为文档、演示文稿、脑图、大纲、卡片等视图，视图和内容均可随时修改，还可导出为传统软件格式或图片。
- **自进化机制**：通过 `discovery` 服务，文件携带意图和部分上下文，自动去 GitHub 等平台查找相似任务和技能，更新后询问用户是否采用，实现能力增强而无需人工干预。
- **协同与共享**：支持单文件打包分享或基于 Git 仓库协同，接收方打开后获得完整内容+操作记录，可继续迭代；多人多 agent 可基于同一 `.self` 文件并行工作，文件自动同步合并。
- **安全与权限**：协议默认排除敏感信息（环境个性化信息、Git 记录、`.ENV` 文件内容），支持文件加密，用户可手动控制打包粒度，后续将通过 hook 机制阻止 agent 读取敏感配置。
- **开放无锁定**：协议完全开放，可自定义后缀名（如 `.videoagi`、`.AJ`），所有 coding agent 均可支持，不与任何平台耦合，数据归用户所有。
- **长程运行记录**：将 agent 长程运行（如 25 小时、20 次迭代）的历史、关键决策、设计意图、技能使用记录打包进 `.self` 文件，他人接收后不仅得到软件界面，还能复刻完整开发能力。
- **经济系统重构**：Remy 提出 Web 4.0 形态——生产力单位从"人工作量"转变为"将 TOKEN 转化为价值的能力"，早期开发者通过 Selfware 技能复用，可从后续使用者产生的红利中持续获利。

### 🎯 关键洞察

**为什么 `.self` 文件比 DOCX/ZIP 更适合 agent 时代**：

DOCX 本质是 ZIP 包（改扩展名为 `.zip` 即可解压），其媒体文件存于临时目录，分享 Markdown/HTML 时图片路径极易断裂。`.self` 文件将所有相关资源自包含封装，解决了传统文件分享时的资源裂开问题，同时支持加密和权限分层管理。

**entry point 设计逻辑**：协议将入口点分为两类——`action`（交互触发）和 `script`（脚本执行），这使得带服务器依赖的复杂程序也能通过 Selfware 封装，本地运行避免了与云服务交互的障碍，依赖直接装入文件包内。

**复杂程序适配性**：少卿明确指出，带服务器的复杂程序反而是 Selfware 的最佳适用场景，因为可以将所有依赖内置，本地运行，彻底绕开云服务配置障碍。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| 核心协议文件 | `selfware.md`（引用 GitHub 协议）+ `manifest` | 定义软件启动、打包范围、安全边界、权限 | manifest 必须明确 entry point 类型（action/script） |
| 数据写入目录 | `content/` 目录，仅 agent 可操作 | 隔离用户数据与软件逻辑 | 用户不得直接写入，需通过 agent 操作 |
| 文件打包分享 | 单文件模式 或 Git 仓库模式 | 单文件：离线分享；Git：自动同步更新 | Git 模式需双方互信，协议默认排除 Git 记录元数据 |
| 安全排除项 | 默认排除：环境个性化信息、Git 记录、`.ENV` 文件 | 防止 API key 等敏感信息泄露 | 后续用 hook 阻止读取 `.ENV`，当前需手动检查 |
| discovery 服务 | 携带意图+部分上下文 → 查询 GitHub 相似技能 → 询问用户确认 | 文件自动进化，无需手动更新 skills | 更新前会询问用户，非强制自动更新（协议明确禁止自动更新） |
| entry point | `action`（交互按钮触发）/ `script`（脚本直接执行） | 降低使用门槛，支持复杂程序封装 | 推荐用 agent 驱动打开，运行失败让 agent 继续尝试 |
| 视图生成 | 文档视图 / 演示文稿视图 / 脑图视图 / 大纲视图 / 卡片视图 | 同一数据源多形态呈现 | 视图即时生成，取决于 agent 能力强弱 |
| 文件加密 | 协议支持对 `.self` 文件加密 | 实现权限管理，控制内容访问 | 具体加密方案需结合软件工程实现 |

### 🛠️ 操作流程

**AJ 实践 Selfware 协议的完整步骤（基于 Claude Code）**：

1. **安装阶段**: 使用 Claude Code 与 AI 对话，执行安装指令，创建 Selfware 实例，了解项目用法（通过 `manifest` 和 `selfware.md` 引导 agent 理解项目结构）。

2. **创建内容**: 让 agent 写文章、创建日记系统，系统以 `.self` 文件形式存在，本地运行后通过 `localhost` 访问，支持 Markdown 格式，随时记录保存。

3. **进化迭代**: 让日记系统自我进化——添加游戏化系统、时光胶囊功能；让 agent 自己写日记，实现人与 agent 共创内容。

4. **分享流程**: 将 `.self` 文件打包（可选择是否包含内容），发送给他人；接收方下载解压，通过 agent 运行（运行失败直接让 agent 继续尝试，无需手动排查）。

5. **协同更新**: 若基于 Git 仓库共享，文件变更自动同步，接收方可在原有基础上继续迭代，操作记录完整保留。

**少卿 Demo 操作流程**：

1. 本地运行 `.selfware` 文件
2. 浏览器打开视图层（`localhost` 端口）
3. 直接修改文档内容 或 指令 agent 更新文档/软件（如插入图片、新增主题）
4. 切换视图（文档→演示文稿→脑图）
5. 打包分享：选择单文件模式或 Git 仓库模式

### 📝 避坑指南

- ⚠️ **API key 暴露风险**：部署涉及第三方服务（如小龙虾）时，协议默认去除敏感信息，但当前版本需手动检查 `.ENV` 文件是否被打包，后续版本将用 hook 强制阻止。
- ⚠️ **文件打不开/运行失败**：不要手动排查，直接让 agent 运行文件，agent 能力足够强，运行失败可指令其继续尝试，通常可自行修复。
- ⚠️ **协议禁止自动更新**：discovery 服务找到新技能后必须询问用户确认，协议明确禁止文件自动更新，防止未经授权的能力注入。
- ⚠️ **视图质量依赖 agent 能力**：同一 `.self` 文件在不同 agent 上的渲染效果不同，每个人的使用体验取决于其所用 agent 的能力上限。
- ⚠️ **Git 协同需双方互信**：基于 Git 仓库的自动同步模式要求协作双方建立信任关系，否则建议使用单文件打包分享模式。
- ⚠️ **content 目录权限**：所有写入操作必须通过 agent 执行，用户不得直接操作 `content/` 目录，否则破坏文件自包含结构。
