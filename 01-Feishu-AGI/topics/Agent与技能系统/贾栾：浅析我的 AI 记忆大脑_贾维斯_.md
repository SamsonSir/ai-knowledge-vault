### 📋 核心分析

**战略价值**: 用 SQLite + MCP Server + Skills 三件套，在 Trae IDE 内搭建五层分级记忆系统，彻底解决"每次对话都要重新交代背景"的痛点，实现真正按需加载的个人 AI 助理。

**核心逻辑**:

- **上下文 = 成本，不是越多越好**：AI 回复质量取决于输入质量，塞满无关记忆反而会"自毁大脑"，导致回复越来越乱（1.0 版的实测教训）
- **1.0 版失败原因**：运行半个多月后，随着记忆条目增多，AI 回复质量下降——根因是"无差别堆砌记忆"，没有分层、没有权重、没有过期机制
- **架构三件套**：SQLite（物理存储）+ MCP Server（实时检索/写入）+ Skills（文件落盘+索引）缺一不可，三者分工明确
- **五层记忆分级**（L0-L4）：L4 宪法层永远置顶 → L3 习惯层按需加载 → L2 知识层自动归档 → L1 对话流水可审计 → L0 临时上下文用完即焚
- **幽灵召回机制**：Active 记忆不足时，自动从 Archived 状态"复活"相关记忆；不需要时绝不出现，避免噪声污染上下文
- **画像注入上限控制**：有 context_hints 时最多注入 `MAX_PROFILE_ITEMS=12` 条 L2/L3（按 weight + last_accessed 排序）；无 hints 时收紧为 `PROFILE_ITEMS_NO_HINTS=3` 条，防止无关常驻记忆冒充场景召回
- **usage_count 语义严格**：只计"被检索命中/明确采纳"次数，系统自动预加载不计入，保证指标真实反映记忆价值
- **相同记忆自动合并**：随使用量增加，重复记忆按算法合并而非新增，防止数据库膨胀
- **与 Clawdbot 的本质差异**：Clawdbot = "文件即真相"（解决持久化）；Jarvis = "记忆变成可调度能力"（解决智能化），走的是"记忆→决策→执行"闭环路线
- **Skills 本质**：按需加载的提示词 + 固化功能脚本，可实现"自进化"（把每次错误记录为下次上下文语料）

### 🎯 关键洞察

**为什么分层是核心设计决策**：

1.0 版的失败证明"存多少"不是关键，"怎么取、何时取、如何保证一致性"才是。L4 永远置顶保证价值观/底线不漂移；L0 用完即焚避免临时噪声污染长期记忆；中间三层按场景动态调度，这是整个系统的灵魂。

**画像注入的反直觉原则**：

`read_user_profile(context_hints="coding")` 这个调用方式说明：场景标签（hints）是触发精准召回的钥匙。没有 hints 时系统主动收紧到 3 条，这是防止"上下文被噪声占满导致真实问题被阉割"的工程保障。

### 📦 配置/工具详表

| 模块 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|------|-------------|---------|-----------|
| SQLite 数据库 | 文件名：`memory.db`，存储记忆、索引、权重 | 物理持久化所有记忆 | 单文件，注意备份 |
| MCP Server | 随 Trae 启动，实时读写 `memory.db` | 聊天时自动检索/存入记忆 | 必须保持常驻，否则记忆系统失效 |
| Skills: knowledge-gardener | `allowed-tools: [Read, Write, Edit, Bash]`，`license: MIT`，`source: local` | 高价值内容落盘为本地文件 | 长内容必须先调此 Skill 落盘，再存路径索引到 MCP |
| 画像注入（有 hints） | `read_user_profile(context_hints="coding")`，上限 `MAX_PROFILE_ITEMS=12` | 精准注入相关 L2/L3 记忆 | 按 weight + last_accessed 排序，非全量注入 |
| 画像注入（无 hints） | 上限 `PROFILE_ITEMS_NO_HINTS=3` | 防噪声，只注入极少量通用记忆 | 全未命中时回退为"通用 Top 3" |
| 资产落盘脚本 | `python scripts/save_asset.py --path "相对路径" --content "内容"` | 自动创建目录、防覆盖、返回绝对路径 | 落盘后必须回调 MCP `update_profile` 登记索引 |
| 记忆驾驶舱 | AI 生成的 H5 页面 | 按层级/内容/标签/文件筛选记忆，查看流水 | 规划中功能：人物画像 |

### 🛠️ 操作流程

**1. 准备阶段**

- 在 Trae IDE 中创建项目，建立目录结构：
  - `20_提效脚本/{分类}/` — 脚本工具
  - `40_知识库/{分类}/` — 知识/SOP
  - `30_内容创作/32_草稿箱/` — 草稿素材
- 初始化 `memory.db`（SQLite）
- 配置并启动 MCP Server（随 Trae 启动）

**2. 配置 Skills（knowledge-gardener）**

在项目根目录创建 Skill 配置：

```yaml
---
name: knowledge-gardener
description: Jarvis 记忆系统的资产化操作员。负责将高价值信息（如脚本、教程、长文、复杂逻辑）物理落盘为本地文件 (Markdown/Python)，并遵循"前店后厂"的目录规范。当 AI 需要保存长内容到记忆时，必须先调用此 Skill 落盘，再将路径索引存入 MCP。
allowed-tools: [Read, Write, Edit, Bash]
license: MIT
source: local
---
```

**3. 资产落盘 SOP（四步必须全走）**

- 资产识别：Python/JS 脚本 > 10 行、完整 SOP/教程、项目配置文件、深度分析报告，触发 knowledge-gardener
- 路径规划：按内容类型选目录（参考 `20_提效脚本/mcp_memory_server/README.md` 中"📂 全景地图"）
- 物理落盘：

```bash
# 保存脚本
python scripts/save_asset.py --path "20_提效脚本/图像工具/gen_art.py" --content "import os..."

# 保存 SOP
python scripts/save_asset.py --path "40_知识库/10_效率手册/SOP_New_Project.md" --content "# SOP..."
```

- 索引登记（落盘后必须执行）：调用 MCP `update_profile`，Layer=`L3_knowledge`，Content="{资产名称} 已归档"，Ref Path={绝对路径}

**4. 日常使用与记忆更新**

- 每天晚上执行日报总结，AI 自动将新记忆归档存储
- 有立刻生效的记忆无需等日报，直接指令要求存入
- 查询历史：直接向 Jarvis 提问历史对话内容，从数据库检索

**5. 验证与优化**

- 用记忆驾驶舱（H5 页面）按层级/标签/文件筛选，验证记忆是否正确归档
- 查看记忆流水，用于历史溯源
- 观察 usage_count 变化，判断哪些记忆真正被采纳

---

### 💡 具体案例/数据

- 1.0 版实测运行超过半个月，记忆条目增多后 AI 回复质量下降，直接触发 2.0 重构
- 画像注入上限：有 hints = 12 条，无 hints = 3 条，两个硬编码参数是防止上下文膨胀的关键
- 日报总结场景：每天晚上固定执行，新记忆自动归档；紧急记忆可随时手动触发存入
- 记忆驾驶舱功能：按层级筛选、按内容/标签/文件搜索、查看全量记忆流水

---

### 📝 避坑指南

- ⚠️ 记忆不分层直接堆砌 = 1.0 版失败的根因，随条目增多回复质量反而下降
- ⚠️ 长内容不能直接存数据库，必须先用 knowledge-gardener 落盘为文件，再存 `ref_path` 索引
- ⚠️ 无 context_hints 时系统只注入 3 条记忆，不要误以为系统"失忆"，这是主动防噪声设计
- ⚠️ `usage_count` 不统计系统自动预加载，只统计被明确检索/采纳的次数，不要用它衡量"曝光量"
- ⚠️ 落盘后必须回调 `update_profile` 登记索引，否则文件存在但大脑不知道，等于白存
- ⚠️ L4 宪法层必须永远置顶，不能被场景召回逻辑覆盖或稀释
