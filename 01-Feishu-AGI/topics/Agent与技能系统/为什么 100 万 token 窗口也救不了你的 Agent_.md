### 📋 核心分析

**战略价值**: Agent 产品落地的核心不是模型能力，而是上下文工程——如何在有限注意力预算下，通过 7 大设计模式让 Agent 完成越来越复杂的任务。

**核心逻辑**:

- **上下文腐化是根本矛盾**：METR 研究显示 Agent 任务长度每 7 个月翻一番，但模型性能随上下文增长而下降。Anthropic 将此命名为"上下文腐化"——LLM 存在"注意力预算"，每个新 token 都在消耗它，100 万 token 窗口解决不了这个管理问题。
- **给 Agent 一台计算机而非一堆工具**：Claude Code 用文件系统（持久化记忆）+ Shell（无限扩展性）替代预设工具集；Manus 用 E2B 沙箱提供完整 Ubuntu + 浏览器 + Python/Node 环境，Agent 可像人类开发者一样使用任意命令行工具。
- **工具数量是性能杀手**：Google Cloud 研究证实单 Agent 系统性能随工具数量增加而下降。GitHub MCP 服务器 35 个工具，光工具定义就消耗 2.6 万 token，还没开始执行任务。Claude Code 只保留约 12 个核心工具，Manus 少于 20 个。
- **CodeAct 是分层行动空间的极致**：Manus 用 Python 代码作为统一动作格式——不调用预设"发送邮件"工具，而是写代码调用邮件 API。好处：减少工具定义 token 消耗、避免中间结果占用上下文、赋予 Agent 组合创新能力。代价：需严格限制每轮一个动作，执行后必须观察结果再决定下一步。
- **渐进式披露实现三层信息架构**：索引层（工具/技能列表，< 1K tokens）→ 摘要层（简短描述，帮助 Agent 判断是否深入）→ 详情层（完整文档，按需加载）。Cursor Agent 把 MCP 工具描述同步到文件夹，初始上下文只有简短列表，Agent 自行读取完整描述。
- **缓存命中率是生产环境最重要指标**：Manus 团队明确结论：高容量模型（Claude Opus）配合缓存，实际成本可能低于低价模型（GPT-3.5）不使用缓存。Claude Code 用户反馈：没有缓存，编码 Agent 成本高到无法商业化。关键约束：保持上下文前缀稳定，新信息只追加到末尾，禁止在历史消息中间插入内容。
- **多 Agent 用 Map-Reduce 模式解决并行与聚焦**：Cursor 2.0 支持最多 8 个并行 Agent，每个在独立工作空间运行。Claude Code 用子 Agent 做代码审查：安全漏洞、性能问题、代码风格各自独立检查，最后汇总。但 Google Cloud 明确警告：多 Agent 带来显著延迟与费用增加，只适用于子任务无强依赖的场景。
- **上下文演化让 Agent 从经验中学习**：Claude Code 将每次会话提炼为"日记条目"，定期反思后更新 CLAUDE.md，形成"执行任务 → 提炼经验 → 更新规则 → 下次更高效"的学习循环。GEPA 方法更结构化：收集执行轨迹 → 评分 → 反思失败案例 → 生成改进提示变体 → 测试迭代，类似超参数调优。
- **卸载 vs 压缩的决策原则**：信息可能再次需要且完整性重要（代码片段、API 响应）→ 卸载到文件；信息仅需概要且不会再次详细访问（长篇文档摘要）→ 压缩。过早压缩是很多 Agent 系统的通病。

### 🎯 关键洞察

**为什么扩大上下文窗口不是解决方案**：

这是一个管理问题，不是容量问题。Andrej Karpathy 的定义精准：上下文工程是"将下一步所需的正确信息填充到上下文窗口中"的艺术和科学。关键词是"正确信息"，而不是"所有信息"。

逻辑链：
- 原因：LLM 注意力是有限预算，信息越多，关键内容被稀释
- 动作：精心设计信息的加载时机、层次、生命周期
- 结果：Agent 在有限注意力下完成复杂任务，且成本可商业化

**"Ralph Wiggum" 循环（长期任务的多 Agent 模式）**：
1. 初始化 Agent 创建计划文件 + 任务追踪文件
2. 执行 Agent 从计划中领取任务并完成
3. 验证 Agent 通过 Stop Hook 检查工作质量
4. 循环迭代直到所有任务完成
5. 上下文通过文件系统和 Git 历史在 Agent 间传递，每个 Agent 只关注当前子任务

### 🛠️ 操作流程

**Anthropic Skills 标准实现渐进式披露**：

1. 准备阶段：为每个技能创建 SKILL.md 文件，结构如下：
```yaml
---
name: skill_name
description: 一句话描述（始终加载到上下文）
---
# 完整技能文档
（仅在 Agent 决定使用时读取）
```

2. 核心执行：Agent 指令中只加载所有技能的 name + description（索引层，< 1K tokens），Agent 根据任务判断是否需要读取完整 SKILL.md。

3. 验证与优化：监控 Agent 读取完整文档的频率，高频读取的技能可考虑提升到摘要层；低频读取的工具定义从初始上下文中移除。

**上下文缓存结构设计**：

1. 准备阶段：将上下文按稳定性排序：
```
[系统提示] → [项目规范] → [工具定义] → [历史对话] → [当前用户输入]
↑_________________________可缓存部分________________________↑  ↑不缓存
```

2. 核心执行：新信息只追加到末尾，绝不在历史消息中间插入，绝不频繁修改系统提示或工具定义。

3. 验证与优化：监控缓存命中率，这是生产环境最重要指标。短会话缓存可能不划算，长会话缓存是必需的。

### 💡 具体案例/数据

- METR 研究：Agent 任务长度每 7 个月翻一番，但模型性能随上下文增长而下降
- GitHub MCP 服务器：35 个工具，工具定义本身消耗 2.6 万 token
- Claude Code：约 12 个核心工具；年收入破 10 亿美元
- Manus：少于 20 个核心工具；基于 E2B 沙箱，每用户独立云端 Ubuntu 环境；Meta 以 20 亿美元收购
- Cursor 2.0：最多支持 8 个并行 Agent，每个在独立工作空间运行
- Manus 团队结论：Claude Opus + 缓存的实际成本可低于 GPT-3.5 不使用缓存

### 📝 避坑指南

- ⚠️ 工具数量陷阱：工具越多不等于能力越强，超过一定数量后模型混淆率上升，性能反而下降。保持核心工具 ≤ 20 个。
- ⚠️ 过早压缩：很多 Agent 系统的通病。代码片段、API 响应等有可能再次需要的内容，应卸载到文件而非压缩摘要。
- ⚠️ 缓存前缀破坏：在历史消息中间插入新内容会使缓存完全失效，导致成本暴增。所有新信息必须追加到末尾。
- ⚠️ 多 Agent 滥用：子任务有强依赖关系时，多 Agent 的协调成本高于收益。Map-Reduce 只适用于彼此独立的子任务。
- ⚠️ 错误经验学习：上下文演化机制中，Agent 可能从失败案例中提炼出错误规则。关键记忆和技能更新建议加入人工审核或增量更新机制，避免大幅变动。
- ⚠️ CodeAct 失控风险：Agent 写代码执行时可能陷入无限循环或产生副作用。必须严格限制每轮一个动作，执行后强制观察结果。
