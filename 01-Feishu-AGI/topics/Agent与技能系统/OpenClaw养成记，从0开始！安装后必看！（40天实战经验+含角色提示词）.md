### 📋 核心分析

**战略价值**: 用纯Markdown文件体系替代复杂编排框架，通过40天持续对话与反馈积累，让8个AI智能体进化成真正懂你的自动化团队。

**核心逻辑**:

- **模型不是变量，文件才是**：第1天和第40天用的是同一个模型。智能体不会因为使用时间变长而变聪明，但围绕它的Markdown文件会越来越精准、越来越贴合你的需求，这才是真正的护城河。
- **三层架构是骨架**：身份层（SOUL.md / IDENTITY.md / USER.md）定义"是谁"，操作层（AGENTS.md / HEARTBEAT.md）定义"怎么干"，知识层（MEMORY.md / 每日日志 / shared-context/）定义"学到了什么"。三层缺一不可。
- **SOUL.md是人格核心**：每个智能体必须有独立的SOUL.md，定义身份、职责、行为原则。Dwight的原则第1天是"找热门趋势"，第10天变成"如果Alex今天无法对此采取行动，跳过"，第20天加入核实步骤——文件在进化，模型没变。
- **USER.md的细节产生复利**：时区（PST）让智能体不会在凌晨3点安排任务；饮食偏好（素食）让Pam起草团队晚餐通讯时不推荐牛排馆。写一次，所有智能体都来读。
- **AGENTS.md强制写入记忆**：智能体在会话之间没有记忆，每次从零开始。AGENTS.md明确规定：每次会话必须先读SOUL.md → USER.md → 今天+昨天的日志 → MEMORY.md。有人说"记住这个"→ 立刻更新文件，不写入文件的纠正下次会话就消失。
- **MEMORY.md的"血泪教训"节是最高价值**：Monica删了一个项目文件夹（2月26日，Ross的gemini-council React应用永久丢失），这个错误从此写入长期记忆，永不重演。一次纠正，存储一次，防止同样错误在未来每次会话中重演。
- **shared-context/是乘数效应**：第20天之前，同样的纠正要对Kelly、Rachel、Ryan、Pam逐个重复。建立FEEDBACK-LOG.md后，写一次，所有智能体都来读。THESIS.md记录当前思维框架，让Dwight知道研究优先级，让Kelly匹配写作思路——所有智能体对齐到同一个真相来源。
- **单写者原则防止协调冲突**：Dwight写intel/DAILY-INTEL.md，Kelly/Rachel/Pam只读不写。永远不要让两个智能体同时写同一个文件。调度顺序：Dwight早8点和下午4点先跑，Kelly和Rachel下午5点后跑——顺序搞错，下游读到的是空文件。
- **每日日志必须修剪**：Kelly的日志一度膨胀到161,000 tokens，输出质量急剧下降，被迫压缩到40,000。规则：每次只加载今天和昨天的日志，定期把反复出现的纠正蒸馏进MEMORY.md。
- **HEARTBEAT.md在第一次故障后才建**：第三周调度器有bug，任务在队列里推进但从未真正执行，好几个小时没发现。之后才建心跳机制。第一天不需要它，在你亲身遇到故障之后再建，你会清楚地知道该监控什么。

### 🎯 关键洞察

**为什么文件体系比prompt工程更有效**：

原因：智能体每次会话从零开始，prompt只影响当次会话，而文件是持久化的外部记忆。

动作：把每一次纠正、每一个偏好、每一个血泪教训写入对应的Markdown文件。

结果：40天后，Kelly的SOUL.md里有具体语气示例、被否决模式列表、"永远不要再建议"专区——这是40天对话的结晶，任何人拿到同一个模型都无法复制。

**shared-context/的杠杆原理**：

在没有FEEDBACK-LOG.md之前，纠正一个习惯需要对4个智能体各说一次，且下次会话还可能失效。有了它之后，一次写入，永久生效，覆盖所有智能体。这是线性成本换指数收益的典型结构。

### 🛠️ 操作流程

**按时间轴的完整SOP**：

1. **今天 — 最小可运行系统**
   - 安装OpenClaw
   - 为主智能体写SOUL.md（参考Dwight模板：核心身份 + 角色 + 3条原则）
   - 写IDENTITY.md（名字/角色/气质/Emoji/灵感来源）
   - 写USER.md（姓名/时区/饮食/背景/写作偏好，禁止项要明确列出）
   - 挑一个最重复的日常任务，设置定时任务让它跑起来

2. **第3天 — 激活记忆写入**
   - 开始给出具体反馈（不是"不好"，而是"不要用破折号，改用冒号或重新组织句子"）
   - 每次反馈后确认智能体已将其写入记忆文件，而不只是停留在聊天记录

3. **第1周 — 建立操作规范**
   - 创建根级AGENTS.md，写入会话启动流程：
     ```
     1. 读取 SOUL.md
     2. 读取 USER.md
     3. 读取 memory/YYYY-MM-DD.md（今天 + 昨天）
     4. 如果在主会话中：读取 MEMORY.md
     ```
   - 添加记忆管理规则："当有人说'记住这个' → 更新记忆文件"
   - 添加安全规则："用回收站而非直接删除；有疑问时先问"

4. **第2周 — 建立长期记忆**
   - 创建MEMORY.md，回顾每日日志
   - 把反复出现的纠正蒸馏成永久条目
   - 必须包含"血泪教训"节和"错误示范"节
   - 此时你会感受到复利开始发生

5. **第3周 — 多智能体协作**
   - 添加第二个智能体，建立基于文件的协作
   - 确定单写者原则：哪个智能体写哪个文件
   - 为每个智能体建立角色专属指南（如Kelly的X-CONTENT-GUIDE.md）
   - 建立shared-context/目录，创建THESIS.md和FEEDBACK-LOG.md
   - 设置调度顺序：数据源智能体（Dwight）先跑，内容智能体后跑

6. **第4周 — 自愈机制**
   - 在第一次遇到故障之后，创建HEARTBEAT.md
   - 监控项：浏览器是否存活（running: false → 启动）；定时任务lastRunAtMs是否超过26小时（超时 → CLI强制触发）
   - 需要监控的任务列表写入HEARTBEAT.md

### 📝 避坑指南

- ⚠️ **不要在一个周末搭完整个系统**：按时间轴逐步建立，过早引入复杂性会让你在调试上花比自己做还多的时间
- ⚠️ **纠正不写入文件等于没纠正**：智能体会话之间没有记忆，只有文件里的内容才能跨会话存活
- ⚠️ **每日日志不修剪会崩**：161,000 tokens的教训——定期蒸馏，只加载今天和昨天
- ⚠️ **调度顺序搞错下游读到空文件**：数据源智能体必须先于内容智能体运行，Dwight 8点/16点，Kelly/Rachel 17点
- ⚠️ **两个智能体同时写同一文件会产生协调冲突**：严格执行单写者原则
- ⚠️ **HEARTBEAT.md不要提前建**：第一次遇到故障后再建，你才知道真正需要监控什么
- ⚠️ **shared-context/不要第一天建**：第20天之前你还不知道哪些纠正是跨智能体通用的
