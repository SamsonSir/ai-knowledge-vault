### 📋 核心分析

**战略价值**: 用 SKILL.md 文件把反复 prompt 的偏好/流程/知识一次性封装，让 Claude 自动按预设工作流执行，彻底消除重复提示成本。

**核心逻辑**:

- **Skill = 文件夹，核心是 SKILL.md**：用 Markdown + YAML frontmatter 编写，本质是 Claude 的"标准操作手册"，一次配置，每次对话自动生效。
- **三层渐进加载，不撑爆上下文**：第一层 YAML frontmatter（name + description）始终在系统提示词中；第二层 SKILL.md 正文仅在 Claude 判断相关时加载；第三层 references/ 文档按需读取。启用几十个 Skill 也不会全部塞进上下文。
- **description 是触发器，写法决定成败**：必须同时包含「做什么 + 何时触发 + 关键能力」三要素，description 太模糊 = Skill 永远不激活。
- **MCP 是厨房，Skill 是菜谱**：MCP 解决"能做什么"（工具/连接），Skill 解决"该怎么做"（最佳实践/工作流）。有 Skill 的 MCP 集成比纯 MCP 集成更易上手，是服务提供商的差异化竞争力。
- **三大场景分类**：① 纯内置能力（文档/设计/PPT）② 多步骤工作流自动化（含验证门控）③ MCP 增强（编排多个 MCP 调用 + 嵌入领域知识）。
- **五大实战模式**：顺序工作流编排、多 MCP 协调、迭代精炼、上下文感知工具选择、领域专业智能——覆盖从简单到复杂的全部场景。
- **关键验证步骤用脚本替代自然语言**：scripts/ 目录下的代码是确定性的，语言理解不是。对于合规检查、数据校验等关键节点，必须用脚本。
- **性能提升有实测数据**：有 Skill vs 无 Skill，对话轮次从 15 轮降到 2 轮，失败 API 调用从 3 次降到 0 次，Token 消耗减半。
- **先单点迭代，再提取经验**：不要一开始追求广覆盖，先在单个困难任务上反复迭代直到成功，再把经验写入 Skill。
- **Anthropic 将 Agent Skills 定位为开放标准**：目标是像 MCP 一样跨平台使用，不局限于 Claude 生态。

### 🎯 关键洞察

**为什么 description 是整个 Skill 的命脉**：

Claude 在每次对话开始时，只读取所有已启用 Skill 的第一层（name + description），据此判断是否激活某个 Skill。如果 description 写得太宽泛（如"帮助处理代码问题"），要么永远不触发，要么什么都触发。正确写法需同时回答三个问题：
1. 这个 Skill 做什么具体的事？
2. 用户说什么话/处于什么场景时应该激活？
3. 它有哪些关键能力是其他 Skill 没有的？

**调试方法**：直接问 Claude "你什么时候会使用 [skill name] 这个 skill？"，它会引用 description 回答，暴露触发逻辑是否符合预期。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| 文件夹命名 | `kebab-case`，如 `my-cool-skill` | 系统正确识别 | 禁止空格、下划线、大写字母 |
| 主文件命名 | 必须精确为 `SKILL.md` | 大小写敏感 | 不能是 `skill.md` 或 `Skill.md` |
| YAML frontmatter | `name` + `description` 两个字段 | 触发判断依据 | description 必须含「做什么+何时触发+关键能力」 |
| 禁用字符 | 禁止 XML 尖括号 `< >` | 防止系统提示词注入 | 用其他符号替代 |
| 保留字 | 名称中不能含 `claude` 或 `anthropic` | 系统保留字冲突 | 会导致 Skill 无法加载 |
| 禁止文件 | 不允许在 Skill 文件夹内放 `README.md` | 避免与主文件混淆 | 参考文档放 `references/` |
| SKILL.md 大小 | 主文件控制在 5000 词以内 | 避免上下文过大变慢 | 详细文档移到 `references/` |
| 关键验证步骤 | 用 `scripts/` 下的脚本执行 | 确定性校验 | 自然语言指令不可靠 |

### 🛠️ 操作流程

**1. 准备阶段：搭建 Skill 文件夹结构**

```
your-skill-name/
  SKILL.md          # 必须，主文件（大小写敏感）
  scripts/          # 可选，可执行脚本（关键验证步骤放这里）
  references/       # 可选，参考文档（详细内容放这里）
  assets/           # 可选，模板、图标等资源
```

**2. 核心执行：编写 SKILL.md**

```markdown
---
name: your-skill-name
description: |
  [做什么] + [何时触发，用户会说的具体话] + [关键能力]
  例：当用户需要审查 Sentry 错误并分析关联 PR 时激活。
  能力：自动拉取 Sentry 错误数据、分析 GitHub PR、给出修复建议。
---

# 正文：完整指令、步骤、示例
```

按场景选择模式：
- 严格顺序流程 → 模式1（顺序工作流编排），明确步骤依赖和失败回滚
- 跨多服务联合 → 模式2（多 MCP 协调），明确阶段间数据传递和集中错误处理
- 需多轮改进 → 模式3（迭代精炼），明确质量标准和停止迭代的判断条件
- 同目标多工具 → 模式4（上下文感知），写清决策树和备选方案
- 嵌入专业知识 → 模式5（领域专业智能），合规先于行动，完整审计追踪

**3. 部署**

- 个人用户：下载 Skill 文件夹 → 压缩 → 上传到 Claude.ai 设置，或放入 Claude Code 的 `skills/` 目录
- 组织级别：管理员在工作区集中部署，支持自动更新
- API 使用：通过 `/v1/skills` 端点管理，Messages API 中用 `container.skills` 参数引入

**4. 验证与优化**

触发测试三步走：
1. 直接请求能触发
2. 换种说法也能触发
3. 无关请求不会误触发

调试命令：直接问 Claude "你什么时候会使用 [skill name] 这个 skill？"

### 📝 避坑指南

- ⚠️ **Skill 不触发**：description 太模糊或缺少触发短语 → 增加具体用户会说的话作为触发词
- ⚠️ **Skill 过度触发**：description 范围太宽 → 添加负向触发条件、缩小范围
- ⚠️ **指令不被遵循**：指令太长/太含糊/关键内容被埋没 → 精简指令、关键信息放最前面、用 scripts/ 做确定性校验
- ⚠️ **上下文过大导致变慢**：SKILL.md 超过 5000 词或同时启用太多 Skill → 详细文档移到 references/，主文件只保留核心指令
- ⚠️ **文件夹命名错误**：含空格/下划线/大写 → 统一用 kebab-case
- ⚠️ **关键验证用自然语言写**：语言理解有不确定性 → 关键验证步骤必须用 scripts/ 下的脚本
