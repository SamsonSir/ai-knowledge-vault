### 📋 核心分析

**战略价值**: 用一张概念地图彻底厘清 Skill / 提示词 / Memory / Projects / MCP / Agent / 工作流 七个概念的边界与协作关系，让你知道该在什么场景选哪个工具。

**核心逻辑**:

- **Skill 本质是"给 AI 写的说明书"**：把你反复教 AI 的规矩、格式、流程打包成一份文档，AI 看一遍就记住，之后自动按规矩执行，无需每次重复。
- **Skill vs 复制粘贴提示词**：手动存文档的提示词会丢失、难查找、换对话要重贴；Skill 存一次全局永久生效，还可内嵌脚本、模板、示例文件，不只是纯文字。
- **Skill vs Projects**：Projects 是"工作空间"，存放某项目专属的背景资料，只在该项目内生效；Skill 是"技能包"，全局生效，存放的是做事方法。判断标准：如果你在多个 Project 里复制同一段指令，就该做成 Skill。
- **Skill vs Memory**：Memory = 记住"你是谁"（姓名、职业、偏好）；Skill = 记住"你想让 AI 怎么干活"（流程、格式、风格）。两者互补，不互斥。
- **Skill vs MCP**：MCP 是"插座"，给 AI 接入外部能力（读数据库、调 API、操作文件）；Skill 是"经验"，告诉 AI 拿到这些数据后怎么分析、出什么格式报告。关系：MCP 是"手"，Skill 是"脑子里的经验"，两者搭档使用。
- **Skill vs Agent**：Agent 是能自主思考、规划、行动的 AI 实体（"人"）；Skill 是这个 Agent 掌握的具体技能。一个 Agent 可同时持有多个 Skill，就像一个人同时会做饭、开车、写代码。
- **Skill vs 工作流**：工作流是固定顺序流程（A→B→C→D），适合重复性、无需判断的任务；Skill 更灵活，AI 自己判断何时触发、如何组合，适合需要上下文判断的任务。
- **Skill 不需要切换**：不像 Custom GPTs 需要手动切换到某个 GPT，Skill 在任意对话中自动判断是否适用，多个 Skill 可同时生效，符合真实工作节奏。
- **Simon Willison（知名开发者）的判断**：Skill 可能比 MCP 更重要——MCP 需要复杂协议规范，Skill 只是 Markdown 加少量配置；几乎所有 MCP 能做的事，用 CLI 工具 + Skill 也能实现。
- **Boris Cherny（Claude Code 创始人）的团队实践**：团队共享一个 Skill 配置文件，所有成员每周多次贡献更新；用 Skill 处理日常高频任务，同时并行运行 5-10 个 Claude 实例协作工作。

### 🎯 关键洞察

Skill 的核心价值不是"功能"，而是**经验的可复用性**。

传统提示词工程的瓶颈在于：好的提示词是个人资产，无法沉淀、无法共享、无法自动触发。Skill 把这个问题从"个人记忆"升级为"组织知识库"——Boris Cherny 团队的做法就是最好的证明：团队共享配置文件 + 每周多次贡献，本质上是在用 Skill 构建一套**团队级 AI 操作规范**，而不只是个人效率工具。

逻辑链：
> 重复指令 → 打包成 Skill → 自动触发 → 多人共享 → 团队 AI 能力标准化 → 并行多个 Claude 实例 → 规模化产出

### 📦 配置/工具详表

| 概念 | 本质 | 作用范围 | 适用场景 | 核心区别 |
|------|------|---------|---------|---------|
| Skill | 做事方法的说明书（Markdown + 少量配置） | 全局，任意对话自动生效 | 高频重复指令、多步骤流程、风格规范 | 自动触发，无需手动调用 |
| 复制粘贴提示词 | 存在文档里的文字 | 手动，每次贴 | 临时一次性任务 | 会丢失，需手动操作 |
| Projects | 项目工作空间 | 仅限该项目内 | 存放项目专属背景资料 | 知识库，不是方法库 |
| Memory | 关于用户的事实记忆 | 全局 | 记住姓名、职业、偏好 | 记"你是谁"，不记"怎么干活" |
| MCP | 连接外部工具的协议插座 | 取决于配置 | 读数据库、调 API、操作文件 | 提供能力，不提供方法 |
| Agent | 自主思考+规划+行动的 AI 实体 | 独立运行 | 复杂多步骤自主任务 | 是"人"，Skill 是"人的技能" |
| 工作流 | 固定顺序流程 A→B→C→D | 按设计路径执行 | 重复性、无需判断的标准化任务 | 死板但稳定，Skill 灵活但需 AI 判断 |
| Custom GPTs | 独立封装的小应用 | 需手动切换进入 | 独立场景的专属助手 | 要切换，Skill 不用切换 |

### 🛠️ 操作流程

1. **创建第一个 Skill（零代码起步）**
   - 在 Claude 或 Coze 对话框直接输入：`帮我创建一个 Skill`
   - AI 会引导你一步步填写：触发关键词、执行流程、输出格式
   - 不需要写代码，一个写得好的说明文档就是有效的 Skill

2. **判断该用 Skill 还是其他工具**
   - 发现自己在多个 Project 里复制同一段指令 → 做成 Skill
   - 是某项目专属的背景资料（如客户信息、项目文档）→ 放 Project
   - 是关于你个人的固定事实（名字、偏好）→ 交给 Memory
   - 需要连接外部系统（数据库、API）→ 配置 MCP，再用 Skill 定义处理方法

3. **复用现成 Skill（直接继承他人经验）**
   - 网上已有数万个现成 Skill（写代码的"代码审查" Skill、自媒体的"爆款标题" Skill、投资的"财报分析" Skill）
   - 直接从 Skill 市场搜索对应场景，导入即用

4. **Skill 不触发时的排查**
   - 原因：描述写得不够具体，Claude 无法匹配触发关键词
   - 解决：用你平时说话时真实使用的关键词来描述触发条件

5. **团队级 Skill 管理（参考 Boris Cherny 团队实践）**
   - 建立团队共享的 Skill 配置文件（单一 source of truth）
   - 所有成员每周多次贡献、更新 Skill
   - 用 Skill 处理日常高频任务，配合并行运行多个 Claude 实例提升产出规模

### 💡 具体案例/数据

- **论文解读 Skill 示例**：丢一篇 PDF → Skill 自动执行：提取要点 → 翻译成中文 → 生成摘要 → 画思维导图，全程无需手动指挥每一步。
- **小红书文案 Skill 示例**：不用每次说"口语化、加 emoji、控制 300 字、开头要有 hook"，直接说"写小红书文案"，Skill 自动套用所有规范。
- **Boris Cherny 团队数据**：同时并行运行 5-10 个 Claude 实例，依赖 Skill 统一行为规范，实现规模化 AI 协作。
- **Simon Willison 的技术判断**：Skill 的技术门槛极低——本质是 Markdown + 少量配置，而 MCP 需要完整的协议规范；有人已用"CLI 工具 + Skill"构建出能自动构建、测试、优化 MCP 工具的完整流程。

### 📝 避坑指南

- ⚠️ **Skill 不触发**：触发关键词描述太抽象，要用你日常真实说话的词，不要用"当用户需要营销内容时"这种书面语。
- ⚠️ **把 Skill 当 Project 用**：Skill 存方法，Project 存资料，混用会导致 Skill 越来越臃肿、难以维护。
- ⚠️ **以为 Skill 需要写代码**：很多高效 Skill 只是一份结构清晰的说明文档，零代码。
- ⚠️ **忽略团队共享价值**：Skill 最大的杠杆不是个人效率，而是团队知识标准化——一个人总结的经验，全团队自动继承。
- ⚠️ **MCP 和 Skill 二选一**：两者不是竞争关系，MCP 给 AI 接工具，Skill 教 AI 用工具，应该配合使用。
