### 📋 核心分析

**战略价值**: OpenAI 官方实战手册，用 Skills + Shell + Compaction 三件套让智能体长期稳定运行，附 Glean（估值 72 亿美元）生产环境数据背书。

**核心逻辑**:

- **Skills = 版本化操作手册**：一组文件 + SKILL.md 清单（含 frontmatter + 指令），平台只暴露名称/描述/路径给模型，模型按需加载完整工作流，核心是"渐进式披露"——不调用不消耗 token。Agent Skills 开放标准由 Anthropic 于 2025 年底发起，已被 OpenAI Codex、GitHub Copilot、Cursor、Google Gemini CLI 等 20+ 工具采纳。
- **Shell 工具 = 真实执行环境**：两种模式——OpenAI 托管容器（Debian 12，预装 Python 3.11 / Node.js 22 / Java 17 / Go 1.23 / Ruby 3.1）；本地 Shell 运行时（自己执行命令，语义相同）。托管 Shell 走 Responses API，支持有状态工作、工具调用、多轮续接、产出物（artifacts）。
- **Compaction = 长对话续命机制**：两种方式——服务端自动压缩（超阈值时在流式输出中自动触发）；独立端点 `/responses/compact`（手动控制时机）。比暴力截断历史更优，保留关键上下文、丢弃噪声。Triple Whale 的智能体 Moby 在单次含 500 万 token + 150 次工具调用的会话中全程保持准确率。
- **三者协同价值**：Skills 消灭"提示词意大利面"（系统提示词膨胀到上万 token 的问题）；Shell 提供完整执行环境；Compaction 保持长期连贯性。组合结果：可重复工作流 + 真实执行力，而非脆弱的巨型系统提示词文档。
- **技能描述 = 模型决策边界**：描述必须回答三个问题：什么时候该用？什么时候不该用？输出和成功标准是什么？实操：在描述中直接放"用于 vs 不用于"块，保持具体（输入、涉及工具、预期产出物）。
- **负面示例是路由精度的关键杠杆**：Glean 实测：引入技能后路由触发率初始下降约 20%，加入负面示例 + 边缘场景覆盖后恢复。多技能并存时，明确说"这个不是你要的"比只说"这个是你要的"更有效。
- **模板/示例放进技能而非系统提示词**：按需加载，不相关查询零 token 消耗。Glean 反馈这是生产环境中质量和延迟提升最大的单一改动。
- **确定性优先于智能**：生产工作流中有明确契约时，直接在提示词里指定"使用 XX 技能"，把模糊路由变成明确契约。灵活场景放权给模型，流水线场景收回控制权。
- **网络访问是高风险组合**：技能 + 开放网络 = 数据泄露高风险路径。合理默认：技能允许、Shell 允许、网络仅在最小白名单下启用。
- **Glean Salesforce 技能案例**：评测准确率从 73% 提升到 85%，首字节延迟降低 18.1%。策略：精细路由 + 负面示例 + 技能内嵌模板和示例。（注：文中未披露评测基准和样本量，数字方向可信，精确度存疑。）

### 🎯 关键洞察

**"单工具调用"到"多工具编排"之间存在准确率断崖**：技能通过让工具推理更程序化来填补这个缺口，同时不让系统提示词膨胀。技能本质上是"活的 SOP"——随组织演进而更新，由智能体一致执行。

**成本盲区**：文中完全未提及成本。托管容器和 Compaction 是 OpenAI 平台专有能力，不要和 Agent Skills 开放标准混淆——前者是平台锁定，后者是跨平台标准。

### 🛠️ 操作流程

**1. 准备阶段：构建技能（Skill）**
- 创建技能目录，包含相关文件 + `SKILL.md` 清单
- `SKILL.md` 写 frontmatter（名称、描述、路径）+ 完整工作流指令
- 描述中必须包含：何时用 / 何时不用 / 预期输出和成功标准
- 加入负面示例和边缘场景（"不要在以下情况调用"）
- 把模板、示例、格式要求放进技能内部，不要放系统提示词

**2. 核心执行：Shell + Skills 挂载**
- 本地开发阶段：用本地 Shell 运行时快速迭代，自己执行 `shell_call` 并返回 `shell_call_output`
- 生产阶段：切换到托管容器（Debian 12 环境），技能保持不变
- 跨步骤复用同一个容器（保留依赖、缓存、中间输出）
- 传递 `previous_response_id` 保持线程连续性
- 产出物统一写入 `/mnt/data/`（报告、清洗数据集、电子表格等）
- 网络访问：先配置组织级白名单（管理员），再按请求设置更窄的子集白名单
- 需要认证的域名：用 `domain_secrets` 配置，边车进程注入真实凭据

**3. 验证与优化**
- 开启 Compaction 作为默认基础设施（不是紧急回退）：优先用服务端自动压缩，需要精细控制时用 `/responses/compact` 端点
- 路由准确率下降时：检查技能描述是否缺少负面示例，补充边缘场景覆盖
- 生产工作流有明确契约时：直接在提示词中指定技能名称，不依赖模型自主路由
- 评估维度：触发率、准确率、首字节延迟、token 消耗

### 💡 具体案例/数据

- **Glean（企业 AI 搜索，估值 72 亿美元）**：
  - 引入技能路由后，初始触发率下降约 20%
  - 加入负面示例 + 边缘场景覆盖后恢复
  - 把模板/示例放入技能后，获得生产环境最大质量和延迟提升
  - Salesforce 专项技能：准确率 73% → 85%，首字节延迟降低 18.1%
  - 用技能编码的企业工作流：客户方案规划、升级分诊、品牌对齐内容生成

- **Triple Whale（电商平台）**：
  - 智能体 Moby 在单次会话中处理 500 万 token + 150 次工具调用，全程保持准确率（依赖 Compaction）

### 📝 避坑指南

- ⚠️ **技能描述不完整 = 路由失效**：只写"用于什么"不够，必须同时写"不用于什么"，否则多技能并存时模型选择困难，触发率下降。
- ⚠️ **模板塞进系统提示词 = 每次都消耗 token**：改放技能内部，按需加载，对不相关查询零消耗。
- ⚠️ **Compaction 当应急方案 = 太晚**：应从架构设计阶段就把压缩作为默认基础设施，而非上下文快满时的紧急补救。
- ⚠️ **技能 + 开放网络 = 数据泄露高风险**：网络白名单必须是"最小化 + 按任务限制"，面向消费者的流程中严禁"开放网络 + 强大操作"组合。
- ⚠️ **请求级白名单超出组织白名单 = 直接报错**：组织白名单是硬上限，请求白名单只能是其子集。
- ⚠️ **混淆平台专有能力和开放标准**：托管容器、Compaction 是 OpenAI 平台专有；Agent Skills 标准是 Anthropic 发起的跨平台开放标准，两者不是同一回事。
- ⚠️ **生产流水线依赖模型自主路由 = 可靠性风险**：有明确契约的工作流直接指定技能名称，不要赌模型自己判断对。
- ⚠️ **容器跨步骤不复用 = 重复安装依赖、丢失中间状态**：长期运行任务必须复用同一容器。
