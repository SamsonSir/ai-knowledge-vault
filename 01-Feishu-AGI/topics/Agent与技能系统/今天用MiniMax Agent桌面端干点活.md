### 📋 核心分析

**战略价值**: MiniMax Agent 2.0 从"被动聊天机器人"进化为"主动嵌入工作环境的 AI 原生工作台"，桌面端 + 自定义 Expert 是其落地 AGI Workspace 愿景的第一步。

**核心逻辑**:

- **放弃 Chatbot 路线的根本原因**：堆砌工程路由（如搜索场景需要硬判断"要不要搜"、"搜几个网页"）在复杂任务下不 work，且在国内搜索/写作赛道无法形成差异化优势，资源有限必须转型。
- **自研 Agent 框架是基础**：从 2025 年 Q1 开始，MiniMax 自建 Agent 框架，现有 Air / Custom / Pro 三种模式均来自该框架；同步建立了自己的 Agent Benchmark，收集现实开放、长程任务（非单一标答），用于评估真实能力。
- **2.0 时代的三个核心命题**：① Domain Knowledge & Memory（理解用户工作偏好 + 穿越周期的长时记忆）；② Environment & Workspace（熟知企业内网散落知识，整合 SOP 应用于工作流）；③ Active（无需指令，通过 Trigger 主动响应环境变化）。
- **桌面端解决 Web 版两大硬伤**：① Web 只能处理上传文件，存在大小/速度/复用限制，Vibe Coding 用户和视频图文创意用户的大型代码仓库或素材无法处理；② Web 托管浏览器频繁被 block，社媒监控/自动发布等实用场景完全不可用。
- **桌面端本地文件处理能力**：支持重命名、整理、批量格式转换、批量总结课程录音；预置视频理解、图片理解工具；敏感数据在本地安全环境处理，记录不上传云端；首次及关键操作会向用户二次确认，有命令和文件夹权限管控。
- **桌面端浏览器托管能力**：Agent 可像人一样操作浏览器（真实点击、填写、提交），不只是查信息；可跨系统自动化，解决重复繁琐网页操作；已在社媒监控、自动发布、自动填表单等场景验证可用。
- **Pro 模式的固化瓶颈促使框架重构**：原 Pro 模式依赖官方固定 Multi-agent 系统（约 10 个 subagent），无法灵活应对用户新需求或专业化场景调优；花费 1-2 个月重构框架，打破固定结构，支持灵活 Router 和按需调用。
- **自定义 Expert 的六大配置维度**：① 指令（行为准则和工作方式）；② 子代理（处理特定任务的专用 Agent）；③ 工具（代码执行、网页搜索、图像生成等）；④ Skill（技能点 + 相关文件）；⑤ 环境变量（外部 API 连接）；⑥ MCP 工具（外部数据和工具获取）。
- **Expert 创建推荐方式**：在左侧用自然语言描述需求，让 MiniMax Agent 自动构建右侧配置（AI 会根据输入知识和文件选择最优配置），再通过预览调优测试，进一步让 AI 理解优化需求；确认后选择可见度（仅自己 / 全部可见）。
- **子代理架构的优势**：每个子代理专精做一件事，相对隔绝上下文，大部分模型在子代理里能有较好适配表现；以 SEO 专家为例，会拆解为 1 个主指令 + 3 个子代理（技术分析、内容策略、分析报告生成）+ 对应 Skill。

### 🎯 关键洞察

**为什么做桌面端而不继续优化 Web**：
- 原因：Web 架构天然限制了"上下文深度"——文件上传有大小上限、托管浏览器被反爬机制 block、无法访问本地文件系统。
- 动作：做桌面端，直接获取本地文件系统权限 + 控制本地浏览器实例（非托管远程浏览器）。
- 结果：Agent 能接触到用户真实工作内容（大型代码仓库、本地素材、企业内网系统），上下文质量质变。

**为什么 Expert 用自然语言构建比手动配置更优**：
- 原因：MiniMax 内部 PGC Expert 的制作过程本身也是"用 AI 来 CPU AI"，人工手动配置子代理逻辑容易遗漏最优结构。
- 动作：输入自然语言需求 + 相关知识文件，让模型自动决策配置结构，再人工预览调优。
- 结果：配置质量更高，且用户无需理解底层 Multi-agent 路由逻辑。

| 工具 | 代码执行、网页搜索、图像生成等按需启用 | 扩展 Agent 能力边界 | 按场景最小化启用，避免工具冗余干扰路由 |

### 🛠️ 操作流程

**场景一：创建自定义 Expert（以 SEO 专家为例）**

1. **准备阶段**: 整理 SEO 相关知识文档（关键词策略、内容规范、技术 SEO checklist），准备好需要连接的外部 API（如 Google Search Console）。
2. **核心执行**:
   - 进入 MiniMax Agent，点击创建 Expert
   - 在左侧自然语言输入框描述需求，例如："创建一个 SEO 专家，能做技术分析、内容策略规划和分析报告生成"
   - 上传相关知识文件，让 Agent 自动生成右侧配置（系统会自动拆解为：1 个主指令 + 3 个子代理：技术分析、内容策略、分析报告生成 + 对应 Skill）
   - 在预览窗口测试对话，验证子代理路由是否符合预期
   - 进一步用自然语言告诉 AI 优化方向，迭代配置
3. **验证与优化**: 确认配置后选择可见度（个人重复性任务选"仅自己可见"；通用场景可选"全部可见"）；后续可复制 PGC Expert 到自己库里做二改，快速微调为个人化版本。

**场景二：桌面端浏览器托管（以 Twitter 监控为例）**

1. **准备阶段**: 下载桌面端（国内版：`https://agent.minimaxi.com/download`；海外版：`https://agent.minimax.io/download`），登录账号，授权浏览器控制权限。
2. **核心执行**: 输入指令，例如："去 Twitter 浏览最新有关 minimax agent 的新 post，整理写入表格给我"；Agent 接管浏览器，自动完成搜索、浏览、数据抓取、写入表格全流程。
3. **验证与优化**: 检查表格数据完整性；如需定期执行，配置定时任务触发器（当前已支持定时任务，更灵活的 Trigger 逻辑在后续版本规划中）。

### 📝 避坑指南

- ⚠️ **Web 版浏览器托管不可用于社媒场景**：Web 托管浏览器会被主流社媒平台 block，社媒监控/自动发布必须用桌面端。
- ⚠️ **固定 Pro 模式无法满足专业化场景**：原 Pro 的 10 个固定 subagent 结构对新需求和专业化调优几乎无法扩展，需切换到自定义 Expert。
- ⚠️ **Expert 创建优先用 AI 构建而非手动配置**：手动配置子代理逻辑容易遗漏最优结构，自然语言描述 + AI 自动生成配置质量更高。
- ⚠️ **主动触发场景依赖 MCP + OAuth 前置配置**：当前主动性功能仅支持定时任务，更灵活的 Trigger 需要 MCP 接入和 OAuth 授权设置完备，否则无法运行。
- ⚠️ **大型企业跨系统取数仍是难题**：数仓分散在不同平台和权限体系下，桌面端和 Agent 目前无法自动打通，需手动配置权限和环境变量。
- ⚠️ **Expert 可见度设置不可忽视**：含个人 SOP、客户数据、竞品分析的 Expert 务必设为"仅自己可见"，避免敏感信息泄露。
