### 📋 核心分析

**战略价值**: Manus 科学家 Peak 系统总结了通用 Agent 在生产环境中 Context Engineering 的五大维度完整实战方案，包含 KV Cache 优化、分层工具空间、多 Agent 架构、错误保留策略等可直接复用的工程决策。

**核心逻辑**:

- **押注 Context Engineering 而非自训模型**：Peak 前司从零训模型，一个训练+评估周期需 1-2 周，迭代速度被拖死，且在未找到 PMF 前就在优化与用户体验无关的基准。Manus 策略：有 PMF 之前不训模型，通用大模型+上下文工程可在数小时内迭代。创业公司要做"船"（随基础模型涨潮水涨船高），不做"柱子"（被水淹没的风险）。
- **输入输出比 100:1，平均 50 次工具调用**：每执行一次工具，上下文就增长一截；理论上模型支持 100 万 token，但实测 200k 左右开始出现渐进式退化（重复、推理变慢、质量下滑）。Manus 通过大量测试确定"预腐烂阈值"，通常在 **128k-200k 之间**，必须在触达此红线前主动干预。
- **文件系统作为终极无限上下文**：不用向量数据库/RAG/知识图谱等有损外部化方案，而是直接用文件系统：无限容量 + 天然持久化 + 命令行直接操作。核心是**可逆性**——URL 在，网页内容可重新 fetch；文件路径在，内容可重新 read；比猜测"10 步后什么信息重要"更务实。
- **紧凑化 = 外部化状态而非压缩数据**：写文件的工具调用执行后，content 字段可安全删除，只保留 path 指针。从模型上下文视角信息消失，但从 Agent 系统视角信息仍可访问。摘要（有损）是最后手段，摘要前先把关键部分卸载到文件，甚至把整个预总结上下文作为日志文件存档留后路；摘要时用结构化 Schema（定义表单字段让 AI 填空，如"修改了哪些文件"、"用户目标"、"执行进度"），而非让 AI 自由发挥；摘要后保留最近几条工具调用/结果完整版本保持连续性。
- **分层动作空间解决工具膨胀问题**：三层架构——核心层 10-20 个原子函数（读写文件、执行 shell 等），边界清晰可组合，直接函数调用；中间层预装在沙盒中的工具（格式转换器、语音识别、mcp-cli 等），通过命令行访问；外层预授权 API（金融数据、3D 设计库等），写 Python 脚本执行（如：获取城市名→获取城市 ID→获取天气，一个脚本搞定）。三层从基础模型视角均通过标准函数调用，接口清晰、缓存友好、函数间正交。
- **KV Cache 命中率是生产 Agent 最重要的单一指标**：以 Claude Sonnet 为例，缓存命中 token 成本 vs 未命中成本差 **10 倍**。优化细节：① 系统提示开头不放时间戳（保持前缀稳定）；② 确保 JSON 键顺序确定性（顺序不稳定会悄无声息破坏缓存）；③ 只追加不修改历史操作记录；④ 如果提供商不支持自动增量前缀缓存，手动插入缓存断点；⑤ 动态添加/移除工具会摧毁 KV Cache（工具定义在上下文前部），改用 **Logit Masking** 在解码过程中遮蔽 token logits，实现自动/必需/指定三种工具调用模式，工具名称设计有意统一前缀（如所有浏览器工具以 `browser_` 开头）以支持按组约束。
- **MCP 通过 CLI 而非函数注入接入**：Manus 开发 `mcp-cli` 工具预装在沙盒中，通过命令行接口调用 MCP，而非注入函数调用空间；输出可直接写文件或分页返回，用 grep、cat 等 Linux 工具实时处理；信任 Agent 可运行 `--help` 自学新工具，不在系统提示里硬编码每个工具用法。保护 KV Cache + 利用 Linux 生态可组合性。
- **反对拟人化 Multi-Agent 架构**：Peak 明确反对按角色划分（设计师 Agent、程序员 Agent、管理者 Agent），认为那是人类公司的运作方式，源于人类的上下文限制。Manus 只保留少量功能性 Agent：通用执行 Agent、规划器 Agent、知识管理 Agent、数据 API 注册 Agent。子 Agent 以 "Agent as a Tool" 实现，有明确输入输出 Schema。两种共享模式：通过通信共享（适合简短清晰任务，子 Agent 上下文只含该指令）vs 通过共享内存通信（适合需要完整历史的任务，如深度研究，但成本更高，无法重用 KV Cache）。
- **保留错误是最有效的行为改善手段**：常见本能是隐藏错误、清理痕迹、重置状态，但擦除失败会移除证据，模型永远无法学会错误恢复。Manus 实践：错误（包含失败动作和堆栈跟踪）保留在上下文中，模型会隐式更新内部"先验"，降低重复犯错概率，比人工设计的纠错机制更有效。错误恢复能力是判断真实 Agent 行为最明显的指标之一，大多数学术基准完全忽略了这一能力。
- **Few-Shot 在 Agent 中可能适得其反**：大语言模型是优秀的模仿者，上下文充满类似的过去动作-观察对时，模型会僵化模仿历史模式。典型案例：审查 20 份简历时 Agent 会陷入僵化节奏，仅因为上下文中看到的模式就重复类似动作。解法：在动作和观察中引入少量结构化变化（不同序列化模板、替代性措辞、顺序或格式上的微小噪音），可控随机性打破模式，迫使模型根据当前状态决策而非照搬历史。
- **六七个月五次重构，最大飞跃来自简化**：Manus 上线后每次大幅改善不是来自增加更多上下文管理层，而是来自移除不必要技巧、更多信任模型。架构质量评估方法：固定架构，切换模型，用"模型能力梯度"评估——若换更强模型性能大幅提升，说明架构未过拟合当前模型局限性，未来可期。设计理念从 "Less Structure, More Intelligence" 到 "Build Less, Understand More"，一脉相承。

### 🎯 关键洞察

**Context Engineering 五维度不是独立优化目标，而是需要动态权衡的系统**：

![](../../_images/2026-01-04/GI8tbdu27odkJexwIDGcaNmxnDd_GI8tbdu2.png)

| 维度 | 手段 | 与其他维度的交互 |
|------|------|----------------|
| 卸载（Offloading） | 数据移到文件系统 | 让缩减更高效（可先卸载再摘要） |
| 缩减（Compaction/Summarization） | 压缩历史（最后手段） | 受卸载支撑；影响缓存效率和输出质量 |
| 检索（Retrieval） | 按需读取文件/URL | 稳定检索让隔离更安全 |
| 隔离（Isolation） | 子 Agent 管理上下文 | 减缓上下文增长，降低缩减频率；但增加成本，影响缓存命中 |
| 缓存（Caching） | KV Cache 命中率优化 | 更多隔离和缩减会影响缓存效率 |

**核心矛盾**：隔离越多，上下文增长越慢，但缓存越难命中；缩减越多，token 越少，但信息损失越大。没有银弹，只有在具体任务场景下的动态平衡。

**"可逆性"是贯穿整个 Context Engineering 的第一原则**：

- 卸载到文件系统 → 可逆（path 在，内容可重取）
- 摘要 → 不可逆（有损），所以是最后手段
- 保留错误 → 不可"逆向清除"（清除即失去证据）
- 保留 todo.md → 任务目标随时可见（注意力可"逆向锚定"到全局计划）

### 🛠️ 操作流程

#### 1. 准备阶段：确定你的"预腐烂阈值"
- 针对你使用的具体模型，设计一批长程任务测试用例
- 从 64k token 开始，逐步增加上下文长度，观察模型是否出现重复、推理变慢、质量下滑
- 记录退化开始出现的 token 数（通常 128k-200k），这就是你的干预红线
- **不要假设模型标称的上下文长度（如 1M token）就是安全边界**

#### 2. 核心执行：搭建五维度 Context 管理体系

**卸载层（优先做）**：
- 所有大块数据（网页内容、文件内容、API 响应）执行后立即写入沙盒文件系统
- 工具调用历史中只保留文件路径，删除 content 字段
- 确保沙盒文件系统在 Agent 生命周期内持久化

**工具空间（架构设计期决定）**：
- 核心函数调用限制在 10-20 个原子操作
- 新能力优先通过沙盒 CLI 工具实现（参考 mcp-cli 模式）
- 需要 Python 脚本的外部 API 统一走脚本层
- 工具命名统一前缀便于 Logit Masking 按组控制

**KV Cache 保护（全程注意）**：
- 系统提示内容固定，不含动态时间戳
- 所有 JSON 序列化显式指定键排序
- 操作历史只追加，严禁修改或删除历史记录
- 确认使用的 LLM 提供商是否支持自动前缀缓存；不支持则手动插入缓存断点

#### 3. 触达红线时：执行紧凑化流程
1. 先将关键中间结果写入文件（可选：将整个当前上下文作为日志文件存档）
2. 用结构化 Schema 进行摘要（固定字段：已修改文件列表/当前用户目标/执行进度/下一步计划）
3. 摘要后**保留**最近 N 条完整工具调用记录（N 根据任务复杂度调整，建议 5-10 条）

#### 4. 验证与优化：评估架构质量
- **黄金测试法**：固定架构，依次接入弱模型→强模型，观察性能提升幅度
  - 提升显著 → 架构健康，模型能力有空间发挥
  - 提升不明显 → 架构可能过度工程，是障碍而非跳板
- 监控 KV Cache 命中率作为生产环境核心指标
- 用户反馈（1-5 星）作为金标准，辅以内部自动化测试（可验证结果类任务）和人工评估（视觉/品味类任务）

### 💡 具体案例/数据

**成本数据（Claude Sonnet）**：
- 缓存命中：0.3 美元/百万 token
- 缓存未命中：3 美元/百万 token
- 差距：**10 倍**
- Agent 输入输出比 100:1，KV Cache 命中率对成本影响极其显著

**Manus 产品数据**：
- 平均每个任务：50 次工具调用
- 平均输入输出 token 比：100:1
- 上线后 6-7 个月内：经历 5 次重构
- 每次最大性能飞跃来源：简化和移除，而非添加

**学术基准 vs 用户反馈错位案例**：
- Manus 早期使用 Gaia 等公开学术基准选模型
- 发现 Gaia 高分模型用户不喜欢
- 原因：大多数基准偏推理任务，完全忽略执行类任务和错误恢复能力
- 结论：创建专注执行任务（而非推理题）的内部数据集

**Few-Shot 僵化具体案例**：
- 任务：审查 20 份简历
- 现象：Agent 在处理前几份后陷入僵化节奏，后续简历处理方式高度雷同
- 原因：上下文中充满类似动作-观察对，模型优先模仿历史模式
- 修复：引入不同序列化模板、替代性措辞、顺序微小噪音

### 📝 避坑指南

- ⚠️ **不要用模型标称上下文长度规划 Agent**：200k 左右实际退化，1M token 的上下文窗口不等于 1M token 的有效工作范围
- ⚠️ **JSON 序列化键顺序**：不显式指定 sort_keys 会导致 KV Cache 被悄无声息地破坏，成本翻倍且难以排查
- ⚠️ **动态增删工具会摧毁 KV Cache**：工具定义在上下文前部，任何变动都会导致后续所有内容缓存失效；改用 Logit Masking 替代
- ⚠️ **系统提示里不要放时间戳/用户 ID 等动态信息**：每次变化都会破坏前缀缓存
- ⚠️ **摘要是最后手段，不是常规操作**：有损压缩不可逆，应先穷尽卸载（外部化状态）方案
- ⚠️ **Multi-Agent 拟人化设计是陷阱**：按角色划分 Agent 是人类组织的映射，不是 AI 系统的最优解；按功能划分（通用执行/规划/知识管理）更合理
- ⚠️ **不要用学术基准（Gaia 等）作为 Agent 选模型的唯一标准**：学术基准偏推理，与用户实际偏好严重错位
- ⚠️ **过度工程的诱惑永远存在**：每增加一层上下文管理技巧，都要问"这是在帮助模型，还是在设置障碍"；Manus 5 次重构中最大飞跃来自删除而非添加
