### 📋 核心分析

**战略价值**: 通过"三层架构"全局配置，让 Claude Code 从"知错不改的职场新人"进化为"有条件反射的执行机器"，同时大幅降低 Token 消耗。

**核心逻辑**:

- **翻车根因一：环境错配**：CC 默认调用"最优雅"方案（如 PowerShell 变量、Python 库），但用户实际环境是 Git Bash 或未配置 Python，导致脚本乱码、卡死，Token 白烧。
- **翻车根因二：跨对话失忆**：每次新对话 AI 重置记忆，上一局踩的坑、花 Token 教会的经验全部清零，下一局照样重蹈覆辙。
- **翻车根因三：规则被"漏看"**：把规则埋在几百字长文档中间，AI 执行任务时追求速度，不会全文阅读，规则形同虚设。
- **Skill（技能包）的局限**：Skill 是预设流程胶囊，若流程内调用的工具与本地环境不匹配（路径错误等），照样卡死——治标不治本。
- **正确顺序**：先立全局规矩（Global Config），再叠加 Skill，顺序不能反。
- **三层架构第一层——铁律**：放在文档最顶端的死命令，例如"必须用中文回复"、"任务结束后必须自动判断是否需要复盘"，确保每次任务收尾都有反思动作。
- **三层架构第二层——索引（图书馆分区机制）**：当 AI 准备调用某类工具（如 Bash/命令行）时，规则自动引导它跳转到对应的"避坑区"，无需翻遍全文，精准定位，减少 Token 消耗、提升执行速度。
- **三层架构第三层——动态经验库（错题本）**：每次踩坑后的正确做法记录在此，旧方法失效则标记过时，新方案自动置顶，实现跨对话的知识持久化。
- **省钱机制**："精准索引"让 AI 每次只读与当前工具相关的几行规则，而非反复吞吐成百上千字，Token 消耗大幅下降。
- **人机共同进化的操作闭环**：发现 AI 犯傻 → 质问原因 → 探讨逻辑漏洞 → 让 AI 自己把修正后的规则写进配置 → 下次生效。这个闭环不依赖编程能力，只依赖逻辑和管理思维。

### 🎯 关键洞察

**为什么"越聪明的 AI 越容易翻车"**：
Opus 级模型脑子里装的是全球最佳编程范式，它的"最优解"是基于理想环境假设的。用户的真实环境（Git Bash、未配 Python、路径异常）与它的预设存在硬性冲突。模型越强，执行"专业方案"的倾向越强，在不适配环境里偏差就越大。

**规则失效的本质**：
AI 执行任务时处于"速度优先"模式，不会线性阅读长文档。规则必须满足两个条件才能被执行：① 位置足够显眼（顶端或索引直达）；② 触发条件足够明确（"当你要用 Bash 时，先看 X 区"）。

**OpenClaw（小龙虾）同理**：
即便是更亲民的智能体界面，底层管理逻辑一致——观察错误 → 引导复盘 → 经验入库。工具界面变了，人的管理方法论不变。

### 🛠️ 操作流程

1. **准备阶段：诊断你的环境**
   - 确认本机 Shell 类型（Git Bash / PowerShell / CMD / zsh）
   - 确认 Python 是否已配置、路径是否正常
   - 把这些环境信息作为"铁律"的一部分写入全局配置顶端，例如："本机使用 Git Bash，禁止调用 PowerShell 变量语法；Python 环境未配置，禁止直接调用 Python 库"

2. **核心执行：写入三层全局配置**
   - 打开 CC 的全局配置文件（`~/.claude/CLAUDE.md` 或对应路径）
   - **顶端写铁律**：语言要求 + 环境约束 + 任务结束强制复盘指令
   - **中段写索引**：按工具类型分区，格式参考："# 工具索引 / 当你使用 [Bash] 时，必须先阅读 [## Bash 经验区]"
   - **末段建经验库**：按工具分区，初始可为空，首次踩坑后填入

3. **验证与优化：主动触发进化闭环**
   - 执行一次真实任务，观察 AI 是否有漏掉规则的行为
   - 若漏掉：直接质问"规则里写了 X，你为什么没执行？"
   - 让 AI 自己分析原因并提出规则修改方案
   - 确认修改方案后，让 AI 直接更新配置文件
   - 重复此闭环直到规则稳定

### 💡 具体案例/数据

- **翻车案例**：整理数百个文件夹的鞋子照片（用于 LoRA 训练），预期几秒完成。实际：CC 让 Bash 和 PowerShell 互相冲突，输出乱码脚本，最终卡死，消耗数千 Token，任务未完成。
- **规则失效案例**：在全局配置长文档中间写了"遇到问题解决后要记录经验"，AI 整理完图片、出错后直接准备结束对话，完全没有执行记录动作。直到作者主动质问才触发复盘。
- **AI 自我诊断原话**："你的规则写得太模糊了，而且埋在几百字的长文档中间。我干活时追求速度，根本不会全文阅读，所以就给'漏掉'了。" —— 这句反馈直接催生了索引层的设计。

### 📝 避坑指南

- ⚠️ **不要只用 Skill 不配全局规则**：Skill 是流程胶囊，环境不匹配照样卡死，必须先有全局规则兜底。
- ⚠️ **规则不能埋在长文档中间**：AI 执行时不线性阅读，规则必须在顶端或通过索引直达，否则等于没写。
- ⚠️ **触发条件必须具体到工具名**：写"遇到问题时查经验库"无效；写"当使用 Bash 时跳转至 Bash 经验区"才有效。
- ⚠️ **经验库需要强制写入机制**：不能依赖 AI 自觉记录，必须在铁律层写明"任务结束后，若本次使用了命令行工具，必须自动检查经验库并更新"。
- ⚠️ **跨对话失忆是系统性问题**：每次新对话都是白板，经验库写入全局配置文件是唯一的持久化手段，不要依赖对话上下文传递经验。
- ⚠️ **环境信息必须显式声明**：不要假设 AI 能自动检测你的环境，Shell 类型、Python 状态、路径规则都要在铁律层明确写出。
