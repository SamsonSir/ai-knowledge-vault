### 📋 核心分析

**战略价值**: 用真实微信小程序项目（资产管理+人生目标）验证了 Claude Code 4个开发阶段的最优 skills 组合拳，结论是精选5个胜过囤积78个。

**核心逻辑**:

- **skills 数量陷阱**：作者装了78个 skills，大多数从未打开过。skills 不是收藏品，是工具，用不上等于零。
- **阶段一（原型）用 `brainstorming`**：宽泛问题只能得到宽泛答案。正确姿势是让 CC 出选择题，逼自己做决策。作者通过十几道选择题，把"做个记账软件"收敛成"22-28岁年轻人的资源分配博弈游戏"。
- **brainstorming 的真实价值**：它能生成你自己想不到的选项，比如"资源分配博弈"这种差异化定位，靠自己头脑风暴大概率想不到。
- **阶段二（UI）先做 playground 再写代码**：直接写代码容易跑偏，先用 playground 实验交互逻辑，确认方向再动手，节省 token 和返工成本。
- **`ui-ux-pro-max` 需要设计约束才能发挥**：不加约束只能出普通样式。必须把产品调性翻译成设计语言：产品类型 + 核心交互 + 目标用户 + 参考风格 + 要避开的雷区，五要素缺一不可。
- **`planning-with-files` 适合多环节项目**：自动生成 `task_plan.md`、`findings.md`、`progress.md` 三文件，随时可问"我现在做到哪了"，对间歇性失忆型开发者极其友好。小项目不必用，涉及云开发+数据库+UI+上架等多环节时必用。
- **阶段三（开发）先写测试再写功能**：微信小程序调试成本高（改完要重新编译+扫码真机测试），`test-driven-development` 让每次重构都能秒级验证，不用手动输数据。
- **Playground→小程序代码转换必有 bug**：HTML+CSS+JS 转 WXML+WXSS+JS，第一遍大概率出错。解法：PUA AI，让它"以完整成熟产品角度修改，有困难先自己搜文档"，它会主动查微信官方文档并自动修复。
- **阶段四（沉淀）用 `insights` 提炼专属 skills**：项目结束前别 exit，让 CC 基于本次对话生成你自己的定制 skills，把开发习惯沉淀成可复用资产。
- **朋友只装5个的反馈**："够用了，很清爽。" 精准 > 堆量。

### 🎯 关键洞察

**为什么要先做 playground？**
原因：Playground 是低成本的交互验证环境，在这里发现逻辑错误的代价接近零；进入正式代码后发现同样的问题，代价是重构+浪费 token+心态崩溃。
动作：用 `ui-ux-pro-max` 生成 playground 界面，跑通交互逻辑后再转小程序代码。
结果：减少中途返工，整体开发效率更高。

**为什么 TDD 在小程序开发里特别重要？**
原因：小程序有体积限制（2MB）、rpx 适配、微信特有 API、审核规范等约束，调试链路比普通前端长得多。
动作：核心算法（如资源分配计算）先写测试用例，再写功能代码，重构时秒级验证。
结果：作者的资源分配算法迭代了三版（简单平均→优先级权重→资金池冻结机制），每次都靠测试兜底，没有裸奔。

### 🛠️ 操作流程

**阶段一：原型阶段**

1. 调用 `brainstorming`，不要问"我该做什么"，而是说"帮我出选择题，逐步确认产品定位"
2. 逐一回答选择题：产品定位 → 目标用户 → 功能范围 → 技术栈偏好
3. 输出物：明确的目标用户画像 + 核心痛点 + 差异化玩法

**阶段二：UI + 规划阶段**

1. 调用 `ui-ux-pro-max`，提供设计约束五要素：
   ```
   产品类型（如：游戏化人生RPG）
   核心交互（如：资源分配博弈）
   目标用户（如：22-28岁年轻人）
   参考风格（如：Monument Valley极简几何 + Duolingo激励感）
   要避开的雷区（如：传统理财软件的严肃蓝绿色）
   ```
2. 生成 playground，验证交互逻辑，确认方向
3. 调用 `planning-with-files`，生成三文件里程碑规划
4. 随时可问："我现在做到哪了？" 获取进度

**阶段三：开发阶段**

1. 调用 `test-driven-development`，先为核心算法写测试用例
2. 对 CC 说："现在根据 playground 的演示，生成真正的小程序代码，前端设计完全按照 playground 来不要自己改"
3. 第一遍必有 bug，用以下 prompt PUA AI：
   ```
   你一定要以一个完整成熟的小程序产品的角度去修改代码，不要从简。
   有什么困难先自己上网搜解决，我相信你有这个能力。
   ```
4. AI 会主动查微信官方文档并自动修复问题

**阶段四：沉淀阶段**

1. 项目结束前，不要 exit，发送：
   ```
   基于我们刚才写小程序的对话，结合 insights 报告给我整理一下我以后可能需要生成的 skills
   ```
2. 根据输出，提炼专属于自己开发习惯的定制 skills

### 💡 具体案例/数据

- 作者项目：资产管理 + 人生目标微信小程序
- 目标用户锁定：22-28岁刚工作年轻人
- 核心痛点：有买房/旅行/学习目标，但不知道每月该存多少，也管不住乱花钱
- 差异化定位：资源分配博弈（像策略游戏一样在多目标间分配资金）
- 资源分配算法迭代三版：
  - 第一版：简单平均分配
  - 第二版：加入目标优先级权重
  - 第三版：加入"资金池冻结"机制（已存的钱不能随意挪用）
- 朋友只装5个 skills 的反馈："够用了，很清爽。"（对比作者78个大多从未打开）

### 📝 避坑指南

- ⚠️ `brainstorming` 问题太宽泛会得到一堆没用的发散想法，必须让它出选择题来收敛
- ⚠️ `ui-ux-pro-max` 不给设计约束，第一次出来的样式就是普通水平，五要素缺一不可
- ⚠️ Playground（HTML+CSS+JS）转小程序（WXML+WXSS+JS）第一遍大概率有 bug，不要期望一次成功
- ⚠️ `planning-with-files` 小项目用了反而增加管理负担，只在多环节复杂项目中使用
- ⚠️ 项目结束后直接 exit 会丢失沉淀机会，先跑 `insights` 再退出
- ⚠️ 微信小程序有 2MB 体积限制，开发前需规划分包加载策略
