### 📋 核心分析

**战略价值**: Claude Code 的核心瓶颈不是提示词质量，而是上下文窗口管理——掌握这 9 条实战策略，可将 AI 编程效率提升 3 倍以上。

**核心逻辑**:

- **上下文窗口是根本约束**：Claude 的"白板"有固定大小，每条消息、每个读取的文件、每条执行命令都会占用空间。白板满了，Claude 开始忘记早期指令、重复自己、犯低级错误——这不是模型变笨，是注意力被稀释。Claude Code 创建者 Boris Cherny 明确指出：用好 Claude Code 的关键就是管理好这块白板。
- **Plan Mode 先于写代码**：按 `Shift+Tab` 切换到计划模式，Claude 只读文件、理清思路，不动代码。跳过计划直接写代码，15 分钟的生成可能解决的根本不是你要的问题，且已消耗大量上下文空间。
- **子代理隔离调研任务**：子代理是独立 Claude 实例，在自己的上下文窗口内工作，结果汇报回主会话但不占用主会话白板。所有"读日志、查源码、分析调用链"的调研任务交给子代理，主会话只做决策和写代码。
- **CLAUDE.md 是持久化规则书**：Claude 每次启动都会读取该文件，一次写入永久生效。每次 Claude 犯错被纠正后，追加一句"更新 CLAUDE.md 以免再犯"，让 Claude 自己把教训写进规则。但文件不能超过约 500 行，否则注意力分散导致规则被忽视。
- **提示词内置验证标准**：不要让自己成为唯一质检员。写需求时同时给出测试用例，要求 Claude 执行后汇报结果；视觉任务粘贴截图并要求 Claude 截图对比差异。
- **提示词要像接口文档一样具体**：明确输入、输出、约束、验证方式。"给 auth.py 加测试"远不如"写 auth.py 的测试，覆盖会话中途过期场景，不允许用 mocks，重点测试令牌看似有效但实际过期的边缘情况"。
- **每 30-45 分钟主动重置上下文**：Boris 团队在复杂任务中的固定节奏。步骤：① 让 Claude 总结当前进度、剩余任务、当前问题；② 开新会话；③ 粘贴总结继续。一份精炼的总结比臃肿的上下文更有效。
- **并行会话 + Git Worktree**：同时开多个 Claude Code 会话，配合 `git worktree` 创建多个独立工作副本互不干扰。典型模式：会话 A 写功能，会话 B 以高级工程师身份审查 A 的产出，找边缘问题和缺陷。有人同时开 3-5 个会话，用 shell 别名（`za`, `zb`, `zc`）一键切换。
- **重复操作封装成 Skill**：每天重复两次以上的操作就该封装。Skill 是保存好的工作流程，调用名字即可触发完整步骤。Boris 团队给 BigQuery 封装了 Skill，团队成员直接在 Claude Code 里做数据分析，无需写 SQL。
- **给真实数据而非猜测**：直接把 Slack 错误对话、CI 失败日志、Docker 输出贴给 Claude，说"fix"即可。不需要手把手描述问题，Claude 自己读真实数据定位故障。Boris 团队接入 Slack MCP 后，Bug 报告直接转给 Claude 处理。

### 🎯 关键洞察

**上下文管理 = 管理 AI 的注意力**，这是独立于编程能力和领域知识之外的第三种技能。

白板满了有三个典型症状：① Claude 忘记一小时前的指令；② 开始重复自己；③ 犯本不该犯的错误。识别这些信号，立即触发上下文重置流程，而不是继续在同一会话里挣扎。

Boris 团队的极端实践：用一个 Claude 写计划，另一个 Claude 以高级工程师身份审查计划——计划阶段的严肃程度直接决定执行阶段的返工量。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| Plan Mode | `Shift+Tab` 切换 | 只读文件不动代码，保护上下文空间 | 必须在写代码前切换，事后切换无意义 |
| 子代理 | 提示词末尾加 `use subagents` | 调研任务在独立上下文完成，主会话不受污染 | 适用于读日志、查源码、分析调用链等重 I/O 任务 |
| CLAUDE.md | 项目根目录创建，Claude 启动自动读取 | 规则持久化，无需每次重复说明 | 不超过 500 行；每条规则必须对应"没有它 Claude 会犯错"的场景 |
| Git Worktree | `git worktree add ../project-b feature-b` | 多会话并行，各自独立工作副本 | 配合 shell 别名（`za/zb/zc`）管理切换 |
| Skill | 在 Claude Code 内定义并命名工作流 | 重复操作一条命令触发 | 每天重复 2 次以上即值得封装 |
| Slack MCP | 接入 Slack MCP，将 Bug 对话直接传给 Claude | 无需人工描述问题，Claude 自读真实数据修复 | 需要配置 MCP 集成 |

### 🛠️ 操作流程

**开始任务前**
- 检查 CLAUDE.md 是否需要更新
- 判断任务复杂度，决定是否启用 Plan Mode（`Shift+Tab`）

**任务执行中**
1. 复杂调研 → 提示词末尾加 `use subagents`，扔给子代理
2. 写代码 → 主会话 + 具体提示词（明确输入/输出/约束/验证）+ 内置测试用例
3. 代码审查 → 开第二个会话，以高级工程师身份审查产出

**每 30-45 分钟**
- 让 Claude 执行："总结我们目前做了什么，还剩哪些任务，当前有哪些问题。"
- 评估是否需要开新会话重置上下文

**任务完成后**
- Claude 犯过的错 → 追加"更新 CLAUDE.md 以免再犯"
- 重复操作 → 封装成 Skill

### 💡 具体案例/数据

- **CLAUDE.md 活规则案例**：作者发现 Claude 在 Obsidian Vault 里反复创建 `.txt` 文件（Obsidian 不显示该格式），纠正一次后追加规则"Vault 内只创建 `.md` 文件"，此后再未复现。
- **子代理效率差距**：作为 Java 后端开发，排查复杂分布式系统问题时，以前一个排查任务就能把整个会话搞废；改用子代理后，主会话只做决策和写代码，效率差距"巨大"。
- **BigQuery Skill**：Boris 团队封装了 BigQuery 数据分析 Skill，团队成员直接在 Claude Code 内查询，无需写 SQL，跨所有项目复用。
- **Slack MCP Bug 修复**：Slack 上报 Bug → 把对话贴给 Claude → 说"fix" → Claude 自己读对话定位问题修复，全程无需人工描述。
- **效率提升量化**：作者自述整套流程让效率提升至少 3 倍。
- **并行会话规模**：有用户同时开 3-5 个会话，配合 `git worktree` 和 shell 别名（`za`, `zb`, `zc`）管理。

### 📝 避坑指南

- ⚠️ **不要跳过计划直接写代码**：15 分钟的错误方向生成，既浪费上下文空间，又产生大量返工。
- ⚠️ **CLAUDE.md 不能无限堆砌**：超过约 500 行后，Claude 注意力分散，部分规则会被忽视。每条规则必须通过"没有它 Claude 会犯错吗"的测试，否则删除。
- ⚠️ **不要舍不得关会话**：臃肿的上下文比精炼的总结更有害。出现"Claude 开始跑偏、反复啰嗦、忘记早期规则"时，立即重置，不要继续在同一会话里挣扎。
- ⚠️ **不要给 Claude 你的猜测，要给真实数据**：描述 Bug 的语言会引入你的主观偏差，直接贴错误日志、Slack 线程、Docker 输出，让 Claude 自己读原始数据。
- ⚠️ **模糊提示词是上下文杀手**：模糊需求导致多轮猜测和纠正，每一轮都在消耗白板空间。一次具体的提示词 > 三次模糊的来回。
