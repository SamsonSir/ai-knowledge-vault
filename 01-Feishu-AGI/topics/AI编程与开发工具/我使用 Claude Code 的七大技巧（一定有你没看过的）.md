### 📋 核心分析

**战略价值**: 通过 Plan 模式、分级 Claude.md、自分形注释、Agent 协作等七项具体技巧，系统性降低 Claude Code 的返工率和上下文过载风险。

**核心逻辑**:

- **先 Plan 后代码**：复杂需求必须先用 `/plan` 命令或 `Shift + Tab` 进入 Plan 模式，AI 会主动追问细节、确认边界，输出任务文档，再开始写代码，减少返工。超大型任务可要求将任务持久化为文档，Claude Code 已支持多会话共享 todo（默认未开启）。
- **上下文过载预警机制**：上下文窗口使用超过 40% 时，AI 指令遵循度开始变差；低于 50% 剩余时就应 `/new` 重开会话。用 `/showline` 命令让状态栏实时展示上下文百分比。
- **关闭上下文压缩**：默认开启的上下文压缩会占用总上下文的 22.5%，即使什么都不做，建议关闭。
- **Claude.md 三条核心指令**：① 称呼规则：每次回复前必须使用"Boss"作为称呼（用于检测指令遵循度，一旦消失说明上下文过载）；② 决策确认：遇到不确定的代码设计问题必须先询问，不得直接行动；③ 代码兼容性：不能写兼容性代码，除非主动要求（防止 AI 用 if 分支堆砌冗余代码）。
- **Claude.md 懒加载**：不要把所有规范都堆进根目录 Claude.md，而是写三条"场景触发"指令，告诉 Claude Code 遇到某场景时去读对应文件，按需加载。
- **自分形三件套**：① 每个业务模块文件夹放一个 Claude.md，记录该模块的地位、逻辑、约束、子模块清单；② 所有源码文件头部写三行注释：`INPUT`（依赖什么）、`OUTPUT`（提供什么）、`POS`（在系统中的地位），放头部是因为 AI 从头部开始加载；③ 每次源码变动后自动逐级更新：源码头部注释 → 当前文件夹 Claude.md → 上一层文件夹 Claude.md。
- **提示词三原则**：① 清晰描述任务最终目的（功能点、页面元素）；② 给参考上下文（接口数据、接口文档、已有页面）；③ 告诉它如何测试（如前端让它用自带 Chrome 验证）。
- **MCP 按需调用**：Claude Code 经常不主动调用 MCP，需要主动指示（如"调用 Context7 查看线上文档"）。MCP 过多会占用上下文窗口，用 `/context` 命令查看详细占用情况，原则是"能本地解决的，别去连云端"。
- **Agent 协作三招**：① 搜索大量代码时让 SubAgent 去搜，SubAgent 是独立上下文，不污染当前会话；② 一条指令串联多 Agent 对抗式纠错（如执行完后自动调用 CodeReview Agent + Test Agent）；③ 多分支并行开发，多开窗口让不同 Agent 同时开发不同模块，再开一个专门处理 PR、代码 Review 和测试验证。
- **Skill 沉淀工作流**：自定义斜杠命令与 Skill 已合并，将高频操作沉淀为 Skill。例如"Commit Skill"：检测待提交代码 → 完善头部注释 → 更新上级 Claude.md → 执行 Git 提交。Skill 支持定制 Prompt、工作流、调度 SubAgent。

### 🎯 关键洞察

**自分形的核心价值在于利用 Claude Code 的分级加载机制**：Claude Code 搜索文件时会自动加载该目录下的 Claude.md。当某文件位于第五层级文件夹时，Claude Code 搜到它的过程中已经依次加载了前四层的 Claude.md，对整个模块的上下文无比清晰。这是一种"被动注入上下文"的架构设计，不依赖用户每次手动提供背景信息。

头部三行注释（INPUT/OUTPUT/POS）的逻辑：AI 加载文件从头部开始，把业务元信息放在最前面，确保每次读取任何文件都能先获取到业务定位，而不是读完整个文件才能理解它的作用。

### 🛠️ 操作流程

1. **项目初始化阶段**:
   - 根目录创建 Claude.md，写入三条核心指令（Boss 称呼、决策确认、禁止兼容性代码）
   - 关闭上下文压缩设置
   - 为每个业务模块文件夹创建 Claude.md，填写地位/逻辑/约束/子模块清单
   - 所有源码文件头部添加 INPUT/OUTPUT/POS 三行注释

2. **任务执行阶段**:
   - 复杂需求先 `Shift + Tab` 进入 Plan 模式，与 AI 确认边界和结构，拿到任务文档
   - 提示词包含：最终目的 + 参考上下文（接口文档/已有页面）+ 测试方法
   - 遇到编译问题或效果偏差，主动指示调用 Context7 查线上文档
   - 搜索大量代码时，指示开 SubAgent 执行，避免污染当前上下文
   - 执行完成后，指示自动调用 CodeReview Agent + Test Agent 进行对抗式验证

3. **上下文管理阶段**:
   - 用 `/showline` 实时监控上下文百分比
   - 剩余低于 50% 时，`/new` 重开会话
   - 发现 AI 回复中没有"Boss"前缀，立即重开会话
   - 用 `/context` 检查 MCP 等占用情况，按需裁减

4. **提交与维护阶段**:
   - 触发 Commit Skill：自动检测变动代码 → 完善头部注释 → 逐级更新 Claude.md → Git 提交
   - 保持自分形结构的自动更新，确保下次 AI 加载时上下文依然准确

### 💡 具体案例/数据

- 上下文压缩默认占用总窗口的 **22.5%**，即使空会话
- 上下文使用超过 **40%** 时，AI 指令遵循度开始变差
- 剩余低于 **50%** 时建议重开会话
- 自分形场景：某文件位于第五层级文件夹，Claude Code 搜到它时已依次加载了前四层 Claude.md，对整个模块上下文完全清晰
- Commit Skill 实际工作流：检测待提交代码 → 完善 INPUT/OUTPUT/POS 注释 → 更新当前文件夹 Claude.md → 更新上一层文件夹 Claude.md → 执行 Git 提交

### 📝 避坑指南

- ⚠️ 不要把所有规范堆进根目录 Claude.md，会直接占满上下文，改用懒加载（场景触发指令指向对应文件）
- ⚠️ 不要安装过多 MCP，每个 MCP 都占用上下文窗口，用 `/context` 定期检查占用情况
- ⚠️ AI 不带"Boss"前缀时不要继续当前会话，这是上下文过载的明确信号
- ⚠️ 迭代任务中不加兼容性约束，AI 会习惯性用 if 分支堆砌代码，越改越乱
- ⚠️ 搜索大量代码不开 SubAgent，会直接把当前会话上下文撑爆，导致主任务还没开始就过载
- ⚠️ 源码头部注释必须放在文件最顶部，放在中间或底部 AI 读取时可能已经截断
