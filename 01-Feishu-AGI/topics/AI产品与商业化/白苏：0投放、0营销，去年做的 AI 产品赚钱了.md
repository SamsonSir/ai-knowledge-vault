### 📋 核心分析

**战略价值**: 一个 1.5 人团队、0 营销预算，用 Vibe Coding + AI-First 运营，从 0 到 1 打造设计 Agent 产品 Mew Design 并实现盈利的完整路径复盘。

**核心逻辑**:

- **产品定位先于技术选型**：Mew Design 定性为"面向非设计师的设计产品"，而非 AI 工具产品。这个定位决定了所有后续取舍——隐藏模型复杂度、不做提示词竞技场、把"可用设计交付"优先于"可配置自由度"。
- **Vibe Coding 的正确姿势是"建规则，放 AI"**：作者开始时连 TypeScript 都没写过，但第一版 demo 的时间分配是：框架选型+技术验证 2 周、原型+表结构 2 周、Design System+组件落地 2 周、编码+测试不到 2 周。时间大头在建规则，规则建完才是 AI 放飞时刻。2025 年全年基本做到一天一版，且全部是 AI commit。
- **执行层给 AI，判断层握在人手里**：作者花最多时间的是思考与决策、规则制定、架构取舍、质量标准，而非写代码。AI 只把下限拉高，上限还是看人。
- **AI-First 运营替代传统 BI 和工程排期**：内部没有传统 BI 系统，用两个 Agent 替代：①日志 Agent 专盯日志、洞察错误、自动提 issues；②数分 Agent 直接拿数据库权限，支持随时追问动态报表（如"今天为什么注册爆了"、"X 国用户付费率低的归因"）。
- **风控 Agent 替代第三方服务**：用 Kimi 超大杯会员写了一个风险用户识别 Agent，自动封禁+自动拉黑，截至写文时识别准确率 100%，比接第三方服务更灵活、更有性价比。
- **主动控流是反常识但正确的决策**：不是没钱才不做营销，而是主动控流。原因有二：①大起大落破坏团队节奏；②非目标用户的高互动反馈会把产品带离主航道，AI 产品转化成本还要叠加 token 成本。
- **冷启动靠社区蹭曝光，不靠预算**：所有预算给了 token 换用户反馈。曝光渠道：WaytoAGI、观猹（国内 AI 产品发现社区）、愿意给露出的朋友、买了一个导航站外链。见过投资人但没有冲动拉曝光，判断是"产品核心价值没站稳，露出越大死得越快"。
- **团队构成极简**：作者（产品+技术）+ 设计朋友 + 海外增长朋友，共 3 人，其余"员工"是 AI。域名 2025 年 1 月 4 日购入，第一版 2025 年 3 月 12 日上线。
- **技术栈选型**：Next.js（非 SPA）、TypeScript、Agent 框架自拆模块、云服务动态路由、CI/CD 逐步接入，SEO 纳入架构考量。开发工具早期同时用 3 个 Cursor 实例并行。
- **产品价值观即护城河**："不管黑猫白猫，能抓老鼠就是好猫"——只交付结果，不交付噱头。这套价值观直接决定了反 AI 产品常识的三个选择：隐藏模型复杂度、社区强调结果质量、交付优先于自由度。

### 🎯 关键洞察

**为什么"反 AI 产品常识"的选择反而是对的？**

逻辑链：目标用户是非设计师（连设计都不一定懂）→ 让他们先做技术决策（选模型、看提示词）= 增加认知负担 → 流失。所以正确路径是：把技术复杂度全部内化，只暴露结果界面。这和 iPhone 当年隐藏文件系统是同一个逻辑。

**为什么 AI 产品要主动控流？**

传统互联网：流量 = 反馈密度 = 迭代速度。AI 产品多了一个变量：token 成本。非目标用户涌入 → 高互动但低价值反馈 → 产品方向被带偏 + token 成本上升 → 双重损耗。控流本质是在保护有限资源的转化效率。

**Vibe Coding 的真实上限在哪？**

作者现在依然不能完整手写 TS，但能深度讨论 Next.js vs SPA 取舍、Agent 框架拆法、模块实现、云服务权衡、SEO、模型动态路由。说明 Vibe Coding 的上限不是"能不能写代码"，而是"能不能定义问题、建立规则、做架构判断"。这是产品思维 + 系统思维，不是编程能力。

### 🛠️ 操作流程

1. **准备阶段（约 6 周）**
   - 方向验证：裸辞后花 2 周做小插件验证"设计 Agent + 只做图文"路线可行性
   - 团队搭建：作者（产品+技术）+ 设计朋友 + 海外增长朋友，共 3 人
   - 域名购入：2025 年 1 月 4 日
   - 框架选型+技术验证：2 周（Next.js、TypeScript、Agent 框架拆法）
   - 原型+表结构设计：2 周

2. **核心执行（约 4 周）**
   - Design System 设计+组件落地：2 周
   - 第一版编码+测试：不到 2 周（3 个 Cursor 并行）
   - 第一版上线：2025 年 3 月 12 日
   - 冷启动：找设计圈朋友做专业反馈，持续迭代；蹭社区（观猹、WaytoAGI）和展会曝光；买一个导航站外链
   - 逐步接入 CI/CD，进入"提需求+聊想法+拍方案"模式

3. **AI-First 运营阶段（持续）**
   - 上线日志 Agent：自动巡检+提 issues
   - 上线数分 Agent：直接给数据库权限，每日动态报表+随时追问
   - 上线风控 Agent：自动识别风险用户、封禁、拉黑
   - 全年保持一天一版节奏，commit 全部由 AI 完成
   - 2025 年最后一天实现盈利

### 📝 避坑指南

- ⚠️ 规则没建好就让 AI 写代码 = 把错误放大。先花时间建规则（架构、模块边界、质量标准），再放 AI 执行。
- ⚠️ 把"看起来在增长"当成"真的在增长"。流量冲高但产品核心价值没站稳，露出越大死得越快。
- ⚠️ 非目标用户的反馈会把产品带离主航道。AI 时代高互动反馈 ≠ 高价值反馈，要主动过滤。
- ⚠️ AI 产品转化成本 = 获客成本 + token 成本，不能只算获客成本。
- ⚠️ 见了投资人就想拉曝光是陷阱。没钱的阶段，打磨产品是第一优先级，少凑热闹。
- ⚠️ 过早搭重基建（BI、固定报表、第三方风控服务）是资源浪费。能用 Agent 替代的流程化工作，先用 Agent 跑通。
- ⚠️ 把 AI 建议当成产品方向。"为什么不能选模型"、"为什么不做提示词说明书"这类建议来自 AI 产品视角，但你的用户可能根本不是 AI 用户。
