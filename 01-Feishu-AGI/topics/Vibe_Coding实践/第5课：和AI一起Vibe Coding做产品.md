### 📋 核心分析

**战略价值**: 用一个「个人主页 + AI工具导航」完整案例，手把手演示从痛点分析→快速demo→可行性讨论→版本规划→PRD→设计→开发→Debug的全链路AI编程工作流。

**核心逻辑**:

- **产品开发是循环，不是线性**：流程是 痛点→demo→PRD→设计→开发→上线→回到PRD，每次迭代都走完整圈，不是一次性做完
- **痛点分析两层思维**：第一层先回答「帮谁解决什么问题」，第二层在做中不断迭代对痛点的洞察；允许一开始回答不清楚，但必须回到用户视角，不能自嗨
- **快速demo的核心价值**：不是做出完美产品，而是验证技术可行性、发现布局问题（如工具列表太长需分类）、发现架构问题（如硬编码不够灵活），这些发现直接影响版本规划和PRD内容
- **可行性讨论必须提前做**：用模板向AI提问5个维度——技术方案（纯前端 or 需要后端）、数据存储（是否需要数据库）、工作量（大概多长时间）、风险（可能的坑）、替代方案（有没有更简单做法）；提前暴露坑，避免做到一半发现需求压根没法干
- **版本规划的核心原则**：V0.1 MVP只做最核心一个痛点，可以丑、可以手动、可以有限制，但必须能跑通；每个版本只解决一个清晰子需求；每个版本必须有明确「验证点」（如何判断这个版本做完了）；明确写「不做什么」避免范围蔓延
- **PRD的三个核心价值**：① 版本管理，功能迭代不抓瞎；② 逼你把产品细节想清楚（AI会问很多问题补充遗漏细节）；③ 减少开发过程中人和AI的理解偏差
- **设计讨论两步走**：先用ASCII图确定布局（结构优先，改起来成本低），再讨论风格和美学（配色、字体、交互）；不要一开始就纠结颜色字体
- **开发三原则**：① 基于PRD每次只干一件事，不要让AI一下子全干完；② 让AI先讲他要怎么做，逻辑合理了再让他做；③ 每一步都要验收，不能跳步
- **Debug四层方法**：① 先把问题变成「可观察的」（打日志暴露关键路径）；② 用事实而不是感觉和AI对话（发生了什么、控制台输出了什么）；③ 基于证据让AI缩小问题范围（用日志排除，不是一次性大改）；④ 修改→立即验证→再反馈，形成闭环

### 🎯 关键洞察

**失败案例的价值**：QuickSave浏览器插件案例说明——即使痛点真实存在（每天看AI资讯想收藏），产品也可能失败。原因是没有在做中迭代对痛点的认知，闷头做完发现方向偏了。这是为什么「快速demo验证」和「版本规划」如此重要的底层逻辑。

**Debug的本质转变**：AI编程时，人的角色不是「修代码的人」，而是「问题定位的导演」。最忌讳的是把「它好像没生效」这种情绪判断喂给AI——AI拿到的必须是执行结果（日志、报错、控制台输出），才能给出有效分析。

### 🛠️ 操作流程

#### 1. 准备阶段：痛点分析 + 快速Demo

**痛点分析**（回答这两个问题）：
- 做一个什么样的产品？
- 帮谁解决什么样的问题？

**个人主页 + AI工具导航案例的痛点**：
- AI工具太多，收藏夹乱七八糟，想用时找不到
- 想展示自己，但纯自我介绍太boring
- 想给别人推荐工具，没有集中展示的地方

**快速Demo目的**：验证能不能做、发现问题（不需要完美，能跑就行）

Demo完成后检查清单：
- ✅ 技术上是否可行
- ✅ 布局是否合理
- ❌ 发现的问题（如：工具列表太长需分类、硬编码需改成可编辑）

#### 2. 可行性讨论（直接用这个模板）

```
我想做一个产品：个人主页 + AI工具导航

我已经做了一个简单demo，现在想做正式版本。

请帮我分析：
1. 技术方案：纯前端 or 需要后端？
2. 数据存储：是否需要数据库做存储？
3. 工作量：大概需要多长时间？
4. 风险：有什么可能的坑？
5. 替代方案：有没有更简单的做法？

我的需求：
- 要能部署上线到github和vercel
- 要能编辑工具列表
- 手机和电脑都要能看
```

#### 3. 版本规划（V0.1 → V1.0）

版本规划提示词核心逻辑（完整提示词见原文）：

- **V0.1 MVP**：只做最核心一个痛点，能跑通即可，不做编辑/云同步/高级功能/美化
- **V0.2-V0.5**：优先级顺序：① 本地增删改查 → ② 数据备份/版本管理 → ③ 跨设备/分享 → ④ 导入导出等进阶功能
- **V1.0**：性能优化、错误处理、UI/UX打磨、文档引导，达到可对外发布质量

每个版本必须包含：功能清单 + 不做什么 + 验证点 + 预计工作量

#### 4. PRD写作（两种方式）

**方式1：提示词写PRD**（云舒版，Gemini 2.5 Pro）

三步确认法：
1. **定框架**：梳理业务流程/用户旅程，分阶段，获得用户确认后才进入下一步
2. **逐故事**：按阶段逐一讨论每个用户故事，UI故事必须画ASCII线框图，每个故事讨论完必须「单点确认」
3. **终稿确认**：回顾全局对齐清单（数据契约、接口流程、性能约束、异常策略、监控指标），用户说「可以生成了」才生成文档

PRD输出格式包含：
- 综述（项目背景 + 核心业务流程/用户旅程地图）
- 用户故事详述（价值陈述 + 业务规则与逻辑 + 验收标准GIVEN-WHEN-THEN + 技术实现概要 + 数据模型/接口契约 + 监控与运营 + 约束与边界 + 开放问题）

**方式2：Skills写PRD**

```bash
# 第一步：下载Skills
请你帮我下载 https://github.com/yunshu0909/yunshu_skillshub 的内容到这个文件夹里

# 第二步：在Claude Code / Codex中使用

```

#### 6. 开发执行

- 每次只给AI一个任务（基于PRD拆分）
- 让AI先说方案，确认逻辑合理再执行
- 每步完成后立即验收，不跳步

#### 7. Debug四层流程

```
第一层：打日志 → 在页面加载、用户交互、数据处理关键节点全部暴露执行路径
第二层：用事实喂给AI → "发生了什么 + 没发生什么 + 控制台输出了什么 + 在哪步中断"
第三层：用日志排除 → 确认哪些步骤正常，哪些没触发，缩小错误范围
第四层：修改→验证→反馈 → 复现问题→修复→再次操作验证，把新日志继续反馈给AI
```

### 📝 避坑指南

- ⚠️ **不要自嗨**：不要沉浸在自己的世界里，要回到用户视角，痛点是否真实存在需要验证
- ⚠️ **不要一次性让AI全干完**：每次只干一件事，否则出了bug不知道是哪步的问题
- ⚠️ **不要跳过验收步骤**：AI修完代码不等于问题解决，必须亲自操作验证
- ⚠️ **不要用情绪语言Debug**：「它好像没生效」对AI无效，必须提供具体的日志和报错信息
- ⚠️ **不要在布局未定时讨论配色**：先ASCII图确定结构，再讨论视觉风格，否则反复返工
- ⚠️ **PRD写作中AI不能自作主张**：所有内容必须源于对话和共识，未确认的细节记录为「开放问题」而不是AI自行填充
