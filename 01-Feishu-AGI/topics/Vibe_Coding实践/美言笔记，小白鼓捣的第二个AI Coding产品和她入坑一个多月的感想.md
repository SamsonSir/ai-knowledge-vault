### 📋 核心分析

**战略价值**: 一个产品经理视角的 AI Coding 入坑实录，从单文件 HTML 到 Cursor 多模块项目，提供可复刻的零基础上手路径与调试经验。

**核心逻辑**:

- **入口选 Claude Artifacts**：零基础起点是 Claude 的 Artifacts 实时预览功能，一句自然语言（如"给我写个贪吃蛇"）即可在框内跑出可交互产品，验证想法无需任何环境配置。
- **第一个可交付形态是单文件 HTML**：提示词模板为「我需要创建一个[需求]，请帮我用美观且响应式的HTML代码来实现」，Artifacts 预览满意后复制代码保存为 `.html` 文件，双击或拖入浏览器即可运行，可直接分享给他人。
- **单文件 HTML 的天花板**：当需求复杂、文件超过约 2000 行时，Cursor 等工具会拒绝应用代码，这是强制进入多文件模块化结构的信号，不是工具的问题，是单文件本身不优美。
- **AI Coding 工具三选一**：主流工具为 Cursor、Trae、Windsurf。作者路径是先用 Trae 国内版盲上，搞不定后转 Cursor，建议了解基础前端知识再上，否则项目一大会完全没有头绪。
- **Cursor 两种模式的切换时机**：
  - **Agent 模式**：项目初期，一句话生成雏形，适合需求描述阶段。
  - **Ask 模式**：项目结构清晰后，用于迭代、问题修复、代码自动补全。
- **先对齐需求再写代码**：复杂项目不要上来就让模型写代码，先用以下提示词对齐：
  > 「我需要创建一个[需求]，请仔细思考，详细描述一下项目实现步骤、所需外部依赖、可能的实现难点和项目风险点，设计清晰的层级结构管理项目目录和文件，不着急写代码。」
  模型输出的规划本身就是编程素养教材，同时帮助自己查漏补缺。
- **提升设计美感的具体操作**：① 搜集优雅设计稿给模型垫图；② 明确要求引用在线图标库（如 Font Awesome），否则模型默认用 emoji，视觉档次差距明显；③ 参考归藏、橘子、云中江树、阿真 Irene、一泽等博主的提示词案例。
- **写代码时要求模块化 + 详细注释**：开始写代码时就要求「按模块化良好地组织代码，对每个模块都尽可能添加详细注释」，便于后续自己理解和定位问题。
- **需求越具体 bug 越多**：模型在需求简单、发挥空间大时（如"一句话生成一套 APP 设计"）一次成型概率高；需求功能点越多越具体，模型越容易打补丁打出新 bug，这是结构性规律，不是偶发现象。
- **AI Coding 工具能覆盖约八成工作量**，剩下两成依然要靠人：看懂逻辑、定位细节 bug、判断根因。

### 🎯 关键洞察

**为什么小白也值得啃 AI Coding**：

- Coding 是 AGI 落地的执行层，是「模型的一只手」，现实世界多数任务可用 Coding 表达。AI Coding 的价值不在于提升程序员，在于**激活创造者**——产品经理、设计师、内容创作者都能把脑中的执念以产品形式交付。
- 技术壁垒正在被系统性打破（GPT-4o 图像能力发布是典型案例，吉卜力风格插画人人可生成），纯技术不再稀缺。价值正在从「事物/产品」转向「人/产品所代表的意义」——独特审美、深刻洞见、鲜明个人品牌、情感联系，这些才是下一阶段的护城河。
- 2025 年模型进化速度远超 2023-24 年，多条主线同时加速：推理模型、多模态（GPT-4o）、垂直 Agent（DeepResearch）、通用 Agent（Manus）、预训练模型。参与 AI Coding 是保持深度参与这个时代的方式之一。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|---|---|---|---|
| Claude Artifacts | 直接对话输入需求，实时预览 | 零配置跑出可交互原型 | 仅适合单文件前端，无法形成可分享产品闭环 |
| 单文件 HTML 导出 | 复制 Artifacts 代码 → 保存为 `.html` → 双击/拖入浏览器 | 可分享的独立小应用 | 超过约 2000 行后 Cursor 拒绝应用，需拆模块 |
| Cursor Agent 模式 | 项目初期使用，一句话描述需求 | 生成项目雏形和目录结构 | 不适合已有结构的迭代阶段 |
| Cursor Ask 模式 | 项目结构清晰后使用 | 问题修复、代码补全 | 需要自己能看懂上下文逻辑 |
| Font Awesome 图标库 | 在提示词中明确要求「使用 Font Awesome 图标库」 | 视觉档次显著高于 emoji | 不主动要求，模型默认用 emoji |
| F12 控制台 | 打开浏览器开发者工具 → Console 查看报错 → 复制报错信息给模型 | 精准定位 bug 根因 | 是小白最有效的 debug 辅助手段之一 |
| DeepSeek V3 换模型 | 新建项目，复制文件，用 V3 重新理解和检查 | 提供全局视角，解决 Claude 反复打补丁解决不了的 bug | 需要重新建项目，有一定操作成本 |

### 🛠️ 操作流程

**阶段一：单文件 HTML 原型（零基础起点）**

1. 打开 Claude，在对话框输入：「我需要创建一个[具体需求]，请帮我用美观且响应式的HTML代码来实现」
2. 在 Artifacts 实时预览框中查看效果，用自然语言持续提需求优化
3. 满意后复制代码，保存为 `.html` 文件
4. 双击或拖入浏览器地址栏，验证运行
5. 可直接将 HTML 文件发给他人，双击即用

**阶段二：进入 AI Coding 工具（需求复杂化后）**

1. 选择工具：Cursor（推荐）/ Trae / Windsurf
2. 建议先了解基础前端知识（HTML/CSS/JS 结构），避免项目变大后完全没有头绪
3. 用 Agent 模式 + 以下提示词对齐需求，不急着写代码：
   > 「我需要创建一个[需求]，请仔细思考，详细描述一下项目实现步骤、所需外部依赖、可能的实现难点和项目风险点，设计清晰的层级结构管理项目目录和文件，不着急写代码。」
4. 确认规划无误后，要求模型开始写代码，同时要求「按模块化良好地组织代码，对每个模块都尽可能添加详细注释」
5. 项目结构稳定后切换到 Ask 模式进行迭代

**阶段三：Debug 流程**

1. 直接描述问题让 Coding 工具修改，模型基于上下文尝试修复
2. 若修复失败：让模型对照代码解释相关段落的实现逻辑，自己寻找逻辑漏洞（不要低估自己，有时人能发现模型没看出来的漏洞）
3. 打开 F12 → Console，复制报错信息粘贴给模型，精准定位问题
4. 若仍无法解决：新建项目，复制文件，换 DeepSeek V3 重新理解全局，尝试解决

### 📝 避坑指南

- ⚠️ 单文件 HTML 超过约 2000 行后必须拆模块，否则 Cursor 等工具无法正常应用代码，强行堆代码只会越来越难维护
- ⚠️ 不主动要求图标库，模型默认用 emoji，视觉效果差距明显，提示词里明确写「使用 Font Awesome」
- ⚠️ 功能点越多越具体的需求，bug 越多，模型打补丁容易拆东墙补西墙，要有心理预期
- ⚠️ 不要上来就让模型写代码，先用「对齐需求」提示词让模型输出规划，再开始写，能减少大量返工
- ⚠️ 模型反复修不好的 bug，不要在同一个对话里死磕，换 DeepSeek V3 新建项目重新看全局，有时能破局
- ⚠️ AI Coding 工具约覆盖八成工作量，剩下两成必须靠人看懂逻辑，不要期待全自动，要有「左脚踩右脚」的心理准备
