### 📋 核心分析

**战略价值**: 一个零代码文科生用1周完成"AI聊天回复助手"微信小程序从选题→调研→开发→上线→变现的完整可复刻路径，含所有实战prompt。

**核心逻辑**:

- **选题验证不靠直觉靠AI调研**：用GPT/Deep Research问"全球已验证的高付费AI细分场景"，得到AI陪聊、AI恋爱助手、AI面试辅导等10个方向，再做二次判断——把"AI恋爱助手"扩展为"聊天回复助手"，理由是场景更高频（职场+社交+家庭）、微信是天然入口、切换成本低。
- **竞品调研用Deep Research一键生成报告**：发现国外Replika/RIZZ月活百万级，国内同类产品问题是功能堆砌、直接套壳GPT、没有场景定制System Prompt、界面复杂。差异化方向：极简聚焦只做一件事 + 深度定制中文场景Prompt + 输出3种风格供选择。
- **技术栈选型原则是"对小白最友好"**：Cursor（AI辅助编程，对话式写代码）+ 微信开发者工具 + 微信云开发（免去服务器部署，自带数据库和云函数）+ Node.js LTS版。
- **开发前必须先做3份文档，不能直接写代码**：①PRD需求文档（Claude生成，20+页，含数据库表结构、页面跳转逻辑、功能优先级）存为`PRD.md`；②`.cursorrules`文件（Claude生成，含命名规范、错误处理原则、微信小程序特殊要求）放项目根目录；③`step_by_step_guide.md`开发流程指南（含每步命令和常见问题）。跳过这步会导致代码质量参差不齐、后期改bug改到崩溃。
- **架构规划用Claude Opus的Plan模式，先规划后动手**：规划内容包括前端页面结构、云函数设计（输入输出）、数据库表设计、第三方API接入方案。修改架构成本远低于修改代码，这步不能省。
- **AI模型选型是最大的坑**：智谱/Deepseek便宜但回复"AI味"太重，同样输入"在吗？"，智谱输出"你好，我在的，请问有什么可以帮助你的吗？"，幻宙输出"在呀，咋啦？"。最终选用幻宙（针对中文情感需求专门训练的模型），API地址：`https://phapi.furina.junmatec.cn/`，模型名：`TIG-3.6-Mirage`。
- **修bug的高效方法是批量处理**：不要一个bug修完再修下一个，把当前所有问题列成清单一次性发给Cursor处理，效率翻倍。看不懂代码也能修bug，直接把报错日志复制给Cursor即可。
- **上线审核有两个关键坑**：①类目必须申请"深度合成-AI问答"而非"效率-工具"，否则被拒；②申请该类目需要提供大模型服务商购买订单，解决方案是去微信服务市场买MiniMax一年免费额度（免费也有订单），截图提交即可通过，微信审核不检查你实际用的是什么模型。
- **个体工商户注册小程序有隐藏福利**：小程序认证费300元，个体工商户优惠价30元（1折）。接入支付必须是公司或个体工商户主体。
- **备案是上线最后一步且容易被忽略**：不备案用户搜索不到小程序，审核需1-3天，要提前准备。

### 🎯 关键洞察

**System Prompt是产品核心竞争力，不是一次写好就完事**

产品"伴聊"的System Prompt设计逻辑分4层：
1. 识别场景类型（职场 / 情感 / 社交）
2. 分层解读：表面文字 + 潜台词 + 真实意图
3. 根据场景制定策略框架
4. 输出3种风格回复：稳健得体（不确定对方态度时）/ 进取破局（想推进关系时）/ 高冷自我（想保持距离时）

这个Prompt经过多轮真实聊天记录测试迭代，不是一版定稿。

**模型选型的判断标准**：不是看价格，是看"像不像人话"。用同一个场景输入测试多个模型，选输出最自然的那个。

### 🛠️ 操作流程

**1. 选题与验证阶段**

用以下prompt发给GPT或Google Deep Research：

```
在AI多模态大模型成熟的今天,全球市场有哪些已经验证过的、
有高付费和高增长的细分领域应用?
对中国市场来说,哪些新机会是个体可以通过AI辅助编程实现的?
请给我10个具体的方向,并说明每个方向的验证依据。
```

然后用Deep Research做竞品调研：

```
请帮我在全球范围搜索目前是否有类似"AI聊天回复助手"的APP/平台,
包括中国APP市场,并做详细的竞品拆解。
重点分析:
1. 这些产品的核心功能和交互流程
2. 用户评价中的优缺点
3. 商业模式(免费/订阅/按次付费)
4. 技术实现方式
然后评估这个想法是否能成立,成立的条件是什么?
最后根据调研结论帮我写出具体的需求文档和技术路径。
```

**2. 准备阶段（开发前必做，不可跳过）**

Step 1 - 生成PRD，发给Claude：

```
基于我之前的调研报告(附上Deep Research的内容),
帮我写一份完整的小程序需求文档,包括:
1. 产品定位和目标用户
2. 核心功能列表和优先级
3. 每个功能的详细交互流程
4. 页面结构和跳转逻辑
5. 数据库表结构设计
6. 技术实现方案
请用Markdown格式输出,结构清晰,便于后续开发参考。
```
→ 保存为`PRD.md`

Step 2 - 生成Cursor Rules，发给Claude：

```
我要用Cursor开发微信小程序,请帮我生成一份Cursor Rules文件。
要求包括:
1. 代码风格规范(命名、注释、缩进)
2. 小程序开发的最佳实践
3. 错误处理原则
4. 代码审查检查项
5. 针对微信小程序的特殊要求(如云函数调用、页面生命周期等)
请直接输出可以复制粘贴使用的.cursorrules文件内容。
```
→ 保存为`.cursorrules`（注意有点），放项目根目录

Step 3 - 生成开发流程指南，发给Claude：

```
基于需求文档和技术栈,给我一份开发流程指南。
从创建项目到最终上线,每一步要做什么,用什么命令,
可能遇到什么问题,如何解决。
要足够详细,让一个从没写过代码的人也能跟着做。
```
→ 保存为`step_by_step_guide.md`

**3. 架构规划阶段**

用Claude Opus，开启Plan模式，发送：

```
基于我的需求文档,请规划出完整的技术架构:
1. 前端页面结构(哪些页面、每个页面的组件)
2. 后端云函数设计(哪些函数、每个函数的输入输出)
3. 数据库表设计(哪些表、字段、关系)
4. 第三方服务接入(AI模型API、支付API)
5. 关键功能的实现思路
请给出详细的架构图和说明,确保整体设计合理后再开发。
```

确认架构无误后再开始写代码。

**4. Logo和Slogan生成**

Logo prompt（发给Gemini Gem或Claude）：

```
产品名:伴聊
定位:AI聊天回复助手,帮用户生成高情商回复
目标用户:职场人士、社交场景不知道怎么回话的人
风格要求:简洁、现代、温暖
请生成3个Logo设计方案,用HTML+CSS直接输出可视化效果。
```

Slogan prompt：

```
产品:伴聊(AI聊天回复助手)
核心价值:当你不知道怎么回消息时,AI给你高情商的回复建议
请生成5个Slogan方案,要求简洁有记忆点,最好能体现场景。
```

**5. System Prompt设计**

发给Claude：

```
我要设计一个AI聊天助手的System Prompt,核心功能是:
用户输入对方发来的消息,AI分析后给出3种风格的回复建议。

要求:
1. 先识别场景类型(职场、情感、社交)
2. 分析对方的表面意图和潜在需求
3. 根据场景制定回复策略
4. 生成3种风格的回复:
   - 稳健得体(适合不确定对方态度时)
   - 进取破局(适合想推进关系时)
   - 高冷自我(适合想保持距离时)
5. 每种回复要符合中文表达习惯,不能太死板

请直接输出完整的System Prompt,可以复制粘贴使用。
```

生成后用真实聊天记录多轮测试，不满意就继续迭代，不要一版定稿。

**6. 开发与修bug阶段**

批量修bug模板（发给Cursor）：

```
目前发现以下问题,请一次性帮我修复:
1. 点击"生成回复"按钮没反应
2. 数据库查询超时
3. 云函数返回数据格式不对
4. 页面加载时有闪烁
5. 支付回调没有触发

请逐一分析原因并给出修复方案。
```

测试幻宙API（发给Gemini）：

```
我想测试幻宙的模型。https://phapi.furina.junmatec.cn/
现在我生成了他们的TIG-3.6-Mirage模型的一个API。
我在哪里可以快速地测试这个模型的对话的回复结果。
```

**7. 上线审核阶段**

- 微信公众平台后台 → 类目管理 → 追加"深度合成-AI问答"类目
- 去微信服务市场购买MiniMax免费额度，截图订单作为资质证明提交
- 开通微信支付商户号 → 配置支付参数 → 小程序后台绑定商户号 → 测试支付流程
- 完成微信认证后提交工信部备案（审核1-3天，提前准备）

### 📝 避坑指南

- ⚠️ 不要跳过`.cursorrules`文件，直接让AI写代码会导致代码风格混乱，后期维护和改bug成本极高
- ⚠️ Node.js必须选LTS版，不要选最新版，兼容性问题会让你踩坑
- ⚠️ 类目申请必须选"深度合成-AI问答"，选"效率-工具"会被拒审
- ⚠️ 深度合成类目资质不需要用你实际接入的模型，买MiniMax免费额度拿订单截图即可，微信不核查实际模型
- ⚠️ 备案不做，小程序上线后用户搜索不到，审核1-3天要提前提交
- ⚠️ 便宜模型（智谱/Deepseek）在情感对话场景AI味重，即使有定制System Prompt也难以掩盖，选模型要用真实场景测试而非只看价格
- ⚠️ System Prompt不是一次写好就完事，要用真实聊天记录多轮测试迭代，作者改了好几版才满意
- ⚠️ 修bug不要一个个修，列清单批量发给Cursor一次性处理，效率差距明显
