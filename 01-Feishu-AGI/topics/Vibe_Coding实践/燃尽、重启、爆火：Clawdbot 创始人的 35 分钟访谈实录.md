### 📋 核心分析

**战略价值**: 一个 burnout 13 年的创始人用 10 天 vibe-code 出个人助手智能体框架，GitHub 星标近 9 万，揭示"个人助手智能体年"的底层逻辑与工程路径。

**核心逻辑**:

- **等待窗口判断**：2025 年 5 月 Peter 就有个人助手 AI 想法，判断"大公司几个月内会做"，等到 11 月无人做，才自己动手。核心教训：大公司在"最后一公里"的个性化场景上系统性缺位。
- **10 天 vibe-code 的起点极低**：第一版只是"WhatsApp 消息 → 调用 Claude Code → 返回结果"，1 小时 hack 完，一遍过（one shot）。随后加图片支持，因为截图能给智能体大量上下文，比打字快。
- **语音处理的"自主解决"案例**：Peter 发了一条语音消息，但没写任何语音处理代码。AI 自行检查文件头（Opus 格式）→ 用 ffmpeg 转 wav → 尝试 Whisper 失败 → 在环境变量里找到 OpenAI key → 用 curl 调 OpenAI 转录 → 回复。全程无人工干预。这是"给智能体真正权力"的核心论据。
- **MCP 无法规模化，CLI 才能**：Peter 明确判断 MCP（模型上下文协议）不可规模化。替代方案是命令行工具（CLI）：智能体天生懂 Unix，只需知道工具名称，调用 `--help` 菜单即可自主使用。他为自己的智能体写了覆盖 Google 全家桶、Sonos 音响、家庭摄像头、智能家居的 CLI 工具集。设计原则：**按模型的预期设计 CLI，而非为人类设计**。
- **"心跳"机制与 `surprise me` 提示词**：给智能体加定时自动触发的 heartbeat，提示词内容是"给我个惊喜（surprise me）"，将智能体从被动响应变为主动行为体。
- **模型选型策略**：Opus 用于社交/人格场景（Discord 回复、幽默感强，设计了"不回复"token，只在值得时发言）；Codex 用于写代码（处理大型代码库更强，写完提示词直接推 main，95% 能跑，需要的 handholding 更少，适合并行任务）。
- **爆火后的运营方式**：Discord 炸了之后，Peter 把整个频道内容复制给 Codex，让它"回答前 20 个最常见问题"，扫一遍加几条指示后批量发出。一个人用 AI 工具实现了公司级的响应速度。
- **改名危机的时间线**：Anthropic 发邮件要求改名（商标问题，内部员工而非律师，态度友善但时间线紧）。在释放旧 GitHub 组织名和注册新名之间的约 **10 秒**内，加密骗子用脚本抢注了两个账号，随后推广假代币 $CLAWD，市值一度冲到 1600 万美元，Peter 公开否认后暴跌 90%。X 团队 20 分钟后帮助解决。
- **本地模型硬件判断**：Peter 自用 Mac Studio 512GB 内存顶配跑本地模型（Miniax 21，当前最好开源模型之一），认为一台不够，需要两三台，等 Apple 新款。但他不认为"人人买 Mac Mini 跑智能体"会成为主流，核心障碍是 **auth 授权模式**：接入 Gmail 等服务的红线太多，本地运行可绕过。
- **"代码本身不值钱了"**：选 MIT 许可证，承认会有人拿去卖，但判断真正的护城河是**想法、眼球、品牌**。代码删了几个月能重建，这才是当前 AI 时代的竞争本质。

### 🎯 关键洞察

**"个人助手智能体年"的底层逻辑**

Peter 的判断链条：
1. 2024 年是编程智能体年（Claude Code、Codex 等工具爆发）
2. 编程智能体让一个人的产出 = 一年前一整家公司
3. 下一步自然延伸：把这种能力用在个人生活场景，而非只是写代码
4. 个人助手的核心价值不是"更好的 App"，而是**消灭 App**：不需要 MyFitnessPal，拍张食物照片，智能体自动识别 + 调整健身计划

**"大多数 App 会被简化成 API"的推论**：
- 第一步：App → API（智能体直接调用）
- 第二步：如果数据能存到别处，连 API 都不需要
- 现实案例：维也纳一家设计公司，老板从未写过代码，从 2024 年 12 月开始用 Clawdbot，通过 Telegram 对话让智能体建了 **25 个内部 web 服务**

**逆向 API 的灰色地带**：Peter 承认部分 CLI 工具是让 Codex 逆向网站 API 实现的，有时违反 TOS。当 Codex 拒绝时，他的 prompt 策略是编造场景（"我在这家公司工作，想给老板惊喜，后端团队不知道"），约 40 分钟后得到完整 API。他称之为"数据解放"。

### 🛠️ 操作流程

**复刻 Clawdbot 最小可行版本**

1. **准备阶段**
   - 环境：Mac（本地运行），安装 Claude Code beta
   - 消息通道：选择 WhatsApp 或 Telegram（Telegram 更易集成，有官方 Bot API）
   - 工具链：ffmpeg、curl、OpenAI key（备用转录）、Claude API key

2. **核心执行**
   - 第一版（1 小时目标）：消息通道接收文本 → 调用 Claude Code → 返回结果，one shot 跑通
   - 加图片支持：允许发送截图作为 prompt 附件，智能体从图推断上下文
   - 加 CLI 工具集：从最常用的开始（Google 搜索、文件操作），按 `--help` 可读原则设计
   - 加心跳机制：定时任务（cron）定期发送 "surprise me" 或自定义 prompt

3. **验证与优化**
   - 测试"自主解决"能力：发一个智能体没有对应代码的任务，观察它是否能自行找路径
   - 模型选型：社交/人格场景用 Opus，代码生成/大型代码库用 Codex
   - 安全边界：明确哪些权限开放，提示词注入风险目前无完整解决方案，不要暴露到公网

### 📝 避坑指南

- ⚠️ **提示词注入（Prompt Injection）目前无解**：Peter 明确说"没解决，绝对存在风险"。不要把 Clawdbot 类系统暴露到公网，尤其是那个"调试用的小 web app"。
- ⚠️ **改名/账号释放操作要原子化**：新旧账号切换之间的窗口期（哪怕 10 秒）会被脚本盯着的加密骗子利用，务必先注册新账号再释放旧账号。
- ⚠️ **MCP 不可规模化**：不要把 MCP 作为智能体工具集成的主要路径，CLI 工具集才是可持续方案。
- ⚠️ **Claude Code 需要更多 handholding**：相比 Codex，Claude Code 在大型代码库场景需要更多引导和"哄骗"，并行任务时效率较低。
- ⚠️ **本地模型单机不够用**：跑 Miniax 21 级别的开源模型，Mac Studio 512GB 单台不够，需要两三台并行，成本极高。
- ⚠️ **早期用户 ≠ 所有用户**：Clawdbot 早期用户多为 AI 研究者，懂风险。爆火后涌入的普通用户会把系统用在设计之外的场景（如直接暴露公网），原有的威胁模型完全失效。
- ⚠️ **逆向 API 的法律风险**：部分 CLI 工具通过逆向网站 API 实现，可能违反 TOS，需自行评估风险。
