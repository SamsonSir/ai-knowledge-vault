### 📋 核心分析

**战略价值**: 编程 Agent 让代码生成成本趋近于零，EPD 三个角色的分工逻辑被彻底重构——瓶颈从「实现」转向「评审」，通才和系统思维者的杠杆率急剧放大。

**核心逻辑**:

- **PRD 流程已死，但需求文档未死**：传统 PRD → 设计稿 → 代码的接力棒模式终结。新流程是：想法 → 原型（Agent 生成）→ 文字意图说明 → EPD 评审。文档的作用从「启动开发」变为「传达意图、辅助评审」。未来 PRD 可能演变为结构化、带版本管理的 prompt。
- **瓶颈转移**：过去瓶颈在「写代码」，现在瓶颈在「评审代码是否够好」。原型生成成本极低，待评审的项目数量急剧膨胀，EPD 三个职能的时间都被评审吞噬。
- **通才杠杆率最高**：能同时覆盖产品、工程、设计的人，过去受限于「实现瓶颈」仍需依赖他人。现在只需和 Agent 沟通，一个人的影响力超过三人团队，因为省掉了全部跨部门沟通成本。
- **使用编程 Agent 是必选项，不是加分项**：上手门槛低，不用的人会被用的人替代。PM 可直接做原型验证想法（不再等工程），设计师可直接在代码里迭代（不再只在 Figma），工程师可把时间从实现转向系统架构思考。
- **好 PM 更好，差 PM 更差**：好的产品思维 → 原型直接有用。差的产品思维 → 带着一个无用原型出现，消耗 EPD 三方评审时间，且「东西都做出来了」的惯性会推动烂功能上线，产品因此变得臃肿。
- **系统思维是核心差异化能力**：执行成本趋零后，「想清楚再动手」的价值倍增。三个方向的系统思维具体指：工程 = 服务架构/API/数据库的清晰心智模型；产品 = 用户真实需要（而非嘴上说的）的清晰心智模型；设计 = 为什么某个东西用起来感觉对的清晰心智模型。
- **产品意识是所有角色的基本要求**：Agent 需要有人告诉它做什么。没有产品意识 → 给别人制造更多待评审的垃圾 → 拖累整个组织。有产品意识 → 评审时只需简要说明即可理解意图，加速沟通和交付。
- **专业化门槛更高，但空间仍在**：专业角色（系统架构工程师、深度产品 PM、UX 设计师）仍有价值，但要求同时具备：本领域顶尖系统思维 + 极快的评审速度 + 极强的沟通能力。这类角色在任何公司都不会多。
- **EPD 正在形成两类人**：建设者（产品思维 + Agent 使用能力 + 基本设计直觉，能把小功能从想法做到上线）和评审者（本领域顶尖系统思维者，评审速度极快）。两类都需要产品意识，区别在于深度 vs 广度的侧重。
- **出身背景不再决定上限**：最有价值的人是「站在文化与深度技术交叉点的双语者」——既知道技术上什么可能，又能分辨哪些文化潮流是真实的。这种人可以来自产品、设计、工程任何方向，但真正的全才极少。

### 🎯 关键洞察

**为什么「好 PM 更好、差 PM 更差」是这篇文章最值得警惕的判断**：

过去，差的产品想法会被「实现成本高」这道天然屏障过滤掉——工程师不会花两周时间去做一个明显没想清楚的功能。现在这道屏障消失了。差的想法可以在几小时内变成原型，然后带着「东西都做出来了」的惯性进入评审流程，消耗 EPD 三方时间，甚至直接合并上线。

逻辑链：实现成本趋零 → 原型数量爆炸 → 评审成为新瓶颈 → 差的产品思维直接转化为组织层面的评审负担 → 产品臃肿、质量下降。

**为什么「PRD 万岁」和「PRD 已死」并不矛盾**：

死掉的是「PRD 作为开发起点」的功能——它不再是触发设计和工程的信号。活着的是「PRD 作为意图载体」的功能——评审者需要知道某个实现是有意为之还是偶然写成的，这个信息只能通过文字传达。形式可以变（比如直接分享生成功能所用的 prompt），但「意图文档」本身不可或缺。

### 📦 角色转型路径对照表

| 当前角色 | 建设者路径 | 评审者路径 | 必备基础能力 |
|---------|-----------|-----------|------------|
| 工程师 | 提升产品意识 + 设计直觉，能独立交付小功能 | 系统架构/API/数据库设计练到极致，能快速评审架构合理性 | 熟练使用编程 Agent |
| 产品经理 | 投入编程 Agent，能直接做原型验证想法 | 建立顶级产品心智模型，主导功能评审 | 产品意识（本职）|
| 设计师 | 投入编程 Agent，直接在代码里迭代而非只在 Figma | 建立顶级用户旅程/交互心智模型，主导设计评审 | 产品意识 + Agent 使用能力 |
| 通才（EPD 均有感觉）| 直接单打独斗，影响力超过三人团队 | — | 和 Agent 沟通即可，无需跨部门协调 |

### 🛠️ 操作流程（新 EPD 工作范式）

1. **想法阶段**: 任何人（PM/设计/工程）产生想法 → 直接用编程 Agent 生成原型，不再等 PRD 审批流程
2. **意图文档**: 原型完成后，配套写一份简要意图说明（替代传统 PRD 的「传达意图」功能），说明哪些是有意为之的设计决策
3. **评审阶段**: EPD 三方围绕原型评审，分别检查：架构合理性（工程）、是否解决真实用户痛点（产品）、是否易用直观（设计）
4. **迭代**: 评审意见反馈给 Agent，快速迭代，直到「足够好」
5. **上线判断**: 警惕「东西都做出来了就合并」的惯性，评审者需主动把关，防止臃肿功能上线

- Vercel：「设计工程师」（Design Engineer）角色已成主流，兼具设计和工程能力。

### 📝 避坑指南

- ⚠️ 不要把「能用 Agent 生成原型」等同于「产品想清楚了」——原型的生成成本趋零，但烂原型消耗的评审成本是真实的，且会随团队规模线性放大。
- ⚠️ 不要因为「东西都做出来了」就推动合并——这是编程 Agent 时代最危险的惯性，会直接导致产品臃肿。
- ⚠️ 专业化不是退路——只懂本领域、不具备产品意识、不会用 Agent 的专业角色，在任何公司的需求都会萎缩。专业化必须叠加在「通用基础能力」之上才有价值。
- ⚠️ 意图文档不能省——评审者无法区分「有意为之」和「偶然写成」，没有意图文档的原型会大幅增加评审成本和误判概率。
- ⚠️ 系统思维无法被 Agent 替代——Agent 降低的是执行成本，不是判断成本。「做什么」和「为什么这样做」仍然是人的责任。
