### 📋 核心分析

**战略价值**: AI 工程效率暴增正在从外部强制终结传统设计流程，设计师必须从"流程守门人"转型为"代码级执行者+方向引导者"，否则将被边缘化。

**核心逻辑**:

- **双钻石模型已死，凶手是工程速度**：当工程师可以同时开 7 个 Claude 实例并行造功能时，设计师再用"调研→发散→收敛→发散→收敛"的老流程根本跟不上节奏。Jenny 在 2025 年 9 月柏林 Hatch Conference 做了"Don't Trust the Design Process"演讲，但仅仅三四个月后（Opus 4.6 发布、大量人在假期发现 Claude Code 之后），她自己就觉得那场演讲内容已经过时了。

- **设计工作分裂为两类，形态都变了**：第一类是"支持执行"——工程师高速出活，任何人都可以提想法让 AI 做粗糙版本，设计师是顾问而非先画稿再交付；第二类是"做愿景方向"——但时间窗口从 2-5 年压缩到 3-6 个月，形式从精美演示文稿变成能指方向的原型。

- **设计师时间分配发生结构性重组**：
  - 几年前：60-70% 做设计稿/原型，20% 配合工程师，10% 开协调会
  - 现在：30-40% 做设计稿/原型，30-40% 直接与工程师配合，新增一块——自己写代码实现
  - 核心变化：画稿时间腰斩，多出来的时间用于配对编程和直接改代码

- **AI 产品必须快速迭代的根本原因是非确定性（Non-deterministic）**：AI 模型同样输入可能产生不同输出，你无法在设计稿里模拟所有状态，也做不出有意义的可点击原型。必须用真实模型、看真实用户怎么用，才能发现真正的使用场景。这是 AI 产品快速发布的技术层面理由，不只是文化偏好。

- **Figma 仍有价值，但原因变了**：代码工具太线性——用 Claude Code 做一个方向就会一直在那个方向深入。好设计需要先想 8-10 种不同做法，把一堆想法甩到墙上再筛选。这种发散式探索 Figma 画布仍做得最好。另一个价值是精细视觉微调：不同排版、字体、样式方向并排比较，比在代码里反复切换高效得多。

- **构建软件最难的部分不是构建本身**：工作中最难的时刻往往不是技术实现，而是你和另一个人在争论"这个功能该不该做""该做成什么样"。AI 可以提供参考意见，但人与人之间的决策分歧 AI 无法替你解决。决策和责任仍然落在人身上——就像 Claude 帮工程师写代码，但工程师仍要为"这段代码对不对、放在产品里合不合适"负责。

- **Co-work 的"10 天"是误导性数字**：10 天是从内部版本到外部发布的冲刺时间。在此之前，团队在不同 Agent 框架上做过大量原型：待办列表怎么展示、多选问题用什么形式、怎么教用户理解使用场景，都试过很多方案。内部原型"Claude Studio"（界面极度密集复杂）里的核心概念——Skills 框架（用 Markdown 文件指导 Claude 完成特定任务）和展示 Claude 计划/待办的 UI——最终被提取进了 Co-work 设计。

- **品牌信任靠快速响应而非完美发布**：真正损害品牌的是"发布了早期版本后什么都不做"。Anthropic 每次发布新版本后，团队成员在 Twitter 上回复用户反馈、快速修复问题、公开展示进展。建立信任的不只是速度，还有让用户感到"我的反馈被听到了、被用上了"。

- **"可读性矩阵"作为信号捕捉工具**：来自 SPC 合伙人 Evan Tana 的框架，两轴为"创始人是否可读"×"想法是否可读"。最有价值的往往是"想法不可读"的象限——别人看不懂但有能量汇聚的方向。Jenny 在 Anthropic Slack 里浏览内部原型时就是在找这类信号，Co-work 的核心概念正是从一个"第一眼看不懂"的内部原型里挖出来的。

- **Claude 目前不够格被当设计师雇佣，但差距在缩小**：Claude 做初稿和展示不同方案还行，但不符合 Jenny 提到的三种设计师原型中的任何一种，没有"这个很特别、值得雇佣"的东西。但过去一年进步了很多，没人知道差距缩小到什么程度会触发行业又一次变化。

### 🎯 关键洞察

**界面范式的演化方向**：聊天不会消失，但未来是"可点击图形界面 + 对话"的结合体。Claude 发布的小组件（天气、股票、多选题等）用户反响好，因为人们仍然喜欢看到 UI、点击它、和它互动，这比打字高效。趋势是：越来越多的 UI 由模型动态生成，而不是工程师逐个手写。Kevin Weil（OpenAI CPO）的观点支撑了聊天的持久性：语言是跨越所有智能水平的界面，随着模型越来越聪明，对话仍然有效。

**Jenny 的 AI 工具栈实际使用方式**：
- Claude Chat → 已基本被 Claude Co-work 取代（使用场景多为长时间运行任务）
- Claude Code → 主要在 VS Code 里用，做前端打磨时需要同时看代码和与 Claude 对话
- 一个具体工作流：有人在 Slack 里说"这个图标偏了"，@ 一下 Claude，Claude 自动改好代码并提交，Jenny 直接合并完成

**从总监回到 IC 的反直觉价值**：Jenny 在 Figma 管过 12-15 人团队加几个设计经理，去 Anthropic 选择做 IC。原因之一是对中层管理未来的真实焦虑——AI 改变工作方式的背景下，管理角色是否会持续存在？结论：设计流程在过去一年变化太快，如果一直做纯管理，根本没时间习得新硬技能。她建议设计管理者做类似工程管理者的"实操轮岗"——先花几个月做 IC 理解技术变化，再回去管团队。
