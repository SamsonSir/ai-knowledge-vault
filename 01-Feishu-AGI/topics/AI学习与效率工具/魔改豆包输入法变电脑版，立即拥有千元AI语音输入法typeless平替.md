### 📋 核心分析

**战略价值**: 用 Python 脚本封装豆包 PC 客户端的语音输入能力，实现「按住说话、松开写入、剪贴板无损」的 Typeless 级体验，且完全免费。

**核心逻辑**:

- **痛点1 — 操作断层**: 豆包原生语音输入需要 `Ctrl+D` → 说话 → 手动按 `Enter` 三步，每次都打断心流；Typeless 的核心价值就是消除这个断层。
- **痛点2 — 剪贴板污染**: 豆包识别完成后会把结果写入剪贴板再 `Cmd+V` 粘贴，导致你之前复制的内容（代码、链接等）被覆盖，无法找回。
- **解决方案本质**: 用 Python 监听「右侧 Command 键」，按下时自动触发 `Ctrl+D`，松开时自动按 `Enter`，同时在触发前用 `pbpaste` 保存剪贴板，插入后 0.2 秒内用 `pbcopy` 恢复。
- **冷启动 vs 热启动延迟差异**: 豆包第一次启动语音识别需初始化 AI 模型，耗时约 **3.5 秒**；后续识别模型已在内存中，耗时约 **1.5 秒**。工具通过 Web 界面滑块让用户自定义这两个延迟值，避免「按 Enter 时识别还没完成」导致插入空白。
- **打包方案选型**: 放弃 Onefile（每次启动需 10 秒解压）和 Electron（携带几百 MB 浏览器内核），选用 **PyInstaller Onedir** 模式（直接加载已解压库，实现秒开）+ **AppDelegate 架构**（在系统 `DidFinishLaunching` 回调时挂载 RunLoop，获取辅助功能权限）。
- **权限依赖**: 全局键盘监听基于 `pynput` + macOS `CGEventTap`，必须授予「辅助功能」权限，否则无法监听全局按键。
- **Web GUI 控制面板**: 工具启动后自动在 `https://localhost:8899` 开启控制界面，提供启动/停止、实时日志、延迟参数配置，无需命令行操作。
- **剪贴板恢复时序**: `Enter` 插入 → 豆包立即执行自动复制 → 脚本等待 **0.2 秒** → 强制用 `pbcopy` 恢复原内容。0.2 秒是关键时间窗口：足够让插入完成，又短到用户无感知。
- **触发键选择右侧 Command**: 左侧 Command 是系统快捷键高频键（`Cmd+C/V/Z`），右侧 Command 日常几乎闲置，冲突概率极低。
- **快捷键固定为 `Ctrl+D`**: 为避免自定义快捷键与其他应用冲突，豆包语音输入快捷键统一设定为 `Control+D`，脚本硬编码触发此组合键。

### 🎯 关键洞察

**为什么 0.2 秒是核心参数**:
豆包的工作流是：识别完成 → 写入剪贴板 → 执行 `Cmd+V` 粘贴。这个写入动作发生在 `Enter` 之后的极短时间内。如果脚本在 `Enter` 后立刻恢复剪贴板，豆包的写入会覆盖恢复的内容；如果等太久，用户会察觉剪贴板短暂变成了语音文字。0.2 秒是实测出的最小安全窗口。

**为什么不自己写语音识别**:
豆包已免费提供语音识别能力，自建需要对接 ASR API（有延迟、有成本、需维护）。本工具的定位是「胶水层」——只做豆包没做好的那部分（自动化按键 + 剪贴板保护），不重复造轮子。

### 🛠️ 操作流程

1. **准备阶段 — 配置豆包**
   - 打开豆包 PC 客户端（非网页版，是桌面 app）
   - 进入「设置」→「快捷键」
   - 将「语音输入」快捷键设为 `Control + D`
   - 建议关闭「语音悬浮球」（设置内可找到）
   - 保存设置

2. **核心执行 — 安装并启动工具**
   - 下载 `阿星AI输入-v2.0-macOS-final.zip`（约 20 MB）并解压，得到 `阿星AI输入.app`
   - 首次打开：右键点击 → 「打开」（绕过 macOS Gatekeeper 安全限制，不能直接双击）
   - 授权辅助功能：系统设置 → 隐私与安全性 → 辅助功能 → 点击 ➕ → 添加 `阿星AI输入.app` → 确保勾选已启用
   - 等待浏览器自动打开 `https://localhost:8899`（首次 5-10 秒）
   - 在 Web 界面点击「🚀 启动语音输入」，日志显示「✓ 语音输入已启动！」即就绪

3. **验证与优化**
   - 在任意输入框（浏览器、编辑器、微信等）按住「右侧 Command 键」说话，松开后文字自动插入
   - 验证剪贴板保护：先复制一段文字 → 用语音输入一句话 → 再粘贴，确认粘贴出来的仍是原来复制的内容
   - 如果插入结果不完整或为空白，进入 `https://localhost:8899` 配置页面，通过滑块适当增大冷启动延迟（从 3.5 秒往上调）
   - 如果热启动插入偶尔不完整，适当增大热启动延迟（从 1.5 秒往上调）

### 💡 具体案例/数据

**剪贴板污染场景复现**:
1. 复制一段代码
2. 用豆包原生语音输入一句话
3. 豆包把识别结果写入剪贴板
4. 去粘贴 → 粘贴出来的是语音文字，代码丢失

**工具修复后的时序**:
```
按下右侧 Command
  → pbpaste 保存剪贴板（瞬间）
  → 触发 Ctrl+D 唤醒豆包语音
  → 说话（用户操作）
松开右侧 Command
  → 自动按 Enter 插入文字
  → 等待 0.2 秒
  → pbcopy 恢复原剪贴板
```

**延迟参数实测值**:
| 场景 | 等待时间 |
|------|---------|
| 冷启动（首次，模型初始化）| 约 3.5 秒 |
| 热启动（后续，模型已在内存）| 约 1.5 秒 |
| 剪贴板恢复等待窗口 | 0.2 秒 |
| App 首次启动时间 | 5-10 秒 |
| App 后续启动时间 | 3-5 秒 |

### 📝 避坑指南

- ⚠️ **不能直接双击打开 .app**：macOS Gatekeeper 会拦截未签名 app，必须右键 → 「打开」才能绕过。
- ⚠️ **辅助功能权限是硬性前提**：不授权则 `pynput` + `CGEventTap` 无法捕获全局按键，工具完全失效，且不会有任何报错提示给用户。
- ⚠️ **豆包快捷键必须固定为 `Ctrl+D`**：脚本硬编码触发此组合键，如果豆包里设的是其他键，触发会失败。
- ⚠️ **冷启动延迟调小会导致插入空白**：豆包模型初始化需要约 3.5 秒，如果脚本在 3.5 秒前就按 Enter，识别还没完成，插入结果为空或不完整。
- ⚠️ **使用的是豆包 PC 客户端，不是网页版**：网页版没有全局语音输入快捷键，工具无法驱动。客户端入口在豆包网页版左下角可以下载。
- ⚠️ **右侧 Command 键专用**：左侧 Command 是 `Cmd+C/V/Z` 等高频快捷键的触发键，脚本只监听右侧，避免与系统快捷键冲突。
