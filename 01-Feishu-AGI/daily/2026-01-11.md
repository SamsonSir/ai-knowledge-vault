### 📋 核心分析

**战略价值**: 用 Vibe Coding（Enter.pro + Supabase）零代码完成一款具备前端申请、后台管理、数据库存储、数据埋点、移动适配的全栈 Web 产品的完整交付流程。

**核心逻辑**:

- **先想后做，流程图是最低成本的验证工具**：在动手前用流程图把学员侧（提交→等待→查看结果）和管理员侧（收单→审核→反馈）两条链路完整模拟一遍，直到每个节点的信息字段都清晰，再开始写 Prompt。
- **MVP 的本质是"最低成本验证假设"**：Granola 早期用人工实时听会议、手写纪要来验证"用户是否需要 AI 纪要"，而不是先造 AI 算法。结论：0-1 阶段不追求完美，只验证核心假设。
- **需求分级 P0/P1 是资源分配的核心决策**：本课案例中"账号收集+工单跟踪"是 P0 必做，"自动充值+品牌透出"是 P1 能做就做，明确优先级避免过度设计。
- **第一版只做纯前端**：先用 Enter.pro 跑通交互流程（无真实数据），验证 UI 路径和字段逻辑没问题，再接数据库。顺序：交互验证 → 数据接入 → 发布 → 迭代。
- **Plan 模式是 Vibe Coding 最重要的技巧之一**：在 Enter.pro 输入框选中 Plan，让 Agent 先输出详细执行计划（将模糊需求具体化 + 拆解实施阶段），确认无误后再执行，避免一把梭出错。
- **Supabase 接入不需要手动建表**：通过 Enter.pro 的 Supabase Connect 插件授权（两步：Organization 授权 → 具体 Project 连接），Agent 自动完成建表、权限配置、数据库访问代码注入。
- **发布 = Publish 一键操作，版本管理是必须有的意识**：Publish 后 30-60s 获得公网域名；每次发布前必须在预览版本测试完整，正式版本出问题可一键回退上一版本。
- **SEO 配置是基础卫生，两步验证**：看浏览器标签页标题是否更新；按 F12 打开 DevTool → Elements，检查 `<meta>` 标签是否有 description/keywords。让 Agent 根据产品实际情况自动补全。
- **数据埋点分两层**：基础层（GA/PostHog 自动采集页面曝光、滚动、通用交互）+ 关键事件层（Custom Event 手动上报，如学员提交申请表单）。原则：宁可多采不可漏采，不要多报重报。
- **移动端适配和品牌升级靠 Prompt 精度**：视觉迭代的瓶颈不是 AI 能力，是人的审美和 Prompt 描述精度。遇到卡点可以借助 Gemini 等工具先生成设计 Prompt，再喂给 Enter.pro 执行。

### 🎯 关键洞察

**产品思维的核心是"真实诉求 vs 表面诉求"**：
- 表面诉求：收集学员邮箱 → 工具：问卷
- 真实诉求：让学员账号持续有钱 → 工具：带状态跟踪的申请系统
- 演进诉求：自动触发充值（余额低于阈值时）→ 未来版本

**Vibe Coding 的天花板不是技术，是想法和执行力**：
- 你不需要会写代码，但你必须能清晰表达"我要什么"
- 0-1 阶段的 Prompt 可以很简洁，产品迭代中 0-1 只占极小比例
- 每一轮迭代都是：明确诉求 → 精准 Prompt → 验证结果 → 下一轮

### 🛠️ 操作流程

**1. 准备阶段：需求梳理与流程设计**

- 识别真实诉求（不是表面诉求），枚举所有解决方案，选出 P0 优先实现
- 画出完整用户流程图（学员侧 + 管理员侧），把每个节点的字段、状态、出口都想清楚
- 整理成需求文档表格，包含：页面模块、所属系统（前端/后台）、需求详情

本课案例的字段设计：

| 字段 | 类型 | 规则 |
|------|------|------|
| 邮箱地址 | 输入框 | 必填，校验邮箱格式 |
| 名称 | 输入框 | 必填，最多20字符，支持中英文 |
| 所学课程 | 下拉框 | 必填，选项从后端配置读取 |
| 工单状态 | 枚举 | 待处理 / 通过 / 驳回（含驳回理由） |

**2. 核心执行：三轮迭代**

- **v1（纯前端验证）**：用简洁 Prompt 让 Enter.pro 生成纯前端版本，验证交互路径是否符合预期
- **v2（交互精修）**：用完整详细的 Prompt（含用户路径、产品需求、字段规则、分页规则）重新生成，验证交互细节
- **v3（接入数据库）**：
  1. Enter.pro 插件 → Supabase Connect → 完成两步授权
  2. 输入框选 Plan，发送"好，我们需要开始接入数据库，不过在你开始编码前，我需要更加详细的执行计划"
  3. 确认计划后执行，Agent 自动建表、配权限、注入代码
  4. 人肉测试四步：学员提交→数据入库 ✅ / 管理员登录→库有新用户 ✅ / 管理员审核→数据同步 ✅ / 学员状态刷新→完整流程 ✅

**3. 发布与上线**

- 右上角 Publish → 等待 30-60s → 获得公网链接
- 左侧时间按钮 → 找 Online 版本 → 加书签（版本管理）
- 用真实手机重新跑一遍完整流程

**4. 验证与优化（SEO + 埋点 + 移动端 + 品牌）**

- SEO：让 Agent 完成配置，F12 验证 `<meta>` 标签和标签页标题
- GA4：
  1. GA 平台 → 数据流设置 → 选网站 → 填正式链接 → 创建
  2. 复制 gtag 代码，通过 Enter.pro 对话接入
  3. GA 平台测试安装 → 显示"正确检测到 Google 代码"
  4. 让 Agent 在提交表单处插入 Custom Event 上报
- PostHog：
  1. 安装 wizard，配置环境变量 `VITE_PUBLIC_POSTHOG_KEY` 和 `VITE_PUBLIC_POSTHOG_HOST`
  2. 初始化 `PostHogProvider` 包裹根组件
  3. 关键事件上报 `posthog.capture()`
  4. PostHog 后台配置授权域名
- 移动端：Prompt 描述移动端场景 → Agent 适配 → 真机验证
- 品牌升级：提供品牌截图 + 设计要求给 Agent；遇到审美卡点，先用 Gemini 生成设计 Prompt 再喂给 Enter.pro

### 📝 避坑指南

- ⚠️ **Supabase 授权必须完成两步**：Organization 授权 + 具体 Project 连接，只做第一步 Agent 无法操作数据库
- ⚠️ **发布前必须在预览版本完整测试**：Publish 后用户立即可访问，未测试的版本不要发布
- ⚠️ **版本必须手动加书签标记**：最新版本 ≠ 正式版本，出问题靠版本回退救场
- ⚠️ **GA 数据 48 小时后才完整**：接入后不要立刻看报表，实时报告可以即时验证是否接入成功
- ⚠️ **PostHog 接入后必须配置授权域名**：否则数据采集会被拦截，在 PostHog 后台填入正式版本链接
- ⚠️ **Course 选项不要写死在前端代码里**：应从后端配置读取，方便后续增减课程不用改代码重新发布
- ⚠️ **移动端适配必须用真机验证**：模拟器不能完全还原真实手机体验，尤其是触摸交互和字体渲染
- ⚠️ **视觉迭代的瓶颈是 Prompt 精度**：遇到 AI 审美卡点，先用 Gemini 等工具生成专业设计 Prompt，再喂给 Enter.pro，效果更可控
- ⚠️ **Admin 入口不应暴露在前端页面**：后台管理入口不要放在用户侧页面，安全隐患

**来源**: 五步框架：把 Workflow 变成可进化的 Skill | **时间**: 2026-01-11 | **原文链接**: https://mp.weixin.qq.com/s/R0WNSWPk...

---

### 📋 核心分析

**战略价值**: 用「拆分→编排→存储→分摊→迭代」五步框架，将任何可视化 Workflow 重构为基于本地文件的 Agent + Skills 架构，获得可进化、可移植、可并行的自动化资产。

**核心逻辑**:

- **Workflow 的三大硬伤**：①表达能力有限，复杂逻辑难以用节点表达；②输入变化即崩溃，为 A 类文档设计的流程遇到 B 类文档直接卡死；③平台锁定，导出导入一通操作还得在对方环境重新调试。
- **Skill ≠ 单一技能**：大多数人把 skill 当翻译/总结这类原子操作，实际上 skill 应是「可组合模块」——用自然语言描述多个 skill 之间的协作关系，本质是「用自然语言编排工作流」。
- **第一步·拆分**：每个 skill/subagent 只做一件事。写作流程拆为：`article-analyzer`（分析素材→输出 analysis.md）、`outliner`（生成 2-3 个提纲方案）、`writer-agent`（根据提纲写草稿，可并行多个）、`polish`（润色定稿）。配图流程拆为：`generate-image`（原子技能，调用图像生成 API）、`article-illustrator`（组合技能，分析文章→识别配图位置→生成插图）、`cover-image`（组合技能，生成 2.35:1 封面图）。写作+配图再组合成完整写作工作流。
- **第二步·编排**：在主 skill 里用自然语言描述整个流程。示例——outliner 的编排描述："先调用 article-analyzer 分析素材，分析完成后保存 analysis.md，然后根据分析结果生成 2-3 个不同风格的提纲方案，为每个方案并行启动 writer-agent 写草稿。" article-illustrator 的编排描述："读取文章内容，识别需要配图的位置（概念抽象处、信息密集处、情感转折处），为每个位置生成图像描述，调用 generate-image 生成图片，最后将图片插入文章对应位置。" 条件分支、并行执行、错误处理均可用自然语言描述，Agent 能理解。
- **第三步·存储**：所有中间结果保存为本地文件。文件链路：`source.md → analysis.md → outline-a.md → draft-outline-a.md → final.md`。三大好处：①可追溯（每步输出有迹可循）；②可断点续传（中途停了下次从上次位置继续）；③可人工干预（手动修改某步结果后让 Agent 继续）。
- **第四步·分摊**：Subagent 之间只传文件路径，不传内容。直接传大段内容会撑满上下文窗口；只传路径则 subagent 自己读文件，上下文保持干净。writer-agent 启动只需三个参数：source 文件路径、analysis 文件路径、outline 文件路径，写完保存到指定路径，返回输出文件路径。此设计支持并行：三个 writer-agent 同时跑，各自处理一个提纲方案，互不干扰。
- **第五步·迭代**：Skill 是文

**战略价值**: YC Winter26 批次结束后，四位合伙人系统复盘 AI 经济格局——模型层竞争格局已发生实质性换位，应用层正进入可复用构建路径的稳定期，创业者的机会窗口在"部署阶段"而非"安装阶段"。

**核心逻辑**:

- **Anthropic 首次超越 OpenAI 成为 YC 创始人首选 API**：Winter26 批次中 Anthropic 占比超过 52%，而 OpenAI 在 2024 年大部分时间和 2025 年初仅徘徊在 20-25%。换位发生在过去 3-6 个月内，核心驱动力是 vibe coding / 编码 agent 类别的爆发，而 Anthropic 模型在该类别表现最优，且这是其内部刻意设定的北极星指标（来源：Tom Brown 内部演讲）。
- **Gemini 快速崛起**：从个位数百分比（约 2-3%）增长至 Winter26 的约 23%。Harj 个人已将 Gemini 作为首选模型，理由是推理能力强 + Groundings API 实现实时信息准确性，在"今天发生了什么"类查询上比 Perplexity 更准确（Perplexity 快但不总准确）。
- **模型选择正在走向编排层抽象**：B 轮级别的 AI 公司已不再忠于单一模型，而是构建编排层，按任务类型动态切换模型。具体案例：某初创公司用 Gemini 3 做上下文工程，输出结果再送入 OpenAI 执行；每次新模型发布都重新评估替换。这套机制的底层是公司自有的垂直领域评估集（非通用 benchmark），因为他们在强监管行业拥有独有数据集。
- **AI 经济已进入稳定期**：Jared 明确指出，2025 年最大的惊喜是 AI 经济"稳定下来了"——模型层、应用层、基础设施层三层分化清晰，每层都有人赚钱，且"如何在模型之上构建 AI 原生公司"已有相对成熟的方案，不再像 2024 年底那样"脚下地基不断移动"。
- **AI 泡沫类比：电信时代而非价值终点**：Garry 和 Diana 援引 Carlota Perez 的技术革命两阶段理论——①安装阶段（重资本支出，感觉像泡沫）→ ②部署阶段（真正产生影响，应用爆发）。当前正处于从安装向部署的转型期。类比：90 年代电信过度建设的暗光纤和管道，最终催生了 YouTube、Google、Facebook。大学生/初创公司不是康卡斯特，他们是 YouTube——基础设施过剩对他们是利好，不是风险。
- **模型商品化趋势利好应用层**：当 Gemini 搜索体验让人感觉"模型在相互商品化"，应用层和初创公司将迎来更好的年份。Claude Code 的发布曾让人担心模型公司要吃掉应用层，但整体趋势仍是模型层竞争激烈 → 算力更便宜 → 应用层受益。
- **垂直领域小模型可以击败大模型**：某 YC 初创公司收集了最优质的医疗保健数据集，通过开源基础模型 + RL 微调，仅用 80 亿参数在医疗保健多项 benchmark 上超越 OpenAI。但风险是：GPT-4.5/5.1 发布后会将微调成果"冲刷殆尽"，必须持续追赶前沿。
- **AI 提升了效率，但并未减少招聘**：去年"创始人独自或只带一人做到百万 ARR"的案例，A 轮后并未演变成"不加人做到千万 ARR"。原因：AI 降低了生产成本和时间，但同时拉高了客户期望，公司需要雇更多人来满足更高标准。瓶颈仍是"能出色执行的人"，而非想法。今年更偏向"AI 提升期望 → 需要更多人"阵营，而非"AI 替代人力"阵营。
- **第一波 AI 公司（如 Harvey）的护城河是资本封锁，而非技术壁垒**：Harvey 早期策略是把沙山路所有能开 1000 万到 1 亿美元支票的约 30 个投资人全部锁定，使后续轮次无人可进，以资本作为护城河。但代价是在无差异化优势的微调上浪费了大量资金，最终受益的是持有更多股份的投资者。第二波公司（Legora、Giga）正在快速追赶。
- **基础设施瓶颈已从算力转向土地和能源**：数据中心建设的核心约束已不是 GPU，而是土地（美国监管严格，加州 CEQA 被滥用）和电力（喷气发动机发电供应链积压，需提前 2-3 年订购）。太空数据中心（StarCloud 2024 年夏被嘲笑，18 个月后谷歌和 Elon 都在做）、太空聚变能源（Zephyr Fusion，国家实验室工程师创立，40 多岁，已完成种子轮）正在成为真实解法。

### 🛠️ 操作流程：构建 AI 原生垂直公司的可复用路径

1. **选模型策略**：不要绑定单一模型。构建编排抽象层，按任务类型（上下文工程 vs 执行 vs 推理）分配不同模型，并建立自有评估集（基于你的垂直领域数据，而非通用 benchmark）。

2. **垂直领域模型微调**：选择开源基础模型 + RL 微调路径，收集垂直领域最优数据集（医疗保健案例：80 亿参数击败 OpenAI）。但必须建立持续追赶前沿的机制，因为大模型新版本会周期性冲刷微调优势。

3. **资本策略（参考 Harvey 案例）**：早期可考虑锁定主要投资人以建立资本护城河，但避免将资金浪费在无差异化优势的微调上。第二波竞争者（Legora vs Harvey，Giga vs Sierra）证明先发优势并非不可逾越。

4. **团队规模预期校准**：不要预设"AI 会大幅减少招聘需求"。当前阶段，AI 提升了客户期望，实际招聘需求并未减少，瓶颈仍是能出色执行的人。A 轮后按正常节奏组建团队。

5. **时机判断**：当前处于安装→部署转型期，应用层是最佳入场时机。不需要参与数据中心、GPU、能源基础设施的竞争，那是大公司的游戏。

### 📝 避坑指南

- ⚠️ **微调陷阱**：在无差异化优势的场景上花大量资金微调，大模型新版本发布后成果归零，最终受益的是持有更多股份的投资者，而非创始人。只在你拥有独有数据集的垂直领域做微调。
- ⚠️ **单模型绑定风险**：绑定单一 API 供应商会在竞争对手发布更优模型时丧失灵活性。B 轮级公司已普遍构建编排层，早期就应设计可替换的模型接口。
- ⚠️ **"AI 省人"预期偏差**：不要基于"AI 会大幅减少人力需求"来规划团队和融资。当前阶段，AI 拉高了客户期望，实际人力需求并未下降，按此预期规划会导致执行力不足。
- ⚠️ **把基础设施泡沫误读为 AI 价值终点**：GPU 过剩、资本密集投入是安装阶段的正常现象，不代表 AI 没有价值。对应用层创始人而言，这是利好而非风险信号。
- ⚠️ **评估集依赖通用 benchmark**：垂直领域 AI 产品必须建立自有评估集。通用 benchmark 无法反映你的实际业务场景，尤其在强监管行业（医疗、法律等）。

## 📔 文章 5

> 文档 ID: `ZNcwwlgxQizrtbkydqTc67x7nFc`

**来源**: AI模型那么多，你是靠什么做选择呢？ | **时间**: 2026-01-11 | **原文链接**: `https://mp.weixin.qq.com/s/wFISf3pZ...`

### 📋 核心分析

**战略价值**: 用 Anthropic 工程实践为蓝本，给 AI 产品/Agent 开发者建立一套可落地的「评估体系」，彻底替代靠感觉选模型的盲飞状态。

**核心逻辑**:

- **评估的本质**：给 AI 一个输入，看输出，判断好不好——但判断的对象必须是「结果（Outcome）」而非「对话记录（Transcript）」。AI 可能幻觉，数据库不会撒谎。
- **三类角色的共同病根**：普通用户说不出好在哪、产品经理换模型后投诉反增、技术负责人面对 10 次 6 成功 4 失败束手无策——根源都是「没有把'什么是好'具体化」。
- **评估是基础设施，不是锦上添花**：没有评估 = 盲飞；有评估的人半小时出换模型结论，没有的人靠感觉猜一周；早期投入成本可见，收益后期复利爆发。
- **三种判断方式按场景选用**：① 有明确对错 → 程序化规则（快/便宜/客观）；② 主观程度问题 → AI 裁判 + 明确评分标准（需定期人工校准）；③ 边缘案例/校准 → 人工兜底。三者组合，不要单押。
- **AI 裁判必须配校准机制**：AI 裁判会打分飘忽、有偏见、产生幻觉（声称看到记录里没有的内容）。必须定期抽样，人工打分与 AI 打分对比，差距过大就调整评分标准。
- **两种评估目的截然不同**：「能力评估（爬坡）」回答「我们能做到什么」，初始通过率低才有区分度；「回归评估（守住）」回答「原来能做的还能做吗」，通过率应接近 100%，骤降即报警。
- **「毕业」机制让评估体系持续进化**：某功能在能力评估里稳定达到 90%+ 后，将其移入回归评估守住成果，同时在能力评估里补充更难任务，持续爬新的山。
- **pass@k vs pass^k 描述两个完全不同的故事**：同一个 60% 单次成功率的 AI，pass@10 ≈ 99.99%（乐观），pass^10 ≈ 0.6%（严格）。代码生成工具（给候选项）用 pass@k；订机票助手（期望一次成功）用 pass^k。选错指标，汇报数据会严重误导决策。
- **三个设计原则防止评估本身出 bug**：① 评判结果不评判路径（AI 走了更聪明的捷径不该被判失败）；② 正反例都要测（只测「该做的」会让 AI 学会「遇事不决就搜索」）；③ 评估环境要干净（每次从零开始，不留上次痕迹——Anthropic 曾发现 Claude 翻 git 历史「开卷考试」导致异常高分）。

### 🎯 关键洞察

**对话记录 vs 结果 的陷阱**

用户说「帮我订明天去上海的机票」，AI 回「已为您预订成功」。
- 只看 AI 回复 → 检查的是「AI 说了什么」
- 真正要查的 → 数据库里有没有这条预订记录

AI 幻觉会让它「说谎」，但环境状态不会。这是评估设计的第一原则。

**小李的教训（换模型后投诉反增）**

只做了能力评估（新模型回答准不准），没做回归评估（原来能处理的退款流程现在还能处理吗）。新模型确实更聪明，但改变了某些措辞，导致下游流程出错。用户感知 = 变差了。

**老张的教训（10 次 6 成 4 败）**

用 pass^k 分析：单次 60% 成功率，连续 3 次都成功的概率只有 22%。这说明 AI 能力够（pass@k 高），但稳定性不行（pass^k 低）。优化方向是提高稳定性，而不是继续拉高能力上限。

## 📓 文章 6

> 文档 ID: `XyCdwsea4iOwzRk8MMycbdk4nDb`

**来源**: 英伟达GDPO论文解读：多奖励强化学习的正确打开方式 | **时间**: 2026-01-11 | **原文链接**: https://mp.weixin.qq.com/s/ilMI3gJV...

---

### 📋 核心分析

**战略价值**: NVIDIA提出GDPO算法，通过"先分别归一化、再加总"替代GRPO的"先加总、再归一化"，从根本上修复多奖励强化学习中的信号崩塌问题，在工具调用、数学推理、代码推理三类任务上实现一致性提升，且无需额外模型或算力。

**核心逻辑**:

- **GRPO的致命缺陷**：GRPO在多奖励场景下先将所有奖励加总，再做组内归一化。导致(0,1)和(0,2)两种完全不同的回答组合，归一化后得到完全相同的优势值(-0.7071, 0.7071)，高质量信号被抹平，称为"奖励信号崩塌（Reward Signal Collapse）"。

- **崩塌的实际后果**：在Qwen2.5-1.5B工具调用训练中，GRPO的正确性奖励在400步后开始下降，训练部分崩溃；数学推理任务中，GRPO训练的模型400步后正确率下滑，同时回答反而变长，模型失去优化方向。

- **GDPO核心改进——解耦归一化**：
  - GRPO做法：`总分 = 奖励1 + 奖励2` → `优势值 = 归一化(总分)`
  - GDPO做法：`优势值1 = 归一化(奖励1)` / `优势值2 = 归一化(奖励2)` → `总优势值 = 优势值1 + 优势值2`
  - 效果：(0,1) → (-0.7071, 0.7071)；(0,2) → (-1.4142, 1.4142)，信息颗粒度被完整保留。

- **信息保留量的量化差距**：随rollout数量增加，GDPO可区分的"不同优势值组合数"远超GRPO：

  | rollout数量 | GRPO不同组合数 | GDPO不同组合数 |
  |------------|--------------|--------------|
  | 4          | 约10          | 约25          |
  | 8          | 约25          | 约100         |
  | 16         | 约50          | 约175         |

- **批次归一化（锦上添花）**：奖励数量增加时，总优势值范围膨胀（3个奖励的范围是2个的1.5倍），导致训练不稳定。GDPO在加总后额外做一次批次级别归一化（batch-wise normalization），实验证明去掉此步骤后部分训练会失败。

- **奖励权重调整几乎无效（重要发现）**：当两个目标难度差异悬殊时（如"长度控制"易满足，"正确性"难满足），即使将长度奖励权重从1.0降到0.25，模型仍优先满足长度约束——因为"性价比最高"。根本解法是条件奖励。

- **条件奖励（Conditioned Reward）**：只有主要目标达成时，才计算次要目标的奖励（如"只有答对了，才计算长度奖励"）。这迫使模型先追求正确性，权重调整才能真正生效。

- **开源即用**：GDPO已开源三个主流框架实现：HF-TRL、verl、Nemo-RL，无需额外模型或计算资源，仅改变归一化顺序。

- **AI对齐意义**：多维度目标（准确、安全、有帮助、不冗长）若被压缩成单一评分，AI学到的是模糊的"好"。GDPO保留各维度区分度，使AI能清楚区分"有帮助但有风险"与"安全但不够用"，是朝更精细、可控AI对齐迈出的一步。

---

### 🎯 关键洞察

**为什么"先分后合"优于"先合后分"**：

归一化本质是在组内做相对排名，抹掉绝对数值差异。一旦先加总，两个奖励的差异信息就永久混合，无法还原。就像把两张试卷的分数加总后再排名，你永远无法知道某人是"语文强数学弱"还是"两科均衡"。GDPO在各奖励维度上分别保留相对排名信息，再汇总，等价于"分科排名再综合"，信息损失最小。

**条件奖励的逻辑**：难度不对称的多目标优化，本质是一个"捷径问题"——模型会找最容易得分的路径。条件奖励通过设置门槛，强制关闭捷径，让模型必须先攻克硬目标。

---

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| GDPO优势计算 | `A_i = (r_i - mean_i) / std_i`，`A_sum = ΣA_i`，`A_final = (A_sum - batch_mean) / batch_std` | 保留各奖励维度信息颗粒度 | 必须做最终的batch-wise归一化，否则奖励数量增加时训练不稳定 |
| GRPO优势计算（对比） | `A_sum = (r_1+r_2+...+r_n - mean) / std` | 信号崩塌，不同组合得到相同优势值 | 多奖励场景下不推荐 |
| 条件奖励设置 | 只有正确性奖励满足时，才激活长度/格式奖励 | 强制模型优先优化主目标 | 不使用条件奖励时，调整权重对模型行为影响极小 |
| 开源框架 | HF-TRL / verl / Nemo-RL | 拿来即用 | 三个框架均已支持，无需额外改动 |

### 🛠️ 操作流程

1. **准备阶段**: 确定任务的多个奖励函数（如格式奖励+正确性奖励），判断各奖励难度是否对称。若难度差异大（如长度易、正确性难），设计条件奖励：仅当主奖励达标时才激活次奖励。

2. **核心执行**: 替换GRPO的优势计算逻辑为GDPO：对每个奖励 $r_i$ 分别在组内做归一化得到 $A_i$，加总得 $A_{sum}$，再对整个batch做一次归一化得 $A_{final}$。从HF-TRL / verl / Nemo-RL中选择对应框架的开源实现直接接入。

3. **验证与优化**: 监控各奖励的收敛曲线（不应出现400步后下降）；检查超长比例和Bug比例是否同步下降；若仍存在目标冲突，调整条件奖励的触发阈值，而非单纯调整权重。

### 📝 避坑指南

- ⚠️ **不能省略batch-wise归一化**：GDPO加总各奖励优势值后，必须再做一次批次级别归一化。奖励数量从2增到3，总优势值范围扩大1.5倍，不做此步骤会导致部分训练直接失败。
- ⚠️ **调权重≠解决目标冲突**：当两个奖励难度差异大时，把次要奖励权重从1.0降到0.25，模型行为几乎不变。必须用条件奖励从机制上解决，而非靠权重调参。
- ⚠️ **条件奖励不能单独使用**：实验数据显示，GRPO+条件奖励虽然准确率提升（53.3%），但超长比例从2.1%飙升至29.2%，说明条件奖励单独使用会引入新的不稳定性，必须配合GDPO使用。
- ⚠️ **GRPO的400步陷阱**：使用GRPO训练多奖励任务时，前期指标可能看起来正常，但400步后会出现奖励下降或回答变长等退化现象，需要提前监控各奖励曲线而非只看总分。
