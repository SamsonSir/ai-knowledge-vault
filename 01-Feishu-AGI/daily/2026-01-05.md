### 📋 核心分析

**战略价值**: Claude Code 创始人 Boris Cherny 亲授 13 个实战技巧，配合 Gemini API 主管 Jaana Dogan 的真实案例，揭示 AI 编程的正确打开方式——不是替代人类探索，而是极速复现已验证的方案。

**核心逻辑**:

- **"一年 vs 一小时"的本质**：Jaana 团队花一年踩坑、验证最佳方案；把结论写进 Prompt 后，Claude Code 1 小时复现玩具原型。AI 压缩的是"执行时间"，不是"探索时间"——创新和试错仍是人的核心价值。
- **并行 5 个本地进程**：Boris 在终端同时跑 5 个 Claude 进程，标签页编号 1-5，用系统通知（iTerm2）监控进度，彻底告别串行等待。
- **本地 + Web 端无缝切换**：本地跑 5 个实例的同时，claude.ai/code 再开 5-10 个；用 `&` 符号或 `--teleport` 命令在终端和 Web 端之间切换；早晨和白天从 Claude iOS App 启动会话，晚些再回来查看结果。
- **坚持用 Opus 4.5 + Thinking 模式**：Sonnet 更快，但 Boris 选 Opus 4.5——原因是工具调用能力更强、干预更少，不需要反复调教，综合效率反而更高。
- **全队共建 CLAUDE.md**：整个团队共用一个 CLAUDE.md 文件，Claude 每次执行前都会读取；谁发现 Claude 犯错就往里记一笔，积累团队代码风格和避坑指南；Code Review 时直接 `@.claude`，让它把修改规则同步进 CLAUDE.md，通过 `/install-github-action` 安装 Claude Code GitHub Action 实现自动化。
- **Plan Mode 是灵魂**：大部分会话从 Plan 模式启动（`Shift+Tab` 按两次），先让 Claude 列出执行计划，对齐后再切自动模式——计划对了基本 One-shot 搞定。
- **斜杠命令消灭重复动作**：把高频操作编成 `/` 命令，存放在 `.claude/commands/` 目录；例如每天使用数十次的 `/commit-push-pr`，命令内嵌脚本让 Claude 自动查 git 状态，避免来回交互。
- **子代理按职能分工**：不让一个 AI 包揽所有任务；`code-simplifier` 专门简化代码，`verify-app` 专门负责测试——按流程职能分工，不是按领域专家分工。
- **PostToolUse 钩子自动格式化**：每次 Claude 写完代码，钩子自动触发格式化，保证代码进测试环境前的最后 10% 质量。
- **权限预设用 `/permissions`**：不用 `--dangerously-skip-permissions` 暴力跳过；而是用 `/permissions` 提前给常用安全的 bash 命令授权，配置提交到 `.claude/settings.json` 全队共享，不再手动点确认。
- **MCP 协议跨平台打工**：通过 MCP 协议让 Claude Code 发 Slack 消息、用 `bq` CLI 跑 BigQuery 查询、从 Sentry 拉错误日志；配置提交到 `.mcp.json` 全队共享。
- **长任务挂机三件套**：(a) 任务完成后让 Claude 通过后台代理自验结果；(b) 用 `agent Stop` 钩子做确定性验证；(c) 用 `ralph-wiggum` 插件自动循环直到完成。沙盒中配合 `--permission-mode=dontAsk` 或 `--dangerously-skip-permissions` 让 Claude 无干扰运行。插件地址：`https://github.com/anthropics/claude-plugins-official/tree/main/plugins/ralph-wiggum`
- **验证闭环是最终杀招**：给 Claude 一个稳定的反馈回路（跑测试、开浏览器测 UI），产出质量可提升 2-3 倍。Boris 的做法是用 Chrome 扩展测试每次改动，Claude 自动打开浏览器、测试界面、持续迭代直到功能正常且 UX 良好。

### 🎯 关键洞察

**"一年 vs 一小时"的正确解读**：

Jaana 后来澄清，这不是 AI 有多牛，而是：
1. 人类花了一年把"最佳方案"探索出来
2. 把结论写进 Prompt，AI 自然能快速复现一个质量不错的原型
3. Claude Code 给的是"玩具版"，不是生产级代码

核心结论：**AI 压缩的是"已知解的执行成本"，人类的核心价值在于"未知解的探索能力"**。

Jaana 的建议：如果你对 Code Agent 持怀疑态度，去你最熟悉的领域，从头构建一个足够复杂的东西，由你自己判断产出是否靠谱。

**Boris 的数据背书**：过去一个月为 Claude Code 贡献 497 次提交、新增约 4 万行代码，全程没打开过一次 IDE，全靠 Claude Code + Opus 4.5 完成。

### 🛠️ 操作流程

1. **准备阶段**:
   - 建立 `CLAUDE.md`，写入团队代码风格和已知坑点
   - 在 `.claude/commands/` 下定义高频斜杠命令（如 `/commit-push-pr`）
   - 用 `/permissions` 预授权常用 bash 命令，提交 `.claude/settings.json`
   - 配置 `.mcp.json`，接入 Slack/BigQuery/Sentry 等工具

2. **核心执行**:
   - 本地开 5 个终端标签（编号 1-5），配置 iTerm2 系统通知
   - 同时在 `claude.ai/code` 开 5-10 个 Web 实例并行运行
   - 每个会话从 Plan Mode 启动（`Shift+Tab` 两次），对齐计划后切自动模式
   - 长任务启用 `ralph-wiggum` 插件 + `--permission-mode=dontAsk`，挂机等结果
   - 复杂任务拆分子代理：`code-simplifier` 简化代码，`verify-app` 负责测试

3. **验证与优化**:
   - 配置 PostToolUse 钩子，写完代码自动格式化

## 📙 文章 4

> 文档 ID: `VnXVwBTaBiun0KkyA4Uco4aYnrf`

**来源**: 如何用 AI + OpenSpec 驱动团队迭代开发 | **时间**: 2026-01-03 | **原文链接**: `https://mp.weixin.qq.com/s/CJPCsYoQ...`

---

### 📋 核心分析

**战略价值**: 通过 OpenSpec 规范构建 AI 可理解的结构化知识体系，将隐性项目知识显性化，使 AI 从"开盲盒"的代码生成器升级为"熟悉业务的开发伙伴"。

**核心逻辑**:

- **AI 在大型项目中降智的根因**：不是模型能力不足，而是缺乏结构化的、AI 可索引的知识管理体系。AI 无法理解历史代码的"为什么"，每次对话都从零开始，导致 Review AI 代码的时间比自己重写还长。
- **OpenSpec 的核心理念**：让 AI 明确知道"知识在哪、如何用、以及为什么这样做"，而非一股脑塞入所有文档。
- **初始化命令只需两步**：`cd /path/to/your-project` → `openspec init`，无需额外编写工具代码。
- **五层知识骨架结构**：`AGENTS.md`（AI 工作规范）、`project.md`（项目上下文+文档索引）、`specs/`（已实现能力规范）、`changes/`（待处理变更提案）、`docs/`（详细文档，需手动创建，非 OpenSpec 原生目录）。
- **双层知识结构设计**：索引层（`AGENTS.md` + `project.md`）只提供"地图"，不写长篇大论；明细层（`openspec/docs/`）存放架构设计、需求文档、业务逻辑等真正干货。
- **AI 工作闭环路径**：接到任务 → 读 `AGENTS.md`（获取开发规范）→ 读 `project.md`（获取业务背景+索引）→ 定位 `docs/` 下具体文档 → 深度理解上下文后开始工作。
- **初始 `project.md` 无需手写**：通过 OpenSpec 提案发起对话，引导 AI 自己梳理项目架构和业务，自动生成初始内容。后续更新 `docs/` 后，需引导 AI 重新总结生成新的 `project.md`，建议定义《文档管理指南》作为准则确保一致性。
- **Change-Driven 变更驱动协作流程**（三步闭环）：
  1. 新需求 → 通过 OpenSpec 创建提案文件 `changes/proposal-xxx.md`，人工审查 Why/What/Impact 三项是否清晰对齐
  2. AI 读取已通过提案 + `specs/` 中已有规范 → 生成/修改代码及测试用例 → 人工 Code Review
  3. 运行 `openspec archive` → AI 自动将本次变更涉及的新知识、新规范更新到 `specs/` 和 `docs/`，完成知识入库
- **微服务多子服务的 OpenSpec 策略**：以知识重合度为判断标准——重合度 <30% 则各子服务独立初始化一套 OpenSpec；重合度 >70% 则共用一套，保证知识一致性和 AI 理解效率。

---

### 🎯 关键洞察

**为什么 AI 在复杂项目中会"降智"**：
- 原因：AI 的上下文窗口有限，面对庞大微服务项目时，无法自动识别哪些历史代码、文档是当前任务的关键上下文。
- 动作：通过 OpenSpec 的索引层，让 AI 每次工作前都能精准定位到相关知识，而非盲目扫描全量代码。
- 结果：AI 从"每次对话开盲盒"变为"有历史经验可追溯的开发伙伴"。

**为什么要用 Change-Driven 流程而非直接让 AI 写代码**：
- 原因：直接让 AI 写代码，知识不会沉淀，下次迭代 AI 仍是"新手"。
- 动作：强制所有变更先走提案（`changes/`），完成后走归档（`openspec archive`），知识强制入库。
- 结果：每次迭代都在为 AI 的"长期记忆"加砖，项目越迭代 AI 越熟练，而非每次重头来过。

### 🛠️ 操作流程

1. **准备阶段（知识骨架搭建）**
   - 进入项目根目录：`cd /path/to/your-project`
   - 执行初始化：`openspec init`
   - 手动创建 `openspec/docs/` 目录
   - 发起 OpenSpec 提案，通过对话引导 AI 梳理现有项目架构和业务，自动生成初始 `project.md`
   - 手动编写 `AGENTS.md`，定义团队开发规范（命名约定、错误码设计、测试策略等）
   - 定义《文档管理指南》存入 `docs/`，作为后续 AI 更新文档的行为准则

2. **核心执行（Change-Driven 迭代）**
   - 新需求到来 → 通过 OpenSpec 创建提案：`openspec/changes/proposal-xxx.md`
   - 人工审查提案的 Why（背景）、What（目标）、Impact（影响）三项，确认人与 AI 理解对齐
   - AI 读取已通过提案 + `specs/` 已有规范 → 生成代码和测试用例
   - 人工执行 Code Review

3. **验证与知识归档**
   - Code Review 通过后，执行：`openspec archive`
   - AI 自动将本次变更涉及的新知识、新规范更新到 `specs/` 和 `docs/`
   - 引导 AI 基于更新后的 `docs/` 重新生成 `project.md`，保持索引层与明细层同步

- ⚠️ `docs/` 目录不是 OpenSpec 原生生成的，`openspec init` 后必须手动创建，否则明细层无处存放。
- ⚠️ 每次更新 `docs/` 后，必须引导 AI 重新生成 `project.md`，否则索引层与明细层脱节，AI 找不到新知识。
- ⚠️ 变更提案的人工审查不能省略——Why/What/Impact 三项必须清晰，这是确保人与 AI 理解对齐的唯一检查点。
- ⚠️ `openspec archive` 必须在每次 Code Review 通过后执行，跳过归档等于让本次迭代的知识凭空消失。
- ⚠️ 微服务项目不要无脑共用或无脑拆分 OpenSpec，以知识重合度为判断标准，灵活选择策略。

**来源**: 马斯克都爱用的Grok定时任务，治好了我的信息焦虑 | **时间**: 2026-01-05 | **原文链接**: https://mp.weixin.qq.com/s/YEX9kMgm...

---

### 📋 核心分析

**战略价值**: 用 Grok Tasks 定时任务替代手动刷信息流，通过精准 Prompt（时间范围 + 热度筛选条件）实现每日自动情报聚合，从根本上消除信息焦虑的触发循环。

**核心逻辑**:

- **信息焦虑的真正根源不是信息量，而是「不确定性」**：不知道自己错过了什么，所以不停刷新。定时任务的价值是把这份不确定性折叠掉，让你知道「今天的变化大概长什么样」。
- **普通 Prompt 抓回来的内容质量差的原因**：没有指定时间范围、热度指标、输出格式，Grok 会随机返回，点赞数明显不是最火那批，内容散且无重点。
- **关键修复：在 Prompt 里加入三个硬约束**：① 时间范围（最近 24h）② 热度指标（点赞 ≥ 30，浏览 ≥ 1 万）③ 严格输出格式（排序 + 代码块包裹 + 真实链接）。这三个约束让 Grok 像 SQL 做精准筛选，缺一不可。
- **关键词构建方法论**：先让 Grok 搜索技术名词的中日英常见写法，再用最近一周高赞现象反推社区在讨论什么，再提炼成监控关键词。这样得到的关键词贴着社区真实用法走，不会跑偏。
- **信息维度切换**：把 Prompt 的信息输入维度从「关键词」切换成「一组大佬账号的 X List」，可以从主题监控升级为人物监控，顺便收罗出一批高质量创作者（一箭多雕）。
- **Grok Tasks 定时任务的核心价值**：设定每天早晚各跑一次，结果以浏览器通知或邮件推送，可在结果上继续追问。把手机大部分通知关掉，只依赖早晚两次总结，换来的是连续思考时间而非碎片刷屏。
- **定时日报即笔记本身**：带时间戳，像每天切下来的薄片。想写某个主题时，翻最近七天日报，选题和案例已经在里面了，不再需要收藏夹/备忘录/todo list（那些最终都会变成信息坟场）。
- **可扩展到其他平台**：同样的定时任务思路可以在元宝（不限数量，但信源含公众号/视频号，时间刻度稍慢）和 Manus 上使用。Manus 有定时功能相当于其他 Agent 都有了，例如在 Manus 里登录 Genspark 可每天定时抓特定资源。
- **冷启动问题的解法**：新号关注大号后用关键词搜索方式先跑一段时间，会自动收罗出高质量创作者账号，再把这些账号加入 List，解决冷启动期看不到有用信息的问题。

---

### 🎯 关键洞察

**为什么加筛选条件是核心而不是锦上添花**：

没有筛选条件的 Prompt → Grok 做的是「召回」，返回相关内容但不保证质量。加了时间范围 + 热度阈值 → Grok 做的是「精排」，等价于 `SELECT * FROM posts WHERE keyword MATCH AND likes >= 30 AND views >= 10000 ORDER BY likes DESC LIMIT 20`。这个差异决定了结果是「有用」还是「够用」。

**定时任务的心理机制**：开始会不安，觉得会漏掉很多信息，但实际上并没有。中间确实可能漏掉几个点，但换来的是更长一段消化和创作的时间。这是一个主动选择「已知的小损失」换取「更大的专注收益」的决策。

### 📝 避坑指南

- ⚠️ **筛选条件不能删**：删掉点赞/浏览量限制，Grok 会退化成随机召回，等于回到原点
- ⚠️ **不要让 Grok 编造链接**：Prompt 里必须明确写「所有链接必须是真实 https://x.com 链接，ID 正确，不要编造数据、用户名、链接」，否则 Grok 可能幻觉生成假链接
- ⚠️ **元宝时间刻度慢**：元宝信源包含公众号和视频号，不适合需要实时性的监控场景，适合对时效要求不高的内容聚合
- ⚠️ **新号冷启动问题**：直接用 List 监控，新号关注大号后一两个月内可能看不到有用信息。解法：先用关键词搜索方式跑一段时间，从结果里识别高质量创作者再加入 List
- ⚠️ **方法越复杂越累**：AI 浏览器、RSS+AI 去重、Deep Research、Claude Code Skills 等方案自动化程度越高反而越累。轻量方案（Grok Tasks）的核心优势是零编程门槛、维护成本极低
