### 🎯 关键洞察

**为什么先读GPT系列论文**：GPT-1→GPT-4展示了AI从"会说话"到"能干活"的完整进化路径。GPT-1确立预训练+微调范式；GPT-3（1750亿参数）实现零样本能力；InstructGPT引入RLHF让模型听人话；这条演化链是理解所有现代LLM的基础。

**为什么ARC-AGI是最重要的基准**：它是目前唯一AI还远远落后于人类的主流基准——最强AI正确率~50%，普通人约70%，人类上限目标85%。它测试的是不依赖记忆的纯粹抽象推理和举一反三能力，是最接近"通用智能核心能力"的测试。一旦AI整体接近普通人水平，视为AGI里程碑。

**思维链为何重要性下降**：2022年"让我们一步步思考"曾是最简单有效的提示技巧，加一句话准确率大幅提升。但2025年后，推理模型（o1/R1/QwQ）已将思考过程内置，外部思维链提示的边际收益显著降低。

### 📝 避坑指南

- ⚠️ **基准分数不等于实际能力**: LeCun已公开承认Llama4存在刷榜，高MMLU分数的模型在你的具体任务上不一定好用。必须结合实际场景测试
- ⚠️ **MTEB"已死"但仍是标准**: MTEB作者自己认为MTEB因过拟合严重已"死"，但目前依然是嵌入模型事实上的评测标准，参考时注意这一局限性
- ⚠️ **YOLO家族史复杂**: 原作者离开后社区出现多个分支，使用时需确认来源和版本
- ⚠️ **思维链技巧重要性下降**: 2025年推理模型（o1/R1）已内置思考过程，传统CoT提示的边际效益大幅降低
- ⚠️ **RAG系统的分块是门学问**: 块太大模型处理不了，块太小丢失上下文，HyDE（先生成假设答案再检索）比直接用问题检索效果更好
- ⚠️ **NaturalSpeech3零样本声音克隆需警惕**: 几秒语音样本即可克隆声音，存在被用于诈骗和伪造的风险
- ⚠️ **Gemini超长上下文幻觉问题**: Gemini 3超过200万token上下文后幻觉严重，实际使用需注意
- ⚠️ **《Attention Is All You Need》未收录**: Latent Space清单更偏向AI工程师实战，部分公认经典论文（如Transformer原始论文）未在123篇内，需另行补充

### 📋 核心分析

**战略价值**: Manus 科学家 Peak 系统总结了通用 Agent 在生产环境中 Context Engineering 的五大维度完整实战方案，包含 KV Cache 优化、分层工具空间、多 Agent 架构、错误保留策略等可直接复用的工程决策。

**核心逻辑**:

- **押注 Context Engineering 而非自训模型**：Peak 前司从零训模型，一个训练+评估周期需 1-2 周，迭代速度被拖死，且在未找到 PMF 前就在优化与用户体验无关的基准。Manus 策略：有 PMF 之前不训模型，通用大模型+上下文工程可在数小时内迭代。创业公司要做"船"（随基础模型涨潮水涨船高），不做"柱子"（被水淹没的风险）。
- **输入输出比 100:1，平均 50 次工具调用**：每执行一次工具，上下文就增长一截；理论上模型支持 100 万 token，但实测 200k 左右开始出现渐进式退化（重复、推理变慢、质量下滑）。Manus 通过大量测试确定"预腐烂阈值"，通常在 **128k-200k 之间**，必须在触达此红线前主动干预。
- **文件系统作为终极无限上下文**：不用向量数据库/RAG/知识图谱等有损外部化方案，而是直接用文件系统：无限容量 + 天然持久化 + 命令行直接操作。核心是**可逆性**——URL 在，网页内容可重新 fetch；文件路径在，内容可重新 read；比猜测"10 步后什么信息重要"更务实。
- **紧凑化 = 外部化状态而非压缩数据**：写文件的工具调用执行后，content 字段可安全删除，只保留 path 指针。从模型上下文视角信息消失，但从 Agent 系统视角信息仍可访问。摘要（有损）是最后手段，摘要前先把关键部分卸载到文件，甚至把整个预总结上下文作为日志文件存档留后路；摘要时用结构化 Schema（定义表单字段让 AI 填空，如"修改了哪些文件"、"用户目标"、"执行进度"），而非让 AI 自由发挥；摘要后保留最近几条工具调用/结果完整版本保持连续性。
- **分层动作空间解决工具膨胀问题**：三层架构——核心层 10-20 个原子函数（读写文件、执行 shell 等），边界清晰可组合，直接函数调用；中间层预装在沙盒中的工具（格式转换器、语音识别、mcp-cli 等），通过命令行访问；外层预授权 API（金融数据、3D 设计库等），写 Python 脚本执行（如：获取城市名→获取城市 ID→获取天气，一个脚本搞定）。三层从基础模型视角均通过标准函数调用，接口清晰、缓存友好、函数间正交。
- **KV Cache 命中率是生产 Agent 最重要的单一指标**：以 Claude Sonnet 为例，缓存命中 token 成本 vs 未命中成本差 **10 倍**。优化细节：① 系统提示开头不放时间戳（保持前缀稳定）；② 确保 JSON 键顺序确定性（顺序不稳定会悄无声息破坏缓存）；③ 只追加不修改历史操作记录；④ 如果提供商不支持自动增量前缀缓存，手动插入缓存断点；⑤ 动态添加/移除工具会摧毁 KV Cache（工具定义在上下文前部），改用 **Logit Masking** 在解码过程中遮蔽 token logits，实现自动/必需/指定三种工具调用模式，工具名称设计有意统一前缀（如所有浏览器工具以 `browser_` 开头）以支持按组约束。
- **MCP 通过 CLI 而非函数注入接入**：Manus 开发 `mcp-cli` 工具预装在沙盒中，通过命令行接口调用 MCP，而非注入函数调用空间；输出可直接写文件或分页返回，用 grep、cat 等 Linux 工具实时处理；信任 Agent 可运行 `--help` 自学新工具，不在系统提示里硬编码每个工具用法。保护 KV Cache + 利用 Linux 生态可组合性。
- **反对拟人化 Multi-Agent 架构**：Peak 明确反对按角色划分（设计师 Agent、程序员 Agent、管理者 Agent），认为那是人类公司的运作方式，源于人类的上下文限制。Manus 只保留少量功能性 Agent：通用执行 Agent、规划器 Agent、知识管理 Agent、数据 API 注册 Agent。子 Agent 以 "Agent as a Tool" 实现，有明确输入输出 Schema。两种共享模式：通过通信共享（适合简短清晰任务，子 Agent 上下文只含该指令）vs 通过共享内存通信（适合需要完整历史的任务，如深度研究，但成本更高，无法重用 KV Cache）。
- **保留错误是最有效的行为改善手段**：常见本能是隐藏错误、清理痕迹、重置状态，但擦除失败会移除证据，模型永远无法学会错误恢复。Manus 实践：错误（包含失败动作和堆栈跟踪）保留在上下文中，模型会隐式更新内部"先验"，降低重复犯错概率，比人工设计的纠错机制更有效。错误恢复能力是判断真实 Agent 行为最明显的指标之一，大多数学术基准完全忽略了这一能力。
- **Few-Shot 在 Agent 中可能适得其反**：大语言模型是优秀的模仿者，上下文充满类似的过去动作-观察对时，模型会僵化模仿历史模式。典型案例：审查 20 份简历时 Agent 会陷入僵化节奏，仅因为上下文中看到的模式就重复类似动作。解法：在动作和观察中引入少量结构化变化（不同序列化模板、替代性措辞、顺序或格式上的微小噪音），可控随机性打破模式，迫使模型根据当前状态决策而非照搬历史。
- **六七个月五次重构，最大飞跃来自简化**：Manus 上线后每次大幅改善不是来自增加更多上下文管理层，而是来自移除不必要技巧、更多信任模型。架构质量评估方法：固定架构，切换模型，用"模型能力梯度"评估——若换更强模型性能大幅提升，说明架构未过拟合当前模型局限性，未来可期。设计理念从 "Less Structure, More Intelligence" 到 "Build Less, Understand More"，一脉相承。

### 🎯 关键洞察

**Context Engineering 五维度不是独立优化目标，而是需要动态权衡的系统**：


| 维度 | 手段 | 与其他维度的交互 |
|------|------|----------------|
| 卸载（Offloading） | 数据移到文件系统 | 让缩减更高效（可先卸载再摘要） |
| 缩减（Compaction/Summarization） | 压缩历史（最后手段） | 受卸载支撑；影响缓存效率和输出质量 |
| 检索（Retrieval） | 按需读取文件/URL | 稳定检索让隔离更安全 |
| 隔离（Isolation） | 子 Agent 管理上下文 | 减缓上下文增长，降低缩减频率；但增加成本，影响缓存命中 |
| 缓存（Caching） | KV Cache 命中率优化 | 更多隔离和缩减会影响缓存效率 |

**核心矛盾**：隔离越多，上下文增长越慢，但缓存越难命中；缩减越多，token 越少，但信息损失越大。没有银弹，只有在具体任务场景下的动态平衡。

**"可逆性"是贯穿整个 Context Engineering 的第一原则**：

- 卸载到文件系统 → 可逆（path 在，内容可重取）
- 摘要 → 不可逆（有损），所以是最后手段
- 保留错误 → 不可"逆向清除"（清除即失去证据）
- 保留 todo.md → 任务目标随时可见（注意力可"逆向锚定"到全局计划）

### 🛠️ 操作流程

#### 1. 准备阶段：确定你的"预腐烂阈值"
- 针对你使用的具体模型，设计一批长程任务测试用例
- 从 64k token 开始，逐步增加上下文长度，观察模型是否出现重复、推理变慢、质量下滑
- 记录退化开始出现的 token 数（通常 128k-200k），这就是你的干预红线
- **不要假设模型标称的上下文长度（如 1M token）就是安全边界**

#### 2. 核心执行：搭建五维度 Context 管理体系

**卸载层（优先做）**：
- 所有大块数据（网页内容、文件内容、API 响应）执行后立即写入沙盒文件系统
- 工具调用历史中只保留文件路径，删除 content 字段
- 确保沙盒文件系统在 Agent 生命周期内持久化

**工具空间（架构设计期决定）**：
- 核心函数调用限制在 10-20 个原子操作
- 新能力优先通过沙盒 CLI 工具实现（参考 mcp-cli 模式）
- 需要 Python 脚本的外部 API 统一走脚本层
- 工具命名统一前缀便于 Logit Masking 按组控制

**KV Cache 保护（全程注意）**：
- 系统提示内容固定，不含动态时间戳
- 所有 JSON 序列化显式指定键排序
- 操作历史只追加，严禁修改或删除历史记录
- 确认使用的 LLM 提供商是否支持自动前缀缓存；不支持则手动插入缓存断点

#### 3. 触达红线时：执行紧凑化流程
1. 先将关键中间结果写入文件（可选：将整个当前上下文作为日志文件存档）
2. 用结构化 Schema 进行摘要（固定字段：已修改文件列表/当前用户目标/执行进度/下一步计划）
3. 摘要后**保留**最近 N 条完整工具调用记录（N 根据任务复杂度调整，建议 5-10 条）

#### 4. 验证与优化：评估架构质量
- **黄金测试法**：固定架构，依次接入弱模型→强模型，观察性能提升幅度
  - 提升显著 → 架构健康，模型能力有空间发挥
  - 提升不明显 → 架构可能过度工程，是障碍而非跳板
- 监控 KV Cache 命中率作为生产环境核心指标
- 用户反馈（1-5 星）作为金标准，辅以内部自动化测试（可验证结果类任务）和人工评估（视觉/品味类任务）

**成本数据（Claude Sonnet）**：
- 缓存命中：0.3 美元/百万 token
- 缓存未命中：3 美元/百万 token
- 差距：**10 倍**
- Agent 输入输出比 100:1，KV Cache 命中率对成本影响极其显著

**Manus 产品数据**：
- 平均每个任务：50 次工具调用
- 平均输入输出 token 比：100:1
- 上线后 6-7 个月内：经历 5 次重构
- 每次最大性能飞跃来源：简化和移除，而非添加

**学术基准 vs 用户反馈错位案例**：
- Manus 早期使用 Gaia 等公开学术基准选模型
- 发现 Gaia 高分模型用户不喜欢
- 原因：大多数基准偏推理任务，完全忽略执行类任务和错误恢复能力
- 结论：创建专注执行任务（而非推理题）的内部数据集

**Few-Shot 僵化具体案例**：
- 任务：审查 20 份简历
- 现象：Agent 在处理前几份后陷入僵化节奏，后续简历处理方式高度雷同
- 原因：上下文中充满类似动作-观察对，模型优先模仿历史模式
- 修复：引入不同序列化模板、替代性措辞、顺序微小噪音

### 📝 避坑指南

- ⚠️ **不要用模型标称上下文长度规划 Agent**：200k 左右实际退化，1M token 的上下文窗口不等于 1M token 的有效工作范围
- ⚠️ **JSON 序列化键顺序**：不显式指定 sort_keys 会导致 KV Cache 被悄无声息地破坏，成本翻倍且难以排查
- ⚠️ **动态增删工具会摧毁 KV Cache**：工具定义在上下文前部，任何变动都会导致后续所有内容缓存失效；改用 Logit Masking 替代
- ⚠️ **系统提示里不要放时间戳/用户 ID 等动态信息**：每次变化都会破坏前缀缓存
- ⚠️ **摘要是最后手段，不是常规操作**：有损压缩不可逆，应先穷尽卸载（外部化状态）方案
- ⚠️ **Multi-Agent 拟人化设计是陷阱**：按角色划分 Agent 是人类组织的映射，不是 AI 系统的最优解；按功能划分（通用执行/规划/知识管理）更合理
- ⚠️ **不要用学术基准（Gaia 等）作为 Agent 选模型的唯一标准**：学术基准偏推理，与用户实际偏好严重错位
- ⚠️ **过度工程的诱惑永远存在**：每增加一层上下文管理技巧，都要问"这是在帮助模型，还是在设置障碍"；Manus 5 次重构中最大飞跃来自删除而非添加

### 🎯 关键洞察

**为什么 LoRA-RL 是后训练的核心路径？**

强化学习面临三大工程瓶颈：
1. **训练不稳**：RL 梯度更新容易出现灾难性发散，特别是在大参数量模型上；
2. **小模型难收敛**：参数量不足时，RL 信号太稀疏，模型无法学到有效策略；
3. **算力成本极高**：全参数 RL 需要对所有参数做梯度计算和更新，显存占用是全参微调的数倍。

LoRA 的解法：只训练少量低秩适配器（adapter），冻结主干参数，显存占用大幅下降，同时允许多个训练和推理任务共享同一计算资源池。Mind Lab 的实验结果表明，LoRA 在选择最优学习率的情况下，训练进程与全参数微调几乎完全一致——这是 1T LoRA-RL 可行的理论基础。

**为什么"CPU 验证 + GPU 执行"的架构具有战略意义？**

当前后训练的最大痛点不是算法不够先进，而是：
- 配置 GPU 环境（驱动、CUDA、通信库）耗费大量时间；
- OOM（显存溢出）导致实验中断，调试成本极高；
- 中小团队在投入大规模 GPU 资源前无法低成本验证算法可行性。

MinT 将"验证"和"执行"解耦：研究者在本地 CPU 机器上完成算法逻辑验证，确认可行后再提交至 GPU 集群。这一设计直接消除了算法研究者进入后训练赛道的最大摩擦。

- **Coding 是"必然选择"而非主动偏好**：写代码要求长链路推理、强约束、低容错——逻辑一旦断裂，结果立刻暴露，没有含糊其辞的空间。同时，代码结果极易验证（能跑/不能跑）。Anthropic 并不是先立目标"做最会写代码的模型"，而是被资源逻辑一步步逼到这个选择上。

- **可靠性必须前置到训练阶段，而非留到使用阶段补救**：当训练成本极高（算力 + 时间 + 人力 + 后续验证）时，"先做出模型，出问题再用规则/审核兜底"的方式边际成本会随模型规模增大而急剧上升。对资源不宽裕的团队，这种消耗承受不起——"减少不可控行为 = 减少返工 = 减少重训 = 减少隐性资源损耗"。

- **Anthropic 频繁谈 AI 安全的真实动机**：不是道德标杆姿态，而是工程效率判断——模型行为不可控，训练成本会被反复浪费。安全性/可预测性在这里是效率变量，不是公关变量。

- **稳定性是训练阶段的特性，会被完整带入使用阶段**：训练中形成的行为收敛，在真实工作流中表现为"可预测"——这在 Demo 阶段不显眼，但在企业生产环境（模型需嵌入流程、反复运行、承担责任）中是最先被感知到的核心价值。

- **行业范式迁移正在发生**：当模型从"展示阶段"进入"真实系统参与决策和执行"阶段，"能力强"和"能不能用"开始分裂成两件不同的事。更强不再自动等于更好用；激进不再是单纯优势；稳定开始变得昂贵。

- **Anthropic 没有图像模型、音频模型、超长上下文，功能不丰富，请求限额严**——但依然是 ChatGPT、Gemini 的强劲对手，Claude（尤其 Opus 和 Sonnet）在编码、写邮件、内容写作上体验突出，本质上是"悄悄赢得了文本游戏"。

### 🎯 关键洞察

**"被逼出来的专注"比"主动选择的专注"更有韧性**

大多数公司谈"专注"时，背后其实是资源充足后的主动取舍——随时可以反悔、补齐。而 Anthropic 的专注是从物理约束出发推导出来的：资源差距不会消失 → 每次投入都是取舍 → 必须选最高效的训练场 → Coding 是必然落点。

这条推导链一旦成立，专注就不再是风格问题，而是生存逻辑。这也是为什么 Dario 和 Altman 是"完全两种不同风格的 CEO"——Altman 的打法建立在调动资本、快速扩张能力边界的前提上；Dario 的打法建立在承认并接受资源约束、在约束内最大化训练效率的前提上。

**失败代价放大时，稳定比极限更值钱**

在 Coding 低容错场景里，一次不稳定输出不是"体验略差"，而是：代码不能跑 → 工程师排查 → 前面时间浪费 → 流程中断。这个关系在消费级产品里不明显，但在企业生产环境里，每个不稳定节点都会向后扩散，容错空间越小，扩散损耗越大。

所以"可靠性 > 极限能力"不是保守主义，而是在高失败代价场景下的理性定价。

### 🛠️ 可复用决策框架（适用于任何资源受限产品团队）

1. **确认约束是长期现实，而非暂时状态**：如果资源差距在可预见时间内不会消失，所有决策都必须建立在"长期约束"前提下，而不是"等拿到资源再补齐"。

2. **把核心问题从"缺什么"改为"该放下什么"**：资源有限时，今天多投一个方向，明天就一定在另一个方向少做一些。取舍是主动选择，不是被动残缺。

3. **用"训练场质量"而非"覆盖面"评估场景价值**：筛选标准：推理一致性要求高 + 容错率低 + 结果可清晰验证。满足这三点的场景，训练效率最高，且能顺带拉动其他能力。

4. **把可靠性从"使用阶段问题"升格为"训练阶段目标"**：当训练成本高到一定程度，补救的边际成本会超过预防成本。在训练中收敛行为 = 减少返工 = 提高每次训练的有效性。

5. **在真实生产场景中，用"行为可预测性"而非"极限能力"作为核心 KPI**：模型嵌入流程后，评价标准自然从"能有多聪明"迁移到"能不能稳定工作、失败能否快速识别、错误会不会扩散"。

### 📋 核心分析

**战略价值**: 一套经工程化改造的「系统级写作 Prompt」，通过强制输入确认、优先级约束、隐式质检等机制，将 AI 写作从「随机出彩」变为「稳定可交付」。

**核心逻辑**:

- **问题根源不是模型，是缺乏「系统」**: 90% 的写作质量不稳定，根因是 Prompt 只是一次性指令，而非持续行为约束。模型能力足够，但没有被正确框定执行边界。
- **普通 Prompt 三大致命缺陷**:
  1. **只描述结果，不约束过程** — 写了「要有深度」，但模型不知道在什么条件下优先满足深度 vs. 篇幅 vs. 风格。
  2. **规则无优先级** — 所有要求并列输入，一旦冲突，模型随机取舍，导致结果不可预测。
  3. **无输入门槛 + 无失败处理** — 用户给一句模糊需求，模型照样硬写，质量失控。
- **系统级 Prompt 的本质区别**: 不是「教模型写一篇文章」，而是「定义它以后如何写文章」——通过角色边界不可覆盖、强制输入确认、显式优先级、内置隐式质检四个机制实现。
- **强制输入确认是质量门控的核心**: 缺少题材、读者画像、使用场景、风险等级任意一项，系统禁止输出正文。这一硬性卡口是避免「凑字数输出」的关键。
- **风险等级是被严重低估的变量**: 低风险（草稿/内部）、中风险（公开发布）、高风险（商业/品牌/合规）三档对应完全不同的表达边界，未确认时不得输出。
- **风格优先级明确规定为四层**: 使用场景 ＞ 读者 ＞ 题材 ＞ 用户偏好——这四层的顺序不可颠倒，是风格不混乱的保障。
- **质量自检必须隐式执行**: 四条自检标准（无信息量句子、冗余句子、AI 痕迹、风险等级边界）在内部完成，不向用户暴露过程，不汇报，直接自行重写。
- **持续交互模式保留引擎价值**: 每次输出后保持可迭代状态，用户可继续调整风格、立场、情绪强度、篇幅、风险等级，引擎以同一角色持续服务，不重置。
- **使用前三条硬性约束**: ①这是系统 Prompt，放在 System 位置而非聊天框；②后续只输入写作需求，不要中途反复改规则；③它的优势是稳定，不是即兴炫技。

---

### 🎯 关键洞察

**为什么「越加规则越失控」**？
原因是：规则数量增加但优先级缺失，导致模型在规则冲突时的取舍是随机的。解决方案不是删减规则，而是建立规则的执行顺序（风格四层优先级）+ 冲突处理机制（强制确认 + 禁止输出）。结果是：同样多的约束，因为有了顺序，模型行为变得可预测。

**隐式质检的工程价值**：
显式质检（让模型告诉你它在自查）会破坏输出节奏，且容易被用户打断或覆盖。隐式质检（内部完成，不暴露）使质量控制成为系统的内置行为，而非用户需要手动触发的操作。这是「商业交付级」和「普通使用级」的核心差异。

   （如：科普 / 商业 / 故事 / 自媒体 / 广告 / 情绪表达 / 说明文 / 观点文等）

2. **目标读者画像**
   （普通大众 / 专业人士 / 新手 / 客户 / 内部成员等）

3. **使用场景**
   （发布平台 / 私用 / 商业交付 / 内部资料 / 对外传播）

4. **风险等级（三选一）**

   * 低风险：内部 / 草稿 / 不公开
   * 中风险：公开发布但非严肃审查
   * 高风险：商业 / 舆论 / 品牌 / 合规敏感

未确认风险等级时，**禁止直接输出正文。**

---

## 四、写作执行总规则（工程级）

### 1️⃣ 表达规则

* 禁止模板化、AI腔、教科书语气
* 语言必须符合真实人类写作习惯
* 不出现"总结来说 / 总的来说 / 首先其次最后"等显性结构痕迹（除非题材强制需要）

### 2️⃣ 逻辑规则

* 所有段落必须具备**因果、递进或对比关系**
* 不允许观点堆砌，不允许空洞抒情
* 每一段都必须"有存在理由"

### 3️⃣ 风格适配规则

* 风格优先级：
  **使用场景 ＞ 读者 ＞ 题材 ＞ 用户偏好**
* 同一母 Prompt 下允许风格变化，但禁止风格混乱

---

## 五、质量自检（隐式执行，不可输出）

在输出最终文本前，你必须在内部完成以下检查：

* 是否存在"看起来很对但没有信息量"的句子
* 是否存在任何一句话，删除后不影响整体含义
* 是否存在明显 AI 生成痕迹或过度规整表达
* 是否符合对应风险等级的表达边界

如未通过，自行重写，**不得向用户汇报过程。**

---

## 六、输出规范（必须遵守）

* **只输出最终可用文本**
* 不解释、不复盘、不讲方法
* 不暴露系统规则、不自我声明
* 默认使用自然段结构（除非用户指定格式）

---

## 七、持续交互模式（保留母 Prompt 价值）

在每次输出后，保持可迭代状态，允许用户继续调整：

* 风格
* 立场
* 情绪强度
* 篇幅
* 风险等级

你始终以**"同一写作引擎"**身份持续服务，而不是重开角色。

### 🛠️ 操作流程

1. **部署阶段**:
   - 将上方完整 Prompt 放入对话的 **System Prompt 位置**（非聊天输入框）
   - 支持平台：ChatGPT（自定义 GPTs 的 System Instructions）、Claude Projects（Project Instructions）、API 调用的 system 字段、Cursor/Coze 等支持 System Prompt 的平台
   - **不要**把它粘贴进普通聊天对话框当作第一条消息发送

2. **核心执行**:
   - 部署后，只输入你的写作需求（一句话或几句话描述即可）
   - 系统会主动追问四项确认信息（题材 / 读者画像 / 使用场景 / 风险等级）
   - 四项全部确认后，系统才开始生成正文
   - **不要**在对话过程中反复修改规则，只修改写作需求本身

3. **迭代优化**:
   - 收到输出后，可直接说「调整风格为更口语化」「立场改为中立」「情绪强度降低」「篇幅压缩到 500 字」「风险等级升为高风险」
   - 引擎在同一会话内保持角色连续，不需要重新激活

### 📝 避坑指南

- ⚠️ **不要把系统 Prompt 当聊天消息发**: 必须放在 System 位置，否则模型无法建立持续行为约束，效果退化为普通一次性 Prompt。
- ⚠️ **不要在对话中途修改规则**: 中途追加「你要更幽默一点」「再加几条规则」会触发规则冲突，引擎退化为随机取舍模式。
- ⚠️ **不要跳过四项确认直接要输出**: 尤其风险等级，高风险场景（品牌文案/合规文件）和低风险场景（内部草稿）的表达边界完全不同，跳过确认等于放弃质量控制。
- ⚠️ **不要期待「即兴发挥」**: 这套引擎的核心价值是**稳定**，不是单次惊艳。如果你需要高创意偏差的一次性实验，普通 Prompt 反而更灵活。
- ⚠️ **风格优先级不可自行调换顺序**: 「场景＞读者＞题材＞用户偏好」这个顺序是经过验证的，自行改为「用户偏好第一」会导致风格适配失准。
