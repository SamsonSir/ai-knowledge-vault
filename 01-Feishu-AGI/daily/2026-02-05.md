### 📋 核心分析

**战略价值**: Anthropic 发布 Opus 4.6，在知识工作、Agent 编码、多学科推理、长上下文四大维度全面超越 GPT-5.2，同步推出 1M 上下文、128K 输出、adaptive thinking、context compaction 等工业级 API 能力，并将产品线延伸至 Excel/PowerPoint/多 Agent 协作。

**核心逻辑**:

- **GDPval-AA 知识工作评测**：Opus 4.6 领先 GPT-5.2 约 144 Elo，领先自家 Opus 4.5 约 190 Elo，换算胜率约 70%，覆盖金融、法律等专业领域子项全面领先
- **Agent 搜索（BrowseComp）**：单 Agent 模式已领先同类，叠加多 Agent 框架后得分达 86.8%，测的是在网络上检索难以找到信息的实际能力
- **Agent 编码（Terminal-Bench 2.0）**：拿下最高分；SWE-bench Verified 平均跑 25 轮，调整 prompt 后最高达 81.42%
- **多学科推理（Humanity's Last Exam）**：跑分时启用 web search + code execution + context compaction（50K token 触发，最大 3M token），使用 max effort + adaptive thinking 组合
- **ARC AGI 2**：使用 max effort + 120K thinking budget 配置
- **长上下文突破（MRCR v2 八针 1M 测试）**：Opus 4.6 得分 76%，Sonnet 4.5 同测试仅 18.5%，差距达 4 倍；这是 Opus 级模型首次支持 1M token 上下文（beta），能在数十万 token 中追踪 Opus 4.5 会漏掉的细节
- **其他专项评测**：OpenRCA（复杂软件故障根因诊断，全匹配得 1 分否则 0 分）、MCP Atlas 长期连贯性（max effort 最高分，high effort 达 62.7%）、CyberGym 网络安全（默认 effort + temperature + top_p，配备 think tool 做多轮交叉思考）、多语言编码、生命科学均有跑分数据
- **内部使用观察**：模型自动将算力集中在任务最难部分，简单部分快速通过；但简单任务上存在「想太多」问题，Anthropic 建议此类场景将 effort 从默认 high 调为 medium
- **Early Access 合作伙伴反馈**：三个核心点——能自主工作无需手把手、能完成前代模型搞不定的任务、改变了团队协作方式
- **安全评估**：对齐偏差率（欺骗、谄媚、配合滥用）与 Opus 4.5 持平；over-refusal 率是近期 Claude 模型最低；首次引入可解释性（interpretability）技术分析模型底层行为；新增 6 个网络安全探针追踪潜在滥用；同时用该模型为开源软件找漏洞打补丁

### 🎯 关键洞察

**长上下文的核心突破**：以往 1M token 上下文的问题不是「支不支持」，而是 context rot——上下文越长，模型表现越差。Opus 4.6 在 MRCR v2 八针 1M 测试中以 76% vs 18.5% 的比分证明这个问题得到了实质性解决，而非仅仅是参数上的扩展。

**Adaptive Thinking 的工程意义**：之前 extended thinking 是二元开关，现在变成了连续控制——模型自己判断何时需要深度推理，何时快速通过。这对 API 调用成本控制有直接影响，开发者不再需要为所有请求支付深度推理的代价。

**Context Compaction 的 Agent 价值**：长对话和 Agent 任务最大的工程痛点是上下文窗口撞顶。Context compaction 自动将旧上下文压缩为摘要替换，触发阈值可配置，这让长时间运行的 Agent 任务在工程上变得可行。

### 🛠️ 操作流程

**1. 准备阶段**

- 确认调用入口：claude.ai、Claude API、AWS Bedrock、GCP Vertex AI、Azure 均已上线
- 模型标识符：`claude-opus-4-6`
- 确认任务复杂度，预判所需 effort 档位（简单任务 → `medium`，复杂推理 → `high` 或 `max`）

**2. 核心执行**

- 普通任务：默认 `high` effort，adaptive thinking 自动介入
- 简单/高频任务：显式设置 `effort: medium`，降低延迟和成本
- 极限推理任务（如 ARC AGI 类）：`effort: max` + 设置 `thinking_budget: 120000`
- 长 Agent 任务：启用 context compaction（beta），配置触发阈值（默认 50K token），上限可设至 3M token
- 超长文档处理：使用 1M 上下文（beta），注意 200K token 以上部分计费变化
- 大输出任务：直接使用，128K 输出上限无需拆分请求

**3. 多 Agent 场景（Claude Code agent teams）**

- 将任务拆分为可独立执行的子任务（如大规模 code review）
- 同时启动多个 Agent 并行工作、自主协调
- 使用 `Shift+Up/Down` 或 `tmux` 随时接管任意子 Agent
- 当前为 research preview 阶段

**4. Excel + PowerPoint 组合工作流**

- 先用 Claude in Excel 处理非结构化数据、推断表结构、设置条件格式和数据验证
- 再用 Claude in PowerPoint（research preview，Max/Team/Enterprise 可用）读取版式/字体/母版，保持品牌一致性，从数据直接生成完整 deck

**5. 验证与优化**

- 长上下文任务：用 MRCR 类多针检索测试验证模型是否真正追踪到关键信息
- Agent 任务：观察是否出现「想太多」现象（延迟异常高），若有则降 effort 档位
- 安全敏感场景：参考 system card 了解对齐偏差率和网络安全探针细节：`https://www.anthropic.com/claude-opus-4-6-system-card`

### 📝 避坑指南

- ⚠️ **简单任务别用默认 effort**：Opus 4.6 默认 `high` effort，简单任务会「想太多」，显著增加延迟和成本，明确设置 `effort: medium`
- ⚠️ **1M 上下文是 beta**：生产环境谨慎依赖，200K token 以上部分计费上浮，评估实际需求再开启
- ⚠️ **Context compaction 会损失细节**：压缩旧上下文本质是摘要替换，对需要精确回溯早期对话内容的任务要谨慎，触发阈值需根据任务特性配置
- ⚠️ **Agent teams 是 research preview**：Claude Code 多 Agent 功能尚未 GA，不建议用于生产关键路径
- ⚠️ **Claude in PowerPoint 是 research preview**：仅 Max/Team/Enterprise 可用，功能尚不稳定
- ⚠️ **网络安全能力提升带来的双刃剑**：Anthropic 额外开发了 6 个新网络安全探针追踪潜在滥用，安全敏感场景需关注 system card 更新
- ⚠️ **US-only inference 有溢价**：需要数据留在美国境内的场景价格为标准价 1.1 倍，按需选择

### 📋 核心分析

**战略价值**: YouMind 是一套基于 IPO 工作流（Input→Process→Output）的 AI 内容创作平台，核心价值在于把「收集素材→消化整理→输出成品」压缩进单一工具，消灭多工具切换成本。

**核心逻辑**:

- **IPO 架构是底层设计哲学**：Input 支持网页、YouTube/B站视频、播客、PDF；Process 由 AI 自动搜索、整理、提炼；Output 直接生成文章、PPT、播客、图片，全链路不出平台
- **AI 建项目功能把选题调研从 2-3 小时压缩到 5 分钟**：输入一句自然语言描述需求（如"WaytoAGI 知识库使用说明书，让第一次使用的新用户也能快速上手"），系统自动完成：理解目标 → 制定计划 → 全网搜索资料 → 输出结构完整文章，无需人工干预
- **YouTube 长视频转中文逐字稿，实测比 Gemini 更完整**：Gemini 遇到长视频会偷懒只给摘要；YouMind 用以下 Prompt 可输出完整逐字稿（需先安装 Chrome 插件）：
  ```
  把这个视频转成中文纯文字，并分好段落
  ```
  实测 7000+ 字的视频内容可完整翻译输出，不截断
- **Skills（原 Shortcuts/快捷指令）是核心生产力乘数**：把多步骤创作工作流封装成一键调用的技能单元，在对话框点击「小立方体」→ 选择技能 → Agent 自动调用对应模型执行，无需跳转其他平台
- **Skills 调用模型是动态匹配的**：例如调用「Blog post」Skill 时系统自动选用 Gemini 2.5 Pro 进行改写，不同 Skill 对应不同擅长写作风格的模型；也可切换到 Ask 模式手动指定模型
- **改写结果以「修订模式」呈现**：改动内容一目了然，确认无误后点「全部接受」一键应用，降低审核成本
- **生图能力内置于对话流**：支持 Nano Banana Pro、GPT Image 1.5、Seedream 4.5 等多个生图模型，可在对话中直接生成推文金句卡、封面、信息卡等，无需切换 Midjourney/Canva 等工具
- **内置 8000+ 生图提示词库**：YouMind 专门搭建了提示词收录网站，覆盖各类生图场景，不需要自己到处找 Prompt
- **Skills 管理入口**：对话框下方小方块 → 点「管理」→ 进入 Skills 管理页，可按分类安装技能（学习、研究、写作、图片、Slides、网页）
- **内容管理以「项目」为单位组织**：每个项目可上传资料或由 AI 起草，适合接陌生选题、行业调研、会议发言准备等场景

### 🎯 关键洞察

**为什么 YouMind 的视频转文字比 Gemini 强？**
Gemini 对长视频有隐性长度限制，超过阈值会自动降级为摘要输出。YouMind 通过 Chrome 插件直接抓取视频字幕/音轨数据，绕过了这个限制，因此 7000+ 字的内容也能完整输出逐字稿。

**Skills 的本质是「个人 SOP 的 AI 化封装」**：
原来的 Shortcuts 只是快捷 Prompt，升级后的 Skills 是完整工作流——可以串联多个步骤（翻译→改写→生图→生成封面），并自动选择最适合当前任务的模型，相当于把你的内容生产 SOP 变成了可复用的 AI Agent。

### 📋 核心分析

**战略价值**: 语雀创始人玉伯以「诚实创业者」视角，拆解从大厂出走到 YouMind 找到 PMF 的完整路径——包含创业三问、产品三问、增长打法，可直接复用于 AI 工具类创业。

**核心逻辑**:

- **创业前必答三问（顺序不可乱）**：① 愿不愿意——方向是否热爱，能否持续干 5-10 年；② 能不能——自己和团队的真实能力边界在哪，差异化在哪（玉伯明确说"我不是杨植麟，所以不做大模型"）；③ 值不值——用户群在哪、商业模式能否正向运转、前两三年艰难期团队是否愿意跟。三问缺一，创业大概率后悔。
- **"一人公司"是陷阱**：AI coding 再强，一人操作等于同时指导 5 个顶级程序员，极速耗竭。6 人团队借 AI 可做出产品，但 6 人打 100 人大厂团队，概率上几乎不可能——核心竞争力必须来自方向差异化，而非人力堆叠。
- **"我想做什么"不重要，用户需要什么才重要**：玉伯最初包装"AI 时代的纸笔 3.0"宏大叙事，聊了八九十家机构后发现是在骗自己——重复说一件事会产生"肯定对"的心理暗示，这是创业者最危险的认知陷阱。
- **800+ 用户访谈提炼出创作者三大痛点**：① 选题难，持续发现好选题极难；② 工具割裂，写一篇公众号要用 5-6 款单点工具（浏览器+搜索+整理+写稿+剪辑）；③ 变现难，副业创作往往倒贴钱。
- **产品切入策略：从最小可验证点切入**：0.1 版本只做一个浏览器插件——一键保存 YouTube 视频或文章，AI 自动总结。从这个最小功能获取真实反馈，再逐步扩展，进入正循环。
- **创作者工作流是非线性的**：不是"选题→找资料→写稿→制作→发布"的线性流程，而是五个模块任意节点切入、反复横跳。因此创作者需要的不是 workflow，而是一个「创作环境」。
- **"稿子"= Long Prompt，这是核心产品哲学**：稿子不是 Draft 也不是 Document，而是融合个人数据（Personal Data）的长提示词。AI 检索公共信息能力强，但个人独特内容无法被检索——只靠公共信息生成的内容必然"千人一面"。个人数据 + Long Prompt = "千人千面"的创作输出。这也解释了为何 YouMind 生成的配图比 Lovart、Gemini 还好——不是模型优化，而是输入是经过加工的 Long Prompt。
- **AI 产品交互形态分类与 YouMind 的定位**：① Chatbot（ChatGPT）；② ChatTask（Manus，定义 AI 完成任务的 GUI）；③ ChatCanvas（Lovart，画布形态适合视觉创作）；YouMind 正在探索第四种：**ChatPage / ChatStudio**。交互形态是 AI 产品差异化和竞争壁垒的关键变量。
- **To C AI 产品增长三步**：① 好看（获客第一步，几个人时就招设计师）→ ② 好用（核心是上下文管理）→ ③ 好玩（网络效应，留住用户的是感觉和氛围，不是具体功能）。
- **全员运营打法**：要求公司每个人定期发视频。副作用是强迫研发人员真正使用产品，形成复利。2025 年 10 月底增长曾躺平，通过全员运营重新拉起增长曲线。

### 🎯 关键洞察

**"AI 味"的本质是缺少个性化输入**

"AI 味"不是模型问题，是输入问题。喂给模型的素材越个性化，输出越去 AI 味。YouMind 的"口喷式写文章"功能（语音输入）验证了这一点：用户毫无章法地跟 AI 聊半小时，最终产出的文章没有 AI 味——因为语音输入本身就是高度个性化的 Long Prompt。

**大厂出来创业的结构性劣势**

大厂思维 = 资源充足假设 + 宏大叙事惯性。玉伯在阿里做了十几年，语雀月访问量 1000 万+，出来创业第一反应仍是"做改变世界的产品"。这种惯性需要外部反馈（融资碰壁、用户访谈）才能打破，不是靠自省。

**沉默用户 > 活跃用户（运营反直觉结论）**

用户群里活跃用户往往藏着大量羊毛党，沉默用户才是大量付费用户。运营策略应该是：谨慎倾听活跃用户反馈，主动触达沉默用户。

**NotebookLM 月访问量 1.4 亿的信号意义**

这个数据说明"人不爱学习"是伪命题。消费者正从被动消费转向主动消费——不想看 2 小时长视频，但愿意用 AI 工具看精华摘要。YouMind 的长期愿景是成为 AI 时代的 YouTube：主动探索式的全球学习与内容平台。

### 🛠️ 操作流程

**复用玉伯创业方法论的 SOP**

1. **创业前自检（三问）**
   - 问自己：这个方向能干 5-10 年吗？（愿不愿意）
   - 列出自己和团队的真实能力清单，找出市场差异化点（能不能）

2. **找方向（避开宏大叙事陷阱）**
   - 融资 pitch 超过 10 次后，重新审视自己的叙事——如果你在重复说同一件事，警惕心理暗示
   - 做至少 50 次潜在用户访谈，提炼真实痛点（玉伯做了 800+）
   - 从最小可验证功能切入，不要上来就做完整产品

3. **产品迭代节奏**
   - 0.1 版本：只做一个核心功能点（玉伯选了"一键保存+AI总结"）
   - 获取真实用户反馈后，识别非线性工作流中的高频痛点节点
   - 围绕"创作环境"而非"工作流"重新设计产品架构

4. **To C 增长三步走**
   - 第一步：招设计师，做好看的产品（哪怕团队只有几个人）
   - 第二步：打磨上下文管理，做好用的产品
   - 第三步：全员运营，每人定期发视频，强迫团队真正使用产品

5. **用户运营反直觉操作**
   - 不要被活跃用户牵着走（羊毛党居多）
   - 主动设计触达沉默用户的机制（问卷、1v1 访谈、邮件）

### 📋 核心分析

**战略价值**: 通过 Clawdbot 2026.2.2 官方飞书插件，将国产大模型（minimax/kimi/GLM 等）+ 飞书群聊打通，实现零翻墙的全国产 AI 助手方案。

**核心逻辑**:
- Clawdbot 2026.2.2 版本正式原生支持飞书 channel，飞书通过长连接（非 Webhook）接收事件，无需公网 IP
- 飞书机器人权限需通过 JSON 批量导入，共 15 项 tenant 权限 + 3 项 user 权限，缺一不可
- 飞书机器人必须手动开启「机器人能力」（填写欢迎语并保存），否则无法收发消息

- 飞书机器人配置必须「创建版本并发布」才能生效，未发布等于所有配置作废
- 最终配对流程：飞书发消息 → 机器人回复 `pair:XXXXXX` → 将配对码粘贴到 Clawdbot Web UI 完成绑定

      "im:message",

      "im:message.p2p_msg:readonly",
      "im:message:readonly",
      "im:message:send_as_bot",
      "im:resource"

## 📔 文章 5

> 文档 ID: `HJQ2wgH61iEZq5kdTJmcI1h5nWf`

**来源**: Wise：AI大模型的下半场：上下文学习，腾讯首席科学家姚顺雨加入后首篇论文CL-bench发布 | **时间**: 2026-03-13 | **原文链接**: `https://mp.weixin.qq.com/s/XMf536eZkZTmgWeVbm5qOQ`

---

### 📋 核心分析

**战略价值**: CL-bench 用实验数据证明当前所有前沿大模型（GPT/Claude/Gemini/Kimi/Qwen）本质上是「参数检索机器」而非「真正的上下文学习者」，平均只能解决 17.2% 的新知识任务，宣告 AI 竞争轴从「模型训练」转向「上下文工程」。

**核心逻辑**:

- **最高分仅 23.7%**：GPT-5.1（High 推理模式）在 CL-bench 中得分 23.7%，是所有测试模型中最高分，但仍意味着 76.3% 的任务失败。所有模型平均分仅 17.2%，说明这不是某个模型的问题，而是整代架构的系统性缺陷。
- **测试规模**：500 个复杂 context、1,899 个任务、31,607 个验证标准，由资深领域专家手工制作，覆盖四类真实世界场景。
- **四类测试场景**：① 领域知识推理、② 规则系统应用（如虚构法律体系、新编程语法）、③ 程序性任务执行（复杂工作流与操作指南）、④ 经验发现与模拟（从实验数据/观测记录/模拟环境中归纳潜在定律）。
- **核心设计约束**：模型必须从 context 中学习「预训练中不存在的新知识」并正确应用——这是唯一判定标准，直接切断了模型靠背诵旧知识蒙混过关的路。
- **失败根因**：模型拿到题目后，第一反应是去预训练知识库里检索相似案例，而不是阅读 context 里的新规则。即使 context 明确标注「这是全新规则」，模型依然倾向于忽略并套用旧知识。本质是「参数推理者」而非「上下文学习者」。
- **发现一——长上下文 ≠ 上下文学习**：能处理长输入（长上下文推理 + 指令遵循）是必要条件但不充分，还需要真正的「从 context 中吸收新信息并泛化」的学习能力，三者缺一不可。
- **发现二——归纳比演绎难得多**：演绎（照说明书执行）尚可，归纳（从数据中自己总结规律）任务解决率普遍低于 10%，且结果波动极大。发现规律远比应用规律难。
- **推理强度的边际效益递减**：GPT-5.1 在管理类和实验数据类任务上，提高推理强度后表现提升约 6%，但其他模型提升有限甚至下滑——「大力出奇迹」的时代已过。
- **姚顺雨（腾讯混元首席科学家）核心判断**：更强的模型，很多时候等价于提供了额外的 Context。CL-bench 是对这一判断的实证化。
- **竞争轴转移**：上半场拼「把模型训练得更聪明」，下半场拼「谁能提供更丰富、更精准的 Context」。工程重心从 Prompt Engineering 转向 Context Engineering。

---

### 🎯 关键洞察

**为什么当前模型会系统性失败？**

原因链：预训练目标是最小化在静态语料上的损失 → 模型学会的是「模式匹配+知识检索」 → 推理时遇到新规则，激活的仍是最相似的旧模式 → 即使 context 明确说明新规则，注意力机制的权重分配仍偏向高频预训练知识 → 结果：context 里的新信息被稀释或忽略。

**归纳推理为何是硬墙？**

演绎任务（给规则→执行）本质是 instruction following，现有 RLHF/SFT 训练对此有大量覆盖。归纳任务（给数据→总结规律）要求模型在 context 窗口内完成「假设生成→验证→修正」的闭环，这在预训练阶段几乎没有对应的监督信号，因此成为系统性短板。

**Context Engineering 的下一个瓶颈**：当 context 窗口无限扩大后，新的挑战变成——如何从海量记忆中提炼有价值信息？用什么机制对抗遗忘？这将是下一代架构竞争的核心战场。

### 🛠️ 操作流程（如何用 CL-bench 思路评估/改进自己的 AI 系统）

1. **准备阶段**：构造「预训练中不存在的新知识」作为 context——可以是虚构的业务规则、自定义 DSL 语法、内部流程文档。确保测试问题的答案无法从模型已有知识中推导出来。

2. **核心执行**：
   - 演绎测试：给出完整规则 → 要求模型执行具体任务，验证 instruction following 质量。
   - 归纳测试：给出多组数据/案例 → 要求模型总结规律，再用新数据验证其归纳结论。
   - 对比测试：同一任务分别用「无 context」和「有 context」两种方式提问，观察模型是否真正利用了 context。

3. **验证与优化**：
   - 若模型在有 context 情况下仍给出与「无 context」相似的答案 → 说明模型在走参数检索路径，需要在 prompt 中强制要求「先逐条引用 context 中的规则，再作答」。
   - 归纳任务失败率高 → 考虑拆解为「先归纳规律（输出中间结论）→ 再应用规律」的两步 chain-of-thought，而非一步到位。
   - 推理强度提升无效 → 不要依赖单纯加大算力，转向优化 context 的结构化程度（分层、标注、去噪）。

### 📝 避坑指南

- ⚠️ 不要用「模型能处理 100K token 上下文」来等价「模型能从上下文中学习」——这是两件事，CL-bench 数据已证伪这一等式。

- **语音录入支持乱序口语**：无需按字段顺序说话，例如一句「张三今天完成了仓库盘点，状态是已完成，发现两箱货包装破损，需要联系供应商，标签打上异常和待处理」，AI 自动拆解人名、状态、多选标签、备注，分别填入 4-5 个字段。
- **支持口语反悔/纠错**：说完后可直接口头更正，例如「预算大概 50 万……不对，改成 80 万」，AI 识别最终意图，以修正后的值录入，无需重新填写。
- **强噪音环境可用**：在机器轰鸣、建筑工地等高噪音场景下，靠近麦克风大声说出指令（如「设备温度 50 度，运转正常，皮带有点松，建议晚班紧固」），仍能准确填入设备名称、状态、处理建议，无乱码。
- **支持多方言**：已验证可识别河南话、四川话、上海话、粤语，适合不会普通话的一线工人使用。
- **AI 粘贴录入两种触发方式**：① 表格视图左上角「添加记录」旁下拉箭头 → 选「AI 粘贴录入」→ 拖拽图片或文字；② 直接在多维表格内按 `Ctrl+V`（Mac 为 `Command+V`），AI 立即启动智能匹配。
- **字段类型自动判断**：AI 会根据当前表格结构，自动将粘贴内容中的「时间」匹配日期列、「金额」匹配数字列、「长段描述」匹配多行文本列，用户只需检查并点击「确认录入」。
- **单选/多选字段智能切换**：面对状态标签（如进行中/已完成）自动识别为单选字段；面对卡片上可能出现多个标签的情况，自动切换为多选字段，直接生成可筛选的结构化数据，无需二次清洗。
- **合并单元格层级还原**：能正确处理跨多行的合并单元格（如项目名称下挂多个任务），还原从属层级结构，不出现字段错位。
- **速度优势**：同一张复杂进度表截图，飞书多维表格比钉钉「万能贴」更快完成识别并填入数据（实测飞书后放先出结果）。

### 🎯 关键洞察

**为什么这两个功能的价值不只是「方便」**：

传统 OCR 和语音转写只做「内容提取」，输出的是原始文字。飞书这两个功能的差异在于多了一层「结构理解」：

- 原因：多维表格 AI 在识别内容的同时，会参照当前表格的字段结构和字段类型，做语义匹配而非字符匹配。
- 动作：识别「进行中/已完成」时判断为单选；识别卡片多标签时判断为多选；识别合并单元格时还原层级；识别口语反悔时取最终值。
- 结果：粘贴/语音后直接得到可筛选、可分析的结构化数据，省去「录入 → 手动建字段 → 清洗数据」的二次处理流程。

**一线场景适配逻辑**：
- 销售外访：拜访完客户走路途中随口录入，支持反悔修正，不需要停下来打字。
- 仓库/工厂：噪音环境 + 方言环境均可用，降低一线工人的数字化门槛。
- 行政/项目管理：复杂进度表截图一键粘贴，合并单元格和状态标签自动还原。
- 电商运营：商品列表截图粘贴后，自动提取商品标题、价格、卖点、品牌标识，分列归类。

### 📝 避坑指南

- ⚠️ 云雾 AI 模型名称必须从官网确认，不能随意填写，否则 API 调用报错
- ⚠️ PPT 提示词中每页的 `NARRATIVE GOAL` 不能省略，省略后图像风格会漂移，失去叙事连贯性
- ⚠️ 生成 PDF 前务必在网页检查文字内容，标题文字会直接渲染到幻灯片中，错别字无法事后修改
- ⚠️ 本地运行需要 Python 环境，Trae 生成代码后直接问 AI "如何运行"，它会给出完整的环境配置步骤
- ⚠️ 云端部署涉及服务器配置，建议先跑通本地版本再提云端需求，避免调试成本叠加
- ⚠️ 不同充值方案折扣不同，按量计费，PPT 生成成本取决于页数和重新生成次数
