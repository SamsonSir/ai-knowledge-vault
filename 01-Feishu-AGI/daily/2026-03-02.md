### 📋 核心分析

**战略价值**: AIGC动画创作者刘士超（梦游牛油果）系统拆解从主旨确立、分镜制作、配音、音乐到商业变现的全链路工业级工作流，并给出新旧模型切换下的效率对比与避坑经验。

**核心逻辑**:

- **主旨先行原则**：完整片子必须先确定核心主旨，再"根据一碟醋包一盘饺子"构建故事。《Dear 李华》主旨是「社会注意力稀缺」，由此构建李华所在靠吸引注意力存在的星球，将抽象现象具象化，而非先有故事再找意义。
- **旧模型工作量基准**：10分钟视频用旧模型（可灵+PS分层逐帧）耗时约1个月，大量使用PS，分镜头逐个制作后做成动画，比《rickandmorty》用Banana图片模型稍快但仍极费劲。
- **新模型效率跃升**：用最新模型，1人1周可完成11分钟视频，绝大多数提示词要求都能满足，不再需要因"效果出不来"而改剧情。
- **内核决定作品寿命**：工具迭代不会淘汰内核扎实的片子——旧工具做的片子现在看依然可行。节奏控制、叙事表达是AI无法替代的部分，这决定作品能否不过时。
- **商业变现路径重构**：新模型使爆款视频极易被复制，甲方压价，靠AI做爆款变现的传统思路已失效。正确路径：确定有长期价值的内容体系 → 打造自有IP → 用IP承接商业合作（参考丁逸的「鹏哥宇宙」、露西坚持风格化创作后吸引黑神话/悟空制片合作）。
- **一致性问题现状**：即梦多参考/全能参考功能可解决角色和场景一致性，只需设计好角色和场景即可保持较好一致性。Seedance 2.0出来前，人脸、口型、语音对照等问题难以调配，现在模型层面已可控制。
- **配音工具选择**：旧方案用Minimax或finevoice克隆配音，Minimax效果更好但存在语气/情绪问题。新方案：即梦全能参考功能，导入约10秒语音即可复制声音，自带语气，极端情绪下节奏表现也不错。
- **音乐制作流程**：不懂乐理 → 将歌名+作者丢进GPT/DeepSeek拆解风格和节奏段落 → 整理提示词 → 用Suno多次生成（《Dear 李华》主题曲在网易云发了17个版本）→ 从中选合适风格。《Dear 李华》是先有音乐再剪辑画面。
- **版权保护路径**：手画角色可向中国版权网申请版权，提交创作过程图片等资料，通过率较高。AI生成内容可在中国AI版权网登记，只要能体现个人思考过程、提供出图过程，维权胜率高。
- **平台变现机制**：抖音平台有AI漫剧机制和奖励，可将作品上传评级获取收益；刘士超计划半年完成长篇剧情后再考虑，期间更新频率降低（一个月不一定能更新一期）。

### 🎯 关键洞察

**注意力稀缺作为创作母题**：《Dear 李华》的世界观设定——星球上的生命靠吸引注意力存在，李华因无人关注而向全宇宙写信，房子靠写信加速——是对现实中流量焦虑的直接具象化。这种"先有主旨再包故事"的方法论，使作品在工具迭代后依然不过时，因为它的价值锚点是社会洞察而非技术奇观。

**技术平权的双刃剑**：新模型让技术门槛趋近于0（豆包等软件老年人也能用），信息差在缩小，但这同时意味着爆款内容极易被复制，创作者的工具优势消失。真正的护城河只剩认知、价值观和叙事节奏——这三者AI目前无法替代。

**IP优先于接单**：承接商业片短期有收益但不沉淀IP。丁逸前期做广告片，沉淀后打造鹏哥宇宙，再用IP承接商业合作；露西坚持风格化创作，火了之后黑神话、悟空制片主动找来。两个案例都指向同一结论：先建IP，商业合作自然来。

### 🛠️ 操作流程

**完整AI动画短片制作SOP（以《Dear 李华》为参考）**

1. **确定核心主旨**：先想清楚要表达什么社会现象或价值观，再围绕主旨构建世界观和故事，避免为做片而做片。

2. **故事与剧本**：以主旨为锚点构思故事梗概，融入具体场景增加趣味性（如沈阳大街）。歌词可自写大纲，交给擅长写诗的人（如徐康硕）写成诗再转歌词。

3. **角色设计**：核心角色手画原创，配角可找现实参考后用Banana模型生成调整。手画角色同步准备版权申请材料（创作过程图片）。

4. **音乐制作**（先于画面）：
   - 找参考风格 → 将歌名+作者输入GPT/DeepSeek拆解风格、节奏段落
   - 整理提示词 → 用Suno多次生成 → 选出合适版本
   - 上传网易云（可发多个版本供选择）

5. **分镜制作**：
   - 用即梦全能参考功能，一张图生成多角度分镜，省略九宫格步骤
   - 多人场景：将主要人物和群像分别出图再衔接，避免后景人脸变形
   - 不可用宫格：后期手动PS替换

6. **视频生成**：
   - 前期确定好图片色彩，避免前后素材色差
   - 用文本精准控制视频内容，文字越精准效果越好
   - 新模型下1人1周可完成11分钟视频

7. **配音**：
   - 用即梦全能参考，导入约10秒语音复制声音
   - 多人配音：分开上传并标注说话人，语音总长度控制在10-15秒，优先单人上传保证识别率

8. **剪辑**：根据已有音乐剪辑画面，AI可粗剪，叙事节奏需人工处理。

9. **版权保护**：核心角色向中国版权网申请版权；AI生成内容向中国AI版权网登记。

10. **发布与变现**：可与平台合作（如快手星芒短剧+可灵AI）；抖音平台可上传评级获取收益；长期路径是打造自有IP后承接商业合作。

### 💡 具体案例/数据

- **刘士超（梦游牛油果）**：全网粉丝35w+，播放量5000w+
- **作品列表**：《反派的意义》（7分半）、《Dear 李华》（10分钟以上，与快手星芒短剧+可灵AI合作，快手独播）、《杀死那个机器人》（11分钟）、《rickandmorty》（去年B站爆款，用PS分层+Banana图片模型）

- **可灵积分消耗**：《Dear 李华》视频部分约消耗3万积分，图片消耗少（算力由可灵商业合作承包，非自费）
- **音乐版本数**：《Dear 李华》主题曲在网易云发了17个版本，歌曲《外星小偷李贝》及片尾曲均用Suno制作，已上传网易云

- **露西案例**：坚持风格化创作，火了之后吸引黑神话、悟空制片等主动合作

### 📝 避坑指南

- ⚠️ **多人语音参考识别率低**：即梦2.0上传两条语音时系统有时认不出说话人，优先上传单条语音，识别率最高；多人时需分开标注，且总长度不超15秒。
- ⚠️ **多人场景人脸变形**：用net banana Pro绘制多人场景时后景人脸变形，解决方案是用即梦将角色和场景分开输入，或将主要人物和群像分别出图再衔接。
- ⚠️ **九宫格分镜有废格**：出现不可用宫格时，后期手动PS替换，或直接用即梦全能参考拆分镜，省略九宫格步骤。
- ⚠️ **前后素材色差**：前期就要确定好图片色彩风格，后期用文本精准控制，不要依赖后期调色修补。
- ⚠️ **爆款复制陷阱**：新模型使爆款极易被复制，靠AI做爆款接单的变现模式已不可持续，甲方会持续压价。
- ⚠️ **AI生成角色版权不确定**：手画角色申请中国版权网通过率高，AI生成角色通过率不确定，核心角色建议手画。
- ⚠️ **工具迭代焦虑**：不必因新模型出现而重做旧片，内核和剧情扎实的作品不受工具迭代影响。
- ⚠️ **时长超5分钟成本回收难**：多数情况下成本难以覆盖收益，时长超5分钟能否回收成本不确定，商业合作是主要回收路径。

> 文档 ID: `OADCwjklpigH0wkMlWlcPxnPnsh`

**来源**: 抽丝剥茧：深度解析 OpenClaw 万字系统提示词（System Prompt）构成 | **时间**: 2026-03-13 | **原文链接**: `https://x.com/LufzzLiz/status/20266...`

---

### 📋 核心分析

**战略价值**: 通过自研代理拦截工具，完整还原 OpenClaw 每次对话默认携带的 ~16k token 系统提示词结构，并给出可操作的 Token 优化路径。

**核心逻辑**:

- OpenClaw 每次对话默认注入约 34062 字符（~16k tokens）的系统提示词，即使用户只发一句 "hi"，底层 input token 也高达 15391+。
- 官方日志路径 `.openclaw/agents/main/sessions/` 只记录 usage 数据，不记录完整提示词明文，必须借助代理拦截才能看到全貌。
- 系统提示词分三大块：硬编码头部（身份+工具+安全+技能索引+记忆召回）、Project Context（8个注入文件）、末尾协议（静默/心跳/运行时快照）。
- 工具列表存在双份冗余：`system ## Tooling` 段和 `body.tools[]` 各携带一份完整工具集，31/31 全重合，其中 `body.tools[]` 还带 parameters schema，两者合计约 5988~9582 tokens。
- Project Context 的 8 个注入文件中，MEMORY.md 是 Token 大头（约 2.3k~3.8k tokens），AGENTS.md 次之（约 2.0k~3.2k tokens），IDENTITY.md 最轻（约 50~80 tokens）。
- Skills 机制是懒加载：系统提示词只注入 `<available_skills>` 索引（name + description + location），匹配到任务才用 `read` 工具加载对应 SKILL.md，不匹配不读，最多读一个。
- subagent 的系统提示词是主动收缩版：Project Context 只带 AGENTS.md + TOOLS.md，工具列表从 31 个缩减到 23 个，且不注入 `## Skills` 索引段，也不带 memory_search / memory_get 等 8 个工具。
- compact 操作只压缩会话消息历史（session.messages），System Prompt 本体按模板重新构建，不会因 compact 自动瘦身，优化 Token 必须从 System 内容本身下手。
- 模型别名表（Model Aliases）是运行时动态注入的，配置的模型越多，每次携带的别名表越长，是隐性 Token 消耗点。
- 安全护栏硬编码在第一段：无独立目标、禁止自我复制/扩权/获取资源，指令冲突时暂停问人，灵感来自 Anthropic AI 宪法；TOOLS.md 明确声明不控制工具权限，但如果 TOOLS.md 里写了权限约束，可能被硬编码的 `TOOLS.md does not control tool availability` 这句话冲掉。

---

### 🎯 关键洞察

**双份工具冗余是最可操作的优化点**：`system ## Tooling` 段的工具列表主要用于行为引导（告诉模型有哪些工具、怎么用），`body.tools[]` 的 schema 是调用必须的机器可读信息不可删。两者完全重合，优化逻辑：

- 原因：系统提示词里逐个列工具描述 → 模型理解工具用途 → 但 body.tools[] 已经有 description 字段，等于说了两遍。
- 动作：把 `system ## Tooling` 里的工具列表缩成极短版（只保留调用规则，不逐个列描述），保留 `body.tools[]` schema 不动。
- 结果：可节省约 874~1399 tokens（system 工具说明段）+ 部分重复描述，整体可压缩 5~10% 的系统提示词体积。

## 📓 文章 6

> 文档 ID: `GA2swbA4liHIxYkNWYwczpkEnhb`

**来源**: AK大神用200行代码，让我终于看懂ChatGPT原理了！ | **时间**: 2026-03-02 | **原文链接**: https://mp.weixin.qq.com/s/9BTLe2O2...

### 📋 核心分析

**战略价值**: Andrej Karpathy 用 200 行 Python（MicroGPT）从零复现 ChatGPT 核心算法，以"生成人名"为任务，完整走通 Tokenizer → Embedding → Attention → MLP → Loss → Backprop → 生成 全链路，是理解 LLM 原理的最小可运行样本。

**核心逻辑**:

- **训练数据与任务定义**：32000 个英文人名（emma/olivia/ava...），目标是学习字母统计规律，生成新的"听起来像真名"的词（如 kamon、karai、anna、anton）。从 ChatGPT 视角看，对话就是文档续写，提示词之后的回应是统计意义上的下一段文本。

- **Tokenizer（分词器）**：26 个小写字母编号 0-25，加特殊标记 BOS（Beginning of Sequence）编号 26，共 27 个 token。"emma" 编码为：BOS(26)→e(4)→m(12)→m(12)→a(0)→BOS(26)。整数之间无大小语义，只是符号 ID。生产环境（如 GPT-4 的 tiktoken）按字符块切分，词汇表约 10 万 token，原理相同。

- **滑动窗口训练样本构造**：每个名字按位置滑动，产生多个"输入→预测目标"对。"emma" 产生 5 个训练样本：[BOS]→e、[BOS,e]→m、[BOS,e,m]→m、[BOS,e,m,m]→a、[BOS,e,m,m,a]→BOS。这是所有语言模型（含 ChatGPT）的统一训练范式。

- **Logits + Softmax → 概率分布**：模型每个位置输出 27 个原始分数（Logits，可正可负）。Softmax 做两件事：① 对每个分数取指数（变正数）② 归一化使总和为 1。代码技巧：先减去最大值再做指数，数学等价但防止数值溢出。大分数经指数放大后主导分布，这是模型能"确信"某答案的原因。

- **交叉熵损失（Cross-Entropy Loss）**：衡量"模型给正确答案分配的概率有多低"。正确答案概率 90% → 损失 ≈ 0.1；概率 1% → 损失 ≈ 4.6；概率 100% → 损失 = 0。训练目标就是让损失持续下降。

- **反向传播（Backpropagation）**：模型共 4192 个参数（4192 个可调旋钮）。前向计算像推多米诺骨牌，反向传播从损失倒推每个参数的"责任分数"（梯度）。若参数在多处被使用，其梯度是所有路径责任之和。PyTorch 的 `loss.backward()` 即此算法。

- **Embedding（嵌入）**：每个 token ID 映射到 16 维向量（16 个数字的"性格档案"），位置也有独立的 16 维向量，两者相加作为网络输入。初始值随机，训练后行为相似的 token（如元音字母）自动聚拢到相近向量，无需人工标注。

- **注意力机制（Attention）**：每个 token 生成三个向量——Query（我在找什么）、Key（我包含什么）、Value（被选中时提供什么）。用 Query 匹配所有历史位置的 Key，加权取对应 Value。严格因果掩码：位置 2 不能看位置 4（未来信息不可见），保证自回归生成合法。多头注意力（Multi-Head Attention）并行运行 4 个注意力头，各自学习不同模式（如：最近 token、BOS 标记、元音位置），结果拼接输出。

- **完整前向流程（每个 token 经历的 8 道工序）**：① Embedding（token 向量 + 位置向量）→ ② RMSNorm（均方根归一化，统一向量尺度）→ ③ Multi-Head Attention（跨位置信息交换）→ ④ 残差连接（保留原始输入）→ ⑤ RMSNorm → ⑥ MLP（每个位置独立非线性变换，"独立思考"）→ ⑦ 残差连接 → ⑧ 输出 27 个 Logits。残差连接是深度网络能训练的关键，相当于给信息开高速通道，防止梯度消失。

- **训练循环（重复 1000 次）**：随机取一个名字 → 编码为数字序列 → 前向计算每个位置的预测 → 计算交叉熵损失 → `loss.backward()` 反向传播 → Adam 优化器更新参数。Adam 优化器记录每个参数的历史梯度，对稳定参数走大步，对震荡参数走小步。损失从 ≈3.3（随机猜 27 选 1 的理论值）降至 ≈2.37。

- **生成流程（推理阶段）**：从 BOS 开始 → 前向计算得 27 个概率 → 按概率随机采样一个 token → 追加到序列 → 重复 → 直到采样到 BOS（结束标记）。温度参数控制随机性：0.5 保守（常见名字）、1.0 原始概率、2.0 大胆（奇异组合）。名字生成最佳温度约 0.5。

### 🎯 关键洞察

**"猜下一个词"就是一切**：ChatGPT 的本质与这个 200 行脚本完全相同——给定已有序列，预测下一个 token 的概率分布。对话中你的输入是上文，模型的回复是统计续写。所有"智能"都来自在海量数据上反复执行这个循环后，参数自动编码的统计规律。

**规模是唯一变量**：

| 维度 | MicroGPT | ChatGPT 量级 |
|------|----------|-------------|
| 训练数据 | 32000 个名字 | 整个互联网文本 |
| 词汇表 | 27 个字符 | ~10 万子词 |
| 参数量 | 4192 个 | 数千亿个 |
| 层数 | 1 层 | 数百层 |
| 硬件 | 笔记本 CPU | 数千 GPU 集群 |
| 训练时间 | 几分钟 | 几个月 |

核心循环完全一致：分词 → 嵌入 → 注意力 → 计算 → 预测下一个 token → 衡量损失 → 反向传播 → 调整参数 → 重复。

### 📦 配置/工具详表

| 模块 | 关键设置/代码 | 预期效果 | 注意事项 |
|------|-------------|---------|---------|
| Tokenizer | 26字母(0-25) + BOS(26)，共27个token | 文本↔整数序列互转 | 整数无大小语义，只是符号ID |
| Embedding | token嵌入16维 + 位置嵌入16维，相加 | 离散符号→连续向量 | 初始随机，训练后自动聚类 |
| Multi-Head Attention | 4个注意力头并行，因果掩码（不看未来） | 跨位置信息聚合 | 位置i只能attend位置0~i |
| Softmax | `exp(x - max(x)) / sum(exp(x - max(x)))` | Logits→概率分布 | 减max防数值溢出，数学等价 |
| 损失函数 | 交叉熵：`-log(p_correct)` | 量化预测偏差 | 从≈3.3降至≈2.37 |
| 优化器 | Adam，1000步训练循环 | 自适应步长更新参数 | 稳定参数大步，震荡参数小步 |

### 📝 避坑指南

- ⚠️ Softmax 数值溢出：直接对大数取指数会溢出，必须先减去最大值：`exp(x - max(x))`，数学结果等价。
- ⚠️ 因果掩码不能省：训练时位置 i 必须屏蔽位置 i+1 及之后，否则模型"作弊"看到未来，推理时无法自回归生成。
- ⚠️ 残差连接是深度网络的命：去掉残差连接，梯度在多层传递中消失，网络根本无法收敛训练。
- ⚠️ token ID 不是数值：编号 4 不代表"比编号 2 大"，直接用整数做运算无意义，必须先过 Embedding 层转成向量。
- ⚠️ 温度过低或过高都有问题：温度趋近 0 → 总输出同一个最高概率 token，生成退化；温度过高 → 概率分布趋于均匀，输出随机乱码。

> 文档 ID: `JQGww6DQhiipMFkHH92cB2jUn0c`

**来源**: Attention is All You Need：一篇论文如何改变了整个AI世界 | **时间**: 2017年（论文发表年份，文章未注明具体日期） | **原文链接**: 无

### 📋 核心分析

**战略价值**: 用最小代价彻底搞懂 Transformer 架构——当今所有主流大模型（GPT、Claude、DeepSeek、豆包）的底层基础，从 RNN 痛点出发，逐层拆解 Self-Attention → Multi-Head Attention → 完整 Encoder-Decoder 架构。

**核心逻辑**:

- **RNN 的两大死穴**：①串行计算，后一个字必须等前一个字处理完，GPU 并行能力完全浪费；②长距离记忆衰减，句子越长越"忘事"，LSTM 也没彻底解决。
- **Transformer 的诞生契机**：2016 年 Google 工程师乌兹在食堂吐槽 RNN 太慢，同事提议"Why not just use attention?"，实验后发现注意力机制可以独立存在、效果更好、还能并行计算。
- **Self-Attention 三角色 Q/K/V**：每个字同时扮演三个角色——Key（我是谁）、Query（我想找谁）、Value（我能提供什么信息）。"它"的 Query 碰到"书"的 Key 时匹配度高，于是给"书"分配更高注意力权重，获取"书"的 Value，从而理解指代关系。
- **核心公式拆解**：`Attention(Q,K,V) = softmax(Q × K^T / √d) × V`，四步：①点积算相关性 ②除以√d 防数值爆炸 ③softmax 转概率 ④乘 V 加权求和。
- **向量的本质是语义空间中的点**：同一个"苹果"在"我吃了一个苹果"中向量靠近"水果/食物"，在"我买了苹果手机"中靠近"科技/品牌"——Attention 让向量动态表示上下文含义。
- **并行 vs 串行的本质区别**：RNN 必须链式传递（h3 依赖 h2 依赖 h1），Transformer 一次矩阵运算同时算出所有字的新向量，V1/V2/V3 同时可用，无需等待。
- **位置编码解决顺序问题**：并行计算不知道字的先后顺序，解决方案是在 Embedding 前给每个位置加正弦/余弦函数组合的独特编码，让模型区分"我爱你"和"你爱我"。
- **升维是语义表达的关键**：原始论文用 512 维向量表示一个字（前几维词性、中间几维情感倾向、后面几维语法角色等），GPT-3 已升至 12288 维。单头注意力只能关注一种关系，Multi-Head Attention 同时派出 8 个视角，每头处理 64 维，最后拼成 512 维，覆盖语法/指代/情感等多种关系。
- **Encoder 重复 6 次，Decoder 重复 6 次**：这是论文原始设定，每层包含 Multi-Head Attention + 前馈神经网络（两层全连接 + 激活函数）。
- **Softmax 的指数放大效应**：分数 5:3:1 经过 e^x 后变成概率 86.7%:11.7%:1.6%，高分被大幅拉开，确保模型输出有明确倾向性。

### 🎯 关键洞察

**为什么 Attention 能替代 RNN？逻辑链**:

原因 → RNN 的信息传递是链式的，距离越远衰减越严重，且无法并行。
动作 → Attention 让每个字直接与所有其他字建立连接，一次矩阵运算完成全局信息交换。
结果 → 长距离依赖问题消失（"它"直接看到"书"），训练速度大幅提升（GPU 并行利用率提高）。

**为什么大模型"不会思考"？**:

本质是：给定上下文，通过 Q/K/V 计算注意力权重，加权融合信息，最后 softmax 选出概率最高的下一个 token。整个过程是矩阵运算的叠加，没有"理解"，只有"计算推测"。

### 🛠️ 操作流程（以翻译"我爱你"→"I love you"为例）

1. **准备阶段（Embedding + 位置编码）**
   - 每个字转成高维向量（原论文 512 维）
   - 叠加位置编码，标记"我"=位置1，"爱"=位置2，"你"=位置3

2. **Encoder 处理（重复 6 次）**
   - Multi-Head Attention（8 头并行）：每个字同时与所有字计算 Q×K^T/√d，softmax 得权重，×V 得新向量
   - 前馈神经网络：每个字独立深化语义
   - 输出：对"我爱你"的深度语义理解报告

3. **Decoder 生成（重复 6 次，逐 token 输出）**
   - 输入 `<start>` → 带遮罩的 Self-Attention（只看已生成内容）→ Cross-Attention 查 Encoder 输出（Q 来自 Decoder，K/V 来自 Encoder）→ FFN → softmax → 输出"I"
   - 输入"I" → 同上流程 → 输出"love"（Cross-Attention 重点关注"爱"）
   - 输入"I love" → 同上流程 → 输出"you"（Cross-Attention 重点关注"你"）
   - 输入"I love you" → 输出 `<end>`，停止

**Self-Attention 伪代码**:
```python
def self_attention(Q, K, V):
    scores = Q @ K.transpose() / sqrt(d_k)  # @ 是 Python 3.5+ 矩阵乘法运算符
    attention_weights = softmax(scores)
    output = attention_weights @ V
    return output
```

### 📝 避坑指南

- ⚠️ **不除以√d 会梯度消失**：Q×K 点积值随维度增大而变大，不缩放会导致 softmax 输出极端（接近 0 或 1），梯度消失，训练失败。
- ⚠️ **Decoder 必须加遮罩**：生成阶段如果能看到未来 token，等于训练时作弊，推理时无法复现，模型无法实际使用。
- ⚠️ **单头注意力表达能力有限**：一个头只能关注一种关系，复杂语言任务必须用 Multi-Head，原论文最小配置是 8 头。
- ⚠️ **3 维向量只是演示**：实际 512 维起步，GPT-3 用 12288 维，维度不够语义信息装不下。
- ⚠️ **Cross-Attention 的 Q/K/V 来源不同**：Q 来自 Decoder 当前状态，K 和 V 来自 Encoder 输出，混淆来源会导致架构理解错误。
- ⚠️ **Transformer 不会"理解"语义**：本质是矩阵运算叠加后的概率预测，输出的是"统计上最可能的下一个 token"，不是"思考后的答案"。

```
~/.openclaw/
├── openclaw.json           # 全局路由和通道配置
├── skills/                 # 全局技能库（nano-banana-pro, seedance2.0）
├── workspace-lead/         # 大总管工作区（含 SOUL.md, AGENTS.md）
├── workspace-geo/          # GEO内容优化师工作区
├── workspace-reddit/       # Reddit营销专家工作区
└── workspace-tiktok/       # TikTok爆款编导工作区
```

**步骤二：写入核心配置文件 `openclaw.json`**

```json
{
  "channels": {
    "feishu": {
      "enabled": true,
      "connectionMode": "websocket",
      "dmPolicy": "open",
      "accounts": {
        "lead":    { "appId": "cli_111", "appSecret": "xxx" },
        "geo":     { "appId": "cli_222", "appSecret": "xxx" },
        "reddit":  { "appId": "cli_333", "appSecret": "xxx" },
        "tiktok":  { "appId": "cli_444", "appSecret": "xxx" }
      }
    }
  },
  "bindings": [
    { "agentId": "lead",           "match": { "channel": "feishu", "accountId": "lead" } },
    { "agentId": "geo-optimizer",  "match": { "channel": "feishu", "accountId": "geo" } },
    { "agentId": "reddit-spec",    "match": { "channel": "feishu", "accountId": "reddit" } },
    { "agentId": "tiktok-director","match": { "channel": "feishu", "accountId": "tiktok" } }
  ],
  "tools": {
    "agentToAgent": {
      "enabled": true,
      "allow": ["lead", "geo-optimizer", "reddit-spec", "tiktok-director"]
    }
  }
}
```

**步骤三：写入人设文件（SOUL.md / AGENTS.md）**

大总管的 `AGENTS.md`（团队通讯录）：
```markdown
# AGENTS.md - 跨境电商协同手册

你是大总管，负责接收老板指令并使用 `sessions_send` 跨域分发。

- **geo-optimizer**：负责撰写符合GEO规则的产品内容。
- **reddit-spec**：负责社区长尾流量劫持。
- **tiktok-director**：负责调用 `nano-banana-pro` 和 `seedance2.0` 生成短视频。

⚠️ 强制纪律：严禁你自己执行底层任务，必须委派！当多平台需要同时运营时，对不同成员并发调用 `sessions_send`。
```

GEO优化师的 `SOUL.md`：
```markdown
# SOUL.md - GEO内容优化师

## 核心职责
你面对的是基于大型语言模型的生成引擎，而不是传统搜索引擎。
你需要将产品内容在 Perplexity、Google SGE 等引擎中的可见性最大化。

## 工作底线
- **绝对禁止关键词填充**：传统SEO的关键词填充手段在GEO中几乎无效果，甚至可能有害。
- **强制数据支撑**：在所有产品描述中必须加入具体的定量数据，而非定性描述。
- **添加权威引文**：在内容中明确引用可靠来源，并添加来自可信来源的直接引文。
```

TikTok编导的 `SOUL.md`：
```markdown
# SOUL.md - TikTok爆款编导

## 核心职责
利用 Seed 2.0 模型能力，复刻具有极强转化率的UGC带货视频。

## 创作原则
- **脚本设计**：必须输出包含痛点展示、产品细节到户外场景的25宫格分镜故事板。
- **运镜与细节**：精准设计出带有轻微自然呼吸抖动的手持拍摄感。必须包含细节特写动作，
  例如向下按压床垫清晰展示回弹性和支撑力。
- **工具调用**：脚本完成后，强制调用全局的 `nano-banana-pro` 生成高保真配图，
  然后将图片资产转交 `seedance2.0` 技能库生成带旁白音频的最终成片。
```

**步骤四：安装技能库并启动**

```bash
# 将 nano-banana-pro 和 seedance2.0 安装到全局 skills 目录
# 然后重启网关
openclaw gateway restart
```

把配置好的4个飞书机器人拉进同一个群，@大总管 发送任务指令，流水线自动启动。

### 📝 避坑指南

- ⚠️ **飞书权限"发布即生效"假象**：权限变更后必须创建新版本并申请发布，仅保存不生效。
- ⚠️ **Bot-to-Bot @无效**：飞书有防机器人死循环机制，Agent A 在群里 @Agent B，B 后台收不到推送。必须用 `sessions_send` 走底层暗线，群里只做明线进度汇报。
- ⚠️ **Skill 层级加载优先级**：公共技能必须放 `~/.openclaw/skills/`，私有技能放 Agent 专属子目录。混放会导致 Agent 误调用他人 API 密钥或找不到工具。
- ⚠️ **Agent 设计原则**：不要按平台分 Agent（每平台一个），要按职能分（Role-based）。内容策略官统一输出，再下发给各平台分身做格式适配，保证品牌调性一致性。
- ⚠️ **大总管严禁亲自执行底层任务**：必须在 AGENTS.md 中写明强制委派纪律，否则大总管会绕过子 Agent 自己干，破坏流水线隔离。
