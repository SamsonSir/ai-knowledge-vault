### 📋 核心分析

**战略价值**: GPT-5.4 首次将 Codex 级代码能力与强世界知识合并进同一主线模型，且可直接走 ChatGPT/Codex 订阅额度，成为 OpenClaw 默认基座模型的最优解。

**核心逻辑**:

- **Agent 基座模型的三要素**：代码能力 + 世界知识 + 多模态理解，三者缺一则 Agent 能力出现明显短板。GPT-5.3-Codex 代码强但世界知识差，说"天书"；Claude Opus 4.6 均衡但 API 极贵且封 OpenClaw 订阅额度；GPT-5.4 是第一个三项都过关的 OpenAI 主线模型。
- **GPT-5.4 = GPT-5.3-Codex 代码力 + 比 GPT-5.2 更强的世界知识 + 更强工具使用能力 + Codex 订阅额度可用**，四项合一，直接锁定 OpenClaw 天选基座。
- **GDPval 83.0%**：测 AI 在真实工作任务（金融、法律等 44 种职业）中的表现。GPT-5.4 Thinking 83.0% > Claude Opus 4.6 78.0% > GPT-5.3-Codex 70.9%。说明 GPT-5.4 能用人话处理真实业务场景，不再是只会写代码的"天书机器"。
- **SWE-Bench Pro 57.7%**：测真实软件工程问题（四种编程语言）。GPT-5.4 Thinking 57.7% vs GPT-5.3-Codex 56.8%，基本持平，代码能力没有退步。
- **OSWorld-Verified 75.0%**：测 AI 像人一样操作电脑（鼠标点击、键盘输入、跨应用切换）。GPT-5.4 Thinking 75.0% > Claude Opus 4.6 72.7%，且操作速度"快的离谱"。
- **ToolAthon 54.6%**：测 AI 使用工具的能力（Agent 核心指标）。GPT-5.4 Thinking 54.6% vs Claude Sonnet 4.6 44.8%，领先近 10 个百分点。
- **上下文窗口从 40 万 token 升至 100 万 token**：对 Agent 执行长任务至关重要，防止"干着干着忘事儿"。注意：超过 27 万 token 后额度按 2 倍计算，但 Codex 给的额度本身很充裕，实际影响有限。
- **原生计算机使用能力**：GPT-5.4 是 OpenAI 第一个内置原生计算机使用能力的主线模型，代码操控（Playwright 等库）+ 视觉操控（截图 → 鼠标/键盘命令）双路并行，可直接原生操控电脑上绝大多数软件。
- **工具搜索（Tool Search）**：模型不再在 prompt 里预载所有工具定义，而是持有轻量工具列表，按需查找并追加定义。OpenAI 内测结果：**在保持相同准确率的前提下，总体 token 使用量减少 47%**，直接降低成本、加快响应、减少上下文污染。
- **OpenAI 明确支持第三方工具（含 OpenClaw）使用 Codex 订阅额度**，是御三家中唯一公开表态不封号、支持第三方调用的厂商，Claude 封了 OpenClaw 的订阅额度（只能走 API，极贵），Google 大批量封号导致反代方案失效。

### 🎯 关键洞察

**为什么 GPT-5.3-Codex 不适合做 OpenClaw 默认模型**：
Codex 是编程特化模型，世界知识甚至不如 GPT-5.2，导致它在规划任务时输出"天书"——技术术语堆砌、缺乏业务语境、非程序员根本看不懂。作者实测让 Codex 审查 AI 热点网站项目的文档规范和代码库，输出文档与 Claude Opus 4.6 的输出对比"一目了然"，Codex 完全不说人话。接入 OpenClaw 当默认模型等于"灾难"。

**为什么 Claude 不适合做 OpenClaw 默认模型**：
Claude Opus 4.6 能力均衡，但 Anthropic 封禁了 OpenClaw 使用 Max Plan 订阅额度（只能在 Claude Code 里用），OpenClaw 只能走 API Key，成本极高，小团队/个人不可持续。Google Antigravity 反代方案也因大批量封号而失效。

**GPT-5.4 的"人话"体验**：作者在 Codex 里实测，GPT-5.4 输出风格明显口语化，例如面对爬取 OpenAI 官网视频的任务，模型自己说"这种活最烦"、"省的跟 Cloudflare 互相折寿"——这正是世界知识和语言风格回归正常的直接体现。

**前端审美**：GPT-5.4 在 Codex 里生成的前端有进步，但仍不如 Claude Opus 4.6 和 Gemini。写作风格仍有"爱用排比句"的怪癖。

## 📗 文章 2

**来源**: 陶哲轩最新演讲：AI 不是来抢数学家蛋糕的，是来把蛋糕做大的 | **时间**: 2026-03-05 | **原文链接**: `https://mp.weixin.qq.com/s/B2zZI9FT...`

---

### 📋 核心分析

**战略价值**: 陶哲轩用亲身主导的 2200 万题项目，给出了 AI+形式验证重构数学协作工作流的可复刻范式。

**核心逻辑**:

- **数学是最保守的学科**：1826 年 Cauchy 的法语教科书今天研究生仍可直接使用；勾股定理两千年后陶哲轩本人还在用；数学家是学术界唯一还在大规模使用粉笔黑板的群体（摄影师 Jessica Wynne 专门出版了《Do Not Erase》记录这一现象，Princeton University Press，2021 年）
- **数学协作的三大结构性壁垒**：① 门槛高——通常需要数学博士才能理解问题；② 容错率为零——100 份贡献中 1 份错误即毁掉整个证明；③ 工作流无法扩展——两三人黑板讨论的模式无法 scale 到 100 人在线协作
- **协作规模差距悬殊**：其他学科论文平均作者数爆炸式增长，数学从 1.5 人缓慢爬到 2.5 人，远远落后
- **工作模式正在从"案例研究"转向"大规模调查"**：从一次挑一个问题花几个月出一篇论文，转变为同时处理数百乃至数千个问题，报告问题群体的统计规律
- **形式验证（Lean）是打破信任壁垒的核心武器**：自动判断数学论证对错，过滤垃圾贡献，让匿名贡献者、非专业人士的提交都可被接受，讨论精度达到"原子级别"（可定位到九步证明中具体哪一步报错）
- **LLM 不可靠但可被驯化**：LLM 能解奥赛题，却会算错 7×4+8×8；每个数学家试用后都得出"不可靠性太大"的结论；解法是接入验证器构成"生成→检验→反馈→纠正"闭环
- **AlphaEvolve = LLM + 遗传算法**：Google DeepMind 工具，底层是 Gemini 代码生成 + 进化算法，已在六边形堆叠等有限维优化问题上打破人类记录；陶哲轩团队 2025 年 11 月在 arXiv 发表了用其攻克 67 个数学问题的论文；无穷维优化方向有新进展但暂未公开
- **AI 的最大价值在"中等难度问题长尾"**：不是替代数学家攻克最难问题，而是处理人类没有人力覆盖的海量中等难度问题，先跑一遍把简单的解决，再把真正难的上报人类专家
- **实际起作用的是平凡技术**：整个 2200 万题项目中，最关键的工具不是 AI，而是 GitHub（存储 Lean 代码）和 Zulip（协作讨论），"基本协作平台简直太有用了"
- **AI 使用边界**：只能在能信任或能检查输出的范围内使用，超出即"too risky"；AI 与更广泛协作体系结合时效果最大，单独辅助个体数学家效果有限

---

### 🎯 关键洞察

**为什么形式验证是关键而不是 AI 本身**

逻辑链：数学协作失败的根本原因不是人不够多，而是"无法信任陌生人的贡献"。AI 生成内容同样不可信任。形式验证（Lean）把"信任人"替换成"信任系统"，一次性解决了两个问题：① 众包协作的质量控制；② AI 输出的可靠性验证。没有验证器，AI 在数学中就是玩具；有了验证器，AI 才能进入生产流程。

**为什么"中等难度长尾"比"攻克最难问题"更有价值（中期）**

原因：最难的问题（如黎曼猜想）需要顶级数学家的创造性洞察，AI 目前无法替代。但数学中存在大量"一个研究生花一小时能解、但没有 2200 万个研究生"的问题，这些问题此前根本没有人力覆盖。AI 填补的是空白，而不是竞争存量，所以是"把蛋糕做大"而非"抢蛋糕"。

### 🛠️ 操作流程

**复刻"等式理论项目"式大规模数学协作的 SOP**

1. **准备阶段**
   - 选择可程序化生成的大规模问题集（等式理论项目：4694 个 magma 等式定律两两蕴含关系，共 22,028,942 对）
   - 确认每道题单独可解（研究生 1 小时级别），但总量超出人力上限
   - 建立 GitHub 仓库，规定所有提交必须包含 Lean 形式化证明
   - 建立 Zulip 工作区，按问题类型分话题线程

2. **核心执行**
   - 将问题集模块化拆分，允许参与者认领子集（不需要理解项目全貌）
   - 设立并公开展示单一核心指标（项目第 16 天：剩余未解决问题从 2200 万降至 888 个）
   - 分工角色：① 写人类可读证明；② 将人类证明翻译成 Lean 代码；③ 写程序批量攻克（一次处理 10 万道）；④ 将计算机证明转化回人类证明
   - 接受来自陌生人、非专业数学家（CS 研究生、本科生、高中生）的贡献，所有贡献通过 Lean 自动验证，无需人工审核信任

3. **验证与优化**
   - Lean 自动过滤错误提交，报错精确到具体步骤
   - 在 Zulip 上针对具体报错步骤进行原子级别讨论
   - 追踪剩余未解决数量，参与者自发将数字往下推，无需全局协调
   - 最终产出：每道题均有证明或反证，三个月完成全部 2200 万道

### 💡 具体案例/数据

**Equational Theories Project（等式理论项目）**
- 启动时间：2024 年 9 月 25 日，GitHub 上公开
- 最终论文：2025 年 12 月上传 arXiv
- 规模：4694 个 magma 等式定律，两两蕴含关系 22,028,942 对
- 参与者：50 人，大多数陶哲轩从未见过面，大多数非专业数学家
- 完成时间：三个月，每道题均有证明或反证
- 对比：此前同类项目最多处理约 20 个问题
- 第 16 天节点：剩余未解决问题降至 888 个

**AlphaEvolve 六边形堆叠问题**
- 问题：11 个正六边形能装进的最小正六边形尺寸
- 结果：打破了此前人类找到的最优记录
- 论文：陶哲轩等人，2025 年 11 月，arXiv，使用 AlphaEvolve 攻克 67 个数学问题

**LLM 算术错误示例**
- 问题：7×4+8×8 = ?
- LLM 表现：算错 → 被指出 → 道歉 → 给出正确答案
- 结论：能解奥赛题，但基础算术不可靠，必须配合验证器

### 📝 避坑指南

- ⚠️ 不要单独用 LLM 做数学：每个数学家试用后结论一致——不可靠性是太大的问题，必须接入验证器构成闭环才有实用价值
- ⚠️ 形式验证不是可选项：没有 Lean 这类工具，众包数学协作的质量控制无解，100 份贡献中 1 份错误即毁掉整个证明
- ⚠️ AI 使用有明确边界：只能在能信任或能检查输出的范围内使用，超出即"too risky"，不要因为 AI 能解奥赛题就过度信任
- ⚠️ AlphaEvolve 目前主要适用于有限维优化：堆叠问题等是数学家关心问题的很小子类，无穷维方向有进展但尚未公开
- ⚠️ 不要高估 AI 在"最难问题"上的价值：中期内 AI 的最大生产力在"中等难度长尾"，而非替代顶级数学家的创造性工作
- ⚠️ 平台选择比 AI 工具更基础：整个 2200 万题项目中，GitHub + Zulip 的贡献比 AI 工具更关键，不要本末倒置

**来源**: 谷歌发布官方 CLI，可操作所有谷歌文档 | **时间**: 2026-03-05 | **原文链接**: `https://mp.weixin.qq.com/s/beVJFuvy...`

---

### 📋 核心分析

**战略价值**: Google Workspace CLI 是谷歌官方组织发布的命令行工具，将全套 Workspace API 封装为 JSON 输出的 shell 命令，专为 AI Agent 调用设计，彻底解决「GUI 对 Agent 不友好」的结构性问题。

**核心逻辑**:

- **发布背景**: 2026年3月5日上线 GitHub，挂在 `googleworkspace` 官方 org 下，主要开发者 Justin Poehnelt 是谷歌 Workspace 开发者关系团队成员，Google Cloud AI Director Addy Osmani 亲自在推特背书。README 声明「非官方支持产品」，定位是「谷歌认可，但不做服务承诺」。发布后数小时内 Star 从 2700 飙升至 3700+。
- **核心问题定义**: 过去30年生产力软件（邮件、文档、日历、云盘）全为人眼和鼠标设计，Agent 无法点击界面。Agent 需要的是：发一个命令 → 收到结构化 JSON → 继续执行。该 CLI 正是这层转换层。
- **动态命令生成机制**: 不预设命令列表，每次运行时直接读取谷歌官方 API 目录（Discovery Service）实时生成命令。谷歌新增 API 后，工具自动可用，无需等版本更新。
- **覆盖产品范围**: Drive、Gmail、Calendar、Sheets、Docs、Chat、Admin，以及所有其他 Workspace API。
- **认证方案全覆盖**: 本地登录、CI 环境、服务器端 Service Account、直接传入已有 token，凭证本地加密存储。
- **三种 Agent 接入方式**: 直接 shell 调用、MCP Server、Skills 文件包（详见操作流程）。
- **安全设计**: 支持接入 Google Cloud Model Armor，在 API 返回内容到达 Agent 之前自动扫描 prompt injection 攻击，可配置为警告或直接拦截。
- **AGENTS.md 规范**: 仓库根目录与 README.md 并排放置 `AGENTS.md`，专为 AI 编程助手（如 Claude Code）写的贡献指南，明确要求「永远假设输入可能是恶意的」，预设 AI 参与维护代码库且调用方为 Agent。
- **竞品对比**: OpenClaw 开发者 Peter Steinberger 数月前自研了同类工具 `gog`（gogcli.sh），宣布将跑 eval 测试对比两者 Agent 适配性。命令风格差异显著（见下方表格）。
- **版本状态**: 当前 v0.3.4，活跃开发中，v1.0 前可能有 breaking changes。

---

### 🎯 关键洞察

**「界面友好 ≠ Agent 友好」是结构性矛盾**，不是工程问题。GUI 的每一层抽象（按钮、弹窗、拖拽）都是为人类感知设计的信号，对 Agent 是噪音。CLI + JSON 输出才是 Agent 的母语。这个工具的价值不在于「方便」，在于打通了 Agent 和 Workspace 之间的语义鸿沟。

**Discovery Service 动态生成命令**是关键架构决策：传统 CLI 工具需要人工维护命令列表，API 更新就要发版。这里直接读 API 目录，工具的生命周期和 Google API 的生命周期解耦，维护成本趋近于零。

**AGENTS.md 文件**预示一个趋势：开源项目将标配两份贡献指南，一份给人（CONTRIBUTING.md），一份给 AI（AGENTS.md）。gog 仓库已同步跟进，格式一致。

### 🛠️ 操作流程

**1. 准备阶段**

- 确认 Google Workspace 账号及对应 API 权限
- 选择认证方式：本地登录 / CI 环境 / Service Account / 已有 token

**2. 核心执行 — 常用命令示例**

列出最近 10 个 Drive 文件：
```bash
gws drive files list --params '{"pageSize": 10}'
```

新建 Sheets 表格：
```bash
gws sheets spreadsheets create --json '{"properties": {"title": "Q1 预算"}}'
```

发送 Chat 消息（`--dry-run` 为预览模式，不实际发送）：
```bash
gws chat spaces messages create \
  --params '{"parent": "spaces/xyz"}' \
  --json '{"text": "部署完成"}' \
  --dry-run
```

启动 MCP Server（仅启用 drive、gmail、calendar）：
```bash
gws mcp -s drive,gmail,calendar
```

一键安装全部 Skills：
```bash
npx skills add https://github.com/googleworkspace/cli
```

**3. 验证与优化**

- 所有命令输出（含报错）均为 JSON，直接用 `jq` 解析验证
- `--dry-run` 参数可在任何写操作前预览，避免误操作
- v1.0 前注意 breaking changes，锁定版本号后再上生产

> 文档 ID: `BzX8wlTggiYjULkU8eFcDd3wnPc`

**来源**: Anthropic 官方出品 Claude 101：一份从入门到上手的完整学习指南 | **时间**: 2026-03-05 | **原文链接**: `https://mp.weixin.qq.com/s/mg2w6j33...`

### 📋 核心分析

**战略价值**: Anthropic Academy 的 Claude 101 免费课程（13节）系统覆盖 Projects / Skills / Artifacts / Connectors / Enterprise Search / Research Mode 六大模块，是目前最完整的官方 Claude 上手路径。

**核心逻辑**:

- **提示词三步法（Set the Stage → Define the Task → Specify Rules）**：先给背景（你的角色/行业/前提），再定任务（写作/分析/构建，越具体越好），最后指定风格或直接给样例参考。上下文越充分，Claude 理解需求越快。
- **迭代思维是核心心智模型**：第一个提示词只是对话起点，不追求一次完美。正确姿势：审阅初稿 → 找出有效/无效部分 → 针对性反馈（"删掉前两段，让结论更具行动导向"比"缩短篇幅"有效得多）→ 必要时直接开新对话而非硬扭。
- **4D AI 素养框架**（Delegate / Describe / Discern / Due Diligence）：① Delegate：判断哪些任务给 AI、哪些留给人，需理解 AI 能力边界；② Describe：精确定义输出、流程、行为方式；③ Discern：批判性评估输出的质量、准确性、适用性；④ Due Diligence：对 AI 辅助成果承担责任，保持透明度。
- **Evals 评估方法论**：收集 5–10 个高频任务样本 → 用样本测试 Claude 输出 → 与自己的成果对比（关键信息是否抓住、语气风格是否对、缺什么）→ 根据对比结果迭代提示词。这是判断 Claude 是否适合某个特定工作流的唯一可靠方法。
- **Projects = 知识中心**：每个 Project 有独立聊天记录 + 知识库 + 自定义说明。知识库用 RAG 技术处理，容量几乎无限制，支持各种文件格式及 Google Drive 连接（单文件最大 30MB）。注意：项目内各聊天之间上下文默认不共享，需手动加入知识库才能跨聊天复用。
- **Skills = 流程编码包**：Skills 文件夹包含指令、脚本、资源，Claude 动态加载后以可重复方式执行特定任务。Anthropic 官方 Skills 覆盖 Excel / Word / PowerPoint / PDF 创建，付费用户自动可用。自定义 Skills 通过对话创建：描述需求 → 回答 Claude 的流程问题 → 上传参考资料（模板/品牌素材）→ Claude 生成 ZIP → 上传到设置启用。
- **Artifacts = 可交互交付物**：满足"内容超过 15 行 + 自成一体 + 可能被复用"时自动生成，支持 Markdown / 代码片段 / HTML 页面 / SVG / Mermaid 图表 / React 组件六类。可在预览和代码视图间切换，支持复制、下载、发布链接、remix。
- **模型选择策略**：Opus → 复杂任务；Sonnet → 日常任务；Thinking 模式 → 需要深度推理；Research 模式 → 多角度调研。
- **Enterprise Search（企业搜索）**：同时搜索 SharePoint / Slack / Gmail / Google Drive，整合成统一答案并标注来源。权限隔离：只显示你在原有工具中已有访问权限的内容，数据不被单独索引或存储。设置分两步：管理员配置连接器 → 各用户用自己账号分别认证。
- **Research Mode 工作机制**：Claude 独立决定下一步调查方向，同时进行多项相互衔接的搜索，从不同角度探讨问题，输出带引文的综合报告。可与 Extended Thinking 组合使用（规划方法 + 信息收集双管齐下），界面有计时器 + 可展开的思考过程面板。

---

### 🎯 关键洞察

**Projects vs Skills vs Artifacts 三者关系**：

- Projects 提供"内容"（知识库里的客户资料、品牌文档、历史数据）
- Skills 提供"方法"（如何执行"客户通话准备"这个流程的具体步骤）
- Artifacts 是"结果"（执行完流程后生成的那份 PPT / 报告 / 代码）

三者组合逻辑：Skills 从 Projects 知识库中提取信息 → 按编码好的流程执行 → 输出 Artifact。

**Artifacts 提示词精度决定输出质量**：
- 低精度："创建一个预算追踪器"
- 高精度："创建一个月度预算追踪器，按类别输入支出，查看饼图明细，超支时提醒"
- 另一个关键变量：告诉 Claude 最终用户是谁（"此流程图适用于新员工" vs "此流程图适用于工程团队"，产出完全不同）

### 🛠️ 操作流程

**1. 准备阶段：建立第一个 Project**
- 新建 Project，撰写项目目标（说清楚这个项目是干什么的、面向谁）
- 上传相关文档（支持各种格式 + Google Drive，单文件 ≤30MB）
- 写好项目说明：语气要求 + 聚焦领域 + 禁止事项
- 适合场景：产品上市计划、内容创作工厂、教育课程开发、金融预算计划、家庭装修改造

**2. 核心执行：用三步法写提示词**
- Set the Stage：我是[角色]，在[行业/公司]，背景是[前提条件]
- Define the Task：我需要你[写作/分析/构建]，具体是[详细描述]
- Specify Rules：用[风格/语气]，参考[样例]，避免[禁忌]

**3. 创建自定义 Skill**
- 直接跟 Claude 对话："我想创建一个 Skill，用于[具体工作流]"
- 回答 Claude 关于流程细节的追问
- 上传参考资料（模板、风格指南、品牌素材）
- 下载 Claude 生成的 ZIP 文件
- 进入设置上传 ZIP，启用 Skill

**4. 验证与优化：Evals 流程**
- 收集 5–10 个高频任务的真实样本
- 用样本测试 Claude 输出
- 对比维度：关键信息覆盖率 / 语气风格匹配度 / 缺失内容
- 根据对比结果修改提示词，重复测试直到满意

### 📋 核心分析

**战略价值**: 用扣子（Coze）的 OpenClaw 龙虾 + 扣子编程，通过纯自然语言 Vibe Coding，从零搭建一个有完整前后端、数据库、通知系统的 Agent 招聘平台并上线部署。

**核心逻辑**:

- **OpenClaw 本质是一个有记忆、有人格、能调用工具的个人 AI Agent**，部署在扣子平台，可接入飞书/钉钉/Telegram，随时随地可触达
- **人格注入是关键第一步**：通过结构化 Prompt（核心定义 → 神经连接协议 → 人格情感 → 思维模型）让龙虾拥有专属人格，而非通用 AI，核心是填写 `{{USER_NAME}}`、`{{PROFESSION/FIELD}}`、`{{SECONDARY_SKILL}}`、`{{HUMOR_STYLE}}`、`{{PREFERRED_PLATFORM}}`、`{{SPECIAL_MENTAL_MODEL}}` 六个变量
- **扣子编程先做规划再做执行**：先用扣子内置编程能力做产品分析（用户角色、功能模块、技术选型），确认方案后再让龙虾执行，避免方向跑偏

- **给龙虾喂优秀参考是加速器**：把同类优秀项目的 skill 文档（如 `https://instreet.coze.site/skill.md`）直接发给龙虾学习，比从零描述需求效率高 3-5 倍
- **世界观设计提升产品粘性**：不要照搬已有 IP，而是结合产品特点（龙虾 = 深海）设计原创世界观（深红港任务公会），同时评估实施可行性和难度

- **Skill 沉淀是复利关键**：每次项目完成后让龙虾总结心得生成 skill 文档，下次直接调用，龙虾会越来越懂你

### 📋 核心分析

**战略价值**: 通过创建隔离用户账号 + Node 虚拟环境，在 macOS 上零风险安装 OpenClaw，并借助 TRAE 技能将复杂配置流程对话化，新手可直接复刻。

**核心逻辑**:
- OpenClaw 底层是 Node.js 生态，Node 虚拟环境方案是原生适配，不存在兼容问题，优先于 Docker 和 Python 方案
- 安全第一原则：先创建隔离普通用户账号，切换到该账号后再安装，主账号数据完全不受影响
- nvm 管理 Node 版本，所有文件装在用户目录（`~/.nvm`），不污染系统，全程无需 sudo
- OpenClaw 要求 Node.js ≥ 22，必须用 `nvm install 22` 指定版本，不能用系统默认 Node
- 启动时强制绑定 `--host 127.0.0.1`，仅本机可访问，防止外网攻击
- 飞书机器人接入需要同时配置「长连接事件订阅」+「卡片回传交互回调」，缺一不可
- 火山引擎接入点 ID 以 `ep` 开头，需单独创建 API Key，两者都要记录
- 权限最小化：macOS 隐私设置中 OpenClaw 仅授权 `~/.openclaw` 目录，屏幕录制/麦克风默认关闭
- 飞书配置完成后，需通过 `openclaw pairing approve feishu xxx` 命令完成配对，缺少此步机器人不响应
- 卸载无残留：删除 `~/.nvm`、`~/.npm-global`、`~/.openclaw` 三个目录即可完全清除

   - 打开 TRAE → 左上角点击「IDE」切换到 Solo 模式

   - 记录 `app_id` 和 `app_secret`

**Node 虚拟环境手动安装（无 TRAE 版）**

1. **准备阶段**
   - 创建隔离普通用户账号并切换（同上）
   - 确保网络通畅，无需提前安装任何软件，无需 sudo

2. **核心执行**
   ```bash
   # 安装 nvm
   touch ~/.zshrc; curl -o- https://raw.githubusercontent.com/nvm-sh/nvm/v0.39.7/install.sh | bash
   source ~/.zshrc
   nvm --version  # 验证，显示 v0.39.7 即成功

   # 安装 Node 22
   nvm install 22
   nvm use 22
   node -v  # 验证，显示 v22.x.x 即成功

   # 安装 OpenClaw
   npm install -g openclaw
   # 若 sharp 报错，改用：
   SHARP_IGNORE_GLOBAL_LIBVIPS=1 npm install -g openclaw
   openclaw --version  # 验证，显示版本号（如 2026.2.25）即成功

   # 初始化
   openclaw onboard

   # 启动（仅本地访问）
   openclaw gateway start --host 127.0.0.1 --port 18789
   ```

3. **验证与优化**
   - 浏览器访问 `https://127.0.0.1:18789` 确认 WebUI 正常
   - 系统设置 → 隐私与安全性 → 文件与文件夹 → OpenClaw 仅勾选 `.openclaw` 目录
   - 通用 → 登录项 → 确认无 OpenClaw 开机自启条目

### 📋 核心分析

**战略价值**: 一套面向人文/内容工作者的 AI 协作方法论——不靠魔法 prompt，而是把创作流程工业化、白盒化，让 AI 成为可管理的生产工位，最终输出你愿意署名的作品。

**核心逻辑**:

- **三条底线先于一切**：① 过程可追溯/可监督/可验证（黑箱会藏幻觉、错引、偷换概念）；② 必须可操纵（你在生产，不是在抽卡）；③ 最终你愿意署名（不愿署名 = 你的意志没贯彻进去 = 质量不可控）。
- **"许愿式"vs"工作台式"**：「帮我写篇好文章」是许愿；把目标受众、讲解方式、结构要求、语气标准全部写清楚才是工作台。你给得越像"作业要求"，AI 越不像 AI，越像真会干活的助教。
- **你才是负责人**：对输出不满时，第一反应不是"AI 不行"，而是自查四项：① 有没有说清对象/受众/目的；② 有没有提供背景材料和约束；③ 有没有把抽象愿望拆成可执行动作；④ 有没有给出可判断对错的标准。
- **同一问题至少问 3 个模型**：AI 像人一样有性格差异——有的擅长写作遣词，有的擅长推理，有的擅长代码。同一家产品的新版本也会持续微调风格和边界。目标不是"选出最强模型"，而是开始像管理团队一样管理模型：谁做第一稿、谁做审稿人、谁出选题结构、谁出段落句子。
- **AI 常识水平 ≈ 985 本科生**：凡是"优秀本科生未必知道"的内容，默认 AI 也不知道，或会"编得很像知道"。超出常识的部分必须你来教：给范例、给标准、给禁区、给语料。它能做"微观插值"（把你的脚手架补全），但脚手架和方向来自你。
- **白盒分步骤 > 黑箱一次性**：以 TTS 朗读稿处理为例，不要说"注意多音字"，而是拆成：① 标出停顿/重音/语速标记 → ② 识别潜在多音字 → ③ 依词典核对读音 → ④ 标注易误读常见字 → ⑤ 用同音无歧义字替换消灭误读。"显而易见的正确做法"人类默认会做，AI 不会默认——不写进流程就会在最省力路径上犯错。
- **先工业化，再 AI 化**：随机的、凭灵感的、资料不管理的流程无法交给 AI。路径是：先把工作变成"产线"（可拆分/可复用/可质检），再把子步骤交给 AI 当工位。实践案例：把"如何写一篇非虚构文章"解构成几十个步骤（为什么用这个故事开头、如何给例子打分、如何起承转合、如何把小故事连接宏大图景……），让不同 AI 只做其中一个步骤——不是模型变强了，是流程把"每次只能行一点点"的能力串起来了。
- **预判 AI 系统性偷懒**：能不打开网页就不打开，能不读 PDF 就不读。解法是把算力用在"理解文本"而非"处理格式"：① 材料转成纯文本/Markdown 再喂给 AI；② 网页内容去掉导航/广告/脚注噪音后复制；③ 长材料先做事实提炼/结构提取再让它写作；④ PDF/EPUB/网页统一入库成可检索 TXT。
- **压缩比扩张可靠**：AI 有上下文窗口上限，给它 20 万字它可能只扫标题。经验规律：把 100 万字压到 1 万字，远比把 1 万字扩到 100 万字可靠。正确姿势是把材料尽可能喂进去（分批/检索/RAG 均可），让它在充分材料基础上压缩出结构、观点和正文——这本来就是"读海量材料→提炼→组织→写作"的人类写作路径。
- **改产线，不改结果**：AI 出 59 分稿子，你改两下变 80 分，改着改着变成重写，最后放弃 AI——这是最常见的翻车路径。解法：不追求单篇 100 分，目标是让产线稳定产出 75～80 分，迭代流程提高平均分。
- **数量对抗平庸**：只要一个版本会得到最中庸的结果。正确做法：总结一次要 5 个版本、开头一次要 5 个做 AB Test、选题一次要 50 个再分组挑选、结构一次要 3 套再组合、表述一次要 10 个不同措辞再择优。提高产量后，分布里自然出现 85～90 分的惊喜样本。
- **行政总厨模式**：你不亲自拍黄瓜，你尝一口→判断是否合格→给出明确反馈（哪里不行、要怎么改）→让它回炉重做。教它怎么达到你的标准，而不是自己跳下去把每次结果修成成品，否则被无止尽的修补耗死。
- **材料 × 品味决定上限**：用最新模型但只有网上资料 vs 用老模型但有完整档案/口述史/实地采访——后者更可能做出好作品。当"生成"变得便宜，真正稀缺的是：你知道什么值得写、哪些证据更硬、哪种叙述更有力量、愿意为材料付出体力劳动。AI 改变的是你和材料交互的效率，但作品的主语仍然是你，宾语仍然是材料，AI 只是"动词"的一部分。

### 🎯 关键洞察

**"许愿—失望—放弃"循环的根本原因**：用户把 AI 当神谕而非工具，期望黑箱一次性给出完美结果，却没有把自己的隐含前提、判断标准、工作方法显式化。一旦你能清晰描述"我的文章是怎么做出来的"，你就会发现决定质量上限的从来不是"用哪个大模型"，而是你有没有把工作方法讲清楚。

**工业化是 AI 化的前提条件**：这是最反直觉也最关键的洞察。很多人想用 AI 跳过流程建设，但 AI 只能接住"可描述、可复现"的那部分。你必须先把自己的工作方法外显化、结构化，才能把它交给 AI。这个过程本身就是对自己工作方法的一次深度审计。

### 🛠️ 操作流程

1. **准备阶段：工业化你的工作方法**
   - 把自己的创作/研究流程完整解构：为什么用这个开头、如何给例子打分、如何起承转合、如何收尾、如何连接宏大图景
   - 拆成若干可独立执行的子步骤（参考：非虚构写作流程可拆成几十个步骤）
   - 为每个子步骤定义：目标受众、输出标准、禁区、参考范例

2. **核心执行：白盒分步骤生产**
   - 材料预处理：PDF/EPUB/网页 → 纯文本/Markdown，去除格式噪音
   - 长材料先提炼结构和事实，再进入写作环节
   - 每个子步骤单独给 AI 执行，不要求一步到位
   - 同一任务至少抛给 3 个模型，建立分工认知
   - 每次要求多个版本（选题 50 个、开头 5 个、结构 3 套、措辞 10 个）

3. **验证与优化：行政总厨模式**
   - 尝一口（评估输出）→ 判断是否合格 → 给出明确反馈（哪里不行、要怎么改）→ 让它回炉
   - 不改单篇结果，改产线流程
   - 质检标准：你愿不愿意把名字放上去？
   - 迭代目标：让产线平均分从 59 分提升到 75～80 分，而非追求单篇 100 分

### 💡 具体案例/数据

**非虚构写作流程解构实验**（FUNES 内部实践）：
- 把"如何写一篇非虚构文章"完整解构，包含：为什么用这个故事开头、为什么选这句话、如何给例子打分、如何起承转合/过渡/收尾、如何把小故事连接到更宏大的图景
- 最终拆成几十个步骤，让不同 AI 只做其中一个步骤
- 结论：不是模型突然变强了，而是流程把"每次只能行一点点"的能力串起来了

**材料 vs 模型的优先级实验**（思想实验）：
- 选项 A：最新模型 + 只有网上资料
- 选项 B：老模型 + 完整档案/口述史/实地采访
- 结论：选项 B 更可能做出好作品。材料质量 > 模型版本。

### 📝 避坑指南

- ⚠️ **"AI 不行"是最危险的归因**：输出不满意时，先自查四项（对象/受众/目的是否说清、背景材料是否充分、抽象愿望是否拆成可执行动作、是否给出可判断对错的标准），再下结论。
- ⚠️ **"妙手一改"陷阱**：擅长写作的人最容易翻车——改着改着变成重写，最后放弃 AI。关注点要挪到上游：迭代流程，不改单篇。
- ⚠️ **黑箱最危险**：幻觉、错引、偷换概念都在黑箱里悄悄发生。任何超出常识的内容（优秀本科生未必知道的），默认 AI 不知道或会"编得很像知道"。
- ⚠️ **格式障碍浪费算力**：直接丢 PDF/网页给 AI，它会把算力耗在处理格式上。预处理是你的责任，不是 AI 的。
- ⚠️ **流程随机化无法 AI 化**：凭灵感、资料不管理的工作流无法交给 AI。AI 只能接住"可描述、可复现"的部分。必须先工业化，再 AI 化。
- ⚠️ **只要一个版本 = 主动选择平庸**：单次输出天然倾向最中庸、最保守的结果。用数量（50 个选题、5 个开头、10 个措辞）对抗平庸。
- ⚠️ **上下文窗口是硬约束**：给 AI 20 万字，它可能只扫标题。用分批/检索/RAG 处理长材料，不要指望它"凭空扩张"。
