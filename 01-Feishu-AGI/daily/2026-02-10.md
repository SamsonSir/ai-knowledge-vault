### 📋 核心分析

**战略价值**: Qwen-Image-2.0 将生图与编辑两条技术支线合并为单一 omni 模型，在文字渲染精度（1k token 指令）、2K 写实质感、图像编辑一致性三个维度同步突破，且以 7B 轻量架构实现。

**核心逻辑**:

- **技术路线合并**：2025年5月项目启动后分两条支线并行——生图支线（8月 Qwen-Image 文字渲染 → 12月 Qwen-Image-2512 细节质感）、编辑支线（8月单图编辑 → 9月多图编辑 → 12月 Qwen-Image-Layered 图层拆分 → 12月 Qwen-Image-Edit-2511 一致性提升），2026年2月10日 Qwen-Image-2.0 将两条支线合二为一。
- **文字渲染特性①「准」**：支持复杂"画中画"渲染，例如在 PPT 时间轴节点内嵌套组图（戴帽子小狗 → 去帽小狗），模型不仅完成渲染还保持了图内角色一致性。
- **文字渲染特性②「多」**：支持 1k token 指令长度，可直出包含多栏数据表、流程图、统计显著性检验（p<0.05, 95% CI, Cohen's d=0.32）等复杂信息图表；复杂 Prompt 可通过 LLM 改写获得（用户输入简短意图 → LLM 扩写为详细描述 → 喂给 Qwen-Image-2.0）。
- **文字渲染特性③「美」**：生成图文混排时，模型自动将文字渲染在画面空白区域，不遮盖图像主体；支持多种书法字体，包括瘦金体（宋徽宗赵佶风格）、王羲之小楷、田英章硬笔、赵孟頫楷书等，且能在水墨长卷中完整渲染《兰亭集序》全文（极个别字有误差，整体准确）。
- **文字渲染特性④「真」**：可同时渲染多种介质上的文字——玻璃白板上的手写体、衣服上的 Logo 印刷体、杂志封面的排版字体，各介质材质、空间斜度、光影反射均有差异化处理，增强真实感；电影海报场景中，密集文字元素（出品方、主演、制作信息）与画面光影自然融合。
- **文字渲染特性⑤「齐」**：漫画对话框内文字自动居中对齐；信息图中相似段落自动对齐；日历格内多行文字（农历日期 + 公历日期 + 节假日标注）排布整齐。
- **写实质感提升**：2K 分辨率（2048×2048）原生支持；森林场景可区分 23 种以上不同明度/饱和度/冷暖倾向/材质（蜡质、绒面、革质、胶质）的绿色，丁达尔光束、苔藓露珠、叶脉半透明感均有精细刻画。
- **omni 模型增益传导**：生图能力的文字渲染和写实质感提升，直接传导给图像编辑能力——可在已有图片上题写书法诗词（指定字体、排列方向）、九宫格姿势组图生成、双图合成自然合照（指定镜头参数：等效50mm, f/4.0, 1/160s, ISO 200）、跨次元编辑（真实城市照片 + 卡通形象融合，保持照片真实性）。
- **模型规模**：7B 参数，架构为 8B Qwen3-VL Encoder → 7B Diffusion Decoder → 2048×2048 像素输出，在速度与画质间取得平衡。
- **AI Arena 盲测**：在文生图和图生图两个基准中均取得优越性能（具体数值见原文图表）。

### 🎯 关键洞察

**LLM 改写 Prompt 的工作流是实用关键**：用户无需手写千字 Prompt，只需输入简短意图（如"帮我生成一个手绘风格的杭州两日禅意人文之旅双语海报"），交由任意 LLM 扩写为包含具体时间、地点、人物、色调、排版、双语对照的详细描述，再输入 Qwen-Image-2.0。这一工作流将模型的 1k token 指令上限真正用满，是解锁复杂渲染能力的核心路径。

**omni 架构的核心价值**：单模型同时覆盖文生图 + 图像编辑，避免了多模型切换的工程复杂度；更重要的是，生图侧的能力提升（文字渲染精度、写实质感）自动惠及编辑侧，无需分别优化两套模型。

### 🛠️ 操作流程

1. **准备阶段 — Prompt 构建**
   - 输入简短意图到任意 LLM（如 Qwen Chat 对话模式）
   - 要求 LLM 扩写为包含：背景色调、构图方式、文字内容（含双语对照）、字体风格、排版方向、光影描述的完整 Prompt
   - 控制扩写后 Prompt 在 1k token 以内

2. **核心执行 — 生图/编辑**
   - 文生图：将扩写后 Prompt 输入 chat.qwen.ai "生成图像" 功能或百炼 API
   - 图像编辑（题词）：上传原图 + 指令，例如：`在图片的左上角加上从右到左，从上到下写着的赵孟頫楷书"[诗词全文]"`
   - 双图合成：上传图1、图2，Prompt 中明确指定：两人站位关系、服装细节、背景统一来源（取图1或图2）、镜头参数（焦距/光圈/快门/ISO）
   - 跨次元编辑：上传真实照片（图1）+ 卡通形象（图2），Prompt 中明确：`请勿更改照片中的真实建筑、街道、车辆或人物，保持照片的真实性`，再描述卡通形象的位置关系

3. **验证与优化**
   - 检查文字渲染准确性（重点核查生僻字、长文本末尾部分）
   - 检查多介质场景中各介质的光影/透视是否自然
   - 若文字对齐/排版不理想，在 Prompt 中补充明确的对齐指令（如"居中对齐"、"自右向左竖排"）

### 💡 具体案例/数据

**案例1 — PPT 时间轴生成**：完整 Prompt 约 500 字，包含深蓝渐变背景、发光时间轴、两条支线（生图支线/编辑支线）、各节点日期（2025-05-06 至 2026-02-10）、节点内嵌套组图描述。模型输出的 PPT 直接用于官方发布文章，无人工后期。

**案例3 — 小楷压测**：《兰亭集序》全文（约 324 字）以王羲之小楷竖排渲染，结论：除极个别字外基本准确。

### 📝 避坑指南

- ⚠️ 长文本渲染（如《兰亭集序》全文）存在极个别字错误，重要场景需人工校对
- ⚠️ 复杂 Prompt 直接手写效率低，务必先用 LLM 改写扩充，再输入模型
- ⚠️ 双图编辑时若不明确指定背景来源，模型可能自行混合两张图的背景，需在 Prompt 中显式声明"背景统一为图X中的[具体场景]"
- ⚠️ 跨次元编辑需显式声明"请勿更改照片中的真实元素"，否则模型可能对真实照片部分进行风格化处理
- ⚠️ API 当前为邀测阶段，生产环境接入需提前申请百炼资格

**核心逻辑**:

- **人物一致性**是 Seedance 2.0 的核心优势，多镜头同一角色不崩脸，打斗场面符合物理逻辑，无 AI 突兀感

### 📋 核心分析

**战略价值**: Anthropic 用真实案例数据论证"编程正从人写代码转向人编排 Agent 写代码"，并给出 2026 年八大可操作趋势，指导工程师和团队完成角色转型。

**核心逻辑**:

- **AI 使用率 vs 放权率存在巨大落差**：开发者在 60% 的工作中用到 AI，但能"完全甩手"的任务只有 0-20%。这意味着 AI 是协作伙伴，不是外包对象，需要持续的 Prompting、监督和人类判断。
- **开发周期单位从"周/月"压缩到"小时/天"**：Agent 接管编码、测试、文档生成后，传统 SDLC 各阶段仍存在，但时间密度极度压缩，监控反馈直接驱动快速迭代。
- **新员工熟悉陌生代码库的时间从"几周"缩短到"几小时"**：Augment Code 案例——某企业 CTO 原估 4-8 个月的项目，用 Augment Code 两周完成，直接改变人才调配逻辑。
- **工程师角色从"写代码"变成"编排 Agent + 验收产出"**：时间分配反转——以前 80% 写代码 + 20% 思考，现在 80% 想清楚要什么/怎么拆/怎么验收 + 20% 编码。
- **多 Agent 层级架构突破单 Agent 上下文瓶颈**：一个"指挥官 Agent"拆解任务、分配工作、合成结果；多个"专家 Agent"并行负责架构/实现/测试/文档。Fountain 案例：筛选速度 +50%，入职速度 +40%，候选人转化率翻倍，72 小时内招满一个新运营中心（原来需要一周以上）。
- **长时间自主运行已成现实**：乐天（Rakuten）让 Claude Code 在 1250 万行代码的 vLLM 库中自主工作 7 小时，一次运行完成全部任务，数值准确率达参考方法的 99.9%。
- **AI 生产力提升体现在"产出量增加"而非"同样的事更快"**：约 27% 的 AI 辅助工作是没有 AI 就根本不会去做的事——扩展项目、交互式仪表盘、试探性实验、顺手修掉低优先级的"小毛病"（papercuts）。
- **经验越丰富，从 AI 协作中获益越大**：Anthropic 工程师原话："我主要在我知道答案应该是什么样子的情况下使用 AI。这种判断力是我通过'笨办法'做软件工程积累出来的。" AI 放大已有能力，不凭空创造能力。
- **非技术用户正在成为 Agent 编程的新主体**：Legora 让不懂编程的律师创建复杂自动化工作流；Anthropic 法律团队用 Claude 把营销审查周期从 2-3 天压缩到 24 小时；Zapier 89% 员工在用 AI，内部部署 800+ AI Agent。
- **安全是双刃剑，必须设计期嵌入而非事后补丁**：攻击者同样会用 Agent 扩大攻击规模，防御必须达到毫秒级自动检测和响应，提前布局的团队面对同样技术武装的攻击者时有结构性优势。

### 🎯 关键洞察

**洞察 1：协作悖论——AI 做得越多，人越不敢完全放手**

越是概念复杂、依赖设计判断的任务，工程师越倾向于紧密协作而非完全委托。原因是这类任务的验收标准本身就需要人类经验来定义。这不是 AI 能力不足，而是"什么是对的"这个问题本身需要人来回答。实践含义：不要试图最大化 AI 自主度，而要最大化"人类判断力 × AI 执行力"的乘积。

**洞察 2：27% 的增量工作是真正的生产力释放**

TELUS 创建超过 13000 个定制 AI 解决方案，代码发布速度 +30%，累计节省 50 万小时以上。这些数字背后的逻辑是：开发周期从周变天后，大量原本"不划算"的项目变得可行。生产力提升不是线性的，是"Agent 能力 × 编排方式 × 人类经验"三者叠加的阶跃式跳跃。

**洞察 3：技术债清理窗口正在打开**

以前技术债积压的根本原因是投入产出比太低——没人愿意花时间重构一堆能跑但很丑的老代码。现在这个成本大幅下降，Agent 可以系统性地处理。这是 2026 年很多团队可以立即行动的低风险高价值机会。

### 🛠️ 操作流程：2026 年四大立即行动项

1. **掌握多 Agent 协同**
   - 学习任务拆解方法论：把复杂项目分解为可并行的子任务
   - 为不同子任务设计专门的 Agent 角色（架构/实现/测试/文档）
   - 建立协调协议：指挥官 Agent 如何合成各专家 Agent 的输出
   - 工具参考：Claude Code 多会话并行、支持多 Agent 状态显示的开发环境

2. **扩展人机监督机制**
   - 建立 AI 自动审查系统，用 Agent 分析 AI 生成代码的安全漏洞和架构一致性
   - 定义"举手规则"：哪些情况 Agent 必须暂停并请求人类判断（高风险操作、商业影响决策、不确定性超过阈值）
   - 把人类注意力集中在：真正新颖的问题、边界案例、战略决策

3. **把 Agent 编程能力扩展到工程团队之外**
   - 识别组织内最了解业务痛点但没有编程能力的角色（法律/运营/设计/销售）
   - 为他们提供 Agent 编程工具和基础培训
   - 优先解决那些"不值得占用工程师时间但一直没人处理"的边缘问题

4. **安全左移：设计期嵌入安全架构**
   - 在 Agent 系统设计最早期就定义安全边界和权限模型
   - 建立自动化安全审查流水线
   - 规划毫秒级检测和响应能力，应对机器速度的攻击

### 📝 避坑指南

- ⚠️ **不要试图完全甩手**：能"完全放权"的任务只有 0-20%，强行甩手会导致低质量输出积累，验收成本反而更高。

### 📋 核心分析

**战略价值**: 用六把刀砍掉互联网时代的六个核心认知框架（DAU/平台/SaaS/AI应用/注意力经济/出海），再用四块基石重建 Agent 时代的新操作系统。

**核心逻辑**:

- **DAU 是负债不是资产**：AI 产品每多服务一个用户就多烧一份推理成本；ChatGPT 是星型拓扑，没有网络效应，靠 DAU 驱动指数增长在 AI 时代根本不成立。Claude 已明确拒绝广告模式。
- **工具→社区→平台三级火箭路径已死**：互联网时代工具不够强所以需要人来补，AI 时代工具本身就能给出完美结果，社区的本质是"人帮人"，当 AI 比人更能帮人时，社区价值基础坍塌。能做平台的只有大模型公司——因为它本身就是星型拓扑的中心节点。
- **SaaS 没死，但主人换了**：人类用户增量已饱和（移动互联网渗透率见顶），Agent 对 API 的调用量正在指数增长。SaaS 从 2B/2C 转向 2A（面向 Agent），软件公司从"产品"变成"基础设施"。
- **"AI 应用"这个词锁死思维**：Application 天然暗示使用者是人，用这个词就会不自觉地去想界面设计、交互优化、用户留存——全是面向人的思考。换词：不做 AI 应用，做 Agent 基础设施。
- **注意力经济→生产力经济**：注意力经济本质是零和博弈（平台赚的是你浪费的时间），生产力经济是正和博弈（你付费拿结果，AI 公司拿收入，双方都创造价值）。产品目标从"让你花更多时间"变成"让你花更少时间拿更好结果"。
- **"出海"是面向人的思维**：Agent 的世界没有地理边界，不需要翻译、本地支付、当地推广。面向 Agent 只需要：API 做好、文档写清楚、协议对接好，全球 Agent 自动找到你。
- **算力马太效应已启动**：更多算力→更好结果→更多收入→买得起更多算力，正反馈循环一旦转起来差距只会越来越大。Claude Fast 模式：2.5 倍推理速度，5 倍 Token 费用，一天总消耗可达普通模式的 12 倍以上。
- **燃烧 Token 的速度决定进化速度**：用顶级模型 vs 用垃圾模型，一年后认知差是 100 倍（对比：用谷歌 vs 用百度，一年后认知差是 2 倍）。买 Token 不是消费，是投资自己。当前燃烧 Token 最快的三类产品：AI Coding、AI Agent、AI Video。
- **Agent 是新世界的人口红利**：一个人可能有 10~100 个 Agent 为他工作，每个 Agent 每天调用外部接口数千至数万次，总调用量远超人类点击屏幕次数。增长飞轮：先被发现（早发布 Skills、写好文档、做好 SEO）→再被依赖（稳定、准确、快速）。
- **人的价值重新定位**：Agent 有能力/理性/耐心，人类有欲望/情感/想象。Agent 能把任何想法变成现实但自己不产生想法。新世界人的价值不是亲自干活，是"决定干什么、为什么干"。人与人的差距取决于能驱动多少 Agent，而不是自己能做什么。

**拓扑结构决定商业模式**：
- 微信 = 网状拓扑 → 每增加一个节点，所有节点受益 → 网络效应 → DAU 是资产
- ChatGPT = 星型拓扑 → 每增加一个节点，只有中心节点受益 → 无网络效应 → DAU 是负债

| 产品目标 | 用户停留时长 | 结果交付效率 |

### 🛠️ 操作流程：如何在 Agent 时代构建产品

1. **重新定义用户**: 把产品文档里所有"用户"替换成"Agent"，重新审视每一个功能的必要性。界面、交互、推送——如果 Agent 不需要，砍掉。

2. **API 优先架构**: 产品核心能力必须通过 API 暴露，不能只有 GUI。Agent 不会点击按钮，只会调用接口。检查清单：
   - API 文档是否机器可读（OpenAPI/Swagger 格式）
   - 返回结果是否结构化（JSON，无歧义）
   - 错误码是否清晰（Agent 需要自动处理异常）
   - 是否支持 API Key 自助开通（Agent 不会打电话给销售）

3. **Agent 可发现性建设**: 
   - 早发布 Skills/Plugin（占据 Agent 工具库的先发位置）
   - 文档写给 Agent 看：精确、无歧义、有示例调用
   - SEO 针对 Agent 搜索习惯优化（关键词精准，结构清晰）
   - 接入主流 Agent 框架的工具市场（如 MCP、LangChain Tools 等）

4. **依赖度建设（让 Agent 用了离不开）**:
   - 稳定性：SLA 99.9% 以上，Agent 工作流不能因你的服务抖动而中断
   - 准确性：返回结果质量持续优于竞品，Agent 会自动选择最优工具
   - 速度：响应延迟直接影响 Agent 工作流效率，P99 延迟需严格控制

5. **算力投入策略**: 
   - 个人/团队：优先购买顶级模型 Token，不要为省钱降级模型（认知差一年后是 100 倍）
   - 产品侧：把算力成本视为研发投入，而非运营成本
   - 重点押注三类高 Token 消耗场景：AI Coding、AI Agent、AI Video

### 🛠️ 操作流程：AI落地三部曲

1. **拆解——把业务"原子化"**
   - 不看业务整体，只看每个环节的**输入**和**输出**
   - 示例（钻石画）：
     - 第一步：输入=用户照片 → 输出=带色号的像素底图
     - 第二步：输入=像素点图 → 输出=对应钻号图
   - 把整条SOP拆成最小粒度的原子步骤，每步都明确输入/输出

2. **诊断——寻找"阻力最大的点"**
   - 问三个问题：
     - 哪个环节是上下游的主要卡点？
     - 哪个环节消耗人力最多？
     - 哪个环节限制了你从1做到100？
   - 阻力最大的那个原子环节=你的支点
   - 一套业务中这样的点有多个，全部记录下来，必须自己亲身盘点，无法外包给顾问

3. **替换——用AI定向替换该环节**
   - 冷静判断两个条件：
     - 当前AI模型（推理/生图/工具调用）的能力边界能否覆盖这个原子环节？
     - AI的成本是否低于人工？
   - 两个条件都满足→立即替换，不要犹豫
   - 判断是否自造轮子：如果该环节需要大量外部知识（行业调研、竞品分析、实时政策），**不要自己爬数据**，直接调用百度搜索API或Deep Research Agent
   - 单点替换稳定后，再向上下游延伸打通

**来源**: 一万字提示词+10个文件，给你的AI造一个"数字灵魂" | **时间**: 2026-02-10 | **原文链接**: https://mp.weixin.qq.com/s/SDD9iJ62...

### 📋 核心分析

**战略价值**: 用 TELOS 10文件体系 + 三层记忆架构 + Hooks/Skills，把 AI 从无状态对话工具升级为持久化、可编程的个人基础设施。

**核心逻辑**:

- **范式转变的本质**：当前 AI 是"自动售货机"（无状态、问答即止）；PAI 要把它变成"操作系统"（持久运行、记住设置、管理文件）。两者的根本差异在于：有无跨会话的持久状态。
- **TELOS = 10个 Markdown 文件构成的数字身份档案**，覆盖：核心身份(00)、价值观(01)、背景(02)、技能(03)、沟通风格(04)、目标(05)、知识领域(06)、工作方式(07)、关系与背景(08)、兴趣(09)。每个文件独立维护，持续迭代，不是一次性问卷。
- **数据主权设计**：TELOS 全部用 Markdown 存本地，可用 Git 做版本控制（`git diff` 追踪自我认知演变），不锁定任何平台，随时迁移。
- **三层记忆架构模拟人脑**：① 短期记忆（当前对话上下文，实时动态）；② 情景记忆（具体对话记录和项目历史，可精确回溯）；③ 语义记忆（从经验提炼的模式和偏好，如"你通常用函数式风格"）。检索机制用向量数据库 + 相似度搜索，实现联想式调取，而非线性翻记录。
- **Hooks = 给 AI 装传感器**：在特定事件（提问前、回复后、工具调用时）自动触发脚本，让 AI 主动感知工作环境，而非被动等待输入。三类典型 Hook 见下方配置表。
- **Skills = 可组合的命令式能力插件**：每个 Skill 是命名化、参数化的 Prompt 模板，通过 `/skill-name` 调用，Skills 之间可相互调用构建复杂工作流，可在团队间共享复用，类比 VS Code 扩展市场。
- **工程三原则**：① 确定性（所有配置文件化，行为可追溯可回滚）；② 模块化（TELOS/Memory/Hooks/Skills/Tools 各自独立可替换）；③ 可追溯性（Markdown + YAML + JSON Lines 全开放格式，`grep` 可搜，任何文本工具可处理）。
- **数据流是明确的单向管道**：用户输入 → Hooks预处理 → TELOS加载 → Memory检索相关上下文 → 构建完整Prompt → 调用LLM → Skills执行 → Memory存储新交互 → Hooks后处理 → 返回响应。每步可配置、可观察。
- **存储方案全部开放格式**：TELOS 用 Markdown、对话历史用 JSON Lines、向量库用 SQLite+向量扩展或专用向量库、配置用 YAML。
- **冷启动是最大门槛**：需要填写10个 TELOS 文件（深度自我反思）+ 积累交互历史（时间投入）+ 配置 Hooks/Skills（技术门槛）。类比搭建个人知识管理系统：前期痛苦，长期回报。

### 🎯 关键洞察

**为什么 TELOS 用10个独立文件而不是一个大文档？**
模块化设计：每个维度独立更新，不互相污染。你的技能(03)提升了，不需要重写价值观(01)。Git 追踪时 diff 粒度更细，能清晰看到"这个月我的目标(05)发生了什么变化"。

**为什么三层记忆比单一聊天记录强？**
原因：人类大脑不是录像机，是压缩机。逐字存储所有对话会导致上下文爆炸（更慢、更贵）且信噪比极低。
动作：语义记忆层做提炼压缩，只保留模式和偏好；情景记忆保留关键事件；短期记忆处理当下。
结果：检索时精准调取相关片段，而非加载全部历史，在"完整上下文"和"效率"之间取得平衡。

**为什么选 CLI 优先而非 GUI？**

### 📝 避坑指南

- ⚠️ **冷启动成本被低估**：10个 TELOS 文件不是填表，是深度自我反思。建议分批完成，先填 `00`（核心身份）和 `05`（目标），其余逐步补充，不要试图一次性完成。

### 📋 核心分析

**战略价值**: 花 $400+ Token 实测飞书 200+ API 后，将所有踩坑经验封装成 9 个可直接加载的 AI Agent Skill（Markdown 文档），让后来者零试错成本复用。

**核心逻辑**:

- **Skill 的本质**：一份写给 AI 看的 Markdown「避坑手册」，AI 调用 API 前先读它，等同于老员工给新人写的操作规范，不是代码/SDK/插件
- **省钱逻辑量化**：无 Skill 时写日期字段需试 4 次、耗 $1.8 Token、15 分钟；有 Skill 后一次成功、3 秒完成——核心原因是 Skill 里明确写了「日期字段必须转换为 13 位毫秒级时间戳」
- **安装方式**：两种路径，方式1下载 Zip 解压复制；方式2 git clone 后 `cp -r` 到 `~/.openclaw/workspace/skills/`，最后统一执行 `openclaw gateway restart` 重载
- **9 个 Skill 覆盖范围**：云文档(doc-writer)、多维表格(bitable)、日历(calendar)、消息与群管理(im)、任务(task)、知识库(wiki)、文件管理(drive)、组织架构(contact)、审批(approval)
- **最高频踩坑类型**：时间戳精度混用（Calendar 用 10 位秒级字符串，Bitable/Task 用 13 位毫秒级字符串，两者不可互换）
- **权限黑盒是最大隐患**：机器人创建的文件/群/空间默认归属机器人，用户不可见——必须在每次创建后立即补调权限接口
- **并发写入是文档乱序根源**：飞书 Docx API 对并发极敏感，所有 Block 必须放在单一 `children` 数组一次性 POST，或严格串行等待
- **ID 类型错传是审批/消息失败主因**：OpenID/UserID/UnionID 三者不可混用，所有接口必须显式声明 `user_id_type`
- **开源地址**：`https://github.com/alextangson/feishu_skills`，Star 数量影响更新速度
- **200+ API 仍在持续测试**，9 个为第一批开源，后续陆续释出

**Skill 的知识传递范式转变**：传统靠文档传递知识、靠 SDK 传递功能；Skill 把「人的试错成本」直接封装给 AI 继承。一个人踩过的坑，通过 Skill 让所有接入的 Agent 永远不再踩。

### 📝 避坑指南

- ⚠️ **时间戳单位混用**：Calendar = 10 位秒级字符串；Bitable/Task = 13 位毫秒级字符串。传错直接导致时间显示 1970 年或 400 报错
- ⚠️ **并发写文档**：飞书 Docx API 并发写入会导致 Block 顺序随机错乱，必须单一 Batch 或串行
- ⚠️ **群创建后不可见**：创建群成功后必须立即补调「拉人入群」API，否则群在手机端不显示
- ⚠️ **卡片 JSON 未字符串化**：`content` 直接传 JSON 对象会被飞书当纯文本输出为代码字符串，必须先 `JSON.stringify()`
- ⚠️ **审批控件 ID 硬编码**：`widget177...` 是随机生成的，不同审批流同名控件 ID 不同，必须先调 `get_approval_definition` 动态获取
- ⚠️ **Wiki 权限 403**：机器人无法直接创建顶级空间，需手动建群→在 Wiki 空间设置中将群组设为管理员
- ⚠️ **Drive 文件失踪**：机器人创建的文件默认归属机器人，用户不可见，必须「写后必授」调 `permissions` 接口
- ⚠️ **Contact 搜不到用户**：不是用户不存在，是机器人「可见范围」未包含该用户，去开发者后台改「可见范围」为全员
- ⚠️ **Calendar 邀约静默失败**：创建日程时 `attendees` 字段会被忽略，必须拿到 `event_id` 后补调「添加参与人」接口
- ⚠️ **node_token vs obj_token**：Wiki 的 `node_token` 是目录位置，`obj_token` 才是真实文档，写内容必须用 `obj_token`
