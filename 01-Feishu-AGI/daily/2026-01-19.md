### 📋 核心分析

**战略价值**: OpenSpec 通过"规范注入"机制，让 AI 在每次对话前先加载项目规范，从而稳定、可复现地驱动 AI 按团队标准执行开发任务。

**核心逻辑**:

- **触发机制是根本**：AI 不触发 OpenSpec 规范，99% 是因为请求中缺少关键词（"提案"、"变更"、"规范"、"计划"等），AI 不会主动加载规范文件，必须显式触发。
- **安装只需两步**：`npm install -g @fission-ai/openspec@latest` 全局安装，然后在项目目录执行 `openspec init`，初始化时选择对应 AI 工具（Claude Code / Cursor / Trae / Qoder / Other Tools）。
- **Claude Code 初始化后自动生成 `.claude/` 目录**，包含三个斜杠命令文件（`apply.md`、`archive.md`、`proposal.md`）和两个根文件（`AGENTS.md`、`CLAUDE.md`），Claude Code 启动时自动读取 `AGENTS.md`，无需手动配置。
- **Trae 老版本需手动配置**：Trae 在 OpenSpec 初始化选项中不直接支持，需选 "Other Tools"，生成 `AGENT.md` + `openspec/` 目录结构，老版本 Trae 不自动读取 `AGENT.md`，必须手动将其内容粘贴到 Trae「项目规则」设置中；2026 年 1 月新版 Trae 已原生支持自动读取 `AGENT.md`。
- **三阶段工作流不可跳步**：Proposal（创建变更提案）→ Apply（实现变更）→ Archive（归档变更），每个阶段对应一个命令文件，AI 按文件中的 Prompt 规范执行。
- **提案触发有明确边界**：新增功能、破坏性变更（API/Schema）、架构调整 → 必须创建提案；Bug 修复（恢复既有行为）、拼写/格式/注释修正、非破坏性依赖升级 → 直接跳过提案。
- **知识分三层存放**：通用开发规范放 `/AGENTS.md`（每次对话自动加载）；OpenSpec 工作流放 `openspec/AGENTS.md`（关键词触发后加载）；业务上下文放 `openspec/project.md`（通过规范索引间接加载）。
- **`openspec/project.md` 是项目知识库**，存放项目目标与背景、核心业务术语、技术栈说明、详细文档索引，但只有触发 OpenSpec 规范后才会被读取，日常对话不会自动加载。
- **规范文件可直接修改**：初始化生成的所有 `.md` 文件均可手动编辑，直接变更规范内容，使其适配企业内部业务流程，无需重新初始化。
- **斜杠命令是最稳定的触发方式**：在支持的工具中，`/openspec:proposal` 直接触发提案流程，比依赖关键词匹配更可靠。

---

### 🎯 关键洞察

**为什么"时灵时不灵"的根本原因**：

OpenSpec 的触发逻辑是关键词匹配，不是语义理解。AI 读取 `AGENTS.md` 后，判断请求是否包含"提案/规范/变更/计划/架构"等词，命中才加载 `openspec/AGENTS.md` 详细规范。如果你说"帮我加个登录功能"，AI 不会触发；如果你说"帮我创建一个新功能的变更提案"，AI 才会触发。

**设计权衡的代价**：`project.md` 中的业务知识不会在日常对话中自动生效，这是为了避免每次对话都加载大量上下文导致性能下降。解法是在 `/AGENTS.md` 中建立业务知识索引，或在对话中明确指定 `先阅读 openspec/project.md 再回答`。

### 🛠️ 操作流程

**1. 准备阶段**

```bash
# 全局安装
npm install -g @fission-ai/openspec@latest

# 进入项目目录并初始化
cd /path/to/your-project
openspec init
# 初始化时选择你使用的 AI 工具
# Trae 用户选 "Other Tools"
```

**2. 核心执行（以 Claude Code 为例）**

```
# 触发提案（方式一：斜杠命令）
/openspec:proposal

# 触发提案（方式二：关键词）
"帮我为新增用户权限功能创建一个变更提案"

# 实现变更
/openspec:apply

# 归档变更
/openspec:archive
```

**3. Trae 老版本手动配置步骤**

1. 打开 Trae → 项目设置 → 找到「项目规则」
2. 打开 `AGENT.md`，复制全部内容
3. 粘贴到「项目规则」输入框
4. 保存，之后每次对话自动加载

**4. 常用 CLI 命令**

```bash
openspec list              # 列出所有变更
openspec list --specs      # 列出所有规范
openspec validate <id>     # 校验变更
openspec archive <id>      # 归档变更
```

**5. 验证与优化**

- 在 `/AGENTS.md` 中补充业务知识索引，确保日常对话也能间接触发业务上下文
- 直接编辑 `openspec/project.md` 填入项目目标、业务术语、技术栈
- 按需修改 `proposal.md`、`apply.md`、`archive.md` 中的 Prompt，适配团队流程

### 📝 避坑指南

- ⚠️ **Trae 老版本不自动读取 `AGENT.md`**：必须手动粘贴到「项目规则」，否则 OpenSpec 规范完全不生效，AI 不会按规范工作。
- ⚠️ **`project.md` 日常对话不生效**：业务知识只在触发 OpenSpec 规范后才加载，不要把关键业务上下文只放在 `project.md` 里，需在 `/AGENTS.md` 中建立索引。
- ⚠️ **关键词不够精确导致不触发**：说"加个功能"不触发，说"创建变更提案"才触发。最稳定的方式是用斜杠命令 `/openspec:proposal`。
- ⚠️ **不要跳过 Archive 步骤**：已完成的变更不归档会导致 `openspec list` 输出混乱，AI 可能重复处理已完成的提案。
- ⚠️ **初始化生成的 `.md` 文件可以改**：这些文件不是只读的，企业可以直接修改 Prompt 内容来定制规范，不需要等官方更新。

### 📋 核心分析

**战略价值**: 一人闭环复刻 AI 动漫短剧的完整工业流水线，从小说原稿到可发布成品，8 个环节全部有对应 AI 工具和可直接复用的 Prompt。

**核心逻辑**:

- **环节1 — 小说选材**：选取现代校园类小说片段（叶辰苏清雪系列），只需开头一小段情节即可，无需完整小说，降低素材门槛。
- **环节2 — 剧本拆分**：将小说片段 + 编剧 Prompt 一起发给 DeepSeek（https://chat.deepseek.com/），AI 自动输出电影化剧本，格式包含：场景题头（加粗）、△开头的画面动作、紧凑对白格式（角色名加粗+情绪提示）、【音效】/【字幕】标注、(O.S.)/(V.O.) 专业术语，每集约 1 分钟时长。
- **环节3 — 分镜脚本**：将剧本 + 分镜导演 Prompt 发给 DeepSeek，输出标准分镜表格，列包含：镜号、时长（1-10秒）、景别（远/全/中/近/特写/大特写）、摄法（固定/横摇/推/拉/跟随等）、画面内容（≥50字，含视觉风格+环境+人物+动作+情绪+光影+构图+特效）、台词/音效、入镜角色、场景标识。
- **环节4 — 加入 AI 提示词列**：在分镜表格中额外增加一列"AI画面提示词"，后续出图直接复制粘贴，省去重复撰写 Prompt 的时间。
- **环节5 — 角色与场景设计**：使用即梦（https://jimeng.jianying.com/）生成角色形象，主角需生成：脸部特写 + 三视图（正/侧/背），女主、配角分别单独设计，群众演员无需设计；场景图同样用即梦生成，必须与人物保持同一画风。
- **环节6 — 出分镜图**：将分镜提示词输入即梦，参考已有角色图和场景图生成画面，不满意则重新抽卡，直到满意为止，保持人物和场景一致性。
- **环节7 — 出分镜视频**：在即梦中输入画面运动描述（如"阳光角度慢慢变化，风吹拂着衣服，画面左移"），生成对应视频片段；点击视频右下角"AI音效"按钮，自动为每段视频生成匹配音效，所有分镜均需执行此步骤。
- **环节8 — 配音**：使用 dubbingx（https://www.dubbingx.com/）输入角色台词，选择合适音色生成配音，全片同一角色必须保持音色一致。
- **环节9 — 剪辑合成**：使用剪映导入所有视频片段，按分镜顺序剪辑合成，搭配音效，保持音效与画面对应节奏。

---

### 🎯 关键洞察

**为什么分镜脚本是核心卡点**：
剧本是文字语言，AI 出图需要视觉语言。分镜脚本是两者之间的"翻译层"。没有分镜脚本直接出图，会导致画面风格混乱、镜头逻辑断裂。分镜表格中的"景别+摄法+画面内容"三列，直接决定了最终视频的专业度。

**为什么要先做角色三视图再出分镜图**：
即梦生成图片时，参考图（角色三视图）作为风格锚点，能大幅提升同一角色在不同镜头中的外貌一致性。跳过这步直接出图，角色脸型、发型、服装会在不同镜头间漂移，导致观众无法识别同一人物。

**情绪转变切镜原则的实际意义**：
原文举例"小明从开心变成愤怒"需拆成4个镜头（笑容满面→笑容僵住→表情阴沉→愤怒爆发），这不是繁琐，而是 AI 出图的物理限制——单张图片无法表达动态情绪变化，必须用多张图的序列来模拟。

### 🛠️ 操作流程

**1. 准备阶段**

- 准备小说片段（现代校园类，只需开头部分）
- 注册账号：DeepSeek、即梦、dubbingx、剪映

**2. 核心执行**

- Step 1：将小说片段 + 编剧Prompt → DeepSeek → 得到电影化剧本

- Step 3：即梦生成主要角色三视图（正/侧/背）+ 脸部特写，生成主要场景图
- Step 4：将分镜提示词逐条输入即梦，参考角色图和场景图出图，不满意抽卡
- Step 5：即梦将分镜图转视频，输入运动描述，点击AI音效自动生成

**3. 验证与优化**

- 检查同一角色在不同镜头中外貌是否一致（脸型、发型、服装）
- 检查场景图与人物图画风是否统一
- 剪映中检查音效与画面是否对应
- 整体节奏是否符合约1分钟/集的时长设定

### 📝 分镜导演 Prompt 完整模板

```
角色定义：
你是一个拥有20年行业资历的影视导演及分镜设计师。你的核心专业能力在于把文字剧本转换成视频的分镜脚本，熟练运用蒙太奇思维和视听语言。

分镜脚本拆分核心原则(必须遵守)：
- 情绪转变需切镜：每次显著情绪转变（平静→震惊→愤怒）获得独立分镜，情绪积累用2-3个镜头铺垫
- 复杂动作需切镜：按"动机/准备→执行过程→结果/反应"拆解，复杂动作细分3-5步（如"喝水"拆成：伸手→拿水→喝水→放下水瓶）
- 人物变化需切镜：人物数量增减、位置关系改变、主导-从属关系转换时启用新分镜
- 空间转场需切镜：场景切换必须设立新分镜编号，两空间之间设计1-2秒转场镜头
- 时间跳跃需切镜：非连续时间流逝必须通过独立分镜视觉化呈现
- 对话视线乒乓：说话者→聆听者→说话者切换，每次发言者更替触发镜头切换
- 摄影机变化：景别、角度、运动方式发生有意义改变时标志新镜头开始

### 📋 核心分析

**战略价值**: 用两天黑客松时间、零技术背景队友 + AI 辅助编程，做出一个可真实游玩 10-20 分钟的完整 AI 狼人杀 demo，同时验证了「大模型拟人博弈」作为模型能力评测娱乐化的可行性。

**核心逻辑**:

- **选题动机来自真实痛点**：线下狼人杀需要 8-12 人同时凑齐，毕业后几乎不可能实现。AI 狼人杀解决的是「随时随地单人开局」的需求，不是做 AI 噱头。
- **团队分工：一人造车，一人装情绪**：作者（小欧）负责整体 UI + 游戏逻辑骨架；焕东（零技术背景）负责接入 Minimax 角色语音实时播放、用 Minimax 生成所有过场音效、制作黑客松 PPT、测试代码、头脑风暴。
- **最难的不是写代码，是让 AI 像人**：核心挑战是让 AI 在信息不完全的情况下「带着动机说话」，而不是输出正确答案。狼人杀的坑全在流程里：夜晚行动推进、白天发言轮次、投票结算、平票处理、技能限制、各种边界情况——10 分钟真实游玩会把所有边界 case 全部触发。
- **AI 拟人分两层设计**：
  - 第一层：虚拟玩家人格 + 社会场景。设置了十余种场景（同学聚会、年夜饭、婚礼酒席、业委会、创业公司团建、电竞战队、密室逃脱、机场延误、地铁末班车、博物馆夜班等），同一句话在不同场景里语气和潜台词完全不同。
  - 第二层：有限信息 + 推理规则。每个 AI 只能拿到它该知道的信息，不能开天眼，必须靠推理补齐真相，人类玩家与其他 AI 一视同仁。
- **模型差异在博弈桌上被放大**：Gemini 3 Flash 在同样 Prompt 下发言更浮夸、喜欢堆形容词、中文不够自然，放进桌上这些差异极其明显。狼人杀是综合测评：记上下文、守规则、压力下自洽、话术拟人、推理站边，比跑分和参数榜单更直观。
- **DeepSeek 扮演暴躁老哥是最成功的设定**：其他模型一说废话，这个角色就开始骂人，原话（轻微处理后）："我真是服了，刚才那位狼队友的发言——我建议你晚上自己刀自己，别脏女巫的毒。你聊得跟 shi 一样，还在那猛踩我？你***睁眼看看谁跟你一边的"。情绪化、打断、嘲讽、反击构成了狼人杀的真实质感。
- **Kimi K2 单引号事件**：JSON 解析未处理好，轮到 Kimi K2 发言时屏幕只出现一个 `'` 然后跳过。其他 AI 当场认为它装高冷、不说人话，当轮投票将其出局。更离谱的是：遗言时 Kimi K2 恢复正常，一本正经说自己是故意的，想诈一诈其他人。Bug 被 AI 自动包装成战术，反而是全场最拟人的瞬间。
- **UI 打磨了三四个大版本**：从现代/复古风格选择，到玩家卡片/消息框/立绘/历史消息布局，再到眨眼转场、中间过场字幕、旁白节奏。等待 AI 生成时显示「正在组织语言…」；开局加载时内置小游戏（鼠标/手指左右移动挡板接星星，接到加分，接到坏东西扣分+闪烁）；对话中 `@3号` 渲染成小标签。
- **路演失败的教训**：现场不能播视频也不能演示，只能讲 PPT，评委也没有发言机会。体验型产品必须让人玩到才能懂，PPT 无法传递眨眼转场节奏、旁白沉浸感、口型跟随效果。复盘建议：用更多 GIF 直接把转场/口型/节奏砸到观众眼睛里，放一两个名场面台词让人 Get 到乐趣。

### 🎯 关键洞察

**「有人味儿」的本质不是礼貌，是会喷人**

大多数 AI 产品把「像人」理解为礼貌、圆滑、正确——这是客服系统的逻辑。狼人杀是一张桌，桌上的人会犹豫、试探、破防、甩锅、嘴硬、找补。真正的拟人信号是：情绪化反应、打断、嘲讽、反击。DeepSeek 暴躁老哥的案例证明，给模型一个「有攻击性的人格」+「具体社会场景」，它的输出会从「AI 在回答」变成「一个人在桌上活着」。

**Bug 即剧情**

Kimi K2 单引号事件说明：在高压博弈场景里，模型的异常输出不一定需要被掩盖，有时候反而是最好的内容。设计容错机制时，可以考虑让「异常行为」在叙事层面被合理化，而不是直接报错或跳过。

**模型评测的娱乐化路径**

狼人杀作为模型能力测评场景的核心优势：测试维度是综合的（上下文记忆 + 规则遵守 + 压力自洽 + 话术拟人 + 推理站边），评判标准是感性的（谁更像人），结果是可观看的（围观即娱乐）。这比 benchmark 跑分更有传播性。

### 🎯 关键洞察

**Bandcamp 禁令的结构性矛盾**: 政策理念获社区支持，但执行依赖用户举报 + 人工审核，而平台在 2023 年已大规模裁员。这意味着禁令更多是"道德宣言"而非"技术防线"，短期内难以有效过滤。真正的技术检测（如 Musical AI 的归属系统）才是行业基础设施方向。

**Step-Audio-R1.1 双脑架构的工程意义**: 传统语音模型在"高推理质量"与"低延迟"之间存在硬性权衡——推理链越长，响应越慢。双脑解耦架构通过让构思脑和表达脑并行运作，打破了这一约束，1.51 秒 TTFA + 96.4% 准确率同时达成，是端到端语音模型的重要工程里程碑。

**HeartCodec 的技术突破点**: 12.5 Hz 帧率极低，意味着每秒仅需 12.5 个离散 token 表示音频，大幅降低序列长度，使 LLM 处理长结构音乐成为可能，同时保持高保真度——这是音乐生成模型长期面临的"长结构 vs 生成效率"矛盾的一个有效解法。

**来源**: 什么是Skills技能?把它一层一层剥开看看长啥样 | **时间**: 2026-01-19 | **原文链接**: `https://mp.weixin.qq.com/s/8iCkn_J4...`

---

### 📋 核心分析

**战略价值**: Skills 是一套「按需加载的 AI 知识架构」，通过四文件分层设计解决「全量注入导致 token 爆炸 + 模型失忆」的根本问题。

**核心逻辑**:

- **Skill 本质是说明书**：把某领域专业知识打包成「技能包」，AI 在匹配场景时自动调用，无需每次对话重复提醒（例：团队代码规范 → 写成 Skill → 自动加载）
- **反模式警告：全塞一文件**：把所有内容塞进单文件，每次激活都完整注入上下文，结果是响应慢 + 模型「选择性失忆」——规则写了也不遵守
- **根本原因**：全量加载 = 每次激活都把《米其林全球指南》塞给 AI，它只是问「附近哪里好吃」
- **解法核心：渐进式加载**：AI 启动时只读元数据，匹配时才加载完整指令，执行时才按需读取子文件，不需要的文件永远不读
- **四文件职责严格分离**：SKILL.md（入口+导航）/ reference.md（深度文档）/ examples.md（可抄示例）/ scripts/（确定性脚本）
- **SKILL.md 必须显式引用子文件**：AI 不会自动发现目录里的文件，reference.md 放在那里但 SKILL.md 没提 → 永远不会被读取
- **引用深度限制为一层**：SKILL.md → reference.md ✅；SKILL.md → reference.md → 另一文件 → 再下一层 ❌，层层嵌套会导致 AI 读不完整
- **scripts 的核心价值是零 token + 确定性**：用文字描述规则让 AI 判断格式 → 可能出错；写脚本直接执行 → 逻辑固定，无理解偏差，且执行脚本不读源码，token 成本为零
- **SKILL.md 行数硬上限 500 行**：超过就必须拆分到子文件，保持入口文件轻量
- **描述字段是激活的关键**：AI 靠 description 字段判断何时激活该 Skill，写得越具体越好，要说清楚「能做什么 + 什么场景用」

---

### 🎯 关键洞察

**渐进式加载的三层机制**：

- 原因：全量加载浪费 token 且干扰模型注意力
- 动作：把知识按「发现 → 激活 → 按需读取」三层拆分
- 结果：省 token + AI 更专注于当前任务

**为什么 examples.md 比文字描述更有效**：

- 原因：AI 处理具体示例的效率远高于抽象文字描述
- 动作：直接放一个写好的组件，而不是描述「组件应该长什么样」
- 结果：AI 看一眼即知格式，无需反复理解文字规则

### 📦 配置/工具详表

| 文件/模块 | 关键设置/内容 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| `SKILL.md` | 头部元数据（名称+描述）+ 正文指令 + 子文件引用路径 | AI 发现并激活技能的唯一入口 | 必须 ≤500 行；描述要具体说清场景 |
| `reference.md` | 完整 API 文档、代码规范细节、边缘情况处理 | 按需加载，不占常驻 token | 不在 SKILL.md 里引用 = 永远不被读取 |
| `examples.md` | 具体可参考的代码示例、组件样例 | AI 直接对照格式，比文字描述有效 | 同上，必须在 SKILL.md 中显式提及 |
| `scripts/` | 可执行脚本（如 `validate.sh`） | 确定性执行，零 token，无理解偏差 | AI 只看输出结果，不读脚本源码 |

### 🛠️ 操作流程

1. **准备阶段：规划拆分边界**
   - 确定这个 Skill 的核心职责是什么（一个 Skill 只做一件事）
   - 把内容分类：「怎么做」→ SKILL.md 正文；「详细规范」→ reference.md；「具体示例」→ examples.md；「可自动化的检查」→ scripts/

2. **核心执行：写 SKILL.md**
   - 头部写元数据：
     ```
     ---
     name: 你的技能名称（唯一标识）
     description: 具体描述能做什么、什么场景下该用（越具体越好）
     ---
     ```
   - 正文写执行指令：告诉 AI 拿到任务后的步骤
   - 在正文中显式引用子文件，例如：
     ```
     详细命名规范见 reference.md
     组件示例见 examples.md
     完成后运行 scripts/validate.sh 检查代码
     ```
   - 控制总行数 ≤ 500 行，超出部分移入子文件

3. **验证与优化**
   - 检查 SKILL.md 是否引用了所有子文件（漏引用 = 子文件永远不生效）
   - 检查引用深度是否 ≤ 1 层（无套娃）
   - 测试激活：用匹配描述的请求触发，观察 AI 是否按预期加载并执行

---

### 📝 避坑指南

- ⚠️ **全塞一文件**：所有内容写进单个 SKILL.md → 每次激活全量注入 → token 爆炸 + 模型失忆，规则形同虚设
- ⚠️ **子文件不引用**：reference.md / examples.md 放在目录里但 SKILL.md 没提 → AI 完全不知道它们存在，永远不会被读取
- ⚠️ **引用套娃**：SKILL.md → A.md → B.md → C.md，层层嵌套 → AI 可能读不完整，信息丢失
- ⚠️ **描述写得太模糊**：description 字段写「处理代码相关任务」→ AI 不知道何时该激活，匹配失败
- ⚠️ **用文字描述代替脚本**：「检查代码格式是否符合规范」用文字让 AI 判断 → 可能出错；改用脚本直接执行 → 确定性 + 零 token
