### 📋 核心分析

**战略价值**: 2026年2月国内开源大模型爆发式集中释放，MoE架构、全模态、扩散LLM、端侧量化四条技术路线同步推进，是判断国内AI格局的关键截面。

**核心逻辑**:

- **DeepSeek鲶鱼效应持续发酵**：GLM、MiniMax、Qwen均在春节前夕密集发布，Qwen3.5甚至除夕当天上线，节奏已完全打破常规发布周期。
- **阶跃 Step 3.5 Flash（2月3日）**：196B总参数，激活11B，45层Transformer blocks中前3层为Dense层、后续为MoE层；浅层Dense设计目的是强化底层特征提取能力，是一种工程上的架构折中。
- **智谱 GLM-OCR（2月3日）**：0.9B轻量模型，切入OCR垂直赛道，体积小、部署成本低，适合端侧/边缘OCR场景。
- **面壁 MiniCPM-o 4.5（2月4日）**：9B全双工全模态模型，打破传统轮次交互概念，模型可主动发起交流，是对话范式的结构性变化。
- **上海AI实验室 Intern-S1-Pro（2月5日）**：1T参数量多模态科学推理模型，专攻AI4Science领域，参数规模为本月最大单体模型。
- **蚂蚁连续四发**：
  - LLaDA 2.1（2月11日）：扩散大语言模型，100B+16B双版本，提出可纠错编辑机制，推理速度达 **892 TPS**，是扩散路线的重要里程碑。
  - Ming-flash-omni-2.0（2月11日）：全模态，100B-A6B规格。
  - Ring-2.5-1T（2月13日）：全球首个基于混合线性注意力架构的万亿参数思考模型。
  - Ming-omni-tts（2月14日）：音频生成模型，0.5B和16.8B-A3B两档规格。
- **智谱 GLM-5（2月12日）**：744B总参数，激活40B，训练数据 **28.5T tokens**，采用DeepSeek Sparse Attention机制，定位后端/推理侧专家模型。
- **千问 Qwen3.5 多波次发布**：
  - 2月4日：Qwen3-Coder-Next，基于Qwen3-Next-80B-A3B-Base增量训练，专攻代码。
  - 2月16日：Qwen3.5-397B-A17B原生多模态，视觉理解能力大幅提升，标志Qwen正式转向原生多模态路线。
  - 2月24日：中尺寸系列三连发——Qwen3.5-35B-A3B、Qwen3.5-122B-A10B、Qwen3.5-27B。
- **腾讯混元 HY-1.8B-2Bit（2月11日）**：端侧量化模型，2Bit量化，边缘设备无压力部署，是端侧压缩路线的代表。
- **新入局者**：京东（JoyAI-LLM-Flash，2月15日）、BOSS直聘（Nanbeige4.1-3B，2月13日，同尺寸登顶）首次出现在开源模型阵营。
- **字节 Seedance 2.0**：本月最炸裂的闭源模型，字节坚持不开源路线依然保持竞争力。

### 🎯 关键洞察

**MoE架构已成标配**：Step 3.5 Flash、GLM-5、Qwen3.5系列、蚂蚁Ming系列全部采用MoE或混合稀疏注意力，激活参数普遍控制在总参数的5%-15%区间，推理成本大幅压缩。

**数据壁垒开始显现**：美团LongCat-Flash-Lite（68.5B）推出DeepResearch功能，作者明确指出"越发体现数据壁垒的重要性"——垂直数据积累正在成为模型差异化的核心护城河。

### 📋 核心分析

**战略价值**: 2026年2月27日，Anthropic 因拒绝删除合同中两条 AI 使用红线而遭特朗普政府"供应链风险"认定，同夜 OpenAI 却以包含相同条款的协议与五角大楼达成合作——揭示这场冲突的本质不是条款内容，而是"谁有权说不"的权力博弈。

**核心逻辑**:

- **Anthropic 是美国政府最深度的 AI 合作方**：首家将模型部署到军方保密云（classified cloud）的公司，首家为国家安全目的定制模型的公司，已覆盖情报界和军方的网络安全、战斗支援等领域，2025年7月通过 Palantir 签下五角大楼 2 亿美元合同。
- **争议仅涉及 1-2% 的使用场景**：Anthropic 对五角大楼 98%-99% 的使用场景无异议，两条红线为：① 国内大规模监控（mass domestic surveillance）；② 全自主武器（fully autonomous weapons，即完全无人类参与的武器系统）。据 Amodei 称，这两条限制在实际军事任务中从未被触发过。
- **五角大楼给出 3 天最后通牒**：要么接受条件，要么被列为供应链风险或面临《国防生产法》（Defense Production Act）强制征用。谈判期间发来的"妥协"条文充斥"如果五角大楼认为适当"、"或按照法律规定做任何事"等模糊表述，实质上未做任何让步。五角大楼首席发言人 Sean Parnell 在截止日前一天仍在推文重申"我们只允许所有合法用途（all lawful use）"。
- **"供应链风险"认定史无前例**：此前该标签仅用于卡巴斯基实验室（俄罗斯政府关联网络安全公司）和外国芯片供应商，从未用于美国本土公司。Amodei 将其定性为"报复性和惩罚性的（punitive and retaliatory）"。
- **政府通知方式：只有推文，没有正式文件**：截至采访时，Anthropic 未收到任何正式政府通知，所有信息来自特朗普 Truth Social 帖子和 Hegseth 的 X 帖子。Amodei 指出 Hegseth 推文存在法律错误——Hegseth 称"任何与军方做生意的公司不能与 Anthropic 进行任何商业活动"，但法律实际只规定在军事合同范围内不能使用 Anthropic，两者差距巨大，目的是"制造恐惧、不确定和怀疑（FUD）"。
- **国内大规模监控红线的技术逻辑**：私人公司收集的美国人位置数据、个人信息、政治归属，政府可合法购买后用 AI 大规模分析——这在法律上不违法，但 AI 时代之前因技术限制不实用，所以法律从未覆盖此场景。第四修正案的司法解释和国会立法均未追赶上来。现实案例：ICE 已使用 Clearview AI 面部识别和 NEC Mobile Fortify 系统追踪个人，ICE 最大承包商 Geo Group 2025年获超 8 亿美元合同用于商业数据库挖掘。
- **全自主武器红线的两重担忧**：① 技术可靠性——AI 模型存在基本不可预测性，可能瞄错目标射杀平民或造成友军误伤；② 监督机制缺失——1000 万架无人机由一人或少数人协调控制，现有问责链条完全失效，"谁在按按钮、谁有权说不"的讨论尚未发生。Amodei 透露曾主动提出在沙盒环境中原型化全自主武器技术，但五角大楼拒绝，除非"从一开始就能想做什么就做什么"。
- **同夜最大讽刺——OpenAI 拿到了相同条款**：Anthropic 被封杀数小时后，Sam Altman 在 X 宣布 OpenAI 与五角大楼达成协议，声称写入了"禁止国内大规模监控"和"人类对使用武力的责任（含自主武器系统）"两项原则，"战争部同意这些原则，在法律和政策中体现了这些原则"。CNN 报道"目前不清楚 OpenAI 的协议与 Anthropic 想要的有什么不同"。
- **R Street Institute 分析师 Mark Dalton 点出根本矛盾**：本周早些时候五角大楼认为 Anthropic 技术不可或缺到要动用《国防生产法》强制获取，现在又说它是"供应链风险"——一个东西不能既是"不可或缺的国家安全资产"又是"供应链风险"，除非真正的问题是谁有权说"不"。
- **Anthropic 商业影响有限但连锁风险存在**：2026年预计收入至少 180 亿美元，估值约 3800 亿美元，被取消的五角大楼合同 2 亿美元占比极小。更大风险是供应链认定引发与五角大楼有业务往来的企业客户出于风险规避减少使用 Claude。

### 🎯 关键洞察

**核心矛盾：条款内容 vs. 解释权归属**

Fortune 分析指出关键差异可能在于措辞方式：Anthropic 试图在合同中明确写出限制条款（即 Anthropic 有权拒绝执行），而 OpenAI 同意五角大楼可将技术用于"任何合法目的"，同时 Altman 也声称将限制写入了协议。Axios 知情人士透露，OpenAI 协议中的限制"反映的是现有美国法律和五角大楼的政策"，意图不是"发明新的法律标准"。

这正是 Anthropic 的核心关切所在：现有法律已经跟不上 AI 的能力了。两家公司的本质分歧不是"要不要这两条红线"，而是"谁有最终解释权"——是供应商还是政府。

**权力博弈的升级路径**：
- 五角大楼立场："所有合法用途（all lawful use）"= 政府自己定义合法边界
- Anthropic 立场：现有法律存在空白，AI 能力已超越法律覆盖范围，在立法到位前供应商有责任充当守门人
- Amodei 的让步与坚守：长期而言应由国会立法（承认政府权威），但国会行动慢，且只需立法一次覆盖这两个具体议题即可

**行业连锁反应**：
- 超过 430 名 Google 和 OpenAI 员工签署公开信支持 Anthropic
- Google DeepMind 首席科学家 Jeff Dean 个人表态反对政府大规模监控
- xAI Elon Musk 站政府一边，称"Anthropic 仇恨西方文明"
- 参议院军事委员会高级参议员在最后通牒到期前写私人信函给双方，承认国防部"无意进行大规模监控或在没有人类参与的情况下使用自主武器"，但也承认"合法使用"问题"需要所有利益相关方进一步努力"
- Altman 在帖子末尾要求五角大楼"将同样的条款提供给所有 AI 公司"

### 📋 核心分析

**核心逻辑**:

- **统一视角公式**：坐在当地 → 食物在桌上（近景）→ 城市地标在窗外（中景）→ 自然山河在最远处（远景），三层景深强制分离，构图不崩
- **提示词结构固定为 7 个模块**：①诗意引言（情绪钩子）②视角定位（时间+地点+朝向）③近景桌面（写实食物清单）④中景城市（地标+建筑写实）⑤远景自然（山脉/水体/天空）⑥光线（时段+方向+反光细节）⑦情绪（时间跨度对比句）
- **近景写实要求**：所有食物「写实如美食纪录片镜头」，具体到每道菜的状态（如「牛油锅底表面花椒和干辣椒密密浮着，气泡不断翻上来」），不能只写菜名
- **中景地标要求**：必须写「真实轮廓」「比例准确」，避免 AI 乱编建筑；同时要写新旧城市断面并置（如北京胡同灰瓦 + 国贸 CBD 同框）
- **光线设定决定氛围**：北京用「深秋午后斜阳右侧低角度」，重庆用「秋末傍晚六点太阳刚落山」，广州用「华南午前漫射白光」，不同城市时段不同，光线参数要具体
- **情绪段落用时间跨度制造张力**：北京「灰瓦一千年、鼓楼七百年、CBD 三十年、长城两千年，全在同一窗框里」；重庆「吊脚楼三百年、解放碑七十年、火锅四十分钟」，用时间落差放大城市厚度
- **固定收尾参数**：每张图统一加「右上角城市名，白色细体，简洁 / 8K，电影级景深，3:4 纵向，纪录片摄影质感」，保证出图风格一致
- **批量跑图工具**：使用 Lovart（https://www.lovart.ai/）接入 NB2，闲时 0 积分，单次可批量跑 100+ 张不断线
- **二次加工流程**：对不满意的图，丢进 Lovart 一次性 roll 10 个变体，从中选最优；纵图改横图用 21:9 比例，适合作公众号封面

### 🎯 关键洞察

**为什么这套框架能批量稳定出图**：

提示词把「不确定性」全部锁死——视角固定（临窗坐姿）、构图固定（三层景深）、食物状态固定（纪录片写实）、光线固定（具体时段+方向）、收尾参数固定（8K/3:4/纪录片质感）。AI 只需要填入城市变量，其余全是约束，所以 300+ 城市跑下来风格不漂移。

**情绪段落的作用不是装饰**：它在提示词末尾给模型一个「叙事锚点」，让画面有时间纵深感而不是单纯的食物摆拍。去掉情绪段，出图会变成普通美食图；加上时间跨度对比，画面会自然带上「历史感」的光线和色调倾向。

### 📋 核心分析

**战略价值**: 从零构建一个「剧本→视频分镜提示词」Skill 的完整工业级 SOP，涵盖结构设计、脚本分工、提示词规范到封装测试的全链路复刻方法。

**核心逻辑**:

- **Skill 四层结构**：`SKILL.md`（定位与入口）+ `scripts/`（执行脚本）+ `references/`（规范知识库）+ `assets/`（交付模板），四层职责严格分离，互不干扰。
- **SKILL.md 是唯一必须存在的文件**，其余目录可按需增减，命名也可自定义。
- **主线思路三步走**：先明确"输入→流程→输出"，再拆分中间步骤，最后统一格式规范，避免做着做着跑偏。
- **七步完整工作流不可跳步**：剧本解析 → 角色提取 → 场景分析 → 分镜生成 → 提示词优化 → 一致性检查 → 导出。
- **统一结构规范是核心防线**：分镜表字段（镜头编号、景别、画面描述、构图、运镜、光线、色调、氛围、时长、镜头角度、提示词）必须在第四步就定死，所有后续脚本都读写同一模板，防止跨幕格式错乱。
- **脚本单独拆分的原因**：脚本负责执行，references 负责规则，assets 负责格式，三者职责不同；规则变了只改 references，逻辑变了只改 scripts，排错成本最低。
- **references 和 assets 建议多轮对话单独优化**，优化到满意后再放入目录，再让脚本调用指定路径，而不是一次性生成完就用。
- **修改时最省 token 的方法**：每次只让 AI 改一件事，明确说"只改哪个文件、不动其他文件、改完告诉我改了哪几行、怎么验证"，避免 AI 到处读文件大改代码。
- **SKILL.md 是否需要改的判断法**：用户使用方式变了 → 改 SKILL.md；只是内部实现变了 → 不用改。
- **提示词优化脚本的关键动作**：术语标准化 + 去重复提示词 + 补质量词（高质量、电影感、清晰对焦等万能词）+ 输出质量打分，让所有镜头风格统一，适配视频模型。
- **一致性检查脚本输出三件事**：问题 + 修复建议 + 复用规范提示词，不只报错，还给解法。

### 🎯 关键洞察

**为什么要先定结构再写脚本**：如果先写脚本再定格式，每个脚本的输出字段会各自为政，后期合并时字段对不上，修复成本极高。先在第四步把分镜表字段锁死，后面所有脚本都读写同一个模板，相当于给整个流程装了一条"轨道"，不会跑偏。

**references vs assets 的本质区别**：
- references = "规则说明书"，告诉脚本"怎么做才对"（术语、模板、规范、指南）
- assets = "空白表格"，提供可直接套用的格式文件（CSV 模板、HTML 模板、速查表）

两者都是为了"复用"，但 references 复用的是判断标准，assets 复用的是交付外观。

### 🛠️ 操作流程

**前期：制定标准**

1. **明确总工作流** — 锁定七步顺序：剧本解析 → 角色提取 → 场景分析 → 分镜生成 → 提示词优化 → 一致性检查 → 导出。没有特别要求时永远执行完整七步，不跳步不合并。

2. **确定输入格式** — 支持 txt/md/docx/pdf/fdx。最低输入：一段话甚至一句话（但需提示用户风格一致性不可控）。推荐输入：剧本 + 角色参考图 + 风格参考图 + 画幅要求 + 时长限制。用户提供的参考图永远优先于 AI 推断。

3. **明确最终产出** — 五样东西：角色设定文档 + 场景设定文档 + 完整分镜提示词表 + 一致性检查报告 + 多格式导出文件（CSV/Markdown/Excel/HTML）。先写清楚产出，防止中途越做越偏。

4. **设计统一结构规范** — 分镜表字段必须在这步锁死：`镜头编号 | 景别 | 画面描述 | 构图 | 运镜 | 光线 | 色调 | 氛围 | 时长 | 镜头角度 | 提示词`。生成 `assets/storyboard_template.csv` 作为所有脚本的读写基准。

**中期：优化功能模块**

5. **准备 references 内容** — 单独生成并优化以下文档，每个独立成文件：`screenplay_format_spec.md` / `character_template.md` / `scene_template.md` / `shot_terminology.md` / `mood_keywords_library.md` / `video_style_guide.md` / `consistency_control.md` / `prompt_patterns.md`。

6. **准备 assets 内容** — 基于第4步结构规范生成：`storyboard_template.csv` / 角色提示词模板 / `export_template.html` / 速查表。

7. **实现剧本解析脚本** (`parse_script.py`) — 读取剧本文件，识别场景标题（INT/EXT 或"第X场/场景X"）、角色名、对白、动作、转场，输出 JSON 字段：`标题`、`场景列表`、`全角色`、`全地点`、`总时长秒数`、`元数据`（含场景数/角色数/地点数）。规则：上一行是角色名则下一行优先判定为对白，否则判定为动作；没有场景标题时自动创建默认场景；空行跳过。优先保证稳健性，任何文本都能读入并形成场景列表。

8. **实现角色提取脚本** (`character_extractor.py`) — 从第7步 JSON 提取角色档案：性别/年龄段/体型/发型/关键词/服装（支持多场次变化追踪）/气质/标志性道具。每个角色输出一段可直接用于 AI 生图的完整提示词描述。

9. **实现场景分析脚本** (`scene_analyzer.py`) — 从每个场景提取：地点/室内外/时间段/光线类型与方向/色彩基调/氛围/天气/季节，为每个场景生成一条基础视觉提示词。

10. **实现分镜生成脚本** (`storyboard_generator.py`) — 按场景自动拆镜头，至少包含建立镜头/角色出场/对话镜头，每个镜头严格按第4步字段输出，先保证"有镜头可用"再优化艺术性。

11. **实现提示词优化脚本** (`prompt_optimizer.py`) — 统一处理：术语标准化（调用 `shot_terminology.md`）+ 去重复提示词 + 补质量词（高质量、电影感、清晰对焦等）+ 输出质量打分。参考 `references/prompt_patterns.md`。

12. **实现一致性检查脚本** (`consistency_checker.py`) — 检查：角色跨镜头提示词漂移 / 场景连续性跳变 / 光影风格突变。输出格式：问题 + 修复建议 + 复用规范提示词。参考 `references/consistency_control.md`。

**后期：测试优化与封装**

13. **确认导出格式** — 默认支持 JSON + CSV + Markdown，HTML 基于 `assets/export_template.html` 生成，支持按场次/角色/场景分类导出。

14. **上传剧本测试** — 用 2-3 份不同类型剧本跑全流程（可 AI 生成）。重点检查：字段是否缺失 / 镜头是否正确生成 / 提示词是否符合 references 规范。发现问题只修改对应脚本，不动其他文件。

15. **封装 SKILL.md** — 生成完整 SKILL.md，包含：定位说明 / 触发词 / 完整七步流程 / 输入要求 / 输出目标 / 文件目录结构。最终目录：`SKILL.md` + `scripts/` + `references/` + `assets/`。

### 📝 避坑指南

- ⚠️ **不要先写脚本再定格式**：字段没锁死就开始写脚本，各脚本输出字段会各自为政，后期合并时对不上，修复成本极高。
- ⚠️ **references 和 assets 不要一次性生成完就用**：建议多轮对话单独优化到满意，再放入目录让脚本调用，否则规范质量低会污染所有下游脚本。
- ⚠️ **修改时不要让 AI 一次改多个文件**：每次只指定一个文件路径，明确说"只改这里、不动其他文件、改完告诉我改了哪几行"，否则 AI 会到处读文件大改代码，token 消耗高且结果不稳定。
- ⚠️ **SKILL.md 不是每次都要改**：只有用户使用方式变了才改 SKILL.md；内部实现变了只改对应脚本。
- ⚠️ **输入不规范时要提前告知用户**：只输入一句话或一段话时，必须告知用户"风格一致性和走向可能不完全可控，建议补充更多信息"，不能静默处理。
- ⚠️ **提示词优化不能只靠脚本**：如果对视频提示词有特定要求，需要单独制作提示词参考文档（放 references）或格式模板（放 assets），并在脚本中显式调用指定路径，不能依赖 AI 自行推断。

**核心逻辑**:

- **无头模式是批量工程化的核心**：`claude -p` 配合 `--allowedTools` 和 `--output-format json` 可以完全脚本化，适合 CI/CD 流水线、大规模迁移任务，无需人工干预。
- **Fan-out 模式解锁大规模迁移**：用 `for file in $(find src -name "*.tsx")` 循环调用 `claude -p`，可以并行处理数百个文件的 Class→Hooks 迁移、测试生成、代码审查，这是单次对话无法完成的规模。
- **Unix 管道让 Claude 成为工具链节点**：`git diff main | claude -p "审查改动"` 或 `npm test 2>&1 | claude -p "分析失败原因"`，Claude 可以无缝嵌入任何现有的 shell 工作流。
- **"Interview Me" 技巧前置需求澄清**：在开发前用 `AskUserQuestion` 工具让 Claude 采访你，生成 SPEC.md，避免"边做边改需求"的返工成本——需求越复杂，这个技巧的价值越高。
- **MCP 的核心价值是零上下文切换**：连上 Slack MCP 后，粘贴 bug 讨论链接直接说"fix"，Claude 读完讨论自动定位并修复代码；连上 BigQuery MCP 后可以直接在 Claude Code 里跑 SQL，Boris 团队 6 个月没手写过 SQL。
- **能用 CLI 替代的 MCP 就替代**：MCP 的工具描述会占用上下文空间，GitHub 操作用 `gh CLI + Skill` 更高效，部署（Vercel/Railway）用 CLI + Skill 即可，只有需要持续连接或实时交互的场景（数据库、浏览器控制、Slack）才值得上 MCP。
- **LSP Plugin 是最值得安装的 Plugin 类型**：`pyright-lsp`、`typescript-lsp`、`rust-lsp` 提供实时类型检查，Claude 编辑完代码立即得到反馈，打断"编辑→构建→报错→修复"的低效循环。

### 🎯 关键洞察

**无头模式的工程化价值**：`claude -p` 不只是"不用打开终端"，它的本质是把 Claude 变成一个可以被脚本调用的函数。`--output-format json` 让输出可以被 `jq` 解析，`--max-budget-usd 2` 让成本可控，`--allowedTools` 让权限最小化。这三个参数组合起来，Claude 就可以安全地跑在 CI/CD 里。

**MCP 配置的常见坑**：MCP 配置必须放在 `.mcp.json`，不是 `settings.json`。`settings.json` 不支持 `mcpServers` 字段，会报 schema 校验错误。这是一个高频踩坑点。

**安全的根本逻辑**：Moltbook 泄露的根本原因是"vibe-coded"——大量 AI 生成代码，缺乏人工安全审查。AI 生成代码的速度越快，安全审查的重要性越高，不是越低。`--network=none` 的 Docker 隔离是最简单有效的防线：被入侵的 agent 无法联网外传数据。

**反向注入防护的原理**：在 Skill 文件中加入安全护栏注释，其权威性高于外部加载的内容，形成对抗恶意注入的"抗体"。这是一个利用 Claude 上下文优先级机制的防护技巧。

### 🛠️ 操作流程

**1. 无头模式批量处理**

```bash
# 批量迁移 React Class 组件到 Hooks
for file in $(find src -name "*.tsx" -type f); do
  claude -p "把 $file 从 Class 组件迁移到 Hooks。保持功能不变。" \
    --allowedTools "Read,Edit" \
    --output-format json
done

# 批量生成单元测试
cat untested-files.txt | while read file; do
  claude -p "为 $file 生成单元测试，覆盖所有边界情况和错误路径。" \
    --allowedTools "Read,Write,Bash(npm test *)"
done

# 批量代码审查（只看本次 PR 改动）
git diff --name-only main | while read file; do
  claude -p "审查 $file 的改动，关注安全和性能问题" \
    --allowedTools "Read,Grep" \
    --output-format json
done
```

# 生成 API 文档
