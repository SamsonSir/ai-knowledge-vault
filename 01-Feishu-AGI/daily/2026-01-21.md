### 📋 核心分析

**战略价值**: 国产开源模型（GLM 4.7 / MiniMax M2.1 / Kimi K2）突破代码能力阈值，使 Claude Code 可脱离官方封号风险落地使用，同时 CC 正推动"Selfware"范式——任何人无需编程即可为自己定制软件，重构个人效率工具链。

**核心逻辑**:

- **封号问题已有解法**：Anthropic CEO 对中国用户封号严重（一月三次），官方订阅不可靠；现在 GLM 4.7、MiniMax M2.1、Kimi K2 三个国产模型均有 Coding Plan，可完全替代官方服务，价格约为官方百分之一
- **CC 不只是代码工具**：它是通用 Agent，能操控文件系统、读写文件、分析数据、合成视频、拆分工资条——29 秒完成一个月的重复性财务工作
- **文件夹是核心设计理念**：CC 启动时绑定一个文件夹作为上下文"游乐场"，每个任务独立文件夹互不干扰，是工作流隔离的基础单元
- **Claude.md 是 CC 的宪法**：每次启动自动加载，记录任务目标和规则，是 CC 的长期记忆和行为约束文件，可让 CC 自动生成
- **危险模式是效率关键**：不开启则每步操作都需确认，极度影响效率；开启后 CC 全自动操控，但必须配合文件夹隔离 + 备份使用
- **Skill 是可复用工作流**：前人验证好的工作流封装，一句 `npx skills-installer install` 即可加载，无需从零构建
- **SaaS 护城河正在崩塌**：以前功能复杂性是护城河，现在独立开发者用 AI 两周可复制 Zendesk 80% 核心功能；SaaS 股票下跌、公司转型 AI 基础设施是市场信号
- **Selfware 范式转移**：软件从"标准化服务"转向"完全贴合个人习惯"——设计师的素材自动分类器、销售的话术效果追踪器、作家的知识图谱助手，这些需求太个性化，任何 SaaS 都不会做，但 CC 可以
- **AI 创业核心竞争力转移**：技术门槛降至历史最低，竞争焦点从"技术 vs 技术"变为"洞察 vs 洞察"；一个人 + 四天 + Claude Code + Vercel + OpenAI API 可复制传统需要三人团队三个月的产品
- **数据护城河是新壁垒**：AI 模型可复制，私有数据不可复制；帮企业把私有数据用好、把特定场景打磨好是可持续的创业方向

### 🎯 关键洞察

**为什么现在是转折点（三重叠加）**:

1. AI 能力：GPT-4 / Claude 3.5 Sonnet 已能理解复杂业务逻辑，无需自训模型，只需会用 API
2. 开发工具：Claude Code / Cursor / Replit Agent 让"描述需求"替代"写代码"
3. 云基础设施：Vercel / Railway / Fly.io 让 `git push` 即完成部署，无需管服务器

**Selfware vs 低代码平台的本质区别**:
- 低代码：在平台框架内搭积木，能做什么受限于平台给了什么
- Selfware（CC）：你想做什么就做什么，没有框架，只有可能性
- 低代码会说"对不起，我们没有这个数据源"；CC 会说"好的，我来写代码从你的 App 拉数据"

**Selfware 三核心特征**:
- 完全贴合个人习惯（软件适应你，不是你适应软件）
- 持续进化（随时告诉 CC "改成这样"，动态迭代）
- 不可复制（你的工具对别人可能完全用不了，但对你是最完美的）

### 🛠️ 操作流程

**1. 准备阶段**
- 准备科学网络环境
- 下载安装 Node.js（`https://nodejs.org/en/download/`）
- Windows 用户额外安装 Git for Windows（`https://git-scm.com/install/windows`）
- 注册智谱开放平台账号（`https://open.bigmodel.cn/`）
- 在 `https://bigmodel.cn/usercenter/proj-mgmt/apikeys` 创建 API Key 并复制备用
- 购买 Coding 套餐（`https://www.bigmodel.cn/glm-coding?ic=KSGMCBOXUT`）

**2. 核心执行**
- 打开终端，执行 `npm install -g @anthropic-ai/claude-code`
- 若报错，macOS/Linux 用 `curl -fsSL https://claude.ai/install.sh | bash`，Windows 用 `irm https://claude.ai/install.ps1 | iex`
- 执行 `claude --version` 验证安装成功
- 执行 `npx @z_ai/coding-helper`，输入 Y，按中文界面提示粘贴 API Key，完成服务器配置
- 建立专属 Claude Code 文件夹，每个任务建子文件夹
- 终端输入 `claude` 启动，或安装 Claude Code Now（`https://claudecodenow.com/`）用启动器

**3. 验证与优化**
- 启动后打招呼测试连通性
- 让 CC 自动创建 Claude.md 文件，把重要规则和任务背景告诉它
- 拖拽文件/文件夹到 CC 测试文件读取
- 用 `Control + V` 测试图片粘贴
- 尝试第一个真实任务（如拆分表格、整理数据）
- 按需安装 Skill：`npx skills-installer install @anthropics/claude-code/[skill名] --client claude-code`

### 📝 避坑指南

- ⚠️ 危险模式必须配合文件夹隔离使用，CC 全自动操控可能删除或修改重要文件，操作前务必备份
- ⚠️ 已安装 Cursor 等 AI 编程工具的用户，强烈建议在这些工具内部终端安装 CC，避免环境冲突报错
- ⚠️ 图片粘贴快捷键是 `Control + V` 而非 `Cmd + V`，终端环境与系统剪贴板行为不同
- ⚠️ 官方 CC 订阅封号严重（一月三次），中国用户不推荐使用官方订阅，优先选国产模型 Coding 套餐
- ⚠️ Selfware 工具维护成本自担，工具出问题需自己或问 CC 修复，复杂工具维护成本较高
- ⚠️ Selfware 深度依赖 AI 平台 API，平台涨价/改规则/停服会影响工具可用性，需有备用方案
- ⚠️ 创业方向避免同质化：门槛低导致竞争者多，今天的点子明天可能有三人在做，需靠垂直领域深度洞察而非技术能力建立壁垒

### 📋 核心分析

**战略价值**: 用一个「个人主页 + AI工具导航」完整案例，手把手演示从痛点分析→快速demo→可行性讨论→版本规划→PRD→设计→开发→Debug的全链路AI编程工作流。

**核心逻辑**:

- **产品开发是循环，不是线性**：流程是 痛点→demo→PRD→设计→开发→上线→回到PRD，每次迭代都走完整圈，不是一次性做完
- **痛点分析两层思维**：第一层先回答「帮谁解决什么问题」，第二层在做中不断迭代对痛点的洞察；允许一开始回答不清楚，但必须回到用户视角，不能自嗨
- **快速demo的核心价值**：不是做出完美产品，而是验证技术可行性、发现布局问题（如工具列表太长需分类）、发现架构问题（如硬编码不够灵活），这些发现直接影响版本规划和PRD内容
- **可行性讨论必须提前做**：用模板向AI提问5个维度——技术方案（纯前端 or 需要后端）、数据存储（是否需要数据库）、工作量（大概多长时间）、风险（可能的坑）、替代方案（有没有更简单做法）；提前暴露坑，避免做到一半发现需求压根没法干
- **版本规划的核心原则**：V0.1 MVP只做最核心一个痛点，可以丑、可以手动、可以有限制，但必须能跑通；每个版本只解决一个清晰子需求；每个版本必须有明确「验证点」（如何判断这个版本做完了）；明确写「不做什么」避免范围蔓延
- **PRD的三个核心价值**：① 版本管理，功能迭代不抓瞎；② 逼你把产品细节想清楚（AI会问很多问题补充遗漏细节）；③ 减少开发过程中人和AI的理解偏差
- **设计讨论两步走**：先用ASCII图确定布局（结构优先，改起来成本低），再讨论风格和美学（配色、字体、交互）；不要一开始就纠结颜色字体
- **开发三原则**：① 基于PRD每次只干一件事，不要让AI一下子全干完；② 让AI先讲他要怎么做，逻辑合理了再让他做；③ 每一步都要验收，不能跳步
- **Debug四层方法**：① 先把问题变成「可观察的」（打日志暴露关键路径）；② 用事实而不是感觉和AI对话（发生了什么、控制台输出了什么）；③ 基于证据让AI缩小问题范围（用日志排除，不是一次性大改）；④ 修改→立即验证→再反馈，形成闭环

### 🎯 关键洞察

**失败案例的价值**：QuickSave浏览器插件案例说明——即使痛点真实存在（每天看AI资讯想收藏），产品也可能失败。原因是没有在做中迭代对痛点的认知，闷头做完发现方向偏了。这是为什么「快速demo验证」和「版本规划」如此重要的底层逻辑。

**Debug的本质转变**：AI编程时，人的角色不是「修代码的人」，而是「问题定位的导演」。最忌讳的是把「它好像没生效」这种情绪判断喂给AI——AI拿到的必须是执行结果（日志、报错、控制台输出），才能给出有效分析。

### 🛠️ 操作流程

#### 1. 准备阶段：痛点分析 + 快速Demo

**痛点分析**（回答这两个问题）：
- 做一个什么样的产品？
- 帮谁解决什么样的问题？

**个人主页 + AI工具导航案例的痛点**：
- AI工具太多，收藏夹乱七八糟，想用时找不到
- 想展示自己，但纯自我介绍太boring
- 想给别人推荐工具，没有集中展示的地方

**快速Demo目的**：验证能不能做、发现问题（不需要完美，能跑就行）

Demo完成后检查清单：
- ✅ 技术上是否可行
- ✅ 布局是否合理
- ❌ 发现的问题（如：工具列表太长需分类、硬编码需改成可编辑）

#### 2. 可行性讨论（直接用这个模板）

```
我想做一个产品：个人主页 + AI工具导航

我已经做了一个简单demo，现在想做正式版本。

请帮我分析：
1. 技术方案：纯前端 or 需要后端？
2. 数据存储：是否需要数据库做存储？
3. 工作量：大概需要多长时间？
4. 风险：有什么可能的坑？
5. 替代方案：有没有更简单的做法？

我的需求：
- 要能部署上线到github和vercel
- 要能编辑工具列表
- 手机和电脑都要能看
```

#### 3. 版本规划（V0.1 → V1.0）

版本规划提示词核心逻辑（完整提示词见原文）：

- **V0.1 MVP**：只做最核心一个痛点，能跑通即可，不做编辑/云同步/高级功能/美化
- **V0.2-V0.5**：优先级顺序：① 本地增删改查 → ② 数据备份/版本管理 → ③ 跨设备/分享 → ④ 导入导出等进阶功能
- **V1.0**：性能优化、错误处理、UI/UX打磨、文档引导，达到可对外发布质量

每个版本必须包含：功能清单 + 不做什么 + 验证点 + 预计工作量

#### 4. PRD写作（两种方式）

**方式1：提示词写PRD**（云舒版，Gemini 2.5 Pro）

三步确认法：
1. **定框架**：梳理业务流程/用户旅程，分阶段，获得用户确认后才进入下一步
2. **逐故事**：按阶段逐一讨论每个用户故事，UI故事必须画ASCII线框图，每个故事讨论完必须「单点确认」
3. **终稿确认**：回顾全局对齐清单（数据契约、接口流程、性能约束、异常策略、监控指标），用户说「可以生成了」才生成文档

PRD输出格式包含：
- 综述（项目背景 + 核心业务流程/用户旅程地图）

### 📋 核心分析

**战略价值**: 用 AI 搜索 GitHub 开源项目 → 用 skill-creator 一键 Skill 化 → 注入 Agent，让任何人都能把人类30年开源积累变成自己的私人武器库。

**核心逻辑**:

- **重复造轮子是反效率的**：GitHub 上几乎所有常见需求都有经过时间鞭打的成熟开源项目，稳定性和成功率远超临时让 AI 写的代码。
- **Skills 的结构优势**：Skills 可以把脚本 + Prompt 打包在一起，这是单 Prompt 或单脚本做不到的，天然适合封装开源项目。
- **支持平台**：Coze（扣子）、OpenCode、Claude Code，只要装了 `skill-creator` 这个官方 Skill，均可执行 GitHub 项目 Skill 化。
- **AI 搜索替代 GitHub 检索门槛**：不懂 GitHub 的普通人，直接用 ChatGPT（推荐 GPT-5.2 Thinking，搜索能力强、幻觉低）提问「有没有做 XX 功能的 GitHub 开源项目」，AI 会直接推荐最优解。
- **构建 Skill 时推荐 Claude 4.5 Opus**，首次运行调试时推荐 **GPT 5.2 Codex**（运行体验比 Claude 4.5 Opus 好 N 倍），后续运行模型无所谓。
- **先 Plan 后 Dev 的两阶段策略**：先让 Agent 进入 Plan 模式规划封装方案，确认无误后再切换开发模式，成功率更高、后期稳定性更强。
- **首次运行必然有 BUG，要迭代固化**：第一次运行会遇到环境问题（如 YouTube 防爬需要浏览器扩展导出 Cookie），AI 会全程指导解决，解决后把经验更新回 Skill 文件，后续运行从几分钟压缩到十几秒。
- **Skill 自我管理**：卡兹克自建了一个「管理 Skills 的 Skill」，可直接对本地所有 Skill 执行卸载、删除、修改、优化操作，避免每次进文件夹手动管理。
- **Skill 化的本质是能力固化**：每个 Skill 经过「首次运行 → 发现问题 → 迭代更新」后固化，成为 Agent 中一个可靠的、可复用的技能节点。

结果：Skill 的成功率、稳定性、效率全面碾压临时代码。

**为什么普通人之前用不了 GitHub 开源项目？**
两道门槛：① 不知道有哪些项目；② 部署需要命令行，环境配置卡死大多数人。

**核心逻辑**:

- **痛点定位**：手动找图库耗时，AI 直出图需每次手写提示词 + 人工点击，流程碎片化，只适合封面场景，正文配图几乎无法规模化。


### 📋 核心分析

**战略价值**: 麻衣画布是一款节点式AI多模态创作工具，支持图文/图像/视频/音乐全链路生成，完全免费，仅需自备API算力。

**核心逻辑**:

- **安装极简**：下载 exe（Windows）或 dmg（Mac）双击即开，无需安装依赖，连接API后即用；本地接收器为可选项，不用也能正常生成，但需手动保存图片/视频/json文件
- **API接入方式**：双击exe → 右上角「API设置」→ 将密钥粘贴至「全局KEY」；也可留空全局KEY，在各节点单独配置分组密钥；API自带中转，测试失败先检查令牌或关闭梯子
- **推荐算力平台两家**：贞贞（`https://ai.t8star.cn/register?aff=HrOY49381`）和 comfy（`https://ai.comfly.chat/register?aff=DSr267109`），comfy为源头，贞贞每1000点算力价格更高，按需选择
- **画布核心操作**：双击左键 = 新建/选择节点；滚轮 = 缩放；左键拖拽节点框 = 移动；Ctrl+S 保存；Ctrl+Z 撤销；Alt+左键框选 = 打组；框选后 Shift+左键拖动 = 批量移动连接线；P键 = 查看性能帧数
- **3.1.0新增关键功能**（2026-01-17）：极简UI重设计、智能分镜表全新改版、节点轮盘联动下一窗口、打组功能（Alt+框选）、批量选择移动（Shift+拖动）、自定义下载默认地址、内存可视化窗口、VEO3.1-4K发送格式、URL可自定义默认地址
- **节点工作流逻辑**：所有操作均以「节点连线」为核心——双击画布呼出节点 → 选择对应功能节点 → 填写参数/上传素材 → 连线 → 运行；节点之间的连接顺序直接影响首尾帧、多图顺序等结果
- **视频生成两种模式**：单帧生成（1张图→视频）和首尾帧生成（2张图分别作为起止帧→视频），首尾帧需注意连线先后顺序，避免首尾帧错乱
- **Sora角色库机制**：需先将角色视频（5秒）上传获取URL，在Sora角色库中创建角色；调用时选择Sora2模型后自动出现角色列表，点击角色头像出现 `@c61a766a1.daisyspr` 类似提示即为调用成功；仅适用于Sora2系列，其他模型不可用
- **多模态串联工作流**：文生图节点 → 图生视频节点 → 视频拆解/提示词反推节点 → 音乐生成节点，可全链路串联，也可单独调用任意环节
- **提示词优化原则**：中文提示词效果不佳时翻译成英文（模型为国外训练，英文理解更准确）；视频生成需反复抽卡，不断优化提示词直到满意

### 🎯 关键洞察

**节点连线顺序是核心变量**：多图生视频、首尾帧生视频时，连线顺序直接决定AI理解的「第一帧」和「最后帧」，顺序错误会导致画面逻辑混乱。操作时务必按照「图1→图2→AI视频节点」的顺序依次连线。

**视频反推提示词的正确姿势**：将已生成视频发送到对话框，使用提示词「反推视频提示词，精确到秒，以便用于AI生成视频」，可获得逐秒分镜描述，用于复刻或二次创作。

**4K图片编辑的内存限制**：同时编辑多个4K图片可能报错，建议分批处理或降低分辨率后再编辑。

### 🛠️ 操作流程

#### 文生图

1. **准备阶段**: 确认API密钥已填入全局KEY，算力账户有余额
2. **核心执行**:
   - 双击画布空白处 → 选择「AI绘画节点」
   - 输入提示词 → 选择图像模型（如FLUX、Z-IMAGE、香蕉等）→ 设置分辨率和尺寸比例
   - 点击运行
3. **验证与优化**: 在侧边栏「生成历史」模块查看效果图；可连接「预览窗口节点」实时预览

#### 单图生视频

1. **准备阶段**: 准备好源图片
2. **核心执行**:
   - 双击画布 → 选择「图片输入节点」→ 上传图片
   - 双击画布 → 选择「AI视频节点」→ 选择模型（推荐Veo3）和参数
   - 将图片输入节点连线至AI视频节点 → 运行
3. **验证与优化**: 预览窗口右下角三个点直接下载；或在侧边栏生成历史双击视频后下载

#### 首尾帧生视频

1. **准备阶段**: 准备首帧图片和尾帧图片各一张
2. **核心执行**:
   - 双击画布 → 调出两个「图片输入节点」，分别上传首帧和尾帧图片
   - 调出「AI视频节点」
   - 按顺序连线：首帧图片节点先连，尾帧图片节点后连，均连入AI视频节点
   - 选择支持首尾帧的模型（Grok3支持首尾帧控制）→ 运行
3. **验证与优化**: 检查视频首尾是否与上传图片一致，不一致则检查连线顺序

#### 多图生视频

1. 同首尾帧流程，三图则调出三个图片输入节点
2. 连线顺序严格按照画面时序：第1帧→第2帧→第3帧→AI视频节点
3. 注意节点框连接先后顺序，避免首尾帧图片错乱

#### Sora角色调用

1. 按上表方法1或方法2创建角色，获得角色ID（格式如 `@c61a766a1.daisyspr`）
2. AI视频节点选择「Sora2」或「Sora2 Pro」模型
3. 模型选择后自动出现已创建角色列表 → 点击角色头像 → 提示词中出现 `@xxx` 标记即调用成功
4. 多角色调用：在提示词中依次添加多个 `@角色ID`

#### 视频反推分镜

- **方法A（对话框）**: 侧边栏生成历史 → 右键视频 → 发送到当前对话 → 输入提示词：`反推视频提示词，精确到秒，以便用于AI生成视频`
- **方法B（节点）**: 双击画布 → 「视频输入节点」上传视频 → 连接「视频拆解/提示词反推节点」→ 运行获得逐帧分镜描述

#### AI音乐生成

1. 双击画布 → 调出「AI音乐生成节点」
2. 选择「灵感模式」（AI自动生成）或「自定义模式」（手动填写风格/歌词等）
3. 根据需求设置参数后运行
4. 注意：Suno纯音乐模式下也可能出现语言文字，属正常现象，多抽卡即可
