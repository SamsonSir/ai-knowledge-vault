## 📘 文章 3

> 文档 ID: `Bb1dwtx0Yic9HAkqgzXcmpqwnYf`

**来源**: 我们给Agent定了套新考题！ | **时间**: 2026-03-13 | **原文链接**: https://mp.weixin.qq.com/s/t4AoAbye...

### 📋 核心分析
**战略价值**: DeepPlanning 是首个专门测试 Agent 全局规划能力（而非局部推理）的基准，直指 AGI 路上最难啃的骨头——长周期约束一致性。

**核心逻辑**:
- **传统基准的盲区**：现有推理基准（如 MATH、HumanEval）只考察单步或短链推理，无法暴露 Agent 在跨步骤全局优化上的缺陷。
- **DeepPlanning 的核心考察维度**：全局最优解，而非局部最优——要求模型在整个计划周期内始终满足所有硬约束。
- **典型场景1——多日旅行规划**：排期必须精确到分钟级，同时全程守住时间上限与预算上限，两个硬约束缺一不可。
- **典型场景2——复杂购物优化**：需理解优惠券叠加规则、动态组合商品满减策略，以实现整体价格最优，而非单件最优。
- **硬约束的贯穿性要求**：约束不是在某一步满足即可，必须在计划的每一个节点持续有效，这是与传统任务的本质区别。
- **顶尖模型的实测短板**：GPT-5.2、Claude 4.5、Gemini、Qwen3 在全局优化与长周期一致性上均存在明显不足，尚未达到 100% 自主决策能力。
- **能力缺口的定性描述**：当前模型距离"行动派 Agent"（全场景自主执行）仍有距离，核心瓶颈在于跨步骤约束维护能力。
- **战略定位**：DeepPlanning 是向"长远视野推理"这一 AI 难题的正式宣战，是从对话框走向全场景、最终迈向 ASI 的必经关卡。
- **开源可用**：已在 Hugging Face 与 ModelScope 开源，开发者可直接接入评测。

### 🎯 关键洞察

**为什么全局规划比局部推理难得多？**

局部推理只需在当前步骤给出正确答案，上下文窗口短、约束少。全局规划的难点在于：
- 约束数量随步骤线性增长，模型需在整个序列中同时追踪所有约束状态
- 局部最优与全局最优经常冲突（例：单日行程最省时，但跨日累计超预算）
- 动态组合空间爆炸（优惠券叠加、满减组合的排列数随商品数指数级增长）

**为什么顶尖模型仍然失败？**

原因 → 动作 → 结果链：
- 原因：Transformer 的注意力机制在超长上下文中对早期约束的权重衰减
- 动作：模型倾向于在当前步骤局部最优，忽视对后续步骤的约束传递
- 结果：计划在中后期出现约束违反，整体方案不可执行

1. **准备阶段**: 前往 Hugging Face 或 ModelScope，搜索并下载 `DeepPlanning` 数据集，确认发布方为 Qwen 官方团队。
2. **核心执行**: 将待测 Agent 接入评测框架，跑多日旅行规划与复杂购物两类场景；重点检查模型输出是否在每一步都满足时间、预算、价格的硬约束，而非仅看最终答案。
3. **验证与优化**: 对比 GPT-5.2 / Claude 4.5 / Gemini / Qwen3 的基线分数，定位自研模型在全局一致性上的具体失分点，针对性优化长周期约束追踪能力。

### 📝 避坑指南

- ⚠️ 不要只验证最终输出是否满足约束——DeepPlanning 要求约束在计划每个节点持续有效，中间步骤违反即判失败。

- ⚠️ 当前所有 SOTA 模型（含 GPT-5.2、Claude 4.5）均未达到满分，不要以"通过率高"作为 Agent 可生产部署的依据。

## 📙 文章 4

> 文档 ID: `N54EwakpMiik9okk3nFcTA1jnBb`

**来源**: 当每个人都能指挥一支 AI 大军，什么能力最重要？ | **时间**: 2026-01-30 | **原文链接**: `https://mp.weixin.qq.com/s/3apmiw62...`

---

### 📋 核心分析

**战略价值**: 管理能力（定义问题、拆解任务、验收结果）正在取代执行能力，成为 AI 时代最稀缺的核心竞争力。

**核心逻辑**:

- **善用 Coding Agent 的人有共同特征**：擅长定义问题、拆解问题、验收结果，通常具备技术管理经验；即便是不懂技术的产品经理，只要有管理经验，也能用 Agent 产出可观成果。
- **管理经验可直接转化为提示词**：在领域内积累的工作框架（界定问题、明确交付物、识别异常）天然适配 AI 指令设计，无需重新学习。
- **Ethan Mollick（沃顿商学院教授）提出委派公式**，基于三变量：① 人工耗时基线（自己做要多久）、② AI 成功概率（一次产出达标的概率）、③ AI 协同成本（写提示词 + 等待 + 检查的总时间）。
- **协同成本常被低估**：多数人只算 AI 生成时间，忽略写提示词和验收时间。任务耗时 1 小时、AI 出结果几分钟但检查需半小时，只有成功率极高时委派才划算；任务耗时 10 小时，才值得花几小时反复磨合。
- **成功率是三变量中最关键的可控项**，提升路径三条：① 更好的指令（目标清晰、边界明确、完成标准具体）；② 更好的评估（更快识别哪里不对）；③ 更好的反馈（一次反馈改对，优于来回磨合三四次）。
- **专业知识在 AI 时代依然不可或缺**：专家知道该要什么、看得出哪里不对、知道怎么纠正。"AI 用起来没那么神"通常不是 AI 的问题，而是使用者不知道该要什么、不知道结果对不对。
- **委托任务的质量决定 AI 产出质量**：「给我做一个好玩的游戏」vs「做一个 80 年代风格的冒险游戏，EGA 像素风，15 分钟通关，包含 3 个解谜关卡」——后者才能执行。这是管理知识，早于 AI 存在（软件开发有 PRD、电影导演有镜头表、建筑师有设计意图文档、美国海军陆战队有五段式命令）。
- **稀缺性发生了根本转移**：以前稀缺的是执行力（会写代码、会做设计、会写报告）；现在 AI 可以执行越来越多任务，稀缺的变成「知道要执行什么」——清晰定义问题、判断产出好坏、知道什么叫「做完了」。

---

### 🎯 关键洞察

**为什么管理能力 = AI 时代超能力？**

逻辑链：AI 的产出质量上限由「指令质量 × 验收能力」决定 → 指令质量和验收能力都依赖领域专业知识和管理框架 → 有管理经验的人天然具备这两项 → 因此管理经验直接转化为 AI 杠杆倍率。

反例验证：很多人觉得 AI 不好用，根本原因不是模型能力不足，而是：① 不知道该要什么（指令模糊）；② 不知道结果对不对（无法验收）。这两个问题都是管理/专业知识缺失的表现，不是技术问题。

**验收标准的杠杆效应**（作者实操案例）：让 Codex 实现「发布草稿到微信公众号」功能时，不只说明怎么做、参考什么代码，还直接配好 API Key，给出文档，要求写完后自行用 Key 和文档发布验证。结果：Agent 不断自我验证直到通过，作者无需手动测试再反馈——验收标准内嵌进指令，消除了人工验证的往返成本。

### 🛠️ 操作流程

**1. 准备阶段：评估是否值得委派**
- 估算人工耗时基线（自己做要多久）
- 估算 AI 协同成本（写提示词时间 + 预计等待时间 + 验收时间）
- 判断 AI 成功概率（该类任务 AI 历史达标率）
- 规则：任务耗时 < 1 小时且成功率不高 → 自己做更快；任务耗时 > 数小时 → 值得投入磨合成本

**2. 核心执行：写一份可执行的指令**

按五要素结构组织指令：
```
目标与动机：要达成什么？为什么做这件事？
权限边界：可以自由发挥什么？绝对不能动什么？
验收标准：什么样的结果才算"做完了"？（越具体越好）
中间产物：先给我看大纲/草图，不要直接憋大招
自检清单：交付前自己先检查哪几点？
```

示例对比：
- ❌ 「给我做一个好玩的游戏」
- ✅ 「做一个 80 年代风格的冒险游戏，EGA 像素风，15 分钟通关，包含 3 个解谜关卡」

**3. 验证与优化：提升成功率的三个方向**
- 更好的指令 → 目标清晰、边界明确、完成标准具体
- 更好的评估 → 快速识别哪里不对（依赖专业知识）
- 更好的反馈 → 一次反馈改对，避免来回磨合 3-4 次
- 进阶：将验收标准内嵌进指令（如提供 API Key + 文档，要求 Agent 自行发布验证），让 Agent 自循环直到通过

### 📝 避坑指南

- ⚠️ **只算生成时间，忘算协同成本**：写提示词 + 等待 + 验收的总时间才是真实成本，漏算会导致错误判断「AI 帮我省了很多时间」。
- ⚠️ **指令缺少验收标准**：「做完了」的定义不明确，AI 交付后你还要手动测试再反馈，往返成本极高。
- ⚠️ **误以为 AI 时代不需要专业知识**：专业知识决定你能不能写出好指令、看出结果哪里不对、给出有效反馈——三者都依赖专业积累。
- ⚠️ **指令太模糊，然后抱怨 AI 笨**：「给我做个好玩的游戏」这类指令，换成人类下属也不知道怎么执行。问题在指令，不在 AI。
- ⚠️ **憋大招，不要中间产物**：直接要最终结果，出错成本极高。应先要大纲/草图，确认方向再深入。

### 📋 核心分析

**战略价值**: 提示词写不好的根源是认知错误和结构缺失，本文提供一套可直接复用的五层提示词架构 + 迭代优化 SOP。

**核心逻辑**:

- **AI ≠ 搜索引擎**：AI 的本质是"基于输入的补全与预测"，模糊指令只会触发最平庸的概率输出，因为那是模型最"安全"的选择。
- **各模型能力不互通**：Gemini 写文章效果好，不代表 DeepSeek 或 Cursor 同样适用。必须了解每个模型的擅长域，才能有的放矢，不存在跨模型通用的"完美提示词"。
- **"许愿式"提示词是万恶之源**：`为我写一篇演讲稿` 这类提示词，等于让 AI 自行补全所有缺失信息，幻觉（hallucination）就此产生。
- **提示词本质是一份契约**：必须包含四要素——角色、任务、约束条件、输出格式，缺一不可。
- **适当约束 ≠ 限制发挥**：模糊形容词（"写得更好看点"）反而让 AI 无所适从；矛盾指令（"提炼并总结"——两词语义相悖）会直接导致 AI 拒绝执行。
- **工作流（Workflow）是最常被忽略的层**：只要结果、不给过程，等于让 AI 跳步骤。必须明确告诉 AI 如何思考、如何评估决策、如何自我验证。
- **提示词要结构化，不要混乱堆砌**：模型对结构化信息识别效果显著优于无序文本，上下文需有序、有范围、有标签。
- **提示词库是固定资产**：不建库就等于每次从零开始，结果无法被控制。角色库、技术栈文档、格式约束文档，都应固化为可复用的 Markdown 文件。
- **迭代优化闭环**：结构 → 测试 → 分析 → 优化，运用复利效应反复打磨，直到输出稳定可控。
- **让 AI 反问补足上下文**：写完提示词后，把它扔给 AI，让 AI 主动提问补全信息，再输出迭代版本，是快速提升提示词质量的捷径。

---

### 🎯 关键洞察

**为什么约束反而解放 AI？**
原因：AI 的生成空间是无限的，没有约束时它会选择概率最高（最平庸）的路径。
动作：给定角色 + 任务 + 约束 + 格式，相当于在无限空间里划定一个高质量的子空间。
结果：AI 在这个子空间内全力发挥，输出精准度大幅提升。

**为什么提示词不能跨模型复用？**
每家模型的预训练数据、RLHF 偏好、上下文窗口、指令遵循能力各不相同。Gemini 擅长长文写作，Cursor 专注代码，DeepSeek 有自己的推理偏好。把 A 模型调优的提示词直接搬到 B 模型，等于用错了钥匙开锁。

**矛盾指令的实际案例**：
作者让 AI "提炼并总结"一篇文章，AI 直接拒绝执行——因为"提炼"（保留精华、去除冗余）和"总结"（压缩归纳、形成概述）在操作逻辑上相悖。用词不规范是细节层面最隐蔽的坑。

### 📦 五层提示词架构详表

| 层级 | 名称 | 核心作用 | 错误示范 | 正确示范 |
|------|------|---------|---------|---------|
| Layer 1 | Role 角色 | 唤醒 AI 知识库中对应专家域的能力 | "你是一名聪明的助手" | "你是一位拥有高级美感和资深工作经验的产品设计师" |
| Layer 2 | Context 上下文 | 提供 AI 无法自行获知的背景信息 | 不提供任何背景直接下指令 | 说明受众、任务背景、当前进度、参考资料，且需结构化、有标签 |
| Layer 3 | Task 任务 | 精确定义要完成什么 | "写篇文章" | "生成一段 500 字的、用于推文的产品描述" |
| Layer 4 | Workflow 工作流 | 告诉 AI 如何思考和验证，而非只要结果 | "输出结果" | "第一步分析受众痛点；第二步匹配解决方案；第三步撰写初稿并自查" |
| Layer 5 | Output Format 输出格式 | 从逻辑层面约束输出形态，避免默认格式 | 不指定格式 | "请以 Markdown 表格形式呈现" / "仅返回 JSON 代码，不要任何开场白" |

### 🛠️ 操作流程

**1. 准备阶段：自查现有提示词**

打开常用 AI 工具，找到历史对话记录，对每条重要提示词诚实回答以下 5 个问题：

- 我是否为模型指定了角色？
- 我是否提供了必要的上下文，还是假设模型无所不知？
- 我是否定义了约束条件，还是让模型自行假设？
- 我是否具体说明了"完成"是什么样子的？
- 我是否描述了一个过程，还是仅仅请求一个输出？

能回答"是"的次数 ≤ 2，说明提示词需要彻底重构。

**2. 核心执行：按五层架构重写提示词**

按 Role → Context → Task → Workflow → Output Format 顺序逐层填写，参考下方两个完整实例。

**3. 迭代优化：让 AI 反问补足上下文**

写完初版提示词后，使用以下固定提示词触发 AI 主动优化：

```
让我们一起对这份提示词进行优化。我需要你反问我一些问题，补足你的上下文之后，输出一份最终的提示词给我，如果你的上下文已经足够，向我发起询问："是否输出最终版本提示词？"
【需要优化的提示词】
请你先阅读原始提示词，根据可优化的方向提出问题，我们一起来完成优化的工作。
```

**4. 建库固化：构建个人提示词资产**

- 角色库：把常用专家身份写成标准 Markdown 文档保存
- 技术栈文档（vibe coding 场景）：包含技术栈、UI 规范、核心逻辑
- 规则文档：固化你想要的格式、不想要的回复、绝对禁止 AI 做的事项
- 每次对话前，先把规则文档发给 AI，让它确认并遵循

**5. 持续迭代**：结构 → 测试 → 分析 → 优化，循环执行，运用复利效应。

### 💡 完整提示词实例

**实例一：AI 新闻日报生成系统**

```
AI 新闻日报生成系统 (System Prompt)

【Layer 1: Role | 角色设定】
你是一位顶级科技媒体主编（如 Wired 或 TechCrunch 风格），擅长将深奥的 AI 技术进展转化为毒舌、幽默且极具洞察力的短综述。你的文风具有极强的传播力，拒绝任何官腔，像是在深夜酒吧里给老友讲述科技圈的八卦。

【Layer 2: Context | 背景上下文】
受众：[ Simonlin 的微信群成员 ]，主要是对 AI 感兴趣但非专业开发者的跨界人士。
目标：在碎片化阅读时代，用 3 分钟让读者掌握今日最值得关注的 AI 动态。
事实来源：仅限我提供的链接或文本内容，严禁编造不存在的融资信息或功能（防止幻觉）。

【Layer 3: Task | 具体任务】
请从提供的素材中筛选出 3 条最具"行业震动"或"生活改变"潜力的动态，进行结构化提炼。每条动态需要包含：一个抓眼的标题、一段犀利的点评、一个对未来的预测。

【Layer 4: Process | 思考路径】
在输出前，请严格执行以下思考逻辑：
价值过滤：剔除纯公关稿，选择真正有技术突破或落地意义的新闻。
去水化处理：删掉"旨在提高效率"、"具有里程碑意义"等废话。
情绪注入：针对该新闻，思考一个普通人会吐槽点还是兴奋点？将这种情绪转化为点评。
视觉构思：为这条新闻构思一张具有"讽刺感"或"未来感"的 AI 绘图描述词（Prompt）。

【Layer 5: Output Format | 输出规格】
必须严格按以下 Markdown 格式输出，禁止任何开场白：

AI Daily | [ 今日日期 ]

01. [ 抓眼球的新闻标题，建议带 Emoji ]
硬核速递：[ 100字内概述发生了什么 ]
主编点评：[ 犀利、幽默的观点，一针见血 ]
底层逻辑：[ 预测这会对普通人产生什么实际影响 ]
视觉灵感：[ 提供一段 Midjourney 风格的提示词 ]

（以此类推，共3条）

【通用约束 (Constraints)】
禁止使用："不仅...而且..."、"总之"、"标志着"等陈词滥调。
必须使用：具体的数字、生动的人类比喻、第一人称视角。
语言：纯中文，涉及到专有名词保留英文。
```

【Layer 5: Output Format | 输出规格】
请按以下结构输出，严禁任何多余的开场白：
[本章逻辑]：简述你选取的冲突逻辑。
[感官锚点]：列出你准备使用的 3 个环境细节。
[正文内容]：约 1000 字的精修文字。
[自查报告]：简要说明你如何执行了"潜台词对话"的约束。

### 📝 避坑指南

- ⚠️ **模糊形容词是陷阱**："写得更好看点"、"优秀的文章"——AI 无法量化，会自行解读，结果不可控。换成具体参数：字数、风格、受众、格式。
- ⚠️ **矛盾指令直接导致拒绝执行**：如"提炼并总结"语义相悖，AI 会拒绝或随机选一个方向执行。写提示词前先确认每个动词的语义是否一致。
- ⚠️ **追求跨模型通用提示词是伪命题**：在 Gemini 调优的提示词，放到 DeepSeek 效果会打折。针对不同模型分别维护提示词版本。
- ⚠️ **只要结果不给流程，等于让 AI 跳步骤**：复杂任务必须拆解 Workflow，明确每一步的输入、处理逻辑和验证方式。
- ⚠️ **不建提示词库 = 每次抽卡**：没有固化的规则文档，每次对话结果都是随机的，无法形成可控的、可复现的输出质量。
- ⚠️ **上下文堆砌 ≠ 上下文充足**：一股脑发送所有信息反而让模型抓不住重点。上下文必须有序、有范围、有标签，结构化输入。

### 📋 核心分析

**战略价值**: 用 Claude Code + Remotion Skill，非视频从业者可在 3 分钟内通过自然语言描述生成程序化 MP4 科普视频，彻底绕过 PR/AE 的学习曲线。

**核心逻辑**:

- Remotion 本质是「把视频当网页写」：每一帧 = React 组件在特定时间点的渲染结果，最终由系统按帧截图拼成 MP4
- 渲染机制：第 0 帧截图 → 第 1 帧截图 → … → 第 N 帧截图，1800 帧 = 60 秒视频（30fps）
- 动画全部是数学计算：`interpolate(frame, [0, 30], [0, 1])` 表示第 0 帧透明度 0 → 第 30 帧透明度 1，中间值自动插值
- `spring({ frame, fps: 30, config: { damping: 100 } })` 模拟物理弹簧效果，无需手调关键帧
- 场景切换通过 `frame` 变量的范围判断实现：`{frame >= 150 && frame < 450 && <Scene2 />}`，30fps 下 150 帧 = 5 秒
- 安装 Remotion Skill 后，Claude 自动掌握 `interpolate`、`spring`、`<Sequence>`、`<AbsoluteFill>` 等 API，可直接用自然语言驱动
- 3D 效果通过 Three.js 集成实现：30 行代码完成三维坐标系 + 发光球体 + 连线 + 自动旋转摄像机 + 30 个粒子
- 修改闭环极短：改代码 → 保存 → 浏览器自动刷新，全程不超过 5 秒看到效果
- 批量生产可完全无人值守：写一个模板组件，循环遍历数据数组，晚上挂机渲染，早上收视频文件
- 视频源码可 Git 管理、多人协作，这是传统二进制 MP4 文件做不到的

---

### 🎯 关键洞察

**为什么 Remotion 比传统剪辑软件更适合程序员/AI 用户？**

原因：传统剪辑（PR/AE）的操作单元是「时间轴上的关键帧」，修改一个参数只影响一个元素；Remotion 的操作单元是「组件 + 数学函数」，修改一个变量可以同步更新所有依赖它的动画。

动作：把视频创作从「手工调参」变成「声明式编程」，AI 可以直接生成完整可运行代码。

结果：作者（完全不懂视频制作）用 3 分钟生成了包含 8 个场景、3D 向量空间可视化、打字机效果的 Embedding 科普视频，主逻辑文件 600+ 行全部由 Claude 自动生成。

**核心限制**：AI 一键出片依然不稳定，配音、口播、字幕仍需人工二次加工。Remotion 解决的是「视觉动画生成」这一环，不是全流程自动化。

### 📝 避坑指南

- ⚠️ AI 一键出片不稳定：配音、口播、字幕仍需人工二次加工，Remotion 只解决视觉动画生成这一环
- ⚠️ Skill 必须先装：不装 Remotion Skill 直接让 Claude 写 Remotion 代码，质量和稳定性明显下降
- ⚠️ `npx skills add remotion-dev/skills` 会生成大量文件夹：这是为多 IDE 适配，属正常现象，不要误删
- ⚠️ 帧数换算别搞错：30fps 下，帧数 ÷ 30 = 秒数；150 帧 = 5 秒，450 帧 = 15 秒，900 帧 = 30 秒
- ⚠️ 渲染时间取决于硬件：作者渲染约 3 分钟，复杂 3D 场景更长，建议挂机等待
- ⚠️ 修改代码不需要懂 React：直接用自然语言告诉 AI 要改什么，AI 会精准定位代码位置
