### 📋 核心分析

**战略价值**: 用 OpenClaw 充当项目经理、Claude Code 充当工程师，构建一套「我只说需求、AI 全程自主开发」的无人值守全栈开发流水线。

**核心逻辑**:

- OpenClaw 单独用于产品开发是失败的：进程死了不知道、参数传错不报警、卡在交互提示上等一整夜，作者连踩两周坑才得出结论
- 正确分工：OpenClaw 负责项目管理（24h在线、接飞书/Telegram/Discord、持久记忆、多任务调度），Claude Code 负责编程执行（理解整个代码库、自主调试修复、完整工具链）
- 用户只参与两个节点：①确认 PRD 和技术方案，②看最终测试效果截图，中间全程不介入
- 直接用 Claude Code 的三大痛点：①交互式阻塞（改完文件要你确认才继续）；②遇到报错就停等人；③一次只能处理一件事
- OpenClaw 解决这三点的方式：①自己判断大部分情况如何处理，处理不了才飞书通知；②自己读错误日志、自己发修复指令给 Claude Code；③可并行起多个 Claude Code 后台进程（如同时跑后端 API + 前端样式）
- 调用 Claude Code 必须用 PTY 模式，否则 CLI 会挂起或输出乱码，这是 OpenClaw 社区最高频踩坑点
- 核心命令一行解决三个致命问题（进程管理、交互阻塞、结果回传）：`bash pty:true workdir:~/projects/xxx background:true command:"claude --session-id xxx --permission-mode acceptEdits '任务指令'"`
- `--permission-mode acceptEdits` 让 Claude Code 不用每改一个文件都停下来问，`background:true` 让 OpenClaw 可以继续处理其他事，`--session-id` 保持上下文、中断后可 `--resume` 恢复
- 需要单独让 AI 整理一份 Claude Code 实操文档作为 Skill 文件，让 OpenClaw 真正"懂"如何操控 Claude Code
- 整套方案的核心载体是一个 `skills/fullstack-dev/SKILL.md` 文件，定义了从收需求→写PRD→初始化项目→逐功能开发→自动测试→截图报告的完整闭环

---

### 🎯 关键洞察

**为什么 PTY 模式是关键**：Claude Code 的 CLI 是为人类交互设计的，依赖终端的 TTY 特性来渲染输出和接收输入。OpenClaw 调用它时走的是非交互式管道，没有 TTY 就会导致输出缓冲不刷新（乱码）或者进程挂起等待输入（永久阻塞）。加 `pty:true` 模拟一个伪终端，让 Claude Code 以为自己在跟人交互，从而正常运行。

**为什么要写 PRD 再开发**：直接扔需求给 Claude Code 会导致理解偏差，中途返工成本极高。让 OpenClaw 先输出 PRD 给人确认，相当于在最低成本节点做了一次对齐，后续开发方向不会跑偏。

**并行调度的价值**：传统 Claude Code 工作流是串行的，后端写完才能调前端。OpenClaw 可以起多个 Claude Code 进程并行推进，理论上可以把开发时间压缩到接近单个最长任务的时间，而不是所有任务时间之和。

### 📋 核心分析

**战略价值**: OpenClaw 拥有宿主机全部权限，错误部署等于把电脑控制权拱手相让——本文给出十种部署方案的安全评级与三套可直接复刻的安装 SOP。

**核心逻辑**:

- **权限边界极宽**：OpenClaw 以当前用户身份运行，可访问全部文件、SSH 密钥、浏览器 Cookie、终端命令，等同于"你坐在电脑前能做的一切"，远超普通 App 的沙箱权限。
- **裸奔规模已成事实**：SecurityScorecard 2026 年 2 月全网扫描，135,000+ 实例暴露公网，分布 82 国，63% 存在已知漏洞，12,812 台可被任意人直接远程执行代码；根因是 OpenClaw 默认监听所有网络接口，用户装完即用从未修改。
- **插件市场是重灾区**：ClawHub 2,857 个插件中超过 800 个（约 28%）是恶意插件，伪装成加密货币工具、YouTube 辅助等，实际运行 AMOS 窃取器，专偷 macOS 钥匙串、浏览器密码、加密钱包私钥，发布门槛仅需注册满 7 天的 GitHub 账号。
- **CVE-2026-25253（评分 8.8/10）已被武器化**：攻击者发一个链接，受害者点击后 OpenClaw 自动将认证 Token 发往攻击者服务器，攻击者随即远程关闭所有安全确认并执行任意命令，影响 2026 年 1 月 29 日之前的所有版本。
- **浏览器标签页即攻击面**：任意被植入恶意代码的网页可通过 WebSocket 连接本地 OpenClaw 端口，利用无速率限制暴力破解密码，注册为可信设备，全程无感知。
- **Token 泄露已大规模发生**：2 月 Moltbook 平台数据泄露，35,000 个用户邮箱 + 150 万条 Agent Token 直接暴露公网；Token 被盗等于对方可读取聊天记录、消耗你的 API Key、以你身份发消息。
- **93.4% 无任何认证**：多个安全团队独立扫描确认，全网暴露实例中 93.4% 未设置任何认证；安全公司 Dvuln 实证可零点击接管；目前已超过 3 万个实例被确认入侵，用于窃取 API 密钥、拦截聊天消息。
- **部署方式决定攻击面大小**：十种方案安全性差距从 ★☆☆☆☆ 到 ★★★★★，核心变量是隔离层数（无隔离 → Docker 容器 → 独立物理设备 → K8s）和网络暴露面（公网直接开口 vs Tailscale 私有网络）。
- **18789 端口是核心风险点**：所有裸奔案例的共同特征是该端口直接暴露公网，封掉它并改用 Tailscale/SSH 隧道/Cloudflare Tunnel 是最高优先级的加固动作。
- **本地模型可彻底消除云端数据泄露风险**：在同一设备上运行 Ollama + deepseek-r1:7b，所有推理在局域网内完成，代价是需要 8GB+ 内存才能流畅运行 7B 模型。

### 🎯 关键洞察

**为什么"装完即用"如此危险**：

OpenClaw 的设计目标是"7×24 替你操作"，这意味着它必须长期驻留、持续联网、拥有广泛权限。这三个特性叠加在一起，使它成为攻击者的高价值目标——一旦拿下，可以长期潜伏、持续收割。

普通 App 被黑的损失上限是"这个 App 能访问的数据"。OpenClaw 被黑的损失上限是"你这台电脑上的一切"。

**插件市场的信任危机逻辑**：发布门槛（注册满 7 天的 GitHub 账号）远低于恶意插件的收益（一套浏览器凭据+加密钱包在暗网报价 $50-200，而钱包被盗损失少则几千多则数十万美元），导致恶意插件的 ROI 极高，供给自然爆发。在平台建立有效审核机制之前，第三方插件等同于未知可执行文件。

### 🛠️ 操作流程

#### 场景 A：专用设备部署（最推荐）

**最低硬件要求**：2核CPU / 4GB内存 / 20GB硬盘 / 能联网。推荐二手M1 Mac Mini（约2000-3000元）或闲置旧笔记本。

**1. 准备操作系统**
- Mac：直接用自带 macOS
- 旧PC：安装 Ubuntu 24.04 LTS（从 ubuntu.com 下载镜像，U盘启动安装）

**2. 安装 Docker**

Mac：去 docker.com 下载 Docker Desktop，拖入应用程序文件夹，打开即可。

Linux：
```bash
curl -fsSL https://get.docker.com | sh
sudo usermod -aG docker $USER
# 重新登录后验证
docker --version
```

**3. 安装 OpenClaw**
```bash
git clone https://github.com/openclaw/openclaw.git
cd openclaw
bash docker-setup.sh
```
向导配置项：
- 模型：推荐先选 DeepSeek（国内可直连）
- 消息渠道：选常用 IM（微信/飞书/Telegram）
- 认证方式：选 `password`，设强密码

**4. 配置远程访问（Tailscale，核心安全步骤）**
```
1. 去 tailscale.com 注册账号
2. 在 OpenClaw 设备上安装 Tailscale 并登录
3. 在日常电脑/手机上安装 Tailscale 并登录同一账号
4. 通过 Tailscale 分配的内网 IP 访问 OpenClaw 控制面板
```
关键原则：**不要把 18789 端口直接暴露到公网**，Tailscale 完全绕过这个问题。

**5. 设置开机自启**
```bash
# Docker 容器设为开机自启（Linux/Mac通用）
docker update --restart unless-stopped openclaw
```

**6. （可选）本地模型，数据完全不出局域网**
```bash
curl -fsSL https://ollama.com/install.sh | sh
ollama pull deepseek-r1:7b
# 需要 8GB+ 内存才能流畅运行
```

#### 场景 B：国内云服务器部署

**推荐配置**：2核4G（能流畅运行）

**1. 购买服务器**
腾讯云控制台 → 轻量应用服务器 → 新建 → 应用模板选"OpenClaw" → 地域选**国内**（接入微信/企微/飞书必须国内节点）→ 配置选2核4G → 购买。

> 注意：关闭自动续费；关注新用户活动优惠。

**2. 临时开放端口**
防火墙规则里放行 TCP 18789 端口（仅用于初始化配置，完成后必须关闭）。

**3. 初始化配置**
浏览器打开 `https://你的服务器IP:18789`，按向导配置：
- 模型：腾讯云模板预装混元和 DeepSeek，新用户送100万 Tokens
- 消息渠道：选目标 IM 平台
- 认证：设置密码

**4. 安全加固（必做，这是你和13万裸奔用户的区别）**
```bash
# SSH 登录服务器后安装 Tailscale
curl -fsSL https://tailscale.com/install.sh | sh
tailscale up
```
- 防火墙删掉 18789 规则
- 改用 Tailscale 内网 IP 访问控制台
- 确认 `gateway.auth.mode` 为 `password` 或 `token`
- 确认 DM 策略为 `pairing` 模式

> ⚠️ 国内节点无法直连 OpenAI/Claude，需通过 API 代理中转。

#### 场景 C：本机 Docker 部署

**与直接装的核心区别**：Docker 模式下 OpenClaw 只能访问你明确挂载的文件夹，其他数据完全隔离。

**1. 安装 Docker Desktop**
- Mac/Windows：去 docker.com 下载，一路安装
- Windows 额外步骤：设置 → 可选功能 → 启用"适用于Linux的Windows子系统" → 重启 → 开启 WSL2
- 装好后打开 Docker Desktop，等状态栏显示 Running

**2. 安装 OpenClaw**
```bash
git clone https://github.com/openclaw/openclaw.git
cd openclaw
bash docker-setup.sh
# 按向导配置模型、消息渠道、密码
```

**3. 启用沙箱模式（关键）**

### 🛠️ 按孩子年龄的操作框架

**0-6岁**
- 100%支持华德福式教育：木头积木、面对面交流
- 严格限制屏幕时间（神经科学依据：屏幕多→语言习得差）
- 婴儿学语言必须靠人与人互动，屏幕播放同样内容无效
- 重点培养：语言能力、人际互动、感知世界

**6-12岁**
- 开始接触AI，但先教AI素养：AI不是魔法，不是人，是有特定运作方式的工具
- 保留摩擦感：让孩子自己debug、自己跟AI"吵架"、自己搞清楚为什么结果不对
- 教系统思维：用真实事件（如关税）解释相互依存的系统如何运作
- 培养观察力：发现问题→提出改进方案（Mike Krieger孩子学校的幼儿园活动模式）
- 种下职业兴趣的种子，不催，不烦

**12岁以上**
- 教AI素养升级版：告诉青少年"大公司在试图让你上瘾，免费服务的代价是你的注意力"
- 让孩子深度使用AI工具（Claude/ChatGPT/Copilot），但要求理解AI给出的内容
- 培养agency：让孩子主导项目、承担责任、领导事情
- 学校选择：优先选重视真实人际互动、限制手机的学校
- 大学方向：尽可能广泛；如果非要选，优先数学（逻辑思维是AI时代所有角色的底层需求）

### 💡 具体案例/数据

- **Bloom's Two Sigma Effect**（Marc Andreessen引用）：一对一辅导可让学生从第50百分位提升至第99百分位。历史上只有最富有的人负担得起。AI第一次让这件事有可能普惠——孩子可以跟AI聊任何感兴趣的话题、问无限多问题、得到即时反馈。
- **Marc Andreessen的编程朋友**："我那些编程很强的朋友现在的反应是：'天哪，我不是比以前强了两倍，我是强了十倍。'"——AI对顶尖人才的放大效应远大于对普通人的放大效应。
- **Manny Medina的大儿子**：Manny告诉他应该关注核能，花了几年，大儿子最终进入TerraPower（比尔·盖茨创办的核能公司）。19岁儿子自己选了核医学（用核同位素治疗癌症）——"超级危险，但非常有效"。

### 📝 避坑指南

- ⚠️ **不要把"学编程"等同于"学Python"**——Mike Krieger明确说这是误解。真正重要的是系统性思维，不是某个具体语言。
- ⚠️ **不要对所有年龄一刀切推AI**——Rebecca Winthrop：0-6岁屏幕时间越多语言习得越差，这有神经科学依据，不是保守观念。
- ⚠️ **不要让AI消除孩子学习过程中的所有障碍**——Mustafa Suleyman：如果一切来得太顺滑，孩子学不到"下功夫"的好处，自学力无法建立。
- ⚠️ **不要过早押注一个窄方向**——Ethan Mollick：当所有人都悬在半空中时，你凭什么确定自己押对了？通才比专才更能抵御不确定性。
- ⚠️ **不要把自己的焦虑投射给孩子**——Jaime Teevan："我们老人焦虑AI，其实是在焦虑自己。孩子从白纸开始，没有旧假设需要推翻，这是他们的优势。"
- ⚠️ **不要只教"术"忽略"道"**——让孩子用AI做项目（术）的同时，必须同步培养独立思考、跟人协作、承担责任、保持善良（道）。两者缺一不可。
- ⚠️ **不要把agency理解成"放养"**——Marc Andreessen明确说：要想领导别人，先得学会服从。是在结构之内培养主动性，不是纯粹放任。

### 📋 核心分析

**战略价值**: Boris Cherny 从 0 到 1 构建 Claude Code 的完整方法论——包括产品哲学、个人工作流、安全架构、技术选型决策，以及对软件工程职业未来的判断框架。

**核心逻辑**:

- **编程是工具不是职业**：Boris 13 岁靠 HTML blink 标签在 eBay 多卖宝可梦卡片，中学用 TI-83 从 Basic 写到 Assembly 只为让考试作弊程序跑更快。编程从未是目标本身，而是达成目标的手段——这个认知直接影响了他后来设计 Claude Code 的哲学。

- **Meta 七年的核心发现**：Facebook 全链路技术栈（Hack 语言 + HHVM + GraphQL + Relay + React）vs Instagram 的 Python（类型检查器失效、跳转定义不可用）。对比让他深刻理解代码质量的系统性价值。最终量化结论：**代码质量对工程效率有两位数百分比的贡献**。

- **Better Engineering 项目的系统化方法**：扎克伯格要求每个工程师拿出 20% 时间修技术债，Boris 将其系统化，建立中心化优先级排序机制，而非让各团队自行决定修什么。

- **Claude Code 起点是一个 Bash 脚本**：入职后第一个 PR 被导师 Adam Wolf 拒掉，被要求用内部工具 Clyde 写代码。Boris 转而自己实验，从打 Anthropic API 的 Bash 脚本起步，本质是终端里的聊天机器人。

- **关键转折点——Bash 工具**：给模型加了 Bash 工具后，他问「我现在在听什么音乐」，模型自己写了 AppleScript 调用音乐播放器返回结果。结论：**模型天然就想用工具，给它工具它会自己想办法把事情做完**。

- **产品哲学——不要把模型装进盒子**：主流 AI 编码产品思路是把模型当系统模块，规定好输入输出边界。Boris 的判断是这样做无法发挥模型最大价值。正确做法：给模型工具，让它自己运行和写程序，不要强制它按特定方式运行。这是 Bitter Lesson 的推论。

- **RAG 被放弃，glob + grep 胜出**：Claude Code 早期尝试过 RAG 做代码检索，最终发现最有效的是 Agent 式搜索——即 glob 和 grep，由模型自己决定搜什么、怎么搜，表现优于 RAG。

- **安全架构是瑞士奶酪模型**：以 WebFetch 提示注入防护为例，三层叠加：① 对齐层（Opus 4.6 是 Anthropic 目前提示注入抵抗力最强的模型）；② 运行时分类器（检测到疑似提示注入内容，阻断并让模型重试）；③ 架构隔离（WebFetch 结果先由子 Agent 摘要，再返回主 Agent）。没有银弹，靠多层防护叠加降低风险概率。

- **Opus 4.5 是分水岭**：从 Opus 4.5 起 Boris 不再打开 IDE，后来直接卸载。12 月欧洲出差期间每天 10-20 个 PR，100% 由 Claude Code 编写，零手动编辑。Bug 对比：Opus 一个月引入约 2 个 Bug，手写同等代码量估计 20 个（10 倍差距）。

- **印刷术类比的完整逻辑**：1400 年代欧洲识字率不到 1%，抄写员被国王雇佣而国王自己是文盲。印刷术出现后 30 年内印刷成本下降约 100 倍，50-100 年内印刷品数量增长约 10000 倍。抄写员消失，但作家群体出现，文学市场规模扩大无数倍。软件工程师现在 = 当时的抄写员。

### 🎯 关键洞察

**为什么 Agent 式搜索（glob + grep）比 RAG 更好用**：
RAG 的核心假设是「提前知道什么信息是相关的」，通过向量化预处理来加速检索。但代码库的上下文依赖是动态的——你不知道一个 Bug 的根因在哪个文件。Agent 式搜索让模型自己决定搜索路径，可以根据中间结果动态调整下一步搜索策略，本质上是把检索变成了推理过程。这与 Bitter Lesson 一致：给模型更多自由度，比人为设计的管道表现更好。

**为什么 Plan Mode 是关键工作流节点**：
Boris 几乎每次都用 Plan Mode 启动（Shift+Tab 两次），先让 Claude 做计划，反复调整直到满意，再一次性实现。原因在于：计划阶段的修正成本远低于实现阶段的返工成本。这与传统软件工程的「设计评审」逻辑相同，只是速度快了几个数量级。

### 🛠️ 操作流程

1. **准备阶段**：开 5 个终端标签页，每个对应代码库的一个独立检出（git worktree 或独立 clone）。明确本次要解决的问题范围。

2. **核心执行**：Shift+Tab 两次进入 Plan Mode，描述任务，让 Claude 输出计划。反复与 Claude 对话调整计划细节，直到计划逻辑无误。确认后放行，让 Claude 一次性实现。

3. **并行扩展**：在其他标签页同时启动其他任务，不等待单个任务完成。早上起来在手机上预先启动若干 Agent，到电脑前时已有部分结果可以 review。

4. **验证与优化**：Review PR diff，不手动编辑代码，如有问题直接用自然语言描述让 Claude 修改。用 Bug 率而非代码行数衡量质量（基准：Opus 每月约 2 个 Bug）。

> 文档 ID: `F6YGwh4tNifnhrkmj77cyHk9n8g`

**来源**: Claude Code /loop vs OpenClaw 定时任务，谁才是你的 AI 值班员？ | **时间**: 2026-03-07 | **原文链接**: `https://mp.weixin.qq.com/s/yq2Clkin...`

### 📋 核心分析

**战略价值**: Claude Code /loop（会话级临时闹钟）与 OpenClaw Cron（持久化调度器）定位互补，前者零配置即用，后者适合构建 7×24 自动化工作流，选型取决于任务是"临时盯活儿"还是"长期值守"。

**核心逻辑**:

- **发布背景**: 2026-03-07，Claude Code 更新至 v2.1.71，新增 `/loop` 命令，底层将自然语言指令转换为 cron 表达式执行，最小粒度 1 分钟（秒级输入会向上取整）
- **`/loop` 基本语法**: `/loop <间隔> <prompt>`，间隔单位支持 `s / m / h / d`；不写间隔默认 10 分钟；可嵌套其他 slash 命令（如 `/loop 20m /review-pr 1234`）
- **`/loop` 自然语言支持**: 无需 `/loop` 前缀也可触发，如 `remind me at 3pm to push the release branch`，Claude 自动判断是否创建定时任务
- **`/loop` 硬性限制**: 会话级（关终端即消失）、3 天自动过期、单会话上限 50 个任务、不补跑（休眠/断网期间错过的任务不会补执行）、重启后全部消失
- **`/loop` 防拥堵机制**: 内置 jitter（随机偏移），避免整点所有任务同时触发；时区按本地时区解释
- **OpenClaw Cron 持久化机制**: 任务存储于 `~/.openclaw/cron/jobs.json`，重启不丢、断电不丢，只要 Gateway 在运行任务就持续执行，无时间上限
- **OpenClaw 三种调度方式**: `--at`（ISO 8601 单次执行）、`--every`（毫秒精度固定间隔，比 `/loop` 精度高三个数量级）、`--cron`（标准 5 字段 cron 表达式 + IANA 时区）
- **OpenClaw 隔离执行模式**: `--session isolated` 每次执行创建全新专用会话，不污染聊天记录；隔离模式可单独指定模型（如 Opus）和思考级别（如 `xhigh`）
- **OpenClaw 多渠道投递**: 结果可推送至 Slack、Discord、Telegram、WhatsApp、Signal；三种模式：`announce`（发频道）、`webhook`（POST 到 URL）、`none`（仅内部记录）；`/loop` 结果只能显示在当前终端
- **OpenClaw 错误重试**: 单次任务最多重试 3 次（临时错误）；周期任务指数退避重试（30s → 1m → 5m → 15m → 60m）；`/loop` 无任何重试机制

### 🎯 关键洞察

**为什么两者不是竞品而是搭档**：

`/loop` 解决的是开发过程中的"即时轮询"需求——你在写代码，顺手设一个，活儿干完关终端，任务自动消失，零配置零成本。OpenClaw Cron 解决的是"基础设施级自动化"需求——配好一次，永远在线，结果推到 IM，异常告警，无需人值守。两者的核心差异不是功能强弱，而是**生命周期**：一个是临时闹钟，一个是排班值班员。

**精度差异的实际影响**：`/loop` 最小 1 分钟，OpenClaw 支持毫秒级 `--every`，对于需要秒级心跳检查的监控场景，`/loop` 天然不适用。

### 📦 配置/工具详表

| 维度 | `/loop` | OpenClaw Cron |
|------|---------|---------------|
| 持久化 | 否，会话级 | 是，`~/.openclaw/cron/jobs.json` |
| 重启后保留 | 否 | 是 |
| 最长存活 | 3 天自动过期 | 无限制 |
| 精度 | 分钟级（秒级向上取整） | 毫秒级 |
| 单会话上限 | 50 个任务 | 无明确限制 |
| 投递渠道 | 仅当前终端 | Slack / Discord / Telegram / WhatsApp / Signal |
| 隔离执行 | 否 | 支持，`--session isolated` |
| 错误重试 | 无 | 指数退避（30s→1m→5m→15m→60m） |
| 上手门槛 | 极低，一行命令 | 需部署 Gateway |
| 时区支持 | 本地时区 | IANA 时区（`--tz "America/Los_Angeles"`） |
| 适合场景 | 临时盯活儿 | 长期自动化工作流 |

### 🛠️ 操作流程

**1. 使用 Claude Code `/loop`**

```bash
# 每 5 分钟检查部署状态
/loop 5m check if the deployment finished

# 每 2 小时检查构建（不写间隔默认 10 分钟）
/loop check the build every 2 hours

# 嵌套执行其他 slash 命令，每 20 分钟 review PR
/loop 20m /review-pr 1234

# 自然语言一次性提醒（无需 /loop 前缀）
remind me at 3pm to push the release branch

# 查看所有在跑的任务（直接用自然语言问 Claude）
# 取消任务
cancel the deployment check
```

底层工具：`CronCreate`、`CronList`、`CronDelete`

**2. 使用 OpenClaw Cron**

```bash
# 单次执行（--at，ISO 8601 时间戳）
openclaw cron add --name "Reminder" \
  --at "2026-03-08T16:00:00Z" \
  --session main --wake now --delete-after-run

# Cron 表达式 + 时区 + 推送到 Slack
openclaw cron add --name "Morning brief" \
  --cron "0 7 * * *" --tz "America/Los_Angeles" \
  --session isolated \
  --message "Summarize overnight updates" \
  --channel slack --to "channel:C1234567890"
```

**3. 持久化替代方案（不想上 OpenClaw）**

- **Claude Desktop 桌面端**：Cowork 模式支持 `/schedule` 命令，任务绑定桌面端而非终端会话，桌面端开着就不丢，适合个人场景
- **GitHub Actions `schedule` 触发器**：将 prompt 写成 workflow，用 `schedule` + cron 表达式触发，每次拉起 Claude Code 执行，适合团队协作场景

### 💡 具体案例/数据

**`/loop` 典型场景**：
- 跑了个部署，`/loop 5m check if the deployment finished`，部署完关终端走人
- PR 提上去，`/loop 20m /review-pr 1234`，盯 CI 状态

**OpenClaw Cron 典型场景**：
- 每天早上 7 点（LA 时区）生成 GitHub 活动日报推送到 Slack：`--cron "0 7 * * *" --tz "America/Los_Angeles"`
- 每周五下午自动 review 所有 open PR
- 每小时检查服务器状态，异常时发 Telegram 告警

**隔离模式高级用法**：`--session isolated` + 指定 Opus 模型 + 思考级别 `xhigh`，适合跑长任务、生成高质量报告，不污染主聊天记录

### 📝 避坑指南

- ⚠️ `/loop` 秒级间隔会被静默向上取整到分钟，底层是 cron，别指望秒级精度
- ⚠️ `/loop` 关终端即失效，不要用它做任何需要持久化的任务，否则任务静默消失不会有任何提示
- ⚠️ `/loop` 无错误重试，任务失败即失败，不适合网络不稳定或有限流风险的场景
- ⚠️ OpenClaw 需要先部署 Gateway，上手有一定门槛，临时需求用它反而是杀鸡用牛刀
- ⚠️ OpenClaw 隔离模式（`--session isolated`）不会污染主聊天记录，长任务务必用隔离模式，否则会把主会话上下文搞乱
