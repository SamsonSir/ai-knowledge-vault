### 📋 核心分析

**战略价值**: 首份 500 人大样本 AI 付费用户定性调研，揭示「赛博雇佣」底层逻辑，直接指导 AI 产品定价、GTM 策略与留存设计。

**核心逻辑**:

- **付费本质是雇佣关系**：用户 = 雇主，AI 产品 = 数字雇员，订阅费 = 薪资。这套框架能解释几乎所有付费行为的底层动机。
- **赛道决定「起薪天花板」**：聊天/翻译/润色类赛道，用户心理起薪约等于一杯咖啡；法律、金融、深度代码赛道，用户愿意支付数百美金。选错赛道，产品上限可能不如别人的起薪。
- **每个功能岗位只有一个 HC**：用户设备里写代码、生图、润色文案的坑位各只有一个，AI 赛道是「赢家通吃」，非全球 Top 级别产品几乎没有胜算。
- **任务失败即解雇**：付费用户对结果错误容忍度极低，AI 在关键任务上掉链子会产生强烈被背叛感，直接触发取消订阅。
- **初始定价几乎不可逆**：用户习惯初始价格后对涨价极度敏感，AI 聊天助手涨价 5 美元以下就会触发迁移。产品初期必须通过「功能分层」一次性定好不同职级的薪资档位。
- **免费版「够好」但不能「够用」**：免费版太好用直接杀死付费转化。试用版应让用户感受到能力，但不能满足全部需求。
- **最佳付费引导是「限速」而非「断功能」**：当用户长期试用满意后，在 Panic Moment 发现免费版响应变慢、需要排队，会瞬间催生付费冲动。切断功能反而产生反感。
- **付费用户续费率以 70% 为 Benchmark**：超过 7 成已付费美国用户认为「为 AI 付费非常有价值」甚至是「生活必需品」。
- **当前付费仍未发挥最大潜力**：已付费用户平均每月支付 30-35 美金，但大部分人心理钱包上限高于此，说明存在显著的提价空间。
- **月付是主流偏好**：月付给用户「可以用更多 token」的安全感，同时保留下月更换产品的灵活度，优于按次付费模式。

### 🎯 关键洞察

**「赛博雇佣」框架的逻辑推导链**：

- 原因：用户把 AI 视为雇员，其决策逻辑完全遵循职场雇佣规则
- 动作：按「岗位价值」而非「软件定价」来评估是否付费
- 结果：高价值岗位（法律/金融/代码）的 AI 产品可以突破传统 SaaS 定价天花板

**「AI 的新老板」群体的付费逻辑**：只要 AI 能替代一个月薪 4000 美元的初级助理，他们对溢价的接受度远超想象。这类用户关心的不是价格，而是能否 End-to-End 闭环搞定一个业务流。

**技术专家（CS 专家）的反常识行为**：他们不怕产品贵，怕产品「蠢」。夸大营销对这类人群是负分。一旦 AI 在专业任务上掉链子，哪怕产品免费也会立刻被裁退。

### 🛠️ 操作流程（GTM 实战）

1. **定位阶段**：先确认自己产品属于哪个赛道，对应「起薪」区间，再确认目标用户类型（8 类中的哪几类），避免在错误赛道浪费营销预算。

2. **定价阶段**：初始定价一次性设好功能分层（不同职级薪资档位）。聊天/润色类不要定高价；法律/金融/代码类可以大胆上探。记住：初始定价几乎不可逆。

3. **免费版设计**：免费版做到「够好」（让用户感受到能力），但保留关键限制（速度、额度、高级功能），绝对不能让用户觉得「够用」。

4. **付费引导**：在用户的 Panic Moment（高频使用、任务紧急时）触发「限速」而非「断功能」，让用户主动产生付费冲动。

5. **留存设计**：以 70% 续费率为 Benchmark。核心是持续保持「胜任力」领先，用户没有品牌忠诚度，只有对能力的追逐。一旦竞品展现出更高性价比，用户会毫无留恋地迁移。

### 📝 避坑指南

- ⚠️ 免费版做太好 = 自杀：「免费版已经够用了」是 AI 产品最危险的用户反馈，直接导致付费转化归零。
- ⚠️ 初始定价过低无法逆转：用户一旦习惯低价，后期涨价（哪怕 5 美元）就会触发迁移，几乎没有补救机会。
- ⚠️ 对技术专家夸大营销是负分：CS 专家一旦发现 AI 能力不符合宣传，立刻裁退，哪怕产品免费。
- ⚠️ 断功能而非限速会产生反感：付费引导应该用「限制响应速度/排队」而非「直接切断功能访问」。
- ⚠️ 非 Top 级产品在赢家通吃赛道几乎无胜算：AI 雇员可以同时为全球无数人服务，用户只会选最好的那个。
- ⚠️ 聊天助手类产品的长试用期不是坏事：3 个月到 2 年的免费使用期是用户建立依赖的过程，强行缩短会导致流失。

### 📋 核心分析

**战略价值**: 李飞飞从文明级技术视角重新定义AI的边界——语言模型只是过渡，空间感知智能（spatial perceptual intelligence）才是下一个前沿，World Labs正在构建能推理、交互、生成3D/4D世界的前沿模型。

**核心逻辑**:

- **AI发展速度超出业内最狂野预期**：李飞飞本人深耕AI数十年，仍坦言近十年尤其近几年的发展速度"超出了我们很多人最狂野的想象"，整个产业全面拥抱AI的速度"至今仍有点不真实"。

- **AI = 新一代计算方式，覆盖面等同于芯片覆盖面**：灯泡里有芯片、冰箱有芯片、汽车有芯片、飞机有大量芯片——凡是有芯片的地方就有计算，而AI是新一代计算方式，因此"任何依赖芯片、依赖计算的设备、系统或场景，最终都会依赖AI计算"。这是理解AI影响范围的核心框架。

- **AI普惠性不是"多元化"议题，而是文明级技术的必然要求**：医疗、农业、教育、机器人、能源与可持续发展——生活与工作的每一个方面都会被影响，因此全球人口都必须参与其中，这不是政治正确，而是技术覆盖逻辑的必然结论。

- **本土化AI的驱动力来自市场需求，而非政策意图**：以泰语原生AI模型项目Typhoon（SCB10X出品）为例，不同地区的人拥有不同文化、语境、共同经验与情境细微差别，这些差异必须被AI理解。推动本土化AI的真正动力是"人们本身的需求"，而非"自上而下的政策"。

- **语言模型是高度有损的信息表达**：以做一份意大利面为例——15~20分钟的烹饪过程，如何调酱汁、如何把意面放进水里、意面在水中的变化——这些物理细节"几乎不可能只用语言来准确描述"。语言对物理世界的描述本质上是有损压缩。

- **空间感知智能是智能的关键支点**：从进化与人类行为角度看，人类不是只靠"说话"生活的——早上醒来拥抱孩子、做早餐、开车、端起咖啡杯——"日常生活中的绝大多数行为，以及整个人类文明中的工作，都依赖于空间、感知与具身智能"。

- **世界建模（world modeling）让AI从被动推理变为主动具身**：单纯的图像推理仍是被动方式；通过world modeling与空间智能，AI可以变得具有主体性（agentic），能够被计算、被规划，为具身智能体（embodied agents）提供主动能力。

- **World Labs成立不到两年，已发布首个模型与产品原型Marble**：目标是打造能"推理、理解、交互，并生成3D/4D世界"的前沿模型，赋能场景包括：模拟、机器人、创意产业、设计、教育、医疗健康、制造业。

- **游戏产业是World Labs最先落地的商业方向**：Marble发布约两个月后，已有大量游戏开发者在使用，目前以小体量项目为主（非3A级），但已开始实质性赋能开发者创造力。

- **AI的下一章明确定义为空间智能**：李飞飞的职业路径从计算机视觉→Stanford HAI创始主任→World Labs联合创始人兼CEO，整条线索指向同一结论：语言之后，空间智能是AI的下一个前沿。

### 🎯 关键洞察

**为什么空间智能是下一前沿，而不只是"另一个方向"？**

逻辑链：
1. 语言模型的本质是对世界的符号化压缩 → 对物理过程天然有损
2. 人类智能的绝大多数行为发生在三维物理空间中（具身性）→ 语言只是其中一个输出通道
3. 真正的AGI需要能在物理世界中规划、交互、预测 → 这要求world modeling能力，而非更大的语言模型
4. 具身智能体（机器人、自动驾驶、AR/VR）的爆发需要空间推理作为底层能力 → 市场需求已经存在

**World Labs的技术定位**：不是做"更好的图像生成"，而是构建能理解物理规律（碰撞、形变、韧性、材质）的世界模型，让AI从"描述世界"升级为"在世界中行动"。

### 📋 核心分析

**战略价值**: 用 Meta-Prompt 把 AI 变成能自主运行 24 小时的工程 Agent，首次在无人值守条件下完成 GBA 模拟器级别的复杂工程项目，证明「AI 写代码」到「AI 做工程」的跨越已发生。

**核心逻辑**:

- **工程规模定标**: 单 Agent、无并行、JavaScript 从零实现，全程 700+ 次工具调用、800 次上下文切换，连续运行 24 小时，交付可运行 ROM 的完整 GBA 模拟器（含 3D 场景渲染）
- **GBA 模拟器的技术复杂度**: 需同时实现 ARM7TDMI 处理器（两套指令集，数百条指令逐一精确复现）、分区内存系统（各区读写规则不同，地址错误即崩溃）、图形引擎（背景层/精灵/窗口/特效/严格时序）、音频系统，最终嵌入 3D 前端场景渲染
- **Meta-Prompt = 循环程序，不是问答**: 传统 Prompt 是一问一答；Meta-Prompt 定义了一个持续执行的工作循环：`工作 → 测试 → 记录 → 推进`，模型在这个循环里反复执行，而非等待人类指令
- **目标拆解粒度是关键**: 整个项目被拆成若干阶段，每个阶段有明确的完成标准（处理器核心达到什么程度、内存系统过哪些测试、图形渲染到什么状态才能进入下一阶段）。拆太粗→模型不知道当前干什么；拆太细→丧失自主判断空间
- **笔记协议是跨 Session 的唯一通道**: 上下文会被清空，文件不会丢。每个 Session 结束必须写入：做了什么、下一步做什么、卡在哪里。新 Session 启动第一件事：读 `progress`、读 `decisions`、读 `blockers`、检查最近改过的文件，然后从下一个待办项继续
- **循环跳出条件防止死磕**: 每次重试记录到文件并带计数；连续失败 3 次强制换思路；单个问题超过 20 分钟必须记录并跳过或升级求助。这条规则是专门防 AI 钻牛角尖设计的
- **底层逻辑三元结构**: 上下文 = 短期记忆；文件 = 长期记忆；Meta-Prompt = 工作方法。上下文窗口装得下全部代码，但装不下持续数天的工作过程，文件可以
- **两个版本对比验证**: 有参考版（提供 gbajs 开源源码作架构参考，GLM-5 消化后自主决定模块拆分、接口定义、实现顺序）→ 核心功能完成，可加载运行 ROM；零参考版（无参考代码、关闭网络搜索，仅凭训练知识+硬件文档）→ 持续运行超 24 小时，处理器指令集核心模块完成，仍在推进
- **上一代模型对比**: 同样任务在上一代模型上跑过，结果是跑着跑着开始遗忘、陷入循环、逐渐偏离目标、工具调用出错直接卡住。GLM-5 在 800 次上下文切换后，架构规范、编码约定、测试流程仍被严格遵守
- **人的角色转变**: 长任务中人变成「巡视者」，整个 24 小时只介入 2-3 次，其余时间 Agent 自主读笔记、做判断、往前推

### 🎯 关键洞察

**为什么 Meta-Prompt 质量决定一切**

同样的模型，不同人写的 Meta-Prompt 效果天差地别。本质上是在对 AI 做逻辑编程——你必须先理解工程是怎么运作的，才能把工程逻辑编进 Prompt 里。这不是 Prompt 技巧问题，是工程理解问题。

**为什么文件比上下文更重要**

行业过去两年花大量精力扩展上下文窗口（4K → 32K → 128K → 1M），但这个项目证明：上下文窗口的大小不是长任务的瓶颈。800 次切换中上下文被反复清空，项目从未停滞。关键信息持久化在文件里，架构决策跨 Session 衔接，这才是长任务的核心基础设施。

**为什么「写代码」和「做工程」是两件事**

写代码 = 单次生成质量。做工程 = 调研 + 架构设计 + 分阶段实现 + 持续测试 + 遇到问题调整方向 + 每步决策记录以便后续衔接。GBA 模拟器这个项目标志着 AI 开始跨越这道坎。

### 🛠️ 操作流程

1. **准备阶段**: 选定工程项目，将其拆解为若干阶段，每个阶段写明明确的完成标准（可量化的测试通过条件）。如有参考资料（开源代码/硬件文档），准备好供模型读取。定义工作范围边界，明确哪些事超出 Agent 能力范围需要人工介入

2. **编写 Meta-Prompt**: 包含以下四个核心模块：
   - 目标拆解（阶段 + 完成标准）
   - 笔记协议（Session 结束写什么、Session 开始读什么、文件命名规范）
   - 循环跳出条件（重试计数 + 失败 3 次换思路 + 超时 20 分钟跳过/升级）
   - 恢复流程（新 Session 启动的固定第一步序列）

3. **启动与巡视**: 启动 Agent，进入巡视模式。不要持续盯着看，定期检查连续几个 Session 的日志（不是单个 Session）。重点观察：是否在同一问题上跨多个 Session 反复循环？是否在干不了的事上持续浪费？

4. **关键时刻介入**: 发现 AI 钻牛角尖时，提供更好的参考文档或明确的方向指引帮其跳出局部最优。发现 AI 在能力边界外死磕时，直接提供所需资源（如 3D 模型资产）并更新 Prompt 中的求助触发条件

5. **验证与优化**: 检查笔记文件的质量（是否足够具体以支撑下一个 Session 恢复）；检查测试文件（Agent 应自己写测试）；对比阶段完成标准验收交付物

### 💡 具体案例/数据

**坑一：AI 钻牛角尖（Boot 阶段 Stub Bios 错误）**
- 现象：Debug Boot 阶段遇到 Stub Bios 错误，模型开始跨多个 Session 反复修复同一问题，每次用的方法差不多，从单个 Session 看像在正常工作，退后看连续日志才发现在转圈
- 根因：模型没有「退后一步看全局」的本能，越陷越深
- 解法：人工介入，提供更好的参考文档，帮其跳出循环

**坑二：AI 不会喊人（3D 资产搜索失败）**

- 根因：模型天生设定是「自己把问题解决掉」，没有「干不了就找人」的本能，短任务无害，长任务造成巨大浪费
- 解法：提前在 Prompt 里定义工作范围和场外求助的触发条件

### 📋 核心分析

**战略价值**: OpenClaw（大龙虾）的本地向量记忆系统（QMD）默认未启用，导致记忆失效；用 Cursor 作为外部手术刀，4 步手动触发模型下载与索引构建，一次性激活后即可全自动运转。

**核心逻辑**:

- **记忆失效的根本原因**：OpenClaw 2.9 之前的老版本或升级用户，配置文件 `~/.openclaw/openclaw.json` 里 `memory.backend` 默认不是 `qmd`，系统跑在「无记忆模式」，且不会有任何提示。
- **两代记忆架构的本质差异**：v2.2 以前是 MEMORY.md 纯文本全量读取（每次对话消耗大量 Token，文件越大越慢，超出上下文直接截断）；v2.2 之后引入 RAG 向量检索，只取相关片段，效率质变。
- **QMD vs Embedding 两条路的选择逻辑**：官方默认走 Google 云端 Embedding API（快准但收费、数据上云）；QMD 是 GitHub 独立开源项目，全本地运行，三个模型全部开源，下载后断网可用，零费用。
- **QMD 哑火的直接原因**：第一次启动时需要下载约 2.2GB 本地模型，默认超时设置太短，下载未完成就超时报错，系统自动降级到效果差的备用方案（`switching to builtin index`），此后记忆检索形同虚设。
- **为什么不能让 OpenClaw 自己修自己**：在对话框里贴报错让它自修，相当于「喝醉的外科医生给自己做开颅手术」，它没有稳定的外部视角，越修越坏。必须用外部工具（Cursor）站在上帝视角操作。
- **Cursor 的角色定位**：不需要懂代码，Cursor Pro + Claude Sonnet 4.5 替你操作终端、改文件，是对非程序员最友好的外科手术工具。
- **QMD 三模型分工**：`embeddinggemma-300M`（文本向量化）+ `qmd-query-expansion-1.7B`（理解搜索意图）+ `qwen3-reranker-0.6b`（对结果重排序），三者协同才能完整跑通一次搜索。
- **自动更新机制的触发条件**：索引新文件每 5 分钟一次，生成向量每 60 分钟一次，防抖保护 15 秒。手动跑通一次后，后续全自动，无需人工干预。
- **记忆文件是人类可读的 Markdown**：存于 `~/.openclaw/workspace/memory/`，可直接用文本编辑器查看和手动写入，5 分钟后 QMD 自动索引。
- **模型只下载一次，全 Agent 共用缓存**：2.9 版本修复了重复下载问题，所有 Agent 复用 `~/.cache/qmd/models/` 下的缓存。

### 🎯 关键洞察

**为什么「升级用户」是重灾区**：从 v2.9 以前升级上来的用户，配置文件是老版本生成的，不包含 QMD 相关字段。新安装时如果没有手动勾选，同样缺失。OpenClaw 不会主动检测并提示你「记忆系统未激活」，这是一个静默失效的设计缺陷。

**RAG 的本质**：不是「更聪明的 AI」，而是「更聪明的信息检索」。向量化把文字转成数学坐标，语义相近的内容在坐标空间里距离近，搜索时找的是「距离最近的坐标」而不是「关键词匹配」，所以能理解「上周那个项目」和「2026-02-10 的会议记录」是同一件事。

### 🛠️ 操作流程

**准备阶段**:
1. 确认 OpenClaw 已更新至 2.9 版本并完成 onboard
2. 打开 Cursor
3. 在 Cursor 中打开文件夹：`~/.openclaw`
4. 确认使用 Cursor Pro，模型选 Claude Sonnet 4.5

**核心执行**:

**第一步：检查 QMD 状态**

将以下内容发给 Cursor AI：

```
帮我检查一下 QMD 记忆检索的状态：
1. 运行 qmd status 看看索引情况
2. 检查 ~/.openclaw/openclaw.json 里的 memory 配置
3. 看看 ~/.openclaw/logs/gateway.err.log 里有没有 qmd 相关的错误
```

中招判断标准（出现任意一条即需继续）：

| 报错信息 | 含义 |
|---|---|
| `Total: 0 files indexed` | 无索引，记忆失效根本原因 |
| `qmd query failed: timed out` | 超时，模型未下载完 |
| `switching to builtin index` | 已降级到效果差的备用方案 |

---

**第二步：构建索引**

将以下内容发给 Cursor AI：

```
帮我执行 qmd update 构建文本索引，然后执行 qmd embed 生成向量。
第一次运行会自动下载本地模型（大约 2GB），请耐心等待。
```

预期输出：

```

[1/3] memory-root (MEMORY.md)
Indexed: 1 new
[2/3] memory-alt (memory.md)
Indexed: 1 new
[3/3] memory-dir (**/*.md)
Indexed: 36 new
✓ All collections updated.

Embedding 37 documents (111 chunks, 210.8 KB)
Model: embeddinggemma
██████████████████████████████ 100%
✓ Done! Embedded 111 chunks from 37 documents in 9s
```

---

**第三步：触发完整搜索（下载剩余模型）**

将以下内容发给 Cursor AI：

```
帮我运行 qmd query "测试记忆搜索" -n 3
第一次搜索会下载查询扩展模型（约 1.3GB）和重排序模型（约 640MB），请等它下完。
```

三个模型下载详情：

| 模型名 | 大小 | 用途 | 下载时机 |
|---|---|---|---|
| embeddinggemma-300M | ~300MB | 文本向量化 | 第二步已下 |
| qmd-query-expansion-1.7B | ~1.3GB | 理解搜索意图 | 本步骤下载 |
| qwen3-reranker-0.6b | ~640MB | 搜索结果重排序 | 本步骤下载 |

成功标志（看到搜索结果即成功）：

```
qmd://memory-dir/2026-02-10-1548.md
Title: Session: 2026-02-10 15:48 UTC
Score: 72%
...相关内容片段...
```

**第四步：验证**

将以下内容发给 Cursor AI：

```
运行 qmd status 确认索引状态
```

预期输出：

```
Documents
  Total:    38 files indexed    ← 不再是 0 了！
  Vectors:  111 embedded        ← 有向量了！
  Updated:  刚才

Collections

  memory-alt:  1 file
  memory-dir:  36 files
```

**验证与优化**:

激活后的自动运转机制：

| 动作 | 频率 | 说明 |
|---|---|---|
| 索引新文件 | 每 5 分钟 | 自动发现新的记忆文件 |
| 生成向量 | 每 60 分钟 | 自动给新文件生成搜索向量 |
| 防抖保护 | 15 秒 | 频繁写入时不会反复触发 |

完整数据流：
```
你跟大龙虾聊天
    ↓
大龙虾自动写入 ~/.openclaw/workspace/memory/2026-xx-xx.md
    ↓
5 分钟后 QMD 自动扫描到新文件，加入索引
    ↓
60 分钟后自动生成向量
    ↓
下次提问时 QMD 搜索相关记忆，大龙虾带着上下文回答
```

### 💡 具体案例/数据

- 实测搜索耗时：约 2-5 秒（MacBook Air CPU，无需显卡）
- 最大模型参数量：1.7B（qmd-query-expansion），MacBook Air 可跑
- 模型总占用空间：约 2.2GB，仅下载一次
- 索引数据库体积：几 MB（`index.sqlite`）
- 向量化示例：37 个文档，111 个 chunks，210.8KB，耗时 9 秒

---

### 📝 避坑指南

- ⚠️ 不要在 OpenClaw 对话框里贴报错让它自己修，会越修越坏，最终宕机
- ⚠️ 第三步的搜索命令必须跑，不跑的话 `qmd-query-expansion` 和 `qwen3-reranker` 两个模型不会下载，后续搜索质量大打折扣
- ⚠️ 电脑慢或网络差时，务必将 `timeoutMs` 调到 `10000`，否则搜索会持续超时降级
- ⚠️ 硬盘空间不足时，`~/.cache/qmd/models/`（2.2GB）是唯一可以删除的部分，删后下次自动重新下载，索引数据库（几 MB）不建议删
- ⚠️ 手动新建记忆文件后，索引更新有 5 分钟延迟，向量生成有 60 分钟延迟，不是实时生效
