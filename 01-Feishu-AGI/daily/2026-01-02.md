> 文档 ID: `D6mJwq1Pmiaa25kT0I8cJs4AnBc`

**来源**: CY：弄懂SKILLS是什么？怎么创建？如何使用 | **时间**: 2026-01-02 | **原文链接**: https://mp.weixin.qq.com/s/iFM9IyWb...

### 📋 核心分析

**战略价值**: Claude Skills 是将通用 LLM 转化为「领域专家数字员工」的 SOP 封装机制，通过渐进式上下文注入实现按需激活、Token 优化与行为确定性。

**核心逻辑**:

- **Skills 本质是 Agent 的「入职手册」**：以文件夹形式打包指令（SKILL.md）、脚本和资源，为 Claude 提供执行特定任务的标准化流程，等同于企业 SOP 减少人为误差的逻辑。
- **4 级激活架构，每级职责分离**：感知（加载 Metadata）→ 触发（语义路由）→ 激活（注入 SOP）→ 执行（程序化编排），每级独立运作，互不污染。
- **Metadata 极度轻量**：每项技能的 Frontmatter 仅占约 100 Tokens，系统启动时全量加载所有技能名称+描述，不浪费上下文。
- **纯语义路由，无硬编码**：Claude 接收用户指令后，依靠 LLM 推理能力自主匹配最相关技能，不依赖算法分发或关键词触发。
- **激活时秘密注入 SOP**：通过 Bash 读取 SKILL.md 正文，注入隐藏消息（`isMeta: true`），同时可动态修改 Allowed Tools 或切换更强模型（如 Claude Opus）。
- **程序化工具调用（PTC）减少幻觉**：复杂逻辑由 Python 脚本在沙盒 VM 中处理，MB 级中间数据不进入对话上下文，模型只接收精炼结果摘要，可减少约 37% Token 消耗并极大降低幻觉率。
- **MCP vs Skills 职责边界清晰**：MCP 负责「数据接入」（连接数据源），Skills 负责「逻辑控制」（教会 Claude 怎么用数据），两者互补而非替代。
- **description 字段是触发的核心**：Claude 完全依赖 description 的语义描述决定何时激活技能，写得越精准，触发越准确。
- **SKILL.md 建议控制在 500 行以内**：超出部分应拆分到独立参考文件，通过 `{baseDir}` 变量引用，保持上下文效率。
- **技能可打包分发**：`.skill` 文件本质是 `.zip` 压缩包，改后缀即可，便于团队共享和复用。

---

### 🎯 关键洞察

**渐进式披露机制**是 Skills 最核心的设计哲学：系统启动时只加载 ~100 Tokens 的 Metadata，任务触发后才将完整 SOP 注入上下文。这解决了「全量加载导致上下文污染」和「按需加载导致感知缺失」之间的矛盾——Claude 始终知道自己有哪些能力，但只在需要时才调用完整指令。

**PTC（程序化工具调用）** 是 Skills 相较于普通 Prompt 的质变点：传统 Prompt 让 LLM 直接处理复杂数据，Skills 让 LLM 编写 Python 脚本交给沙盒执行，LLM 只负责「编排逻辑」而非「处理数据」，这是行为确定性大幅提升的根本原因。

### 🛠️ 操作流程

**方式一：Create with Claude（推荐新手）**

1. 打开 Claude，进入 Skills 创建入口
2. 直接用自然语言描述你想要的技能（如「帮我创建一个公众号对比叙事写作技能」）
3. 与 Claude 对话完善细节，AI 自动生成技能文件
4. 点击「Copy to your skills」完成添加，或下载文件备用

**方式二：Write skill instructions（推荐已有想法者）**

1. 进入编辑界面，填写三个字段：
   - `Name`：如 `contrast-storytelling`
   - `Description`：如「对比叙事法写作技能，用于公众号/自媒体文章创作，当用户需要写对比类文章时触发」
   - `Instructions`：技能的核心理念、写作流程、模板、示例等（即 SKILL.md `---` 下方的正文内容）
2. 保存，系统自动生成 SKILL.md 文件和文件夹结构

**方式三：Upload a skill（适合导入现成技能）**

1. 准备符合以下结构的文件夹：
```
contrast-storytelling/
├── SKILL.md          ← 必须
├── reference.md      ← 可选
└── scripts/          ← 可选
    └── helper.py
```
2. 将文件夹压缩为 `.zip`，可选择改后缀为 `.skill`
3. 在 Skills 入口选择「Upload a skill」上传

**SKILL.md 标准结构模板**：
```yaml
---
name: your-skill-name
description: 详细描述技能功能及触发场景，Claude 依靠此字段决定何时激活
allowed-tools:
  - Read
  - Write
  - Bash
model: claude-3-5-sonnet-latest
---

## 项目概述
简述技能目的

## 具体步骤（SOP）
1. 分析...
2. 执行...
3. 输出...

## 参考资源
详见 {baseDir}/reference.md

## 输出示例
输入：...
输出：...
```

**使用已创建的技能**：

| 平台 | 调用方式 |
|------|---------|
| Claude.ai 网页版 | 自然语言触发（自动）或明确说「使用 contrast-storytelling 技能帮我写一篇文章」 |
| Claude Code | `/contrast-storytelling` 命令直接调用 |

### 📝 避坑指南

- ⚠️ `description` 写得太泛会导致技能从不触发或频繁误触发，必须明确描述「什么场景下激活」
- ⚠️ `name` 字段不能包含大写字母、空格或特殊字符，否则无法识别
- ⚠️ SKILL.md 超过 500 行会降低上下文效率，复杂内容必须拆分到 `reference.md` 等独立文件
- ⚠️ Upload 方式要求 SKILL.md 必须在压缩包根目录，不能嵌套在子文件夹内
- ⚠️ `.skill` 和 `.zip` 本质相同，分发时可直接改后缀，无需重新打包

### 📋 核心分析

**战略价值**: 一位 27 岁独立创作者用「Build→Learn→Share→Growth」闭环，在离职后一年内同时跑通 AI 自媒体（一泽Eze）与 AI 产品（Chat Memo，10000 用户，Google 精选扩展）两条曲线的完整方法论。

**核心逻辑**:

- **离职后最大的敌人是「无北极星」**：在职时组织替你承担时代浪潮的冲击，离职后账户不再定时增长，AI 行业月月迭代（DeepSeek R1、Manus 在休耕期横空出世），焦虑感对个体的冲击前所未有地真切。解法：给自己设定半年到一年的实验期，先跑起来再说。
- **「做内容」vs「做产品」的抉择困境，本质是信息与信念不足**：不要在路径上死磕，而是用「Build→Learn→Share→Growth」闭环统合精力——以产品方向为引研究技术，把钻研过程做成可分享的内容 Case，再将积累转化为可推广的独立产品，用影响力助推冷启动。一鱼多吃，最大化个人投入效用。
- **Be a builder，not just an influencer**：仅靠观察、测评、仿照，很难理解 AI 在具体场景中的边界与趋势。例：纯 RAG + Prompt 工程（无复杂记忆框架）就能实现人格化 AI 伴侣；一段随意提示词就能让 AI 精准猜出用户 MBTI；「通用 Agent 为核、Skill 为壳」的低成本垂直 Agent 范式已有迹可循。
- **AI 自媒体的内容选题策略**：不追求「最快最全的新品资讯」——Gemini 3 Pro 只是 Coding 模型的正常提升，4o 生图只是 Gemini 2.0 flash imagination 的同项能力进步，远没有 Claude Sonnet 4、Gemini 2.0 flash imagination 本身惊异。范式无变化的节点不值得动笔，把精力留给真正的突破点。
- **Chat Memo 的产品洞察起点**：AI 对话记录是记录认知迭代、情绪起伏、个人好恶的「原始手稿」，是构成「数字自我」的核心数据。向 AI 倾诉的情绪波动是自我和解的线索；被 AI 推翻的思路映射底层思考模式；无意识的提问风格隐含性格特质线索。
- **Chat Memo 的技术路径**：100% AI Coding 开发，零手写代码。核心功能：实时无感本地记录（完全本地，保障隐私）、统一快速检索历史对话、一键导出所有对话记录到本地。支持 GPT、Gemini、Claude、豆包、元宝等平台。已被 Google 认证为精选 AI 记忆扩展，自然增长至 10000 用户量级。
- **一人业务的精力正向循环五步模型**（见下方操作流程）：核心是「个人优势 + 垂类差异化」，而非追赶热点。
- **AI Coding 独立产品的关键认知**：AI Coding 让原本受限于开发成本、ROI 打不正的小产品有了可能性；不是用 AI Coding 就必须做 AI 相关产品；不是所有产品创业都必须融资 to VC；你最大的优势在于了解自己的痛点与兴趣。
- **自媒体差异化定位公式**：个人优势 + 垂类下其他人明显没做好的点 = 差异化竞争力。用内容对外传播自己的方法与思考，让目标受众一眼认可你的突出优势，无论正向还是负向反馈都是好的开始。
- **成功的颗粒度认知**：一人公司不需要获取「大成功」，小成功（有收入/影响力的个人副业）就足够。暂时没取得应有的小成功，可能只是没让外界正确了解到自己的优势。

### 🎯 关键洞察

**Chat Memo 的产品逻辑推导链**：

> 个人笔记（向量数据）+ 调优 Persona 提示 → AI Partner 对话 → 效果奇佳但门槛高（不是人人有笔记）→ 转向：AI 对话记录本身就是最高频的信息出入口 → 把散落在各平台的对话收集起来 → Chat Memo

这条推导链的核心洞察是：**AI 聊天记录 = 这一代人的自我叙事载体**，人通过叙事定义自己，所以管理 AI 对话记录 = 管理数字自我。

**AI 自媒体的时机判断**：25 年下半年起，随着 AI 狂飙一年，大众对新品资讯的讨论已陷入疲惫。泛资讯类账号的红利期已过，新入场者必须走垂直差异化路线。

**「一稿多投、一心两用、一鱼多吃」的具体含义**：同一个研究动作（如测试模型边界），同时产出：①内容文章（IP 增长）②产品功能验证（产品积累）③行业判断（建立圈内口碑）。三个产出共享同一份时间投入。

### 🛠️ 操作流程

**一人业务「个人精力正向循环」五步 SOP**：

1. **确定突破口**：个人优势 + 长期目标 → 推导出短期突破口（不是追热点，是找「垂类下其他人明显没做好的点」）
2. **补齐能力差距**：识别突破所需的能力缺口，定向补齐（如 AI Coding、Prompt 工程、SEO）
3. **实践验证 + 内容分享**：面向目标群体打造早期影响力，用内容传播自己的方法与思考，让目标受众一眼认可你的突出优势
4. **补齐商业资源**：实现早期商业化（内容合作、厂商邀约、产品付费等）

## 📔 文章 5

> 文档 ID: `BIK9wjwb6iBzSlkBKvAcfJxvnvf`

**来源**: 可为与不可为：CDP 视角下的 Browser 控制边界 | **时间**: 2025-12-30 | **原文链接**: `https://mp.weixin.qq.com/s/us_pALuk...`

---

### 📋 核心分析

**战略价值**: 系统梳理 CDP/Puppeteer 在 Browser-Use 场景下的能力边界，帮助架构师精准判断哪些功能「直接支持/间接支持/完全不支持」，避免踩坑并指导技术选型。

**核心逻辑**:

- **能力集合关系**：Computer（全集）⊃ Browser（App 层）⊃ CDP（Debug 层）⊃ Puppeteer（CDP 子集）。Puppeteer 未封装全部 CDP API，所以能力 < CDP。
- **Tabs 是间接支持的高频痛点**：CDP 原生 Tabs API 能力弱，新建/更新/切换/关闭 4 个基础操作都需要组合多个 API 实现；且用户拖拽改变 Tab 顺序后，CDP 无法感知顺序变化。
- **Navigate 是直接支持的稳定能力**：back/forward/reload/goto 4 个操作 Puppeteer 封装完善，可直接调用；但地址栏输入框的联想记录无法通过 CDP 获取。
- **chrome:// 内部设置页可直接 navigate**：所有 `chrome://` 协议页面（history/downloads/extensions 等）本质是网页，CDP 可直接 `Page.goto('chrome://xxx/')` 访问，无需走设置菜单入口。
- **快捷键跨平台适配是工程必做项**：macOS 用 `Meta`（Command），Windows/Linux 用 `Ctrl`；LLM 发出快捷键 action 时，工程侧必须判断 OS 并转换：`hotkey("ctrl+A") → isMacOS ? keyboard('Meta+KeyA') : keyboard('Control+KeyA')`。
- **macOS 下 Meta+Key 快捷键在 CDP 中失效**（Issue #776，2017 年至今未修复）：原因是 CDP 不发送 nativeKeyCodes，且 Chromium 前台时快捷键被地址栏截获。解决方案是用 `Input.dispatchKeyEvent` 的 `commands` 参数替代。
- **CDP 截图只能截网页内容区（绿框），截不到 Chrome UI（红框）**：若 VLM 训练数据是完整 Chrome 截图，直接传 CDP 截图会导致 action 坐标错位，需特别注意模型适配。
- **系统级 Input 控件（Select/Date/Time/Color）无法被 CDP 截图捕获**：迂回方案是通过 JS 注入将系统控件替换为 DOM 控件，再截图。
- **Dialog 弹窗是高优必处理项**：Alert/Confirm/Prompt/Beforeunload 4 类弹窗触发后 JS 引擎挂起，必须响应关闭才能继续流程，CDP 通过 `Page.javascriptDialogOpening` 事件可感知。
- **CDP 覆盖约 95% 业务功能**：书签、翻译、右键系统菜单、标签页分组、阅读模式等甜品功能不支持，但这些在 Browser-Use 场景基本用不到，越简单越健壮。

---

### 🎯 关键洞察

**为什么 macOS 快捷键在 CDP 下失效（根因链路）**：
1. CDP 发送键盘指令时不携带 `nativeKeyCodes` → 系统层不产生真实 OSX 事件
2. 即使修复 nativeKeyCodes，Chromium 需要处于前台才能响应快捷键
3. 即使 Chromium 在前台，快捷键会被地址栏截获而非传递给 Page

→ 结论：不要试图在 macOS 上用 `Meta+Key` 模拟快捷键，改用 `commands` 参数走编辑指令通道。

**为什么 CDP 截图坐标会错位**：
CDP `Page.captureScreenshot` 只截取 viewport 内的网页内容，不含浏览器工具栏/地址栏/标签栏。部分 VLM（如基于完整桌面截图训练的模型）的坐标系是全屏坐标系，传入 CDP 截图后坐标原点偏移，导致 click 等 action 打偏。

### 🛠️ 操作流程

#### 1. 快捷键跨平台适配 SOP

```typescript
// OS 判断 + 快捷键转换
function resolveHotkey(action: string): { key: string; modifier: string } {
  const isMacOS = process.platform === 'darwin';
  const modifier = isMacOS ? 'Meta' : 'Control';
  return { key: action, modifier };
}

// 编辑类快捷键：用 commands 参数（macOS 必须）
await page.keyboard.down('KeyA', { commands: ['SelectAll'] });
await page.keyboard.up('KeyA');

// 功能映射替代高权限快捷键
// 查看历史记录（替代 Cmd+Y / Ctrl+H）
await page.goto('chrome://history/');
// 退出浏览器（替代 Cmd+Q）
await browser.close();
// 返回上页（替代 Cmd+[ / Alt+Left）
await page.goBack();
```

#### 2. 系统 Input 控件替换为 DOM 控件（JS 注入）

```javascript
// 以 Select 为例，注入 JS 替换系统 select 为自定义 DOM 下拉
await page.evaluate(() => {
  document.querySelectorAll('select').forEach(select => {
    // 构建自定义 DOM 下拉替换原生 select
    const wrapper = document.createElement('div');
    wrapper.className = 'custom-select';
    // ... 构建选项 DOM
    select.parentNode.replaceChild(wrapper, select);
  });
});
// 替换后 CDP 截图即可捕获
```

#### 3. Dialog 弹窗处理 SOP

```typescript
// 必须在 page 初始化后立即注册，否则弹窗会阻塞所有后续操作
page.on('dialog', async (dialog) => {
  console.log(`Dialog type: ${dialog.type()}, message: ${dialog.message()}`);
  // 根据业务逻辑决定 accept 或 dismiss
  await dialog.accept(); // 或 dialog.dismiss()
});
```

### 📝 避坑指南

- ⚠️ **Tabs 顺序不可信**：用户手动拖拽 Tab 后，`Browser.pages()` 返回的顺序与视觉顺序不一致，不要依赖 index 做 Tab 定位。

- ⚠️ **文件下载 API 较弱**：`DownloadBehavior` 只能控制策略和路径，无法监听下载进度或完成事件，需自行轮询文件系统。

## 📓 文章 6

> 文档 ID: `CgezwAZfAioC8tkaDJEc8SvZnKf`

**来源**: 让你的照片变成烟花秀[附提示词] | **时间**: 2026-01-02 | **原文链接**: https://mp.weixin.qq.com/s/9YM9E0v7...

### 📋 核心分析

**战略价值**: 用一套结构化 Midjourney Prompt 模板，将任意主体（人像/Logo/地图/文字）渲染成夜空烟花造型的4K长曝光合成摄影图。

**核心逻辑**:

- **核心视觉逻辑**：烟花爆炸的光粒子轨迹「聚合成形」，形成可识别的主体轮廓（人脸、Logo、文字等），悬浮于地标建筑上方，形成「底部锚点建筑 + 顶部光构主体」的垂直构图
- **主体槽位可替换**：Prompt 中 `[Insert Your Subject]` 是模块化插槽，可填入：人像（glowing portrait of a woman）、国家地图（map of a country）、品牌Logo（brand logo）、文字（sparkler writing）
- **底部地标同样可替换**：迪士尼城堡、埃菲尔铁塔、城市天际线均可作为底部锚点建筑
- **长曝光摄影风格关键词**：`long-exposure photography`、`light trails`、`scattering embers`、`particle effects`、`volumetric smoke`、`glowing plasma`、`shimmering light dust`
- **配色逻辑**：背景用深午夜蓝或纯黑（最大化对比度），主体光构用金色/白色/青色，或品牌专属色（如蓝+黄）
- **光照逻辑**：烟花主体作为自发光源（Emissive lighting），向下对建筑产生软性轮廓光（Rim lighting on architecture），增强真实感
- **负空间运用**：深色夜空作为高对比背景，确保烟花形状轮廓清晰可读
- **文字融入方式**：若主体是文字，风格应为「烟花棒书写体」或「霓虹灯管」——粗体、发光、草书或块状字体，置于烟花主体位置本身，或严格作为底部字幕叠加
- **画幅选择**：`--ar 1:1` 适合方图/社交媒体，`--ar 16:9` 适合电影感宽幅
- **渲染风格标签**：`Hyper-realistic Composite Photography`、`Magical Realism`、`Light Painting Art`、`Cinematic Visual Effects (VFX)`

---

### 🛠️ 操作流程

1. **准备阶段**: 确定你的主体内容（人像/Logo/地图/文字），以及底部地标建筑

2. **核心执行**: 将下方完整 Prompt 复制进 Midjourney，替换 `[Insert Your Subject]` 和画幅参数

3. **验证与优化**: 若形状识别度不够，在主体描述前加 `clearly defined silhouette of`；若光感不足，追加 `ultra-bright emissive glow`
