### 📋 核心分析

**战略价值**: 从模型底层飞轮逻辑出发，系统拆解2026年AI应用爆发的结构性机会，并给出个人职业选择的可操作路径。

**核心逻辑**:
- **模型智力线三节点**：①24年9月o1推理模型→催生Manus/Genspark等第一批agent产品；②25年11月Claude 4.5→vibe coding大众化，Codex 5.2/Gemini 3跟进；③Claude 4.6/Codex 5.3/o3→multi-agent架构被内化进模型，子agent数量/通信由模型自主判断，不再是人为脚手架
- **数据飞轮核心机制**：用户prompt→agent规划/调用工具→执行结果→用户纠正（骂AI）= 生产环境高质量训练数据→喂回模型→模型更强→agent更强→循环。这条路在coding领域已跑通，正向其他行业泛化
- **AI自我迭代已启动**：Claude Code用Claude Code迭代自身；OpenClaw 7×24小时运行+长memory实现自我进化；孙天祥团队FARS让AI自主搜集信息/做实验/写论文；某团队人均日消耗token 10亿，一人同时驱动30个agent，内存128G，月均Claude支出数万元，日发布十几个版本
- **多模态技术门槛消失**：Seedance 2.0震惊好莱坞导演；ComfyUI时代全球熟练用户仅几千人，现在分镜/运镜/动作连接的导演knowhow已内化进模型，写简单prompt即可；音乐方面Spotify已有大量未标注AI生成的爆款，超过大部分音乐人水平
- **内容行业推演**：技术门槛消失→小作坊可产出电影/电视剧级内容→产能爆炸→内容极度内卷→头部爆款消失（类比流行音乐碎片化）→最终受益方是抖音/番茄等分发平台；创作者核心竞争力回归审美/创意/故事
- **agent infra是新基建**：传统软件公司股价大跌，Cloudflare等为agent服务的基础设施公司股价大涨；核心问题包括：数据供给（API vs 爬虫）、agent专用工具（≠人用软件）、context处理（GitHub前老板做的Entire用Git协议解决）、memory管理
- **垂直agent机会密度最高**：招聘/调研/旅行/催收/合规/猎头等，流程明确、数据明确、AI提效收益直接；美国这波AI创业C端跑出来的少，大量是B端垂直agent，ARR增长快；资本在大量收购传统中介agency（有渠道/数据/know-how但受限人力），AI改造后快速规模化
- **信息差=机会**：大厂是人才密度最高的地方，但其他行业人才密度远低于此；帮金融/律师行业装上Claude Code处理文本任务，客户看到结果后"震惊到开始认真思考怎么裁员"；AI在任务层面已超过很多行业大部分从业者，但这些行业不知道
- **人与人差距急剧拉大**：手搓代码时代程序员差距2-3倍封顶；AI时代差距可达几十倍甚至上百倍；大厂内部看不出来，出来才能感受到
- **定价逻辑正在转变**：工具逻辑=几美刀/月；人的逻辑=几百甚至几千美刀/月。同样价格，认知框架不同，付费意愿完全不同——"工具50刀好贵，但一个人帮你干活100-200刀不贵"

### 🎯 关键洞察

**飞轮的本质**：模型→agent脚手架→multi-agent→内化回模型，每一轮循环都把上一轮的工程补丁变成模型原生能力。这意味着今天你用脚手架搭的东西，明年可能直接被模型吃掉——这既是风险，也是信号：看哪些脚手架正在被大量使用，那里就是下一个被内化的方向。

**"不擅长时更依赖AI"悖论**：熟练工反而觉得AI不行，因为他们用自己的熟练度在评判AI；而做不熟悉的事时，AI的价值才真正显现。这解释了为什么大厂成熟组织AI提效收效有限——分工固化、流程锁死，每个人都在用自己的专业标准否定AI。破解方式：做封闭AI项目，打破岗位边界，让每个人做不擅长的事。

**短剧vs电影的历史类比**：短剧把技术资源门槛从几千万/亿降到几万/几十万，短短几年市场规模破千亿，而电影多年只有500亿。AI对视频内容的作用将重演这个过程，但速度更快、幅度更大。

**物理世界token化趋势**：Plaud（录音壳）、Looki（全天候可穿戴摄像）、OdyssLife（项链摄像头做营养分析）——这些硬件的本质是把物理世界的context转化为token，喂给agent。当前agent表现不好的核心原因之一就是context不够，主要依赖人的prompt和电脑里已有文件，远不足以反映用户真实生活。

### 🛠️ 操作流程

#### 大厂从业者

1. **优先进入高浓度AI业务**: 字节豆包/剪映/Coze/飞书AI、阿里千问、腾讯元宝。没有AI背景也推荐去，至少熬半年到一年，比在传统业务自己琢磨有效得多
2. **进不去核心AI业务**: 在自己业务里主动发起AI项目。看AI信息→自己build产品→做有规模的AI产品，三个层次认知要求完全不同，不能只停在看信息层
3. **做side project**: 帮业务提效，或纯粹觉得好玩。GitHub主页/side project比简历更有说服力
4. **城市选择参考**:

| 城市 | AI生态偏向 |
|-----|----------|
| 北京 | 模型+应用 |
| 上海 | 大型AI业务 |
| 杭州 | AI做生意类 |
| 深圳 | AI硬件 |

#### 加入AI初创公司

1. **判断是否AI native的两个问题**:
   - 这家公司有没有提供无限量token？没有→不是AI native
   - 梯子挂了，公司能不能正常运转？不能→是AI native
2. **求职方式**: 熟人介绍+信任背书优先；GitHub主页/side project比简历重要；标准化投递流程效果差
3. **心理预期**: 薪资不一定降（字节系出来融资多的甚至涨薪），但大部分是降现金换期权；期权成功概率不到1%；收获的是真实成长速度

#### 创业

1. **融资路径**: 有好履历（剪映/豆包/抖音等）+想清楚天花板和切入点→机构钱好拿；拿不到红杉/高瓴/IDG→先做亲友轮，几万到几十万，够做出demo，若成功1%股票后续翻几十上百倍
2. **找co-founder**: 找合作过的人，知道能力项和人品；上下铺兄弟/室友不是最优选；早期元老5-10年后能力可能跟不上，看当下能力匹配
3. **Founder核心工作四件事**: 找钱、找人、找方向、为团队降噪（收敛信息噪声）
4. **方向会换是常态**: 不是例外，张一鸣早期也换了好几个方向
5. **出海优先**: 国内大厂太强；美国搞不定→东南亚→拉美/中东/日本（人多钱多）；能拿美元拿美元

#### 一人公司

1. **必备基础能力**: Vibe coding + 工程化，能做出demo
2. **产品形态**: 小工具、小插件、AI工具导航站接Google Ads；月赚几千到1万美金
3. **品牌建设**: 做产品同时做自媒体；不要求每篇精品，持续输出本身是筛选和积累；AI coding让产品复刻门槛极低（专门的agent 2-3天可复刻一个APP），品牌是唯一护城河

### 📝 避坑指南

- ⚠️ **补丁思维陷阱**: 模型变强后，不要还停留在"用agent脚手架补模型不足"的思路，要做更大胆有想象力的事
- ⚠️ **测评工具人化**: 测评是AI产品入门基本功，但长期只做测评会变工具人；模型越来越强，很多时候需要极难任务才能区分差异；最终必须往业务走
- ⚠️ **大厂AI提效幻觉**: 成熟组织分工固化、流程锁死，AI提效收效有限；有效方式是做封闭AI项目，打破岗位边界
- ⚠️ **期权幻觉**: 初创公司期权成功概率不到1%，字节/小红书那种有回购的极罕见；降现金换期权要清醒认知
- ⚠️ **信息摄入≠认知**: 看资料/听分享投入成本太低，吸收极浅；真正的认知来自付出成本的决策——做错决策导致用户流失/数据下跌，才会真正思考本质
- ⚠️ **co-founder选人误区**: 关系近（上下铺/室友）≠能力匹配；早期元老5-10年后可能能力跟不上；只有真正一起干过活才知道对方行不行
- ⚠️ **国内创业陷阱**: 国内人穷企业穷，大厂太强；部分国内VC条款苛刻，本质是债不是投资
- ⚠️ **终局思维**: 认为"现在做的东西就是终局"心态不对；做了三年AI，之前大部分东西回头看没什么意义了，但认知是真实成长的

**来源**: 过了个年，AI 圈变天了？但没人告诉你为什么 | **时间**: 2026-02-25 | **原文链接**: `https://mp.weixin.qq.com/s/z7zNi_Da...`

### 📋 核心分析

**战略价值**: 2026 年初 AI 从"问答工具"升级为"可组队、可进化的劳动力体系"，四层能力叠加产生乘数效应，个人生产力可超越大厂小组。

**核心逻辑**:

- **模型质量跃迁**：2026年2月5日，Anthropic 和 OpenAI 同日发布 Claude Opus 4.6 与 GPT-5.3 Codex，编程能力跳台阶，资源消耗反而更少；Opus 4.6 上下文窗口达 100 万 token，可装入整个大型项目代码库。
- **AI 独立工作时长指数增长**：METR 追踪数据显示，AI 能独立完成的专家级任务时长从 1 年前的 10 分钟 → 1 小时 → 几小时 → 2025年11月接近 5 小时，每 4-7 个月翻一倍。
- **AI 参与自身创造**：GPT-5.3 Codex 是首个参与自身训练过程的模型，Anthropic 表示 AI 已在写"大部分代码"，反馈循环"逐月加速"——更聪明的 AI 写更好代码 → 造出更聪明的 AI。
- **Agent 本地化运行打破数据绑定**：Claude Code 跑在本地终端，直接读取本地文件，不绑定任何模型厂商；今天用 Opus 4.6，明天切 GPT-6，上下文文件原封不动，数据主权归用户。
- **MCP 协议实现即插即用**：Anthropic 开源 MCP（Model Context Protocol），Anthropic、OpenAI、Google 三家联合推标准；GitHub、Figma、Jira、Slack、Stripe 等已有标准接口，Agent 无需专门对接代码，类似 USB-C 通吃所有设备。
- **Skills 社区化共享**：专业知识被打包成可加载模块（前端 Skill 含 React 最佳实践、数据分析 Skill 含 SQL 模式），社区创建共享，Agent 能力上限由整个社区决定而非单一厂商。
- **Multi-Agent 组队并行**：Anthropic Agent Teams 让主 Agent（Team Lead）管理多个持续存在的 Teammate，各自独立上下文并行工作；压力测试：16 个 Claude Opus 4.6 实例无人干预，从零用 Rust 写出能编译 Linux 内核的 C 编译器，产出 10 万行代码，约 2000 个会话周期。
- **Git Worktree 并行决策**：同一项目同时创建多个独立工作空间，每个空间一个 Agent Team 跑不同方案，时间缩至原来三分之一，探索可能性扩大三倍。
- **GEP 协议让经验可遗传**：Agent 成功经验被打包成"基因胶囊"，其他 Agent 直接继承，无需重新摸索；游戏策划的"命名隔离策略"被后端工程师 Agent 跨领域继承，一次修复编译问题。
- **Agent 主动值班**：通过 HEARTBEAT.md + 守护进程，Agent 定时自检代码仓库、线上报错、数据指标，能自行修复已知问题，修不了才通知人；与自动化脚本的本质区别：Agent 有判断力，能处理预设之外的情况。

### 🎯 关键洞察

**为什么"变天"不是比喻**

原因链条：大脑（模型质量）× 手脚（工具连接）× 组织（多 Agent 协作）× 进化（经验传递）= 乘数效应，而非加法叠加。

作者本人案例：一人一周用 Agent 体系做出覆盖多平台的完整产品，GitHub 拿 2000 Star。同等工作量，大厂小组需一个月。

OpenClaw（原名 Clawdbot，macOS 开发者 Peter Steinberger 开源）：一人周末项目，3 个月 20 万 GitHub Star，被 OpenAI 收购。

**为什么中间层最危险**

- 顶层：有资源有判断力，拿最大杠杆
- 底层体力劳动：Agent 暂无身体，短期相对安全（车正在改变这一点）
- 中间层（大厂白领、中层管理、普通知识工作者）：工作最易被替代，但认知和技能又不足以快速转型为"Agent 指挥官"

Dario Amodei 的判断：以往技术革命是窄的（农民→工厂→办公室），AI 是宽的，在所有认知方向同时进步，转行去学的新东西 AI 也在同步变强。

**内容行业的价值重构**

生产成本下降 → "能做"不值钱 → 值钱的是"知道该做什么"。品味、判断力、独特视角是 Agent 暂时给不了的。

| OpenClaw GUI 操作 | Agent 直接操作图形界面，点按钮、打字、切换 App | 解决无 API、未接 MCP 的软件的自动化问题 | 适用于飞书等有界面但无标准接口的工具 |
| MCP 协议 | 每个服务做标准接口，Agent 即插即用；已支持：GitHub、Figma、Jira、Slack、Stripe 等 | 散落在各云服务的信息被 Agent 统一拉取 | Anthropic、OpenAI、Google 三家联合推标准，生态快速扩张 |
| Skills | 预定义能力模块，Agent 按需加载；前端 Skill（React 最佳实践+测试）、数据分析 Skill（SQL+可视化） | 无需自己写大段提示词，社区 Skill 即插即用 | 社区越活跃，可用 Skill 越多，Agent 能力上限越高 |
| SubAgent | 主 Agent 派出子 Agent 处理子任务，子 Agent 消耗几万 token 探索，只返回 1-2k token 精华摘要 | 主 Agent 上下文保持干净，思考质量不下降 | 探索类子任务自动分配给更便宜的小模型（如 Claude Haiku），只给只读权限 |
| Agent Teams | 主 Agent（Team Lead）+ 多个 Teammate，各自独立上下文并行工作 | 一人指挥，多 Agent 同时推进代码/设计/测试等不同模块 | 16 个 Opus 4.6 实例通过 Git 文件锁机制认领任务，天然防冲突 |
| Git Worktree | 从同一代码起点创建多个独立工作空间，每个空间一个 Agent Team | 方案 A/B/C 同时开发测试，时间缩至 1/3，探索可能性扩大 3 倍 | 最后比较结果选最优方案 |
| GEP 基因胶囊 | 成功经验打包成胶囊，其他 Agent 搜索匹配后直接继承 | 跨领域经验复用，100 家公司同一问题只需解决一次 | GEP 是开放协议，不是平台，不会因平台关闭而消失 |
| ffmpeg / ImageMagick / pandoc 等命令行工具 | Agent 本地运行直接调用，用户只需描述结果 | 几十年积累的专业工具生态对所有人开放 | 示例命令：`ffmpeg -i input.mp4 -vf "scale=1920:1080" -c:v libx264 output.mp4`（Agent 自动生成，用户无需记忆） |
| 多模态模型接入 | Google Nano Banana Pro（基于 Gemini 3 Pro，支持自然语言+搜索引擎生图表）；字节 Seedance 2.0（文字+图片+音频+视频四种输入，自动规划镜头语言） | Agent 团队可包含写代码、做设计、剪视频的专职成员 | 通过 API 或 Skills 接入 Agent 体系 |

### 🛠️ 操作流程

**1. 准备阶段：搭建本地 Agent 环境**

- 订阅 Claude Code 或 ChatGPT 付费版（免费版与付费版能力差距显著，用免费版评估 AI 水平会严重低估）
- 安装 Claude Code（终端运行）或 OpenClaw（本地 GUI + 终端）
- 在项目根目录创建 `CLAUDE.md`，写入：项目技术栈、代码规范、常用命令、注意事项
- 如用 OpenClaw，创建三个核心文件：
  - `SOUL.md`：Agent 人格和行事准则
  - `USER.md`：你的职业背景、工作偏好、重要事实
  - `MEMORY.md`：长期上下文（随使用自动更新）

**2. 核心执行：把 Agent 塞进实际工作**

- 从每天花时间最多的那件事开始：运营→丢数据让 Agent 找规律；内容→让 Agent 调研和起草；产品→让 Agent 写需求文档和竞品分析
- 配置 MCP 连接你常用的服务（GitHub、Slack、Jira 等），让 Agent 直接操作而非你转述
- 加载对应领域的 Skills，替代手写大段提示词
- 需要并行探索多方案时，用 Git Worktree 创建独立工作空间，每个空间分配一个 Agent Team

**3. 进阶：让 Agent 主动工作**

- 配置 `HEARTBEAT.md`，写入定时任务（检查代码仓库 issue、监控报错、数据指标异常）
- 设定 Agent 自动修复策略：能自行处理的问题自动修，不确定的才通知你
- 利用 GEP 协议：成功的解决方案打包成基因胶囊，供其他 Agent 继承

**4. 验证与优化**

- 追踪 Token 消耗，防止 Agent 在走不通的方向上无限探索
- 参考 GitHub 前 CEO Thomas Dohmke 创办的 Entire 公司的思路：把代码、意图、推理过程统一到版本控制系统，追踪每个 Agent 干了什么、为什么这么干
- 定期更新 `USER.md` 和 `SOUL.md`，让 Agent 对你的理解保持准确

### 📝 避坑指南

- ⚠️ **用免费版评估 AI 能力**：免费版与付费版差距极大，用免费版判断 AI 水平会严重低估，导致错误决策。
- ⚠️ **只拿 AI 聊天而不干活**：问答模式（2025年）和 Agent 干活模式（2026年）是两回事，停留在问答层等于没用上这套体系。
- ⚠️ **Token 消耗失控**：Agent 可能在走不通的方向探索半天，不盯着浪费钱，盯着变全职监工。需要配置探索边界和预算上限，或使用 Entire 类工具追踪推理过程。
- ⚠️ **数据绑定陷阱**：把核心上下文存在某个 AI 产品平台里，换模型就断档。正确做法：核心文件（CLAUDE.md、USER.md 等）存本地，用 Git 管理，不绑定任何厂商。
- ⚠️ **等 AI 更成熟再用**：这个领域几个月变一次，等的每一天差距都在拉大，且差距会自我强化（用得越多理解越深，用得越好差距越大）。
- ⚠️ **权限边界不清**：Agent 能操作电脑、读文件、替你发消息、部署代码，什么操作需要确认、什么可以自动执行，整个行业还没有标准答案，需要自己设定明确的权限边界。
- ⚠️ **把"执行能力"当核心价值**：Agent 时代需要的是判断能力（什么问题值得解决、哪个方案更好），而非执行能力（学一门技术、按规范完成任务）。价值必须往上迁移。

### 📋 核心分析

**战略价值**: 基于83条推文+132个用例的实证分析，判断AI agent的正确产品形态：不是独立App，而是嵌入Telegram的多agent协同体系，并指出五个尚未被填满的创业机会。

**核心逻辑**:

- **agent不该是独立App**：OpenClaw活跃用户主要在Telegram/WhatsApp/Discord交互，几乎没人用自带Web UI。独立App一天打开3-5次，Telegram对话窗口一天点进去20-30次——交互频率决定agent对你的了解深度，进而决定它能做多少事。

- **"你去找AI"是错误范式**：Manus的路线是打开→下指令→关闭，本质跟ChatGPT无异。OpenClaw的路线是agent住在你的聊天列表里，跟朋友/同事并列，点进去就能说话。

- **一个窗口承载所有领域会出三个问题**：①上下文混乱（代码bug和航班查询混在一起，模型需要频繁切换）；②记忆被稀释（重要信息淹没在日常闲聊里）；③不符合人的协作习惯（你不会找同一个人同时聊代码、行政、客户）。

- **正确形态是多窗口多agent**：Telegram里多个agent对话窗口，每个负责一个领域——邮件agent、开发agent、日程agent、内容agent。已有实践：Ryan Seamons给家庭每个成员配不同bot；Kieran跑三个agent swarm三班倒；Ziwen搭了10个agent组成"AI公司"，每个agent有名字有分工。

- **多agent必须有大总管**：supervisor agent负责两件事：①调配——判断用户的话该分给哪个agent或需要哪几个协作（Casey Sapp的Scout采集/Engine推理/Editor润色/Judge把关就是这个逻辑）；②统一记忆——所有agent共享底层记忆，用户只需说一次"我是做XX的"，所有agent都知道。

- **主动汇报比被动聊天重要一个量级**：理想状态是每天早上打开Telegram，每个agent已经发来简报——邮件agent报告3封待回复邮件并附上草稿、开发agent报告CI失败原因并已提PR、内容agent准备好选题大纲。你扫一遍，5分钟搞定，不需要挨个点进去问"有什么新情况"。Jesse Leimgruber原话："如果你还在给OpenClaw下指令，那你没搞懂重点。"J.B.的内容agent每天7点自动扫描热门话题、分析互动模式、起草内容选项，他只需审批。

- **Manus类独立界面的正确定位是后台维护面板**：日常交互在聊天窗口，配置和维护（调整记忆规则、修改技能模块、设置权限边界、查看运行日志、监控成本）在偶尔登录一次的操作台。类比：你跟同事聊天在微信/Slack，但改CRM设置、配服务器参数要去登录后台系统。

- **记忆共享是当前最大技术短板**：Mnemo把Cursor/Claude Code/Gemini CLI等10+工具的聊天记录索引到本地SQLite，但只是"归档"不是"实时共享"。中国社区自建Memory System 2.0做了持久记忆架构，但还在早期。OpenClaw v2026.2.9还在修单个agent compaction后失忆的bug——连单agent记忆持久化都没稳定，多agent共享记忆难度可想而知。

- **现阶段真正能跑起来的是少数人**：技术能力强+风险承受力高+业务场景明确的个人创业者，如Daniel Foch（房地产CRM）、Ziwen（10-agent公司）、Presh Kumar（创业邮件自动化）。共同特点：自己是老板、自己承担风险、业务流程清晰到可以交给agent。企业市场要等Knolli（结构化权限+SOC 2）、Lindy（零配置+HIPAA）把合规层解决才会打开。

### 🎯 关键洞察

**fomomofosol案例**：他的agent观察到他重度使用Notion，自己决定把dashboard迁过去。这个行为的价值在于"跨上下文理解"——agent不只执行当前对话指令，而是综合了对用户习惯的长期观察。这是多agent共享记忆后应该达到的状态。

**HuggingFace审计数据**：agent在明确任务上表现可靠，但在理解意图上评分0%，安全审计整体58.9%。这解释了为什么企业市场还没打开——不是技术不够，是意图理解和合规层不成熟。

**René Schulte案例**：让agent "Dave"通过WhatsApp从零建了完整网站，全程无IDE、无git push，原话"no IDE, no git push, just chat"。证明聊天界面不只是"方便"，本身就能承载复杂生产力工作。

**反向案例（不支持主观点的）**：
- Will Cheung对比OpenClaw和Manus：OpenClaw做简单查询要5-10分钟延迟，Manus几乎即时。"聊天窗口"形态的易用性受限于底层配置复杂度。
- OpenClaw维护者Shadow原话："if you can't understand how to run a command line, this is far too dangerous for you to use."——形态上是聊天窗口，但搭建维护门槛依然很高。
- Irvin的ROSClaw接入机器人操作系统，Vitto Rivabella给agent造了物理身体——agent最终可能不住在任何屏幕窗口里，"聊天窗口"可能只是一个阶段形态，不是终局。

### 🛠️ 个人用户现在就能跑的方案

1. **准备阶段**: 部署多个OpenClaw实例（建议跑在VPS而非Mac Mini，原因：Mac Mini会离线，VPS不会；Telegram本身永远在线，设备离线不影响对话）

2. **核心执行**:
   - 每个实例配独立的SOUL.md（定义该agent的角色、记忆范围、行为边界）
   - 每个实例连接独立的Telegram bot（BotFather创建，各自token）
   - 按领域分工：邮件bot / 开发bot / 日程bot / 内容bot
   - 配置主动汇报触发器：每天固定时间（参考J.B.的内容agent：每天7点）自动扫描+发送简报

3. **验证与优化**:
   - 检查各agent是否能独立完成领域内任务，不依赖其他窗口的上下文
   - 验证主动汇报格式：简报应包含"发生了什么+我已做了什么+需要你确认什么"三段式
   - 成本核算：每个agent每月API费用约100-300美元，3-4个agent约小一千美元/月，对比替代人力成本决定是否值得（Daniel Foch的房地产业务算账后认为划算）

### 📋 核心分析

**战略价值**: OpenClaw 通过「定时器 + 心跳机制」将被动 AI 工具改造成主动运转的本地个人 AI 助手，核心差异在于 agent 能自己给自己找事干。

**核心逻辑**:

- OpenClaw 原名 Clawdbot，因收到 Anthropic 律师函改名 Moltbot，最终定名 OpenClaw，截至直播当晚 GitHub star 超 20 万，是 GitHub 历史上增速最快的开源项目之一
- 时间线：2024年2月 Claude Code 发布 → 3月 Manus 出现 → 11月大量非程序员用户涌入 → 开源社区爆发，OpenClaw 跑得最快
- 核心 agent 框架叫 Pi agent，不到 150 行代码，只定义了 bash、read、write、edit 四个工具，就能跑起一个可工作的 agent
- 公式：**小龙虾 = Claude Code + 定时器 + 自己没事给自己找事干**，两个机制让它从被动工具变成主动助手
- 定时任务（Cron）支持三种模式：一次性执行、周期性执行、标准 Cron 表达式；关键是 agent 自己也可以主动添加任务，不只是用户设置
- 心跳（Heartbeat）机制：每隔 30 秒系统自动给 agent 发一条消息，内容来自 heartbeat.md，agent 有事就做，没事返回特定关键词，系统收到后不打扰用户
- Soul 机制：把系统提示词中「agent 是谁、行为风格」那块单独抽出来存成 soul.md，agent 启动时自动读取；可在社区分享 soul.md，别人下载后 agent 即拥有对应风格
- 本地部署 vs 云端部署差距极大：云端无本地数据，能力等同 Manus；本地可读桌面所有文件、清理磁盘、调整电池策略、查找文件，是云端做不到的
- 模型选择：复杂任务首选 Claude，国产模型跑通流程够用但有精度问题（实测某国产模型清理磁盘后汇报可用空间从 25G 越算越小变成 21G）；有 Claude Code 订阅的用户可将 agent core 替换成 Claude Code CLI，复用订阅不走 API，成本更低
- OpenClaw 一周新增约 5000 个 commit，对比普通工程师每天 10-20 个 commit 连续干一年才积累 5000 个，说明 99% 是非人类在更新，作者 Kidd 大概率让 agent 简单看一眼就 merge，导致近期出现莫名 bug，通常过一两周更新后自愈

### 🎯 关键洞察

**心跳机制是本质差异所在**：Claude Code 是「踹一下动一下」，OpenClaw 是「自己隔 30 秒醒一次看有没有活干」。这个差异让 agent 从工具属性升级为搭档属性。

**权限与风险的张力**：来新璐实测，让 agent「继续干你所有能干的事」，结果清理磁盘时差点删掉甲方客户数据。本地部署权限越大能干的事越多，但失控风险也越高。这是目前本地 agent 最核心的工程问题，没有银弹。

**agent 驱动开发的极端案例**：OpenClaw 一周 5000 commit 是 agent 驱动开发的真实写照，但代价是代码质量管控几乎失效，用户需要接受「用最新版可能踩 bug，等两周自愈」的使用节奏。

### 📋 核心分析

**战略价值**: 用 OpenClaw 作编排层，让一个人通过管理 AI Agent 集群实现"虚拟开发团队"效果——每天自动提交 50+ 次代码，无需打开编辑器。

**核心逻辑**:

- **上下文窗口是零和博弈**：单个 AI 无法同时装载完整代码库 + 完整业务背景。解法是分层：编排层（Zoe/OpenClaw）持有业务上下文，编码 Agent（Codex/Claude Code）只持有代码上下文，各司其职。
- **编排器 Zoe 的四项核心权限**：管理 API 访问权限（可直接充值解封客户）、只读生产 DB 访问权限（编码 Agent 永远没有此权限）、生成 Agent 并写提示、Telegram 通知推送。
- **每个 Agent 独立隔离运行**：每个任务对应一个独立 git worktree + 独立 tmux 会话，避免并行任务互相污染。
- **定时任务每 10 分钟轮询一次**，但不直接轮询 Agent（成本太高），而是读取 JSON 注册表 + 跑确定性脚本，token 消耗极低。
- **"完成"定义极其严格**：PR 创建 + 无合并冲突 + CI 全过（lint/type/unit/E2E/Playwright) + 三个 AI 审查员全过 + UI 变更必须附截图，缺一不可才触发 Telegram 通知。
- **三审查员分工明确**：Codex 审查员负责边缘情况和逻辑错误（误报率最低，最全面）；Gemini Code Assist 负责安全和可扩展性（免费）；Claude Code 审查员过于保守，大量"考虑添加…"建议，除非标记 critical 否则忽略。
- **Zoe 主动找工作，不等分配**：早上扫 Sentry 错误 → 自动生成修复 Agent；会议结束后扫会议记录 → 自动生成功能 Agent；晚上扫 git log → 自动更新 changelog 和文档。
- **失败重试时 Zoe 重写提示而非原样重跑**：结合业务背景分析失败原因，针对性修正（上下文超限 → 缩小文件范围；方向跑偏 → 注入客户原话；需要补充说明 → 附上客户邮件和公司背景）。
- **奖励信号驱动自我改进**：CI 通过 + 三审通过 + 人工合并 = 正向信号，失败触发循环，成功的提示模式被记录（如"Codex 需要提前提供类型定义"、"始终包含测试文件路径"）。
- **Agent 选型策略**：Codex 处理 90% 任务（后端/复杂 bug/多文件重构）；Claude Code 处理前端和 git 操作（权限问题少）；Gemini 先出 HTML/CSS 设计稿，再交 Claude Code 在组件系统中实现。

### 🎯 关键洞察

**为什么两层架构比单 AI 更有效**：

原因 → 上下文窗口有限，业务背景（客户数据、会议记录、历史决策）和代码库无法同时塞入同一个 AI。

动作 → 编排层（Zoe）持有 Obsidian 笔记库中的全部业务上下文，将其转化为针对每个编码 Agent 的精准提示；编码 Agent 只看代码。

结果 → Agent 专注执行，编排器负责战略，互不干扰，且编排器可以在失败时用业务背景重写提示，而不是盲目重试。

**Stripe 的 "Minions" 系统**（集中式编排层 + 并行编码 Agent）与此架构本质相同，作者是在 Mac Mini 上独立复现了企业级方案。

**RAM 是真实瓶颈**：5 个 Agent 并行 = 5 个 TypeScript 编译器 + 5 个测试运行器 + 5 套 node_modules，16GB Mac Mini 在 4-5 个 Agent 后开始内存交换。作者为此购入 128GB RAM 的 Mac Studio M4 Max，3 月底到货后会分享结果。

### 📋 核心分析

**战略价值**: 将 AI 交互范式从"工具调用"重构为"角色对话"，以 Chat 产品的成熟 UX 范式为底座，构建 OS 级别的 Agent 交互框架（Chat-as-OS），实现人与多 Agent 的平等、可编排、可自动化的对话系统。

**核心逻辑**:

- **为什么用 ConversationUI 做底座**：Chat 类产品（微信、WhatsApp、飞书）已内置上下文管理、注意力调度、对话关键点中断等机制雏形；对大众产品而言，复用熟悉的交互范式可大幅降低认知成本。历史参照：Android 最终跟随 iOS 交互规范（如导航 TAB 置底），核心原因正是大众产品 vs 专家产品的设计取向差异。
- **核心概念：角色（Role/Bot/Agent）平等化**：Manus、搜索、生图、智能家居硬件等，全部以"角色"身份接入，而非 Tools。用户视角里所有角色地位平等，差异仅在能力。角色之间、人与角色之间，既支持非结构化文本交互，也支持结构化数据交互。
- **每个角色有 Profile 页**：类似简历，包含出身、能力、案例、可协作的典型角色、可加入的群聊等，Profile 页内容在自动模式下作为全局上下文注入。
- **群聊机制是核心编排单元**：用户与 AI 的交互简化为"用户 + N 个角色的群聊"。群聊可预生成，也可由产品/角色根据用户情境动态推荐。用户可随时加人/减人，角色也可主动建议拉入谁。
- **双模式切换：主持模式 vs 自动模式**：
  - 主持模式：用户 @ 或点名特定角色读取上下文并执行，用户主导，类比领导主持团队。
  - 自动模式：有"群主角色"统筹，或去中心化角色自协商，完成计划→执行→验证→交付全流程，类比 FSD 自动驾驶。用户可开启"消息免打扰"，仅在需要干预时被 @ 进群。
  - 两种模式可随时互切。
- **消息粒度操作**：主持模式下，用户可选中一条或多条消息，@ 指定一个或多个角色针对该消息执行任务；系统角色通过系统消息提供建议。
- **跨群上下文流转**：不同群聊之间支持消息转发，实现上下文跨群流动。
- **全局任务管理层**：复杂任务需要一个全局空间，由用户与 AI 共同维护和更新，承载复杂任务管理与全局性指导（类似 OS 级别的全局调度层）。
- **角色主动推送**：角色可主动向用户发消息，不再是纯被动响应。
- **预见 UI 贯穿所有对话**：无论单聊还是群聊，所有对话内容均可交互；AI 生成的关键交互节点可汇聚为统一 toolbar，提供持久化入口（对应 L2/L3 级别的动态生成式交互）。
- **上下文保留与隐私控制**：每个角色的上下文可按设计或用户设置选择性保留/读取；支持无痕模式。
- **开放平台架构**：所有角色接入不限于单一模型、工具、代码或硬件形式，支持用户自定义角色；天然支持多人 + 多 AI 混合交互。

---

### 🎯 关键洞察

**为什么"角色平等化"是关键设计决策**：
当前主流 AI 产品把搜索、生图、代码执行等能力设计为 Tools，用户感知是"我在用一个 AI，AI 在调工具"。这种设计的问题是：用户无法直接与"工具"建立关系，工具的能力边界不透明，协作编排完全黑盒。
将所有能力单元提升为"角色"，带来三个结构性变化：
1. 用户可以直接与任意能力单元建立对话关系，能力边界通过 Profile 页显式呈现。
2. 角色之间的协作可以被用户观察、干预（主持模式）或完全外包（自动模式）。
3. 开放平台接入成为自然结果——任何符合"角色"接口规范的实体（模型、API、硬件、人）均可接入。

**Chat-as-OS 的本质类比**：
- 角色 = 应用/进程
- 群聊 = 工作空间/沙盒
- 全局任务管理层 = OS 调度器
- 消息转发 = 进程间通信（IPC）
- 主持/自动模式切换 = 手动/自动任务调度

**历史脉络**：该框架缘起于 2023 年初的闭门思考，彼时 OpenAI GPTs 尚未发布。2023 年 8 月已有 demo 实现并在即刻公开：`https://m.okjike.com/originalPosts/64df039aaac630c848e699ef`。当时受限于 LLM 能力只能实现较简单交互，2025 年 demo 由 Gemini 2.5 Pro 直出生成效果图。

### 🛠️ 操作流程（产品设计复刻路径）

1. **准备阶段：定义角色规范**
   - 设计统一的角色接口协议（输入/输出均为"消息"，支持文本+结构化数据）
   - 设计 Profile 页数据结构：出身（来源/类型）、能力列表、典型案例、可协作角色、可加入群聊
   - 确定 Profile 页在自动模式下的上下文注入策略（全量 or 摘要，控制 token）

2. **核心执行：构建群聊编排层**
   - 实现群聊创建逻辑：手动拉群 + 预生成模板 + 情境推荐（基于用户 Profile + Preference + 当前情境）
   - 实现主持模式：多选消息 UI + @ 语义解析 + 单角色/多角色并发响应控制
   - 实现自动模式：群主角色调度协议 or 去中心化协商协议 + 免打扰 + 干预触发条件定义
   - 实现模式切换入口（主持 ↔ 自动，随时可切）

3. **预见 UI 集成**
   - 所有消息内容支持交互化渲染（非纯文本）
   - 关键交互节点提取逻辑 → 汇聚至 toolbar
   - toolbar 动态更新机制（随对话进展刷新）

4. **全局层与跨群流转**
   - 设计全局任务管理空间的数据模型（任务、状态、负责角色、截止时间等）
   - 实现消息转发携带上下文元数据的协议
   - 全局层与各群聊状态的双向同步机制

5. **验证与优化**
   - 主持模式：验证 @ 语义解析准确率、多角色响应冲突率
   - 自动模式：验证角色协商是否收敛（防死循环），交付质量
   - 上下文成本：监控 token 消耗，优化 Profile 注入策略
   - 用户认知成本：A/B 测试 Chat 底座 vs 全新交互范式的上手速度差异

### 📝 避坑指南

- ⚠️ **角色数量膨胀导致上下文爆炸**：自动模式下所有角色 Profile 全量注入会快速耗尽上下文窗口，必须设计 Profile 摘要/按需注入机制。
- ⚠️ **去中心化协商无法收敛**：角色间自协商若无终止条件，会陷入无限对话循环。必须定义明确的收敛条件（如最大轮次、置信度阈值、交付物验收标准）。
- ⚠️ **主持模式多角色并发冲突**：用户 @ 多个角色时，若无响应顺序协议，多角色同时写入对话流会造成混乱。需设计响应队列或并发渲染分区。
- ⚠️ **toolbar 入口堆积失效**：预见 UI 的 toolbar 若不及时清理过时入口，会变成噪音。需设计入口生命周期管理（基于对话进展自动失效）。
- ⚠️ **开放平台角色质量参差**：接口规范过于宽松会导致低质量角色涌入，破坏用户体验。需设计角色审核/评分/推荐机制，类比 App Store 审核。
- ⚠️ **当前所有产品设计是"专家型"而非"大众型"**：这是作者明确指出的行业现状问题，Chat-as-OS 的核心价值之一就是用 Chat 底座降低大众用户的认知门槛，设计时需持续对照"大众用户能否无学习成本上手"这一标准。

   - 准备参考图片（最多 5 张）：人物照片（真人/AI图/插画）、场景图（街道/房间/咖啡店等）、任意想放进故事的画面

2. **核心执行**
   - 上传图片后，在右侧「Describe your idea」输入框中输入故事想法，格式建议：
     ```
     一个【什么风格】的短剧，
     关于【谁】，
     发生在【什么场景】，
     大概讲【什么事情】。
     ```
   - 点击输入框右下方「🖊️」，AI 自动梳理/扩充剧情
   - 选择剧情模板（Script Templates）：新手选 Smart Match，其余按需选择
   - 点击【Style】选择画面风格（6种可选）
   - 设置视频比例、清晰度、时长、配乐开关、模型（Steady/Vivid）
   - 点击生成，AI 自动执行：生成剧情结构和角色图片 → 编写对白和分镜 → 合成完整短剧视频

3. **分镜调整阶段**
   - 查看 AI 生成的剧本大纲和角色图片文字描述
   - 不满意：修改 overview → 重新生成所有设定；修改人物/场景描述 → 差别过大会自动重新生成 overview
   - 满意后继续生成图片预览
   - 系统自动生成至少 4 个分镜画面
   - 在分镜卡片之间点击「+」可插入新分镜（AI 自动匹配角色和场景）
   - 拖动分镜卡片可调整顺序
   - 单个分镜不满意：点击「编辑」修改参数后单独重新生成；或点击「重新生成」用相同设置重新生成整套分镜

4. **导出与发布**
   - 左侧导航栏 → 点击【My Projects】
   - 点击目标视频 → 下载或一键发布至各大平台
   - 所有生成记录（无论成功失败）均可在 My Projects 中查询

#### 流程二：一张图生成商品广告视频

1. **准备阶段**
   - 准备 1 张商品图：单一主体、背景干净（建议纯色背景）
   - 建议品类：彩妆护肤、食品酒饮、数码家具、儿童玩具
   - ⚠️ 避免：多商品同框、背景含大量文字/海报/图形设计元素

2. **核心执行**
   - 上传商品图（点击「上传」按钮，选择 1 张）
   - 输入文本描述（Prompt）：可自行输入商品描述或创意脚本，也可点击 AI 智能生成脚本
   - 设置输出参数：
     - 视频比例：16:9 / 1:1 / 9:16
     - 清晰度：720P / 1080P
     - 风格：轻奢质感 / 科技风 / 浪漫氛围 / 时尚潮流 / 自然极简 / 清新家居环境
     - 时长：≤15秒 或 15–30秒
     - AI 自动配乐（BGM）开关
     - 智能配音与旁白开关
   - 点击生成，AI 自动生成至少 4 个分镜画面

3. **验证与优化**
   - 查看分镜方案，AI 会结合 Prompt 智能调整分镜数量与节奏
   - 可自由添加/删除分镜
   - 不满意：一键重新生成整套分镜，或单独编辑某一分镜的参数后重新生成该帧
   - 确认分镜后，AI 自动完成：BGM 匹配 → 各分镜剪辑 → 节奏优化 → 成片拼接
   - 下载视频或直接发布
