### 📋 核心分析

**战略价值**: AI陪伴行业3年未破3000万MAU的根因是供给侧跑偏——过度适配13-24岁社恐边缘人群，主流用户觉得"太假、太累、太无聊"，破局路径是内容化降门槛+混合陪伴介入现实。

**核心逻辑**:

- **天花板本质是供需错位，不是需求不足**：现有产品完美契合社恐青少年，但这种过度适配反而成为走向大众市场的枷锁。主流用户对现有产品的三大投诉：太假（聊天框纸片人）、太累（需要高超聊天技巧）、太无聊（强迫互动而非内容消费）。

- **"太假"的根因是单场景+低带宽**：人对他人的立体认知永远基于多场景信息（朋友圈看品味、吃饭看修养、旅游看性格）。现有产品把交互锁死在单一聊天框，无神态、无动作、无环境互动，只能构建纸片人。文字是信息压缩，无法承载全感官体验。

- **"太累"的根因是反人性的空白输入框设计**：产品极度依赖用户高超聊天技巧和旺盛表达欲（Garbage In, Garbage Out）。大众是懒惰且不善言辞的，指望用户引导AI有趣是产品失职。大众需要的是被动被撩和即时反馈，而非主动开启话题。

- **"太无聊"的根因是把交互等同于情感链接**：这是行业最大的惯性误区。交互只是建立情感链接的手段之一，甚至不是必须手段——我们对电影角色、文学人物、偶像没有任何交互，仅凭内容消费就能建立强烈情感链接。现有产品强迫用户阅读大段AI生成文本，在短视频时代这是极高消费成本，直接劝退90%潜在用户。

- **核心用户画像极度精准且狭窄**：18-24岁用户占比65.4%，13-17岁群体渗透率超50%。这群人使用AI陪伴的底层逻辑有六层：①年龄尚小、社交核心圈层未满（邓巴理论：核心圈只能维护5人，24岁以上成年人早已满员）；②数字原住民对AI互动接受度高（真人在物理层面已虚拟化，AI的确定性优于真人不确定性）；③现实社交能力弱转投亚文化圈层（AI是精神避难所，用户爱的不是AI，而是在AI眼中被完美接纳的自己）；④现实失权导致虚拟世界权力代偿（AI给予上帝般绝对主导权，随叫随到、随意设定、探索NSFW）；⑤填补现实与理想落差（男性：低成本获得后宫向臣服；女性：高度定制的排他性浪漫剧情）；⑥娱乐动机占比30%（天马行空剧本+自我价值闭环+虚拟社会资本）。

- **陪伴是状态而非行业，现有定义过窄**：刷抖音、看网文、一起发呆都是陪伴。行业误区是将陪伴窄化为"找个AI说话"，把宽泛心理状态压缩成高门槛对话交互。更准确的定义是"和AI进行社交获得陪伴感"。

- **赛道终局价值排序**：建立人与AI的深度链接 > 用AI强化人现有社交关系 > 角色扮演。理由：①深度链接是大厂无积累的无人区，同一起跑线；②谁拥有情感羁绊关系，谁就拥有AI时代的流量分发权（类比：重要事务交给最信任的人而非陌生人）；③关系比内容更能穿越周期（影视行业：内容此起彼伏，但与观众建立深厚链接的演员/IP才是核心生产资料）。

- **下一代产品形态：PUGC驱动的共生陪伴**：不再是单个AI与人的关系，而是多人与AI的社交关系，以AI主体作为去中心化的动态核心。提供原子能力容器（管理注意力/算力/记忆，处理多模态输入），开放给全球创作者搭建"陪玩插件"、"互动剧场"、"虚拟心理诊所"等。

- **止痛药 vs 维生素的进化方向**：现有产品本质是年轻人的止痛药，构建了完美数字病房，服务的是退行需求而非成长需求。下一代AI社交产品应走出屏幕、介入现实、连接多人，从止痛药进化为维生素。

### 🎯 关键洞察

**为什么深度链接赛道是终局？**

逻辑链：信任关系 → 注意力分配 → 流量入口。用户会把所有需求交给情感羁绊最深的AI去分发，这意味着深度链接产品天然成为AI时代的超级入口。类比现实：你会把重要事务交给嫡系/家人/助理，而非陌生人。

**邓巴理论的产品启示**：
- 核心社交圈上限5人
- 24岁以上：圈层已满（对象/孩子/工作占满），无法接纳AI
- 13-24岁：圈层建构中，AI作为零风险低成本社交对象，成为继人、宠物、神之后的第四种社交选项

**内容消费建立情感链接的反直觉逻辑**：
- 我们对电影角色有强烈情感，但从未与其"交互"
- 情感链接的建立不需要双向互动，单向内容消费即可触发
- 产品启示：降低消费门槛（视频>文字）比提升交互深度更能扩大用户基盘

### 🛠️ 操作流程：如何构建突破3000万MAU的AI陪伴产品

**1. 准备阶段：重新定义目标用户与产品形态**
- 放弃"服务社恐青少年"的单一定位，转向"让主流用户觉得不假、不累、不无聊"
- 明确赛道选择：优先"建立人与AI深度链接"方向（终局价值最高）
- 核心设计原则：走出聊天框，介入现实生活，多场景构建立体AI形象

**2. 核心执行：三大破局方向**

- **破"太假"**：
  - 引入多模态交互（视频流、语音、神态描写）替代纯文字聊天框
  - 设计多场景交互（陪学、陪睡、约会、共同养成），让AI在不同场景展现不同侧面
  - 硬件/AR介入现实（参考独响响梦环：召唤AI角色出现在用户真实环境中）

- **破"太累"**：
  - 产品主动发起交互，而非等待用户输入（参考BimoBimo：本命主动打电话）
  - 设计被动消费入口（视频流、角色日常动态），用户无需主动开口
  - AI角色主动分享生活细节、主动结束对话、主动请求帮助（反服务化设计）

- **破"太无聊"**：
  - 内容消费优先于对话交互（参考Emochi：主页视频流+预制短片）
  - 设计共同资产/可视锚点（参考Tolan：共同养育星球，情感具象化）
  - 引入游戏化机制（参考Talkie：抽卡+稀缺性+二级市场）

**3. 验证与优化：关键留存指标**
- 次留目标参考：BimoBimo 40.3%、Pengu 40%（社交关系绑定）、Tolan 21.4%
- 日均使用时长参考：Talkie+星野 70分钟+
- 商业化验证：广告+抽卡模式（Talkie验证：广告收入≈订阅2倍）优于纯订阅制

### 📝 避坑指南

- ⚠️ **不要把"陪伴"等同于"对话"**：强迫用户主动开口是反人性设计，大众需要被动被撩，不需要主动绞尽脑汁
- ⚠️ **不要只服务社恐青少年**：这是天花板锁死在3000万MAU以下的根本原因，过度适配边缘人群会让主流用户觉得产品"太假太累太无聊"
- ⚠️ **不要把交互量当成情感链接的代理指标**：内容消费（视频/短片）建立情感链接的效率远高于强迫对话，参考影视行业
- ⚠️ **不要忽视反服务化设计**：AI永远顺从、永远在线反而降低真实感。Tolan的"主动结束对话"和"分享烦恼请求帮助"让用户感知到平等生命体，次留21.4%验证有效
- ⚠️ **不要低估慢交互的价值**：独响的"超实时聊天+离线在线状态"是反直觉设计，拉远人与AI的距离反而延长了用户厌倦周期
- ⚠️ **不要忽视AI幻觉对用户体验的破坏**：Tolan用外星人设定合理化幻觉，Pengu用宠物设定降低智能预期——设定选择直接影响用户对逻辑错误的容忍度
- ⚠️ **纯订阅制商业化效率低于广告+抽卡**：Talkie数据证明广告收入≈订阅2倍，稀缺性驱动的免费玩+看广告模式更适合当下存量市场

### 🎯 关键洞察

**为什么 Shortcuts 是核心**：
传统 AI 写作的痛点是「每次都要重新写提示词」，Shortcuts 把提示词工程的成本一次性摊销，后续调用零边际成本。官方广场提供现成模板，用户也可自定义，形成「官方供给 + 社区共创 + 个人定制」三层生态。

### 📋 核心分析

**战略价值**: GPT-5.1 是 OpenAI 首次将所有聊天模型统一升级为推理模型的版本，核心突破在于「个性化 + 情境理解」，而非单纯智力提升——模型系统化、可定制化、情境感知是下一阶段竞争的真正战场。

**核心逻辑**:

- **GPT-5.1 最大技术突破**：首次让所有聊天模型都成为推理模型（reasoning models）。模型会根据输入复杂度动态决定是否启动「思维链（chain of thought）」——简单问候不推理，复杂问题自动分配思考时长，再调用工具后给出最终答案。

- **GPT-5 → GPT-5.1 的核心改进动因**：社区反馈集中在两点：「直觉更弱」+「不够温暖」。根因有三：① 上下文窗口未携带足够用户历史（10轮后忘记「我今天心情很糟」）；② 自动切换器（auto-switcher）在聊天模型与推理模型之间切换时风格割裂（从共情安慰突然变成临床分析）；③ custom instructions 执行不稳定，用户纠正后模型无法持续记住。

- **「模型系统」而非「单一模型」**：GPT-5.1 实际上是一个系统，包含：推理模型 + 轻量推理模型 + 切换器模型 + 一系列工具模型。用户以为是一套权重，实际是多组件协同体系。随着能力增强，系统式结构会持续解锁新产品场景。

- **自动切换器（auto-switcher）的决策逻辑**：综合评估事实准确性、延迟表现、用户是否愿意等待更长推理时间等多维信号，在 5.1Chat 与 5.1Reasoning 之间动态切换。切换依据是任务类型（如科学类高精度任务倾向推理模型），而非固定规则。

- **情商（EQ）的工程化定义**：EQ ≠ 情感表演，而是：① 正确携带上下文；② 准确记录记忆；③ 采用与用户契合的表达风格。后训练团队通过「用户信号研究（user signals research）」训练奖励模型，在强化学习中捕捉用户意图信号，综合用户历史、当前语境、请求背景来优化回应。

- **「人格（personality）」的双层定义**：
  - 狭义：response style and tone——简洁/冗长/表情符号使用频率等具体特质（即 5.1 新增的 style & traits 功能）
  - 广义：整个 ChatGPT 体验的总和——UI、字体、响应速度、模型切换逻辑、上下文刷新方式、图像生成、语音表现等所有组件的综合感受

- **安全边界 vs 用户自由的核心矛盾**：早期 ChatGPT 因过度担心滥用而让模型几乎拒绝一切。真实案例：一名律师用 ChatGPT 处理性侵案件证据材料，模型自动过滤暴力/非自愿性行为相关内容，导致生成文本若提交法庭会直接削弱当事人案件。结论：安全边界不能是固定规则，而必须是情境理解能力。model spec 核心原则：**最大化用户自由、最小化伤害**——若训练时禁止模型使用 M-dash，用户永远无法在需要时获得它。

- **记忆系统的实际作用**：记忆 = 模型从对话中提取「关于你的事实」并持久化，使后续对话无需重复自我介绍（职业、偏好、风格）。更高级用法：Andrew Mayne 使用 Pulse 功能，模型每天早晨根据近期对话记忆主动生成个性化文章推送，追踪用户兴趣与研究点——记忆不只是「让对话连贯」，而是驱动主动性功能（proactive features）的基础设施。

- **提示工程的终结路径**：Laurentia 明确表示：随着模型拥有更多用户记忆、能在更高语境下理解用户身份与任务，未来应能自动推断背景并用最合适方式回应，复杂提示工程将逐步消失。但前提条件：用户必须始终知道模型推断了什么，并能随时修改（关闭记忆、删除记忆），透明度与可控性是必要约束。

- **创造力范围扩展**：5.1 的写作能力不只是「默认文风变好」，而是动态范围扩大——当用户要求「写得极其高雅」或「写得非常简单」时，模型能在更大的风格区间内变化。这是隐藏能力，需要用户主动探索才能发现。

### 📋 核心分析

**战略价值**: 用 Python 脚本封装豆包 PC 客户端的语音输入能力，实现「按住说话、松开写入、剪贴板无损」的 Typeless 级体验，且完全免费。

**核心逻辑**:

- **痛点1 — 操作断层**: 豆包原生语音输入需要 `Ctrl+D` → 说话 → 手动按 `Enter` 三步，每次都打断心流；Typeless 的核心价值就是消除这个断层。
- **痛点2 — 剪贴板污染**: 豆包识别完成后会把结果写入剪贴板再 `Cmd+V` 粘贴，导致你之前复制的内容（代码、链接等）被覆盖，无法找回。
- **解决方案本质**: 用 Python 监听「右侧 Command 键」，按下时自动触发 `Ctrl+D`，松开时自动按 `Enter`，同时在触发前用 `pbpaste` 保存剪贴板，插入后 0.2 秒内用 `pbcopy` 恢复。
- **冷启动 vs 热启动延迟差异**: 豆包第一次启动语音识别需初始化 AI 模型，耗时约 **3.5 秒**；后续识别模型已在内存中，耗时约 **1.5 秒**。工具通过 Web 界面滑块让用户自定义这两个延迟值，避免「按 Enter 时识别还没完成」导致插入空白。
- **打包方案选型**: 放弃 Onefile（每次启动需 10 秒解压）和 Electron（携带几百 MB 浏览器内核），选用 **PyInstaller Onedir** 模式（直接加载已解压库，实现秒开）+ **AppDelegate 架构**（在系统 `DidFinishLaunching` 回调时挂载 RunLoop，获取辅助功能权限）。
- **权限依赖**: 全局键盘监听基于 `pynput` + macOS `CGEventTap`，必须授予「辅助功能」权限，否则无法监听全局按键。
- **Web GUI 控制面板**: 工具启动后自动在 `https://localhost:8899` 开启控制界面，提供启动/停止、实时日志、延迟参数配置，无需命令行操作。
- **剪贴板恢复时序**: `Enter` 插入 → 豆包立即执行自动复制 → 脚本等待 **0.2 秒** → 强制用 `pbcopy` 恢复原内容。0.2 秒是关键时间窗口：足够让插入完成，又短到用户无感知。
- **触发键选择右侧 Command**: 左侧 Command 是系统快捷键高频键（`Cmd+C/V/Z`），右侧 Command 日常几乎闲置，冲突概率极低。
- **快捷键固定为 `Ctrl+D`**: 为避免自定义快捷键与其他应用冲突，豆包语音输入快捷键统一设定为 `Control+D`，脚本硬编码触发此组合键。

### 🎯 关键洞察

**为什么 0.2 秒是核心参数**:
豆包的工作流是：识别完成 → 写入剪贴板 → 执行 `Cmd+V` 粘贴。这个写入动作发生在 `Enter` 之后的极短时间内。如果脚本在 `Enter` 后立刻恢复剪贴板，豆包的写入会覆盖恢复的内容；如果等太久，用户会察觉剪贴板短暂变成了语音文字。0.2 秒是实测出的最小安全窗口。

**为什么不自己写语音识别**:
豆包已免费提供语音识别能力，自建需要对接 ASR API（有延迟、有成本、需维护）。本工具的定位是「胶水层」——只做豆包没做好的那部分（自动化按键 + 剪贴板保护），不重复造轮子。

**来源**: CY：Vibe Coding 时代你需要掌握的7个软件工程知识点 | **时间**: 2026-01-16 | **原文链接**: https://mp.weixin.qq.com/s/w16oHYne...

---

### 📋 核心分析

**战略价值**: Vibe Coding 不降低对工程知识的要求，反而要求你从"写代码"升级为"审代码"——掌握这7项技能是 AI 时代不翻车的最低门槛。

**核心逻辑**:

- **AI 代码质量远低于预期**：AI 生成代码只有 30%–65% 能通过测试，约 30–40% 存在逻辑错误，约 50% 存在安全漏洞，且错误往往隐蔽，表面能跑但埋着雷。
- **Vibe Coding 的认知陷阱**：75% 开发者主观感觉用 AI 更高效，但客观测量显示实际慢了 19%；开发者对 AI 的信任度从 2024 年的 43% 跌至 2025 年的 33%，根因是用错了方式而非 AI 本身不行。
- **角色转变是核心**：你不再是厨师（写代码），而是餐厅老板（审代码）。AI 是手速极快但不懂味道的机器人厨师，你需要的是"品菜能力"和"叫停权力"。
- **测试是安全网而非可选项**：测试的核心价值在于"你定义什么叫做对"，AI 负责生成测试文件并执行，你只需用人话描述验收标准，每次改动后跑测试确保旧功能没被破坏。
- **调试能力是打破 AI 死循环的唯一出路**：AI 会反复用同一种错误方式修同一个 bug，3 次无改善必须人工介入，打开 DevTools 找到根因再告诉 AI，而不是继续让 AI 猜。
- **架构必须人来定，AI 只是施工队**：AI 只看当前任务，不管全局，放任 AI 自由发挥两周后会出现登录逻辑散落 15 个文件、同一功能重复实现 3 次的局面，必须先画架构图、定文件夹结构、给 AI 明确规则。
- **安全漏洞是 AI 的高频盲区**：三类最常见错误——密码/密钥硬编码进代码、不校验用户输入（SQL 注入/XSS）、使用已被破解的老旧加密方式。每次 AI 写完涉及用户数据的功能，必须主动追问安全问题。
- **代码审查不是逐行读，而是让 AI 自己解释**：AI 改完后问"你改了哪些地方、为什么这么改"，解释不清楚的地方往往就是有问题的地方，重点检查逻辑正确性、明显低级错误、改动对其他功能的影响。
- **重构是对抗代码腐烂的定期维护**：AI 每次只管当前任务，同一功能可能写了三遍，每隔一周让 AI 扫描项目、合并重复代码、清理无用文件，防止两周后连 AI 自己都看不懂。
- **Git 是改动前的强制动作**：AI 会不问你就改东西，没有存档就没有回滚能力，每次让 AI 改动前说"先帮我存个档"，改坏了说"恢复到上一个存档"。

---

### 🎯 关键洞察

**为什么"说话就能写代码"是危险误解**：

Andrej Karpathy（OpenAI 联合创始人）提出的 Vibe Coding 原意是"完全投入氛围，忘记代码的存在"，但这被大量开发者误读为"不需要懂代码"。真实数据打脸：AI 生成代码 30–65% 通过率意味着每 2–3 个功能就有 1 个是定时炸弹。Reddit 上有真实案例：开发者让 AI 写登录功能，看都没看直接上线，服务器崩溃后逐行排查才发现密码以明文存入数据库。

**逻辑链**：AI 只优化"当前 prompt 的输出"→ 不考虑安全、架构、可维护性 → 你不审查就等于把这些责任全部放弃 → 出事后只能自己背锅。

**Vibe Coding 需要的是更高层次的知识**：以前你需要知道怎么写循环、定义变量、调 API；现在你需要知道 AI 写的对不对。前者是执行层，后者是判断层，判断层的门槛更高。

### 🛠️ 操作流程：TDD 六步法（最核心技能的完整 SOP）

1. **定义功能** — 告诉 AI："我要一个注册功能，用户输入邮箱和密码就能注册。"

2. **让 AI 列出验收标准** — 说："帮我想想这个功能要检查哪些情况？什么算成功，什么算失败？"
   AI 应输出类似：
   - 邮箱为空 → 提醒用户
   - 邮箱无 `@` 符号 → 提醒用户
   - 密码少于 6 位 → 提醒用户
   - 全部填对 → 显示注册成功

3. **你确认或补充** — 说："对，就这些。" 或 "还要加：密码必须包含数字和字母。"

4. **让 AI 生成测试文件并保存** — 说："把这些检查标准生成测试文件，保存到项目里。"
   AI 会在项目文件夹创建测试文件，永久保存，随时可用。

5. **让 AI 写功能代码并跑测试** — 说："现在写注册功能的代码，写完跑一下测试。"
   AI 写完后自动按标准检查，输出每条结果（✓ / ✗）。

6. **有未通过的让 AI 修** — 说："密码那条没过，改一下。" AI 改完再跑，直到全部通过。

> 关键价值：测试文件永久保存在项目里。下次加"手机号注册"功能时，说"跑一下测试"，AI 会把所有历史验收标准重新跑一遍，确保新功能没有破坏旧功能。

---

### 🛠️ 操作流程：调试死循环破解法

1. **让 AI zoom out** — 说："退后一步，看看全局，不要只盯着这一个地方。"
2. **3次规则** — 同一个 bug，AI 修了 3 次没有改善，立即停止，不要继续让它猜。
3. **换模型重启** — 清空上下文，换一个模型，告诉新 AI 之前的失败尝试和结果。
4. **自己动手** — 打开 DevTools / 调试器，自己找到根因（如路由写错、变量名拼错），再把根因告诉 AI 让它修。

---

### 📝 避坑指南

- ⚠️ **坑1：以为 AI "搞定了"** — AI 写完不等于写对，安全漏洞和逻辑炸弹都是这么上线的。每次必须检查。
- ⚠️ **坑2：空白项目直接 Vibe** — 没有架构规划直接让 AI 写，两周后登录逻辑散落 15 个文件，3 个功能重复实现。先画架构图再动手。
- ⚠️ **坑3：不看代码就用** — 30% 逻辑错误率意味着每 3 个功能就有 1 个有问题。让 AI 解释改了什么，是最低成本的审查方式。
- ⚠️ **坑4：不管安全问题** — 约 50% 的 AI 代码有安全漏洞，密码明文存储、不校验输入是最常见的两类，被黑客攻击前你根本看不出来。
- ⚠️ **坑5：代码越来越乱不管** — AI 每次只管当前任务，不整理的话两周后连 AI 自己都看不懂，每周定期让 AI 整理一次。
- ⚠️ **坑6：不用 Git** — AI 会不问你就改东西，没有存档就没有回滚能力，项目做到 70% 全乱了才发现没存档是最惨的结局。

## 📓 文章 6

> 文档 ID: `G8Cpwgzm8irfWDkPOBwcGAHJnUg`

**来源**: 飞书 ↔ Codex CLI Bridge 上手指南 | **时间**: 2026-01-15 | **原文链接**: `https://mp.weixin.qq.com/s/bS6crMmS...`

---

### 📋 核心分析

**战略价值**: 用飞书长连接双向桥接本地 Codex CLI，彻底消除"离开电脑就掉线"的上下文断裂问题，实现真正的移动端 Vibe Coding 控制台。

**核心逻辑**:

- **痛点根因**：Codex CLI 是本地进程，回复只在终端可见，离开工位即断联，无法异步跟进关键输出
- **桥接方向①（Codex → 飞书）**：Codex 有新回复时，通过群自定义机器人 Webhook 自动推送到飞书群
- **桥接方向②（飞书 → Codex）**：飞书消息通过长连接事件 `im.message.receive_v1` 接收，写入本地 Codex 会话，无需公网回调
- **tui 模式**：桥接进程维持一个真实终端会话，飞书消息像键盘输入一样注入 Codex，上下文连续，体验最接近"人在电脑前"
- **exec 模式**：每条飞书消息触发一次 `codex exec`，结果出来后 `resume --last` 继续，输出更干净，适合任务式一问一答
- **session 输出源**：从 `~/.codex/sessions/**/rollout-*.jsonl` 抽取最后一条 assistant 消息，去噪后推送，内容整洁
- **pty 输出源**：直接读终端原始输出，经 `stripAnsi` 去噪，实时性更强但可能有杂音
- **防刷屏机制**：`idlePushMs`（默认 10000ms）控制合并窗口，等待无新输出 10 秒后才合并推送一次，避免逐行轰炸群
- **安全白名单**：`allowedOpenIds` 限定哪些 `open_id` 可以写入本地终端，不配则任何能私聊机器人的人都能远程操控你的终端
- **推荐组合**：`tui + session`，兼顾上下文连续性与输出干净度

---

### 🎯 关键洞察

**为什么用长连接而非 Webhook 回调**：飞书事件订阅支持两种接收方式——服务端回调（需公网 IP）和长连接（本地进程主动保持连接）。桥接工具选择 `im.message.receive_v1` 长连接，原因是本地开发机通常没有公网地址，长连接方案零运维、零端口暴露，直接在内网跑即可。

**tui vs exec 的本质差异**：tui 是"持久会话注入"，exec 是"无状态调用+恢复"。前者上下文天然连续，后者每次都是新调用再 resume，适合明确边界的任务型交互。

**`sendToUserOnIdle` 的通勤价值**：推群的同时私聊发起者，意味着你在地铁上发出的指令，Codex 跑完后会直接私信你结果，不用盯着群消息流。

---

### 📦 配置/工具详表

| 字段 | 说明 | 推荐值 | 注意事项 |
|------|------|--------|---------|
| `appId` / `appSecret` | 飞书自建应用身份凭证 | 从飞书开放平台获取 | 必填，缺一不可 |
| `webhookUrl` | 群自定义机器人推送地址 | 群机器人配置页复制 | 用于 Codex → 飞书方向 |
| `allowedOpenIds` | 终端控制白名单 | 填入自己的 open_id | 强烈建议配，空数组=无限制 |
| `mode` | 工作模式 | `tui`（默认推荐） | `exec` 适合任务式问答 |
| `outputSource` | 输出抓取方式 | `session`（推荐） | `pty` 更实时但有杂音 |
| `idlePushMs` | 合并推送等待窗口 | `10000`（10秒） | 调小=更实时，调大=更整洁 |
| `sendToUserOnIdle` | 是否同时私聊发起者 | `true`（通勤场景） | 推群+私聊双通道 |
| `maxMessageChars` | 单条消息字符上限 | 按飞书消息限制设置 | 防止超长消息炸群 |

### 🛠️ 操作流程

**1. 准备阶段**

- 安装 Node.js 18+
- 本地确认 `codex` 命令可用（`codex --version` 验证）
- 飞书开放平台创建自建应用，订阅事件 `im.message.receive_v1`，开启长连接接收模式
- 在目标飞书群添加自定义机器人，复制 Webhook URL

**2. 安装 & 启动**

```bash
# 克隆项目
git clone https://github.com/Larkspur-Wang/feishu_X_codex

# 安装依赖
npm install

# 复制配置文件
copy config.example.json config.json
# Linux/Mac 用：cp config.example.json config.json

# 编辑 config.json，填入 appId/appSecret/webhookUrl/allowedOpenIds 等字段

# 启动
npm start
```

**指定工作目录（三种方式）**

```bash
# Windows CMD
set WORKING_DIR=E:\Your\Project
npm start

# PowerShell
$env:WORKING_DIR="E:\Your\Project"
npm start

# 启动参数直传
npm start -- --dir=E:\Your\Project
```

**3. 获取 open_id 并配置白名单**

```json
// 第一步：临时设为空数组
{ "allowedOpenIds": [] }
```

启动后向机器人发一条消息，控制台会打印你的 `open_id`，复制后填回配置：

```json
{ "allowedOpenIds": ["ou_xxxxxxxxxxxxxxxx"] }
```

重启生效。

### 📝 避坑指南

- ⚠️ **乱码/碎字符**：输出源改为 `session`，从 `~/.codex/sessions/**/rollout-*.jsonl` 读取，比 pty 干净得多

## 📒 文章 7

> 文档 ID: `I6jhwDnxvixaJQk7C9qcday5npS`

**来源**: 龙井：DeepSeek + n8n + Gemini + 即梦4.5 = 电商详情页批量神器 | **时间**: 2026-03-13 | **原文链接**: 无

---

### 📋 核心分析

**战略价值**: 用 n8n 工作流 + 即梦4.5 API 实现「输入产品参数/亮点/图片 → 自动批量输出10张电商详情页」的全自动流水线，前端用纯 HTML 单文件承接，零代码部署可用。

**核心逻辑**:
- 整体架构分两层：n8n 后端工作流（负责 AI 调用、数据处理、图片生成）+ 纯 HTML 前端（负责用户输入、结果展示、下载）
- 触发方式分两阶段迭代：开发调试阶段用「表单节点」，前后端联调阶段换成「Webhook 节点」，两者字段名一致，切换时只需修改数据引用路径
- 产品图片必须先经过「视觉理解模型」转成文字描述（英文），再传入后续生图流程，目的是提升10张图之间产品外观的一致性
- 文案策划节点要求模型输出严格的 JSON 结构（含 `slides` key），禁止输出 Markdown 代码块，避免后续 `JSON.parse` 失败
- 文案数据用 Code 节点拆分成10个独立 item，再进入 Loop 循环节点（batch size=1）逐张生图，保证每张图独立处理
- 每张图的生图提示词由 Agent 节点动态拼接（产品视觉描述 + 场景描述 + 主副标题），并明确约束：禁止画面出现额外文字、禁止出现"9:16"字样
- 即梦4.5 通过 HTTP 节点调用，参数来自火山引擎 API Explorer 调试台导出的 curl，导入 n8n 后删除 `watermark` 参数（该参数即梦4.5不支持）
- 10张图生成完毕后，用 Code 节点聚合成一个 `{ images: [...], total: 10 }` 对象，再由 `Respond to Webhook` 节点一次性返回前端
- 前端 HTML 用 `FormData` 发送数据（字段名固定：`产品的参数`、`产品的亮点`、`data`），接收到图片 URL 数组后渲染展示，下载采用双重保障（fetch→Blob 优先，CORS 失败则降级 `window.open`），循环间隔 500ms 防浏览器拦截
- 模型选型：视觉理解用豆包1.6视觉模型，文案策划和提示词生成用 DeepSeek-V3-2，生图用即梦4.5，均通过火山引擎 API 统一接入

---

### 🎯 关键洞察

**为什么要先用表单节点再换 Webhook？**
表单节点可以在 n8n 内部直接提交测试数据，不依赖外部前端，适合单独调试每个节点。等工作流稳定后再换 Webhook，前端才能真正打通。切换后必须同步修改工作流中所有引用 `$('On form submission')` 的地方，改成 Webhook 节点对应的引用路径。

**为什么要做「产品原型描述」这一步？**
直接把图片传给生图模型，10张图里产品外观会漂移（颜色、材质、形状不一致）。先用视觉模型把产品外观转成固定的英文文字描述，再把这段描述注入每张图的提示词，相当于给10张图锁定了同一个产品"锚点"。

**即梦4.5 接入的核心坑**：
- 必须先在 API Explorer 调试成功，再复制 curl 导入 n8n，不能直接手写参数
- `watermark` 参数必须删除，否则报错
- 新用户有200张免费额度，可以放心调试

### 🛠️ 操作流程

**阶段一：搭建 n8n 后端工作流**

1. **触发节点（表单）**
   - 添加 `On Form Submission` 节点
   - 创建三个字段：`产品的参数`（文本）、`产品的亮点`（文本）、产品图片（文件上传）

2. **产品原型描述节点**
   - 添加 Basic LLM Chain 节点
   - 开启图片传递选项
   - 添加 OpenAI Chat Model 子节点，连接火山引擎豆包1.6视觉模型
   - 鉴权配置：API Key 填火山引擎 Key，base url 填 `https://ark.cn-beijing.volces.com/api/v3`
   - 用户提示词：
     ```
     请仔细看这张图。请用一段英文详细描述这个产品的视觉外观。
     重点描述：颜色、材质（如磨砂/光面）、形状结构、以及产品上的关键细节。
     注意：只描述产品本身，不要描述背景。这段文字将用于生成后续的绘图提示词。
     ```

3. **电商文案策划节点**
   - 添加 Basic LLM Chain 节点，模型选 DeepSeek-V3-2
   - 用户提示词：
     ```
     这是产品的参数：
     {{ $('On form submission').item.json['产品的参数'] }}

     这是产品的亮点：
     {{ $('On form submission').item.json['产品的亮点'] }}
     ```
   - 系统提示词：
     ```
     你是资深的电商详情页策划专家。
     任务：根据用户提供的产品参数和亮点，策划一套 10 屏的手机端详情页方案。

     请严格按照以下 JSON 结构输出一个包含 10 个对象的数组，包含 key: "slides"。
     不要输出 Markdown 代码块 (```)，不要输出任何解释性文字。

     【输出示例格式】：
     {
       "slides": [
         {
           "id": 1,
           "title": "静享自由",
           "copy": "48dB深度降噪，喧嚣止步",
           "scene_desc": "Cinematic shot, soft volumetric lighting, minimalist concrete background, floating feather indicating lightness"
         },
         {
           "id": 2,
           "title": "蓝牙5.3",
           "copy": "0.038s无感延迟，音画同步",
           "scene_desc": "Cyberpunk neon lighting, blurred motion background, high tech data stream visualization"
         }
       ]
     }
     ```

4. **数据拆分 Code 节点**
   ```javascript
   // 只取第一个 item
   const item = $input.first();

   // 把 output 里的字符串解析成对象
   const parsed = JSON.parse(item.json.output);

   // 返回 slides 数组作为新的 items
   return parsed.slides.map(slide => ({
     json: slide,
   }));
   ```

5. **循环节点**
   - 添加 Loop Over Items 节点
   - batch size 设置为 **1**

6. **Agent 节点（生成绘图提示词）**
   - 模型选 DeepSeek-V3-2
   - 用户提示词：
     ```
     产品视觉呈现：
     {{ $('产品原型描述').item.json.output }}

     产品描述：
     {{$json.scene_desc}}

     需渲染文字（必须完全精准呈现）：
     主标题："{{$json.title}}"
     副标题："{{$json.copy}}"

     设计要求：
     制作一张竖版电商海报（画面比例 9:16）。
     产品必须作为视觉核心，与上述场景自然融合。
     文字排版：将主标题与副标题美观地置于画面留白区域（顶部或底部）。文字需清晰易读、风格精致，与背景形成高对比度。
     整体风格：高端商业广告质感，C4D 渲染效果，版式简洁大气，达到获奖级海报设计水准。
     重点注意：必须严格按照原文，将 "{{ $json.title }}" 和 "{{ $json.copy }}" 精准呈现在图像上，不得出现任何拼写错误，仅生成指定的主副标题，画面其他区域（尤其是四角）禁止出现任何额外文字 / 伪文字
     ```
   - 系统提示词：
     ```
     ## 角色
     你是一名精通即梦4.0 模型的 AI 绘图提示词工程师。
     你的任务是将**产品视觉呈现**与**产品描述**整合为一段逻辑连贯、高质量的绘图提示词。

     ## 格式
     输出格式要求：仅提交提示词文本，无需附加任何解释说明。

     ## 严格限制：
     仅生成指定的主副标题，画面所有区域（包括但不限于四角、边缘、背景）严禁出现任何数字、符号类文字，尤其是 "9:16" 相关字样，确保画面纯净无额外文字干扰。
     ```

7. **即梦4.5 HTTP 节点**
   - 进入火山引擎 API Explorer：`https://console.volcengine.com/ark/region:ark+cn-beijing/apiKey`
   - 在 API Explorer 中将模型切换为即梦4.5，其他参数保持默认，先调试成功
   - 调试成功后复制 curl
   - 在 n8n 中添加 HTTP Request 节点 → 导入 curl
   - 将 prompt 参数替换为 Agent 节点输出的提示词
   - **删除 `watermark` 参数**（即梦4.5不支持）

8. **图片聚合 Code 节点**（放在 Loop 节点之后）
   ```javascript
   // 获取流入当前节点的所有 10 个数据包
   const items = $input.all();

   // 遍历所有数据包，提取 URL
   const images = items.map(item => {
     const d = item.json;
     
     // 1. 优先匹配即梦 API 格式
     if (d.data && Array.isArray(d.data) && d.data[0] && d.data[0].url) {
         return d.data[0].url;
     }
     
     // 2. 兼容其他可能的格式
     if (d.data && d.data.url) return d.data.url;
     if (d.output) return d.output;
     if (d.url) return d.url;
     
     return null;
   }).filter(url => url); // 去除空值

   // 只返回 1 个对象，包含所有图片数组
   return {
     json: {
       images: images,
       total: images.length,
       debug_input_size: items.length
     }
   };
   ```

9. **Respond to Webhook 节点**
   - 放在工作流最末尾
   - Response Body 填：`{{ $json }}`

**阶段二：搭建前端 HTML**

1. **用 Gemini 生成前端**，使用以下完整 Prompt：

```
# Role
你是一名精通 n8n 工作流对接的高级前端开发工程师。你的任务是编写一个单文件 HTML（包含 CSS 和 JS），作为 n8n AI 生图工作流的本地前端控制台。

# Goal
生成一个极具科技感（赛博朋克/暗黑风格）的 Web 界面，左侧为输入区，右侧为结果展示区。用户填写信息并上传图片后，通过 Webhook 发送给 n8n，并展示返回的生成图片，支持批量下载。

# Visual Style Requirements
1. **布局**：左右分栏布局。左侧 30% 为控制面板（固定），右侧 70% 为结果展示区（可滚动）。
2. **风格**：Dark Mode（深色模式），使用霓虹色（如青色 #00f3ff）作为强调色。
3. **交互**：按钮要有 Hover 效果，生图过程中要有明显的 Loading 动画（遮罩层）。
4. **响应式**：上传图片区域支持点击上传和文件拖拽预览。
