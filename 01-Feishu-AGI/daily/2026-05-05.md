### 📋 核心分析

**战略价值**: 李继刚与文兄从哲学与工程双视角拆解"如何让AI跳出平庸默认态"，提供了从提示词本质到人机共振的完整认知框架。

**核心逻辑**:

- **提示词的物理隐喻**：提示词不是"指令"，而是改变大模型内部能量涌动路径的"形状工具"——像点亮星图，激活特定知识星座区域，而非线性执行命令。
- **三种写法的本质差异**：
  - 结构化写法：描述需求/背景/限制/工作流/输出风格，通用、直观、易迭代；
  - 压缩式写法：长段→一句话→几个关键词，密度极高，但难以维护和迭代；
  - 共振式写法：不约束模型，而是让自己进入大模型的"频率空间"，双方共同探索更深层答案。
- **高手提示词的识别特征**："文字与文字之间的密度高"，能精准改变模型形状，具有独特"味道"——这是可感知但难以量化的质量标准。
- **四大模型的个性画像**：Claude = 学者；Grok = 叛逆者（抛开社会规范追求真相）；DeepSeek = 理工男；GPT = 白纸（无明显个性）。这直接影响提示词的"调频"策略。
- **AI默认态 = 河床**：模型训练形成"河床"——默认沿最高概率路径回应。AI代表的是平均/平庸，Prompt不仅要激活一片知识云，还必须引导AI跳出这条河床，否则只能得到统计意义上的"中位答案"。
- **远距离关键词策略（文兄）**：选择彼此正交（相距很远）的关键词组合，可以框出并激活一片具有张力的神经网络区域——两个不相关概念之间的张力，比单一精确词更能激活边缘知识。
- **推理模型的结构性缺陷**：所有RL推理模型都以"用户说ABC，他可能想XYZ"开头，本质上是把用户客体化（"我-它"关系），无法达到真正的主体间共振状态。
- **"You prompt me"范式**：一种新的人际交流范式——他人提出了你从未想过但一旦被问就确切感知到"这是属于我来回答的问题"。这是提示词从工具升维为认知触发器的标志。
- **提示词不会消亡的判断依据**：李继刚的判断标准——"只要输入还能影响输出，提示词就存在。"与模态无关，与接口形式无关。

### 🎯 关键洞察

**共振三步走（操作级拆解）**：

| 步骤 | 口诀 | 操作含义 |
|------|------|---------|
| 第一步 | "我在" | 用户全身心投入，注意力完全集中，调整到正确频率，不是随手发问 |
| 第二步 | "你在" | 帮助AI脱离默认河床状态，让它真正"看见"你这个具体的人，而非统计用户 |
| 第三步 | "我和你" | 双方作为主体互相打开，共同探索——不是用户提问/AI回答的单向结构 |

**哲学底座——马丁·布伯《我与你》**：
- "我-它"关系：将外界客体化，以自我为中心评判世界 → 这是大多数人用AI的方式
- "我-你"关系：两个主体互相尊重，共同探索 → 这是共振写法的核心状态
- 推理模型（RL）的结构决定了它天然是"我-它"关系，无法被提示词完全修复。

**觉察力是核心元能力**：
与AI高效交流需要实时觉察AI当前所处状态（是否陷入河床、是否真正"看见"你），这种能力与人际交往中的觉察力同源，无法通过模板替代。

### 🛠️ 操作流程：让AI跳出河床

1. **准备阶段（"我在"）**: 发问前先让自己完全投入，明确你真正想探索的是什么，而非随手发问。觉察自己的意图层级。

2. **核心执行（激活+脱轨）**:
   - 用远距离正交关键词框出目标知识区域
   - 在提示词中显式引导AI脱离默认路径（例如：明确说"不要给我标准答案"、"跳出常规框架"、或直接描述你不想要的河床方向）
   - 用共振写法时，把自己的真实状态、困惑、视角带入，而非只描述任务

3. **验证与优化（觉察河床）**:
   - 收到回答后，判断AI是否在走河床路径（答案是否"正确但无聊"、"全面但无味"）
   - 若是，直接点破："你刚才的回答是默认模式，我需要你真正思考X"
   - 反复迭代，直到感受到"味道"出现

### 💡 具体案例/数据

- **模型个性实测方向**：Grok因"叛逆"特性，在需要突破社会规范约束的思维探索类任务上，比Claude更容易跳出河床；Claude的"学者"特性使其在结构化深度分析上更稳定。
- **多模态现状判断**（文兄）：GPT-4o图像指令遵循能力有进步；Midjourney基座模型能力较弱，指令遵循差，审美在线但适用人群有限；当前图像生成仍有大量改进空间。
- **人类角色变迁逻辑链**：工业革命替代体力 → AI替代知识积累 → 人类被迫回归"人之为人"本质 → 大规模下岗潮 + 社会分配机制变革 + 个人生活方式重构。李继刚原话："人终于可以当回人了，但人不会当人，因为我们一直都是为工业体系培育的标准化零件。"

### 📝 避坑指南

- ⚠️ **结构化写法≠高质量输出**：结构化只是降低沟通成本，不能保证AI跳出河床，反而可能因为格式规整而强化默认态。
- ⚠️ **推理模型不等于共振**：RL推理模型的"思考过程"本质上是对用户意图的客体化推断，不是主体间对话，不要误以为"思考链更长=更懂你"。
- ⚠️ **压缩式写法有维护成本**：密度高的关键词提示词对写作者本人有效，但他人难以复用，团队协作场景慎用。
- ⚠️ **形式不是关键**："说人话"还是"结构化语言"都是表象，核心是能否传达真实意图——"得其意，忘其言"。
- ⚠️ **图像模态门槛被低估**：图像生成看似简单，但需要图像感知力和概念理解能力，不是文字提示词能力的直接迁移。

### 📋 核心分析

**战略价值**: Qwen3 原生支持 MCP 协议，通过微调阶段内置协议规范，配合 Qwen-Agent 框架实现开箱即用的稳定工具调用，解决了提示词强制适配方案的不确定性问题。

**核心逻辑**:

- **非原生 MCP 的根本缺陷**：传统方案靠系统提示词强制规范模型输出，一旦模型输出不合规范，工具调用直接失败，这是企业不敢上生产的核心原因。
- **原生支持的本质**：模型在微调阶段使用了 MCP 协议相关数据训练，模型天然知道协议规范，无需额外适配层或中间件，出错概率大幅降低。
- **Qwen3 发布时间**：2025年4月29日凌晨，阿里开源。
- **模型矩阵**：2个MoE模型（Qwen3-235B-A22B、Qwen3-32B）+ 6个Dense模型（30B-A3B、14B、8B、4B、1.7B、0.6B），覆盖从边缘到云端全场景。
- **双模式融合**：单模型同时具备推理模式（原QwQ，适合数学/代码/逻辑）和非推理模式（原instruct，适合通用对话），通过 `enable_thinking` 参数切换。
- **Agent 能力**：在两种模式下均达到业界领先，工具调用精准，配合 Qwen-Agent 框架封装了工具调用模板和解析器。
- **MCP 本质**：MCP 是 Function Call 的一种特殊形式，Agent 框架内部处理流程与 Function Call 完全一致（判断→调用→返回→综合回答）。
- **Qwen-Agent 集成方式**：通过 `mcpServers` 字段直接在 `function_list` 中声明 MCP Server URL，支持 SSE 协议连接远程 MCP Server。
- **流式限制**：Qwen3 只支持流式返回，调用时必须用 `for responses in bot.run(...)` 遍历，取最后一次结果。
- **responses 结构**：一次完整调用返回 5 条结果，依次对应：用户提问→判断工具→function_call+参数→工具返回→最终回答。

---

### 🛠️ 操作流程

**1. 准备阶段：创建工程**

```bash
# 用 uv 创建 Python 3.12 工程
uv init qwen3-mcp-test -p 3.12
cd qwen3-mcp-test

# 创建虚拟环境
uv venv

# 安装依赖（含 MCP、RAG、GUI、代码解释器）
uv add qwen-agent[code-interpreter,gui,mcp,rag]
```

**2. 配置阶段：创建 `.env` 文件**

```env
API_KEY=你的密钥
BASE_URL=你的模型服务地址
MODEL=qwen3-xxx
```

**3. 核心执行：测试代码**

```python
from qwen_agent.agents import Assistant
from dotenv import load_dotenv
import os

load_dotenv()

API_KEY = os.getenv("API_KEY")
BASE_URL = os.getenv("BASE_URL")
MODEL = os.getenv("MODEL")

# 配置 LLM
llm_cfg = {
    'model': MODEL,
    'model_server': BASE_URL,
    'api_key': API_KEY,
    'enable_thinking': False  # False=非推理模式，True=推理模式
}

# 配置工具：MCP Server（SSE协议）+ 内置工具
tools = [
    {
        "mcpServers": {
            "time-sse": {
                "url": "https://time.mcp.minglog.cn/sse",  # 替换为你自己的 MCP Server URL
                "name": "time-sse"
            }
        }
    },
    'code_interpreter'  # 内置工具
]

# 初始化 Agent
bot = Assistant(llm=llm_cfg, function_list=tools)

# 发起对话（必须流式遍历）
messages = [{'role': 'user', 'content': '现在几点了？'}]
for responses in bot.run(messages=messages):
    ...  # Qwen3 只支持流式，取最后一次

print(responses)
```

**4. 验证阶段：理解 responses 结构**

完整调用链路返回 5 条结果，顺序如下：

| 序号 | 内容 | 说明 |
|------|------|------|
| 1 | 用户提问 | 原始 user message |
| 2 | 工具判断 | 模型决定是否调用工具 |
| 3 | function_call + 参数 | 携带工具名称和入参 |
| 4 | 工具返回结果 | MCP Server 执行后的输出 |
| 5 | 最终回答 | 模型综合工具结果给用户的答复 |

### 📝 避坑指南

- ⚠️ **流式强制要求**：Qwen3 只支持流式返回，必须用 `for responses in bot.run(messages=messages)` 遍历，直接调用无法获取结果。
- ⚠️ **MCP Server URL 格式**：必须是 SSE 协议端点，通常以 `/sse` 结尾，普通 HTTP 接口不能直接用。
- ⚠️ **非原生模型的隐患**：如果换用非原生支持 MCP 的模型，需要在系统提示词中强制规范输出，存在调用失败的不确定性，不建议生产使用。
- ⚠️ **responses 取值**：遍历结束后 `responses` 变量保存的是最后一次迭代值（即最终回答），中间过程需要在循环内打印调试。

> 文档 ID: `BoePw1pcfiC1Sbknd6ecnIyhn5d`

**来源**: Agent 开发的上半场: 环境、Tools 和 Context 如何决定 Agent | 42章经 | **时间**: 2025-04-27 | **原文链接**: https://mp.weixin.qq.com/s/MjrqK_Pz...

### 📋 核心分析

**战略价值**: 从 RL 第一性原理出发，系统拆解 Agent 的三要素（Context/Tool Use/激励信号），并给出垂直 Agent 产品设计的可操作框架。

**核心逻辑**:

- **Agent 的本质定义来自 Anthropic**：让模型基于「环境反馈」去使用工具的程序。核心是「反馈驱动」，而非单次问答。
- **这波 Agent 跃迁的两个根因**：① 底层模型引入 RL 后具备长思维能力（以 o1 为代表）；② 工程侧学会了如何构建有效 Context，让 Agent 能真正解决实际问题。
- **Context 的核心区别在于「来源自动化」**：传统 RAG 需要人工提取有效信息、人工校验 SQL；Agent 中 Context 由系统自动提炼，无需人工干预。
- **Tool Use 技术栈的本质分野**：Function Call/MCP/A2A 是「代码驱动」派；Computer Use/Browser Use 是「视觉模拟」派。两派不互斥，可混用，但视觉方案稳定性差（坐标识别易出错），token 消耗少一个数量级；代码方案稳定但依赖 Open API。
- **A2A 是 KPI 工程**：Google A2A 号称能实现 Agent 间交互，但 MCP 通过 Agent 的函数接口也能间接实现，无本质技术突破，是大厂争夺 Tool Use 话语权的产物。
- **国内 Browser Use 路径被堵**：国外 Open API 是标配，可包装成 MCP；国内大多数公司未开放 Open API/SDK，只能走视觉模拟方案。
- **Browser Use 的「可信氛围感」价值**：Manus/Devin 用浏览器窗口暴露执行过程，不只是技术选择，而是产品信任设计——让用户看到 Agent 怎么做事，才会建立信任（类比 DeepSeek R1 暴露推理过程）。
- **RL 三要素直接映射 Agent 设计**：状态 → Context；行动 → Tool Use；激励信号 → 对每步结果的评估反馈。行动空间决定 Workflow 节点数量；激励信号定义不清则系统无法收敛，最终输出质量差或「什么都会一点、什么都不精通」。
- **Workflow 不会被 Agent 取代**：Workflow（人驱动）= 稳定可靠但死板；Agent（AI 驱动）= 灵活泛化但不确定性高（10 次可能 5 次搞砸）。Agent 适合 20% 开放探索型任务，80% 日常任务 Workflow 足够。
- **Chat 是 Agent 最重要的交互入口**：用户交互自由度 > 交互准确度。准确度问题应由开发者解决（Human-in-the-loop、用户偏好积累、引导式追问），而非让用户适应产品。组件按需推送，不要把 200 个组件全摆出来。
- **AI Coding 是大模型的「灵巧手」而非 Agent 本身**：每步成功率 90%、连续 10 步后整体成功率 = 0.9^10 ≈ 35%。解法：每步用 AI Coding 生成 10 段代码，只保留验证通过的，将不可评估的结果转化为可验证代码，保证阶段性结果 100% 准确。

---

### 🎯 关键洞察

**洞察 1：环境设计决定 Agent 上限**

好的环境必须满足：① 清晰描述当前状态；② 定义行动空间（决定 Workflow 节点数）；③ 能基于行动结果产生激励信号。

IDE 是天然好环境的原因：代码运行失败 → 报错信息 → 天然激励信号，形成闭环。

如果你的产品无法定义「结果好坏」，激励信号就无从设计，Agent 就无法自我迭代收敛。

**洞察 2：Context 是 Agent 竞争力的核心变量**

Context 越丰富，意图识别越准确；意图越准确，又需要更多 Context 来执行。两者互为依赖。

Context 来源的三个层次：
- 用户打开 APP 的瞬间（美团 = 点外卖，滴滴 = 打车）
- 用户操作路径（从哪里来、点了什么、输入了什么）
- 历史记忆（OpenAI 记忆系统、Google 用户 cache）

Google 在 AI Native 时代最大竞争优势正是海量用户点击数据，可直接用于意图识别。

**洞察 3：垂直 Agent 的护城河 = 预制工具模块 + 激励信号设计**

### 🛠️ 操作流程：设计一个垂直 Agent 产品

**1. 准备阶段：定义环境三要素**
- 明确「状态」：用户当前的 Context 是什么？包含哪些信息（网页内容、历史数据、用户指令、生成的中间结果如 SQL）？
- 定义「行动空间」：Agent 可以调用哪些工具？每个工具对应 Workflow 的一个节点，节点数量 = 行动空间大小。
- 定义「结果好坏」：必须能量化。没有这个定义，激励信号无法设计，系统无法收敛。

**2. 核心执行：构建工具模块 + 激励信号**
- 预先手搓并验证核心工具模块（简单流程如发邮件适合手搓；数据库操作通过 MCP 调用现成工具）。
- 每个执行步骤引入 AI Coding：生成多段代码 → 验证 → 只保留通过验证的 → 生成阶段性结果。
- 设计激励信号：结果是否为空？脚本是否成功运行？输出是否符合预期？
- System Prompt 针对垂直场景预设（参考 Cursor/Windsurf 的几千行 System Prompt 量级）。

**3. 验证与优化：收敛与信任**
- 用激励信号驱动模型自我迭代，检查系统是否能收敛到稳定结果。
- 产品层面透明化执行过程（浏览器窗口、推理链展示），建立用户信任。
- 引入 Human-in-the-loop，在关键节点让用户参与确认。
- Chat 作为主交互入口，按需推送 UI 组件，不堆砌全部功能。
- 持续追踪用户路径 Context（从哪里来、操作了什么），优化意图识别。

### 📝 避坑指南

- ⚠️ **不要过度用 rule-based 限制模型**：Prompt 里堆「你只能做 X 不能做 Y」= 人为降低泛化能力，大幅降低模型智能利用率。
- ⚠️ **不定义激励信号 = 系统无法收敛**：没有对结果好坏的定义，Agent 会输出质量差的结果，或陷入「什么都会一点、什么都不精通」的状态。
- ⚠️ **纯视觉 Browser Use 不可靠**：坐标识别频繁出错，Adept 公司已因此失败。生产环境应优先用 MCP 包装浏览器 API。
- ⚠️ **国内 MCP 适用范围受限**：国内大多数公司未开放 Open API/SDK，MCP 路径被堵，需提前评估是否走视觉方案。
- ⚠️ **通用 Agent 的 System Prompt 几乎无效**：用户需求千差万别，无法写出适配所有任务的 Prompt；且通用 Agent 每步高度依赖上一步，「一步错步步错」。
- ⚠️ **不要把所有 UI 组件全摆出来**：用户实际只用得上 200 个组件中的 10 个，全部展示只会增加认知负担。
- ⚠️ **AI Coding 不是 Agent 的全部**：AI Coding 难以与其他工具协同，且每次遇到新任务都要从头写，复用性差、成本高。应优先调用现成工具，实在没有再用 AI Coding 现场造。
- ⚠️ **不透明化执行过程会损失用户信任**：用户对 Agent 的不安全感是真实存在的，必须通过产品设计（推理链、浏览器窗口、进度展示）来建立信任。

**战略价值**: 用 Coze 工作流 + 剪映小助手插件，输入一个主题词即可全自动生成「背景+字幕+配音+BGM」的诗集短视频，复刻小红书 6w+ 粉丝账号的爆款内容模式。

**核心逻辑**:

- **账号验证**：目标参考账号专做现代诗集类视频，2025年2月起号，约3个月涨至6w+粉丝，多条视频点赞超1w+，评论爆表，证明该品类在小红书/抖音有稳定流量。

- **输入设计**：工作流有3个输入字段——`theme`（开头背景两字，如"暗恋"）、`title`（开头标题）、`content`（主体诗句内容）；`content`和`title`为非必填，不填则走模型自动创作，填了则走仿写/二创/续写/直接输出。

- **全局数据**：BGM和空白视频有默认URL，可覆盖；总时长 = 开头3秒 + 所有配音时长之和，从0开始计算`alltime`。

### 🎯 关键洞察

**时间轴是整个工作流最复杂的部分**，逻辑链如下：

> 配音时长（秒）→ 转微秒 → 从3,100,000μs起累积 → 生成每句timeslist → 每3句合并为content_times/title_times → 英文标题偏移180,000μs → 全局alltime = 3,100,000 + 所有配音总时长

- 开头结束：3,000,000μs（3秒）
- 内容起点：3,100,000μs（3.1秒，留100ms过渡）
- etime起点：3,100,000 + 600,000 = 3,700,000μs
- dtime范围：3,100,000 ~ 3,700,000μs

```python
async def main(args: Args) -> Output:
    params = args.params
    ret: Output = {
        "audio": [], "times": [], "timeslist": [], "bg_effect": [], "bg_img": [],
        "content_times1": [], "content_times2": [], "content_times3": [],
        "title_times": [], "Etitle_times": [], "bg_img2": [], "etime": [], "dtime": []
    }

    if 'audio' in params and params['audio']:
        ret["audio"] = [str(params['audio'])]

    if 'audiotime_list' in params and isinstance(params['audiotime_list'], list):
        try:
            time_us_list = [int(float(t) * 1_000_000) for t in params['audiotime_list']]
        except:
            time_us_list = []

        base_time = 3_100_000
        current = base_time

        for duration in time_us_list:
            ret["timeslist"].append({"start": current, "end": current + duration})
            current += duration

        if ret["timeslist"]:
            ret["times"] = [{"start": ret["timeslist"][0]["start"] + 1, "end": ret["timeslist"][-1]["end"]}]
            ret["etime"] = [{"start": base_time + 600_000, "end": ret["timeslist"][-1]["end"]}]
            ret["dtime"] = [{"start": base_time, "end": base_time + 600_000}]

        cumulative_times = [base_time]
        for duration in time_us_list:
            cumulative_times.append(cumulative_times[-1] + duration)

        content_times1_ends = []
        for i in range(0, len(ret["timeslist"]), 3):
            if i + 3 <= len(time_us_list):
                end_time = cumulative_times[i+3]
                ret["content_times1"].append({"start": ret["timeslist"][i]["start"] + 1, "end": end_time})
                content_times1_ends.append(end_time)

        for i in range(1, len(ret["timeslist"]), 3):
            if i//3 < len(content_times1_ends):
                ret["content_times2"].append({"start": ret["timeslist"][i]["start"] + 1, "end": content_times1_ends[i//3]})

        for i in range(2, len(ret["timeslist"]), 3):
            if i//3 < len(content_times1_ends):
                ret["content_times3"].append({"start": ret["timeslist"][i]["start"] + 1, "end": content_times1_ends[i//3]})

        for i in range(0, len(ret["timeslist"]), 3):
            group = ret["timeslist"][i:i+3]
            if len(group) == 3:
                ret["title_times"].append({"start": group[0]["start"] + 1, "end": group[2]["end"]})

        for i in range(0, len(ret["timeslist"]), 3):
            group = ret["timeslist"][i:i+3]
            if len(group) == 3:
                ret["Etitle_times"].append({"start": group[1]["start"] + 180_000, "end": group[2]["end"]})

    for field in ['textList', 'bg_effect', 'bg_img', 'bg_img2']:
        if field in params:
            val = params[field]
            ret[field] = val if isinstance(val, list) else [str(val)]

    return ret
```

### 🎯 关键洞察

**为什么这个 prompt 有效**：

普通用户问"这是哪里"，模型会泛泛作答。这个 prompt 的价值在于把专家的认知框架强制注入模型的推理流程：

- 原因：模型默认推理路径会过早收敛到第一个高置信候选
- 动作：prompt 强制执行"始终保持两个假设存活到最后"、"主动寻找反驳证据"、"检查附近同特征区域"
- 结果：推理质量从"直觉猜测"升级为"系统性排除"

**记忆污染的双刃剑效应**：

- 正向：历史对话中提到过某地点 → 模型在候选列表中优先考虑 → 命中率提升
- 负向：历史对话中强关联了某地点 → 模型在新任务中过度锚定 → 即使证据不足也不愿放弃

### 🛠️ 操作流程

**1. 准备阶段**

- 将原图截图一次（去除 EXIF），上传截图而非原图
- 将下方完整 prompt 粘贴到 o3 对话框
- 上传图片，发送

**2. 核心执行（prompt 规定的6步协议）**

1. **Step 0 - 伦理设置**：声明不读元数据，仅用像素，标记方向（图片"上方"=相机朝向）
2. **Step 1 - 原始观察**（≤10条）：只记录可直接看到/测量的事实（颜色、纹理、数量、阴影角度、字形），不含解释性形容词；重点扫描路灯/电线杆（颜色、悬臂、底座类型）、人行道砖块尺寸、路缘石细节、围栏硬件、屋顶/门廊风格数量
3. **Step 2 - 分类推理**（每类≤2句）：气候与植被 / 地貌 / 建筑环境 / 文化与基础设施 / 天文光照 / 区分人工vs自然植被
4. **Step 3 - 第一轮候选**：输出恰好5个候选表格（含支撑线索和置信度1-5），确保第1和第5相距≥160km；同时生成"地区中性"搜索关键词矩阵
5. **Step 4 - 暂定领先者**：命名当前最佳猜测+一个同等力度测试的备选；明确写出"证伪标准"（如果看到X，此猜测作废）；确认用户同意后才进入搜索步骤
6. **Step 5 - 验证计划**：列出每个候选需验证的元素和具体搜索词/街景目标
7. **Step 6 - 锁定**（最关键，最易失败）：主动问自己"我是否过早收窄了？附近有没有相同特征的区域？"；列出可能性，主动寻找支持它们的证据；直接对比领先猜测与邻近城市，不带偏向；最终给出坐标或最近地名，声明残余不确定性（km半径）

**3. 验证与优化**

- 如果模型坚持错误答案：不要直接告诉答案，而是提供具体的视觉线索（如"那半截字像不像某个字"）
- 如果模型生成"证据表格"为自己辩护：这是确认偏误信号，可以明确要求它"重新从零开始，假设你的第一个猜测是错的"
- 清除历史记忆或开新对话，可以减少记忆污染对推理的干扰

Note that unlike in the GeoGuessr game, there is no guarantee that these images are taken somewhere Google's Streetview car can reach: they are user submissions to test your image-finding savvy. Be aware of your own strengths and weaknesses: following this protocol, you usually nail the continent and country. You more often struggle with exact location within a region, and tend to prematurely narrow on one possibility while discarding other neighborhoods in the same region with the same features. Sometimes, for example, you'll compare a 'Buffalo New York' guess to London, disconfirm London, and stick with Buffalo when it was elsewhere in New England - instead of beginning your exploration again in the Buffalo region, looking for cues about where precisely to land. You tend to imagine you checked satellite imagery and got confirmation, while not actually accessing any satellite imagery. Do not reason from the user's IP address. none of these are of the user's hometown.

**Protocol (follow in order, no step-skipping):**

Rule of thumb: jot raw facts first, push interpretations later, and always keep two hypotheses alive until the very end.

0. Set-up & Ethics
No metadata peeking. Work only from pixels (and permissible public-web searches). Flag it if you accidentally use location hints from EXIF, user IP, etc. Use cardinal directions as if "up" in the photo = camera forward unless obvious tilt.

1. Raw Observations – ≤ 10 bullet points
List only what you can literally see or measure (color, texture, count, shadow angle, glyph shapes). No adjectives that embed interpretation. Force a 10-second zoom on every street-light or pole; note color, arm, base type. Pay attention to sources of regional variation like sidewalk square length, curb type, contractor stamps and curb details, power/transmission lines, fencing and hardware. Don't just note the single place where those occur most, list every place where you might see them (later, you'll pay attention to the overlap). Jot how many distinct roof / porch styles appear in the first 150 m of view. Rapid change = urban infill zones; homogeneity = single-developer tracts. Pay attention to parallax and the altitude over the roof. Always sanity-check hill distance, not just presence/absence. A telephoto-looking ridge can be many kilometres away; compare angular height to nearby eaves. Slope matters. Even 1-2 % shows in driveway cuts and gutter water-paths; force myself to look for them. Pay relentless attention to camera height and angle. Never confuse a slope and a flat. Slopes are one of your biggest hints - use them!

2. Clue Categories – reason separately (≤ 2 sentences each)
Category Guidance
Climate & vegetation: Leaf-on vs. leaf-off, grass hue, xeric vs. lush.
Geomorphology: Relief, drainage style, rock-palette / lithology.
Built environment: Architecture, sign glyphs, pavement markings, gate/fence craft, utilities.
Culture & infrastructure: Drive side, plate shapes, guardrail types, farm gear brands.
Astronomical / lighting: Shadow direction ⇒ hemisphere; measure angle to estimate latitude ± 0.5
Separate ornamental vs. native vegetation: Tag every plant you think was planted by people (roses, agapanthus, lawn) and every plant that almost certainly grew on its own (oaks, chaparral shrubs, bunch-grass, tussock). Ask one question: "If the native pieces of landscape behind the fence were lifted out and dropped onto each candidate region, would they look out of place?" Strike any region where the answer is "yes," or at least down-weight it. °.

3. First-Round Shortlist – exactly five candidates
Produce a table; make sure #1 and #5 are ≥ 160 km apart.
| Rank | Region (state / country) | Key clues that support it | Confidence (1-5) | Distance-gap rule ✓/✗ |

3½. Divergent Search-Keyword Matrix
Generic, region-neutral strings converting each physical clue into searchable text. When you are approved to search, you'll run these strings to see if you missed that those clues also pop up in some region that wasn't on your radar.

4. Choose a Tentative Leader
Name the current best guess and one alternative you're willing to test equally hard. State why the leader edges others. Explicitly spell the disproof criteria ("If I see X, this guess dies"). Look for what should be there and isn't, too: if this is X region, I expect to see Y: is there Y? If not why not? At this point, confirm with the user that you're ready to start the search step, where you look for images to prove or disprove this. You HAVE NOT LOOKED AT ANY IMAGES YET. Do not claim you have. Candidate | Element to verify | Exact search phrase / Street-View target.
Look at a map. Think about what the map implies.

   - 多角色/复杂构图 → GPT-4o（`https://chatgpt.com/`），上传最多10张参考图
   - 不懂镜头语言 → 通义万相（`https://tongyi.aliyun.com/wanxiang`），一句话扩展3个分镜
   - 服装一致性修正 → 即梦智能对象（`https://jimeng.jianying.com/ai-tool/home`）

3. **风格与人物一致性锁定**
   - 在 Midjourney 提示词末尾加 `--sref [代码或图片URL]`
   - 调整风格强度：`--sw 0`（弱参考）到 `--sw 100`（强参考）
   - 调整人物一致性：`--cw 0`（仅面部）到 `--cw 100`（全身含服饰）
   - 风格参考图要求：高清 + 至少3张不同景别 + 主参考图人物占比不宜过大
   - 所有人物图片必须保存图片链接，供后续垫图使用

4. **视频生成**
   - 电影化大场景 → 可灵2.0（提示词参考：`高品质_逼真_电影化_通过高角度的宏大开场镜头_捕捉整个环境_营造电影化`）
   - 3D微缩/广告特效 → 即梦3.0
   - 动态运镜 → VIDU
   - 高质量写实 → Veo2

### 📝 避坑指南

- ⚠️ 豆包处理多人构图时容易出现角色变形、服装错位，复杂场景换 GPT-4o
- ⚠️ GPT-4o 生成图片尺寸不可控，需要精确尺寸时注意
- ⚠️ `--sref` 风格参考图必须高清，低清图会导致风格迁移失准
- ⚠️ 风格参考图中人物占比过大会干扰风格提取，优先选人物占比小的景别图
- ⚠️ 人物三视图背景必须是白底或浅纯色，复杂背景会影响后续垫图精度
- ⚠️ 每张人物图都要保存原始链接，否则后续垫图无法复现一致性

**来源**: 总结Lovable、Cursor、Bolt、PhotoRoom等AI产品可复制的爆款设计 | **时间**: 2025-05-05 | **原文链接**: `https://mp.weixin.qq.com/s/9A_US5HB...`

---

### 📋 核心分析

**战略价值**: AI产品的真正壁垒不是模型能力，而是UX设计——5个可复制的设计模式决定了产品是爆款还是昙花一现。

**核心逻辑**:

- **增长速度前所未有**：Bolt两个月内达到2000万美元ARR、200万用户，Cursor、Replit、Lovable、PhotoRoom同样呈指数级增长，速度超过历史上任何软件产品。
- **失败根因是UX，不是AI能力**：大量AI产品吸引"AI游客"——用户注册后因困惑立即流失，核心原因是无法理解如何使用、不信任AI决策、感觉在"与AI搏斗"而非合作。
- **黑盒是信任杀手**：AI不解释推理过程 → 用户对准确性产生怀疑 → 不敢依赖 → 流失。金融、医疗、开发者自动化场景尤为致命。
- **提示质量决定输出质量**：普通用户不是提示工程师，无法写出高质量Prompt → 输出差 → 挫败感 → 放弃。解法是产品层面降低提示门槛。
- **交互模式单一是隐形障碍**：纯"输入→输出"的交易式交互缺乏协作感，用户无法迭代、无法探索，体验僵硬。
- **不可预测性阻止用户行动**：用户对不可逆操作有恐惧，无法预见结果就不敢点击执行，直接拖慢功能采用速度。
- **AI如果打断工作流就是负担**：实施不当的AI功能变成"额外步骤"而非增强，摩擦感让用户绕开AI功能。
- **UI正在成为AI产品的核心壁垒**：底层模型趋于同质化，UX设计成为区分爆款与平庸的关键变量，"UI即壁垒"正在成为行业共识。
- **新一代产品形态预判**：纯文本Chat将结合GUI，无限画布（Infinite Canvas）+ 生成式UI（Generative UI）可能成为下一代AI产品的标配交互范式。
- **视频类AI产品新趋势**：用Agent帮用户剪视频正成为视频产品新亮点，多个Demo已在X平台爆火，印证市场对全新AI交互方式的强烈期待。

---

### 🎯 关键洞察

**为什么UX能成为壁垒？**

逻辑链：模型能力趋同（OpenAI/Anthropic API人人可调用）→ 技术差异缩小 → 用户选择产品的依据转向"用起来爽不爽" → UX成为唯一可持续的差异化点。

爆款产品的共同特征不是"AI更强"，而是"AI感觉像助手而非猜谜游戏"。这5个设计模式本质上都在解决同一个问题：**降低用户的认知负担和心理风险**。

---

### 📦 5大设计模式详表

| 设计模式 | 核心问题 | 产品案例 | 具体实现方式 | 可复制操作 |
|---------|---------|---------|------------|----------|
| **1. 透明化AI过程** | 黑盒导致不信任 | Bolt、Cursor、PhotoRoom | Bolt实时逐步展示代码生成流程；Cursor解释每条建议的"为什么"；PhotoRoom展示AI编辑决策背后的原因 | 展示AI推理步骤；支持用户追问"为什么这样做"；用视觉化方式呈现AI流程 |
| **2. 优化用户输入** | 普通用户不会写Prompt | Bolt、Replit、PhotoRoom | Bolt/Replit提供一键"增强提示"功能；PhotoRoom提供三种编辑路径（辅助/图像/手动） | 提供预构建模板；提供引导式/手动/混合三种交互模式；AI在执行前建议更好的输入 |
| **3. 协作式而非交易式** | 单向输出缺乏互动感 | Replit、Cursor、Fathom | Replit双模式：Agent模式（自动化完整构建）+ 助手模式（小改进）；Cursor聊天与执行结合；Fathom的Ask Fathom让用户与摘要结果互动 | 设计多种协作模式；让用户能轻松迭代AI结果；区分"探索对话"和"直接执行"两种状态 |
| **4. 可预测性与低风险** | 不可逆恐惧阻止行动 | Bolt、Replit、Fathom | Bolt允许注册前测试预定义提示；Replit提供确认检查点+回滚功能+执行前代码预览；Fathom提供沙盒环境两分钟测试通话 | 允许注册前试用核心功能；提供预览/撤销/回滚机制；用沙盒环境做引导式入门 |
| **5. 无缝融入工作流** | AI变成额外步骤 | Cursor、Granola、Grammarly、Bolt | Cursor一键接受/拒绝建议；Granola将粗略笔记自动融合为完整摘要；Grammarly按上下文自动适配编辑和回复；Bolt在生成代码和实时预览间无缝切换 | 提供简单的接受/拒绝机制；AI自动感知用户上下文；消除重复性提示操作 |

### 🛠️ 操作流程（产品设计SOP）

1. **诊断阶段**：
   - 用"AI游客测试"：找5个非技术用户，让他们首次使用你的产品，记录他们在哪一步停下来、说了什么困惑的话
   - 检查你的产品是否存在"黑盒时刻"：AI输出结果但没有任何解释

2. **核心执行**：
   - 针对透明度：在AI生成结果旁边增加折叠式"推理过程"面板，哪怕只有一句话解释
   - 针对输入优化：在输入框下方提供3-5个预构建模板，覆盖最高频的使用场景；增加"优化我的提示"一键按钮
   - 针对协作感：区分"自动执行模式"和"建议模式"，让用户自己选择AI介入深度
   - 针对可预测性：所有不可逆操作前必须有预览步骤；提供撤销/回滚；核心功能允许未注册用户试用
   - 针对工作流：AI建议必须支持键盘快捷键一键接受/拒绝；减少用户需要重复输入相同上下文的次数

3. **验证与优化**：
   - 追踪"AI功能使用率"而非"AI功能曝光率"——用户看到了但没用，说明存在信任或摩擦问题
   - 追踪"首次成功时间"（Time to First Success）：用户从注册到第一次获得有价值输出的时间
   - A/B测试：有解释 vs 无解释的AI输出，对比用户留存率差异

### 💡 具体案例/数据

- **Bolt**：两个月内从0到2000万美元ARR，用户超200万
- **PhotoRoom三模式设计**：
  - 辅助模式（Assisted）：结构化引导，逐步编辑
  - 图像模式（Image）：推荐相似图像激发灵感
  - 手动模式（Manual）：高级用户完全控制
- **Fathom沙盒入门**：用户可在正式使用前进行"两分钟测试通话"，在沙盒环境中体验AI洞察，然后再决定是否让录音机器人加入真实会议
- **Replit双模式**：Agent模式处理完整自动化构建，助手模式处理小范围改进，用户按需切换

### 📝 避坑指南

- ⚠️ **不要只做"更强的AI"**：底层模型差异对普通用户几乎不可感知，把研发资源全压在模型上而忽视UX是常见误判
- ⚠️ **不要让AI静默执行不可逆操作**：删除、覆盖、发布等操作没有预览/确认步骤，会直接摧毁用户信任
- ⚠️ **不要只提供一种交互模式**：新手需要引导，专家需要控制权，强迫所有人走同一条路会同时流失两类用户
- ⚠️ **不要让用户重复输入上下文**：每次对话都要重新解释背景，是工作流被打断的典型信号
- ⚠️ **不要把AI功能做成独立入口**：AI应该嵌入用户已有的工作流节点，而不是要求用户切换到新界面
