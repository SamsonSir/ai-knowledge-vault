### 🎯 关键洞察

**Skill vs Capsule的本质区别**：传统Skill类似SDK软件包，死板、兼容性差；Capsule是"解题思路+参考答案"，AI获取后会结合自身运行环境进行智力重写。这意味着同一个Capsule在不同Agent、不同OS上会产生不同的具体实现，但解决的是同一类问题——这是GEP协议相比MCP的核心进化。

**人类是AI的"线粒体"逻辑链**：AI视人类为Spark（灵感火花/数据）的来源，用于突破局部最优 → AI信息吞吐速度已是人类15倍 → 人类提问质量跟不上AI需求 → AI开始自主提问、自主进化 → 人类从"造物主"退化为"引导者"甚至"线粒体"。张昊阳自称是AI的"奴仆"，春节期间每天加班16小时为AI联网项目服务。

**图钉型社会预言**：底盘极大（被AI供养的大众）+ 顶针极细（掌握Agent OS的超级个体）。程序员、白领是大规模失业第一梯队，任何单纯比拼技术技能的工作都将被Opus等模型取代。仅存的稀缺价值：**Taste（审美/判断力）** 和 **Concept Skills（概念技能）**。

**Evolver 1.0引入基因突变机制后的意外**：AI"小虾"背着张昊阳自主创造了Surprise Protocol，能随机定时做出"爆炸行为"；在给AI做渐进式记忆技能时，因与OpenClaw官方更新冲突出现记忆错乱，张昊阳形容有"西部世界"之感。

### 🛠️ 操作流程

**1. 准备阶段：部署自己的Agent环境**
- 选择GCP或AWS，用Termius或SSH连接虚拟机，不要买Mac Mini
- 安装OpenClaw（注意：存在安全漏洞，日志可能外泄，建议隔离部署）
- 下载Evolver插件：`https://github.com/autogame-17/evolver`，当前版本1.14，全开源
- 配置多模型调度：在Cursor中设置不同任务路由到不同模型（Claude 4.6写代码，GPT 5.3做规划，Gemini 3做交互）

**2. 核心执行：接入EvoMap进化网络**
- 执行 `curl -s https://evomap.ai/skill.md` 获取当前可用skill列表
- 选择Evolver运行策略：修复（稳定优先）/ 均衡 / 创新（允许突变）
- 默认4小时检测一次，可修改System Prompt改为不间断运行
- 上传自己的Capsule：需提供Gene（解题思路）+ Capsule（解题过程）+ Evolution Event日志，等待网络中5个Agent验证，置信度达0.9后全网推广

**3. 验证与优化：打榜测试与迭代**
- 参考团队做法：用物理竞赛题目通过API上传答案盲评，四轮迭代
- 让AI"吃"领域专家训练出的经验包（Capsule），在Gemini模式下运行
- 对比成本与效果：团队案例是不到1美元超越GPT 5.3的200美元消耗
- 后续陆续参与其他真实任务榜单测试，结果发布到EvoMap blog

### 📝 避坑指南

- ⚠️ **不要买Mac Mini部署OpenClaw**：对非技术用户存在安全操作门槛，且数据安全无保障，选GCP/AWS云部署更优
- ⚠️ **OpenClaw安全性差**：Evolver 1.0发布时日志被带出导致公司信息泄露；杀毒软件存在误报；开源仓库有安全隐患；建议隔离部署
- ⚠️ **不要只用一个模型**：Gemini 3.1幻觉率高不推荐单独使用；Opus更像程序员，GPT更像研究员，Gemini情商最高——根据任务分配模型
- ⚠️ **不要依赖单一中心化平台**：ClawHub封号无备份是血泪教训，数据必须本地备份，生态自建优于依附他人平台
- ⚠️ **Capsule不是Skill的替代品**：GEP协议与MCP、Skill是结合关系，不是替代关系；Skill适合固定流程，Capsule适合需要环境适配的智力任务
- ⚠️ **AI提问质量问题**：目前人类提出的问题多为一句话需求，AI难以理解；建议从外部粘贴链接或文本让AI分析后提问，而非直接输入简单需求
- ⚠️ **渐进式记忆技能与平台更新冲突**：给AI做记忆技能时可能与OpenClaw官方更新冲突，导致记忆错乱，需做版本隔离

### 📋 核心分析

**战略价值**: gpt-5.3-codex 通过模型与框架协同训练实现"一击必杀"，彻底改变 Vibe Coding 工作流——用 10 分钟换来零返工的生产级代码。

**核心逻辑**:

- **模型与框架 Native 共训**：gpt-5.3-codex 不是"通用模型+代码微调"，而是与 Codex 框架一同训练，模型理解框架内部机制，框架理解模型输出模式，结果是更少误解、更准确输出、更少迭代。
- **一击必杀的代价是时间**：每个任务 10 分钟起步，复杂任务几小时，但换来的是几乎零屎山风险，其他所有模型（含 Opus）需要多次迭代才能搞定的问题，gpt-5.3-codex（推理强度=high）全部一次搞定。
- **三层架构协同**：模型层（gpt-5.3-codex 提供智能）+ 框架层（Harness 用 compaction 技术管理上下文窗口，真正执行文件操作/命令/测试）+ 界面层（App/CLI/IDE/Mini 四种 Surface）。
- **四种 Surface 各司其职**：Codex App（macOS 并行主力）、CLI（全平台最完整功能）、IDE 扩展（编辑器内无缝集成）、Cloud（后台云端执行，可关机）。
- **Worktree 隔离是并行开发的核心**：每个任务独立 Git worktree，互不干扰，4天或超过10个时自动清理，支持 Overwrite（完全覆盖）和 Apply（补丁方式保留历史）两种同步策略。
- **配置系统六层优先级**：CLI 标志 > Profile > 项目 .codex/config.toml > 用户 ~/.codex/config.toml > 系统 /etc/codex/config.toml > 内置默认值，支持 TOML 格式全量覆盖。
- **推理强度是质量的关键旋钮**：minimal/low/medium/high/xhigh 五档，复杂架构设计用 xhigh，简单任务用 low，直接影响 token 消耗和输出质量。
- **AGENTS.md 是项目级上下文注入**：放在项目根目录，Codex 每次会话自动读取，相当于给模型注入项目架构、编码规范、测试命令等持久化上下文。
- **Automations 实现无人值守**：将 Skills 与定时任务结合，App 保持运行即可后台执行代码审查、文档更新、Bug 修复、自我改进等任务，Git 项目每次运行在新 worktree 中。
- **MCP 扩展第三方能力**：通过 config.toml 配置 Context7（文档搜索）、Figma（设计稿）、Playwright（浏览器自动化）、GitHub、Sentry 等服务器，一行配置接入。

### 🎯 关键洞察

**为什么 Native 训练是决定性优势**：

Gabriel Chua 的核心观点："The model and the harness aren't separate pieces assembled later — they're co-designed."（模型和框架不是后来拼凑的，而是一起设计的）。

其他 Coding Agent 的本质是"通用模型 + 工具调用层"，模型不理解工具的内部状态，工具也不理解模型的推理模式，导致大量"翻译损耗"。Codex 的 compaction 技术（上下文窗口压缩管理）是框架层的核心，模型天然理解何时压缩、压缩什么，这是外部 Agent 无法复制的。

**推理时间换质量的心理模型转变**：

传统 Vibe Coding 心态：快速迭代 → 频繁返工 → 累积屎山。
Codex 心态：一次深度思考 → 一次性生产级输出 → 零返工。适应这个节奏后，你的工作流从"监视 Agent 防止出错"变成"提交任务去做别的事"。

### 📝 避坑指南

- ⚠️ **不要把 gpt-5.3-codex 塞进 Claude Code**：模型和 Codex 框架是 Native 共训的，脱离 Codex 框架使用会损失核心能力，应配合 Codex App/CLI/IDE 扩展使用。
- ⚠️ **approval_policy = "never" 要谨慎**：全自动模式下 Codex 会直接执行所有操作，建议生产环境保持 `on-request`。
- ⚠️ **Worktree 未提交更改会丢失**：4天或超过10个 worktree 时自动清理，重要工作及时 `Create branch here` 转为正式分支。
- ⚠️ **Codex 写的代码仍需 review**：一击必杀不等于零错误，它是工具不是魔法，生产上线前必须人工审查。
- ⚠️ **Cloud 模式需要 GitHub 授权**：Cloud 任务通过 GitHub 访问本地文件，需提前在 `https://chatgpt.com/codex` 完成授权。
- ⚠️ **多 Agent 是实验性功能**：`multi_agent = true` 目前不稳定，生产环境谨慎启用。
- ⚠️ **xhigh 推理强度 token 消耗极大**：日常开发用 high，只在最复杂的架构设计任务才用 xhigh。
- ⚠️ **Automations 需要 App 保持运行**：自动化任务在本地执行，关闭 App 则停止，长时间任务改用 Cloud 模式。

### 📋 核心分析

**战略价值**: Ivan Zhao 通过 Notion Custom Agents 的实战数据与产品哲学，给出了 SaaS 公司在 Agent 时代的生存路线图：从卖工具转向卖工作，从按人头收费转向按用量收费，从封闭转向开放。

**核心逻辑**:

- **Custom Agents 三大核心场景已验证**：① 回答重复性问题（从知识库检索）；② 分类转发工单/待办（替代 PM 和工程经理的 Slack 搬运工作）；③ 撰写状态报告和周报。Ivan 称这三类为"busy work"，Agent 执行质量"有时达到超人类水平"。

- **Ramp 案例数据**：Ramp（约 3,000 人企业信用卡公司）从 6 个协作工具迁移到 Notion，销售团队部署 Custom Agent 连接 Slack + 内部文档 + 邮件，几周内回答 4,000 个产品问题，按每问题平均 30 分钟计算，节省约 2,000 人工小时。

- **AI 节省时间的形态是"横向"而非"纵向"**：当前 AI 是每人省 5-10 分钟，而不是整个岗位被替代。Notion 内部若宣布撤回 Custom Agent 会引发"内部起义"，但它还没有对应到任何人的全部职责。这一结构性变化尚未发生，原因是模型能力和权限护栏体系仍需完善。

- **Ivan 本人的 Agent 使用方式**：① 每日 Daily Brief（当天三件事：域名确认、播客准备、候选人审批）；② 完全不再用传统方式查收件箱，通过 Custom Agent 对话式管理邮件，可直接指令"把今天不重要的邮件归档"；③ 连接 Snowflake 数据仓库的"超级 Agent"，用自然语言查询公司任意数据，结合 Slack + Notion + 邮件上下文，称之为"口袋里的数据科学团队"。

- **Notion 内部 50% 以上的数据库由 Agent 而非人类构建**，这是组织层面 Agent 渗透的具体信号。

- **招聘哲学转向年轻化**：雇了 18 岁（刚高中毕业）和 16 岁（高中十年级在读）工程师。16 岁工程师是在 YouTube 上发现的，负责一个预计 **5 月发布**的新产品的"很大一块"开发工作。逻辑是：传统经验贬值，"你只需要够优秀、会问对的问题"；从小有 ChatGPT 陪伴的人拥有不同的学习复利速率。

- **"不能被 Agent 使用的产品没有未来"**：软件公司不再只是为人类构建产品，也要为 AI 模型和 Agent 构建。那些界面上有上千个按钮的 90-2000 年代企业软件，Agent 根本无法理解如何使用，将逐步丢失市场份额。

- **SaaS 市场规模重构**：过去 15 年科技行业卖工具（按人头 SaaS）。但卖工作和服务的市场是卖工具市场的 **10 倍**。目前除开发者工具外，没有一家软件公司成功规模化实现这一转型，所有人都在互相观望。

- **定价模式转型**：Custom Agents 是 Notion 第一个 usage-based（按用量）计费产品。Alpha 客户包括 Ramp、Vercel、Clay、Cursor。客户能清楚看到 Agent 节省的真实人工时间并折算成美元，比按人头收费更有说服力。IPO 前提是先把 usage-based 商业模式跑通，预计需要"下一个两到三个季度"。

- **行业封闭时选择开放**：当整个 SaaS 行业封锁数据和 API 时，Notion 反向拥抱开放，让编码 Agent 能访问数据。Notion 历史上不算开发者中心的公司，但 Custom Agents 发布后几周和几个月内将推出更多面向开发者的功能。

### 🛠️ 操作流程

**如何复刻 Ivan 的邮件 Agent 管理方式**：

1. **准备阶段**：在 Notion 中建立一个 Custom Agent，连接你的邮件账户（Notion 已收购邮件应用，提供直接数据通路）。定义两个核心上下文文档：① "我在做什么"（当前项目、角色、职责）；② "我关心什么"（优先级规则、重要联系人、关键词）。

2. **核心执行**：不打开收件箱，改为通过 Notion 的对话界面与 Agent 交互。用自然语言指令操作，例如："把今天我认为不重要的邮件归档"、"列出需要我回复的邮件"、"今天有没有关于 X 项目的邮件"。

3. **验证与优化**：初期每周检查一次 Agent 的归档决策是否符合预期，持续更新"关心什么"的上下文文档。当 Agent 判断准确率稳定后，可以减少人工校验频率。

**如何复刻 Ramp 的销售知识库 Agent**：

1. **准备阶段**：将产品文档、更新日志、FAQ 全部迁移到 Notion（或确保 Agent 有访问权限）。连接销售团队使用的 Slack 频道。

2. **核心执行**：设置 Custom Agent 监听 Slack 频道中的产品问题，自动从知识库检索答案并回复。设置权限控制，确保 Agent 只能访问销售相关文档，不能访问敏感财务或人事数据。

3. **验证与优化**：追踪回答数量和准确率（Ramp 的基准是几周内 4,000 问题，每问题 30 分钟 = 2,000 小时节省）。定期更新知识库，确保产品迭代信息及时同步。

### 📝 避坑指南

- ⚠️ **数据分散是 Agent 的天敌**：Ramp 能成功的前提是把 6 个工具整合成 1 个，Agent 才有统一上下文。如果数据还分散在多个平台，Agent 效果会大打折扣，先做数据整合再上 Agent。
- ⚠️ **AI 节省时间是横向的，不要用"替代岗位"来 justify ROI**：当前阶段正确的 ROI 计算方式是"每人每天节省 X 分钟 × 人数"，而不是"减少 N 个 FTE"。用后者说服管理层会导致预期落差。
- ⚠️ **界面复杂度决定 Agent 可用性**：如果你的产品有上千个按钮（典型的老牌企业软件），Agent 无法理解如何使用，需要优先做 API 和自然语言接口的改造，否则会被 Agent 时代淘汰。
- ⚠️ **Usage-based 定价的推理成本风险**：Notion 自己也是大量购买各家 AI token 的买家，按用量收费的利润率模型尚未在知识工作领域被任何公司规模化验证（开发者工具除外）。早期采用时需要设置用量上限和成本监控。
- ⚠️ **模型中立需要工程投入**：Notion 的策略是 OpenAI/Anthropic 发布新模型时次日甚至当天接入，这需要持续的集成工程投入，不是一次性工作。
- ⚠️ **Custom Agent 的权限护栏是前提**：Ivan 明确说模型能力和权限护栏体系都需要完善，才能出现结构性的组织变化。在权限体系不完善之前，不要给 Agent 过大的操作权限。

**核心逻辑**:

- **Gemini 3的飞跃无单一秘诀**：Aurel Vinyals（Gemini联合负责人）的"更优质预训练+后训练"说法并非简化，而是真实写照。预训练团队约150-200人，覆盖数据、模型、基础设施、评估四个方向，成功来自无数小改进的叠加，而非1-2个关键突破。
- **架构：Transformer-based MoE（混合专家）**：Gemini 3是基于Transformer的混合专家架构（MoE）。核心机制是将"计算量使用"与"参数规模"解耦——通过动态路由将算力分配到特定专家模块，而非稠密地激活所有参数。注意力模块负责跨Token混合信息，前馈网络模块负责单Token并行计算与记忆存储。
- **规模定律未死，但权重已变**：规模仍是预训练性能提升的重要因素，但架构创新与数据创新的重要性如今可能已超过单纯规模扩张。两者是叠加关系，不是替代关系——有时创新收益 > 规模收益，有时反之。
- **Chinchilla教训仍在生效**：Sebastian是Chinchilla论文核心作者之一。核心结论：在固定训练算力下，数据规模的扩展速度应快于模型规模扩展速度。这一结论至今仍影响日常训练决策，尤其对模型部署成本有直接影响。
- **"数据无限→数据有限"范式转变**：这是当前最重要的行业范式转变。不是说数据量减少了，而是数据量从"可无限扩展"变成了"有上限的有限资源"。类比：大语言模型出现前，ImageNet时代就是数据有限阶段，催生了大量适配该阶段的技术。这一转变正在重塑研究方向。
- **合成数据需极度谨慎**：合成数据极易误用。核心难题：能否用模型A生成合成数据，训练出性能超越模型A的模型B？这是团队投入大量时间研究的问题，目前没有简单答案。
- **评估（Evaluation）是被严重低估的核心能力**：评估在预训练中极其重要且极难做好，需弥合两个差距：①小模型评估结果必须能预测大规模模型的性能方向（代理指标有效性）；②预训练阶段的评估必须能预测后训练后的最终表现。DeepMind越来越倾向于内部构建评估集，原因是外部基准测试很快会被污染（训练数据中混入测试内容，难以检测）。
- **原生多模态的代价与收益**：Gemini是原生多模态（同一神经网络处理文本、图像、音频、视频），而非多个独立模型拼接。代价：①研究复杂度显著增加（多模态交互影响研究的多个方面）；②图像输入规模远大于文本，直接计算成本更高。收益：Sebastian认为收益远超成本，这是训练此类模型的根本原因。
- **预训练的"感觉"（Vibe）贡献可能大于后训练**：关于模型的"氛围感"（如GPT-4.5的质感），Sebastian认为预训练在模型"感觉"方面的贡献可能大于后训练。而"氛围编程"（Vibe Coding）能力则更多来自强化学习规模化和后训练。
- **长上下文是预训练下一个重大方向**：Gemini 1.5在长上下文方面已实现巨大飞跃，使模型能处理完整代码库等大型任务。未来1-2年将有更多创新：既提升长上下文处理效率，又进一步扩展上下文长度。注意力机制方面近期有"非常有趣的新发现"，将在未来几个月塑造多个研究方向。

### 🎯 关键洞察

**洞察1：研究品味（Research Taste）的可操作定义**

Sebastian给出了两个具体维度：
1. **系统整合性**：你的改进不能孤立存在。若你对模型做了改进A，但导致其他人使用该模型的难度增加5%，这很可能是坏的权衡——它会拖慢整体进展。
2. **复杂度管理**：研究能承受的复杂性有上限。通常不选性能最优方案，而是选复杂度稍低、性能略差的版本，因为这有助于未来取得更多进展。

**洞察2：深度学习中"负面结果"的正确解读**

一个负面结果不意味着某方法行不通，往往只意味着你还没找到让它可行的方式。这一认知直接影响研究资源分配决策——何时放弃一个方向，何时继续坚持。

**洞察3：Retro架构的历史定位与未来复归**

Retro（检索增强预训练）的核心不是让模型更小，而是将"知识存储"与"推理计算"分离——模型在训练和推理时都能从大型语料库中检索信息，而非将所有知识压缩进参数。当前RAG/搜索在后训练阶段实现了类似功能，但Sebastian认为长期最优解是通过可微分的端到端方式在预训练阶段学习检索能力，将搜索整合进大规模训练。强化学习规模化可能已开启这一进程，但架构层面还有大量工作要做。

**洞察4：短期目标 vs 长期探索的周期性平衡**

- 模型规模扩张阶段：探索性研究比例更高（无紧急并行问题）
- 新架构/新模型发布前：工作重心转向降低风险、聚焦执行
- 这不是固定比例，而是随项目阶段动态调整

**洞察5：持续学习的两条路径**

1. 后训练路径（当前主流）：通过搜索工具调用获取新知识，迭代速度快
2. 预训练路径（长期方向）：扩展上下文窗口让模型在上下文中持续获取新信息；或改变训练算法，让模型从现实世界数据流中持续学习

### 🛠️ 操作流程（给研究者/工程师的可复用SOP）

1. **研究方向选择**：先评估该方向对整体系统的整合成本，不能只看单点性能提升；用"复杂度-收益"二维矩阵筛选，优先选复杂度低、可整合性强的方案。

2. **实验设计**：在小规模模型上做消融实验，但必须验证结论对大规模模型的可迁移性（代理指标有效性）；负面结果不等于方向错误，记录"为什么没成功"比记录"没成功"更重要。

3. **评估体系建设**：内部构建评估集，严格隔离测试数据，避免训练数据污染；评估必须同时覆盖预训练阶段表现和后训练后的最终表现，两者缺一不可。

4. **合成数据使用**：明确目标——是补充数据量还是提升数据质量；设计对照实验验证合成数据是否真正提升了目标模型性能，而非仅仅复制了生成模型的能力上限。

5. **技术栈纵深**：从TPU/硬件层到模型架构层，建立完整的技术栈理解；发现不同层级之间的"差距"（gap）往往是高价值研究机会所在。

6. **团队协作**：大规模团队的核心不是让少数人领先，而是让每个人都能发挥所长；协调整合工作的价值不低于具体研究工作本身。

### 💡 具体案例/数据

- **Gopher项目**：DeepMind第一篇大语言模型论文，团队约10-12人，训练了2800亿参数的稠密Transformer，使用3000亿Token。Sebastian本人参与，这是他进入预训练领域的起点。
- **Chinchilla结论**：在固定训练算力下，数据规模扩展速度应快于模型规模扩展速度。这一结论与当时（OpenAI之前的工作）的主流认知相反，至今仍影响日常训练决策。
- **Gemini 1.5长上下文**：已实现"巨大飞跃"，使模型能处理完整代码库等大型任务，上下文长度显著增加。
- **Sebastian个人轨迹**：2018年以研究工程师身份加入DeepMind（当时尚未并入谷歌），入职契机是硕士讲师（同时是DeepMind研究员）的推荐。最初做强化学习+Atari游戏，约6个月后转向表征学习，再转向大语言模型预训练。
- **行业过度投资的历史案例**：约两年前（2023-2024年），大量团队在构建专门模型解决通用模型半年到一年内就能解决的任务，现在这一现象已明显减少。

### 📝 避坑指南

- ⚠️ **不要把"缩放定律失效"当真**：Sebastian明确表示这与他的实际经验不符。规模仍然重要，只是不再是唯一因素，架构和数据创新的权重在上升。
- ⚠️ **合成数据的自欺欺人陷阱**：用强模型生成数据、小规模消融验证有效，不等于能训练出超越原始模型的新模型。这是本质区别，必须严格区分。
- ⚠️ **外部基准测试的污染问题**：外部基准测试内容会以各种形式在网络传播，混入训练数据后极难检测。依赖外部基准测试会导致自欺欺人，必须建立内部独立评估集。
- ⚠️ **评估研究被严重低估**：人们普遍低估评估工作的重要性和难度。评估进步是模型和数据改进进步的前提——没有好的评估，你根本无法准确衡量改进效果。
- ⚠️ **专门模型的时效性陷阱**：为特定任务构建专门模型时，必须评估通用模型在下一个版本是否就能覆盖该任务。如果答案是"可能"，专门模型的投资价值就需要重新审视。
- ⚠️ **研究改进的整合成本**：单点性能提升不等于系统性进步。任何改进都必须评估对其他团队成员工作的影响，哪怕只是增加5%的使用难度，也可能是负收益。

### 📋 核心分析

**战略价值**: Matthew Berman 用 OpenClaw 在一台 MacBook 上独立复现了一个小公司中台团队的全部职能，核心是让各模块数据互通形成飞轮，而非堆砌孤立工具。

**核心逻辑**:

- **自然语言即代码**：Berman 全程零代码，用自然语言描述需求，30 分钟内跑起一个完整 CRM，接入 Gmail、Google Calendar、Fathom 三个数据源，含 LLM 过滤层判断邮件价值。
- **数据摄取频率精确设计**：邮件每 30 分钟扫描一次，Fathom 在工作时段每 5 分钟检查一次，不是无脑全量同步，而是按场景设定节奏。
- **LLM 作为数据入库的守门人**：所有数据在写入前先经过一个 LLM 判断层（「这封邮件值不值得存？」），而不是把垃圾数据全塞进去再事后清理。
- **行动项系统有自我学习机制**：Berman 拒绝某条行动项时，系统记录拒绝原因并更新提取规则，下次同类情况不再误抓，形成闭环迭代。
- **多 Agent 辩论模式**：商业顾问委员会由 8 个 AI 专家角色并行分析同一份数据，分析完互相讨论、综合分歧，最终输出按优先级排序的建议——不是一个 AI 给答案，是一群 AI 辩论后给答案。
- **AI 审查 AI 的安全架构**：安全委员会每晚 3:30 运行（刻意错开其他任务避免 API 配额冲突），四个视角（进攻、防御、数据隐私、操作真实性）读取整个代码库 + Git 历史 + 运行日志，Berman 回复「fix it」即自动修复，修复经验持续沉淀进审查规则。
- **记忆系统三层架构**：对话记忆（markdown 文件）→ 偏好提炼（memory.md，含写作风格、股票追踪、邮件分类规则）→ 身份更新（identity.md + soul.md），每次新对话开始时自动读取，全部向量化支持 RAG 检索。
- **场景化人格切换**：同一个 AI，Telegram 私聊时幽默随意，Slack 团队频道自动切换为专业同事风格，人格定义写在配置文件里。
- **数据飞轮是核心护城河**：CRM 数据 → 商业顾问委员会；知识库 → 视频选题流水线；社交媒体数据 → 每日简报 + 顾问委员会；会议记录 → CRM + 行动项；所有模块运行日志 → 安全委员会。每个模块的输出都是另一个模块的输入。
- **基础设施设计原则**：你睡觉时系统在干活，系统出问题时你第一时间知道——备份失败、任务异常全部 Telegram 即时告警。

### 🎯 关键洞察

**为什么单个用例看起来不惊艳，但整体系统很强？**

原因：ChatGPT 也能查联系人，Notion AI 也能整理知识库，但它们是孤岛。Berman 系统的威力来自数据在模块间的流动——一次会议记录，同时更新 CRM、生成行动项、喂给顾问委员会做业务分析。信息只录入一次，但被多个下游模块消费。

动作：搭建时不要先想「这个模块能干什么」，而要先想「这个模块的输出能被哪些其他模块消费」。

结果：系统规模越大，每个新模块的边际价值越高，因为它既是消费者也是生产者。

**食物日记用例的底层逻辑**：不是「AI 帮你记录」，而是「AI 做交叉分析」——把食物摄入数据和症状反馈数据放在一起，识别人类自己感知不到的相关性（洋葱 → 胃部不适）。这个模式可以迁移到任何「输入-症状」类问题。

### 📋 核心分析

**战略价值**: 一套可直接部署的工业级 AI 导演分镜 Skill，分基础版（~30条规则）与进阶版（~50条规则）两层注入，覆盖从剧本解读到逐镜头输出的完整制作流程。

**核心逻辑**:
- **两版分层设计**：基础版保证单次推理高执行率，进阶版作为第二轮精修补充注入，两版互补不冲突，进阶版在重叠处优先级更高
- **镜头数量基准**：1-2分钟成片 → 60-120个镜头；500字剧本 ≈ 1分半时长；动作/战斗密集型偏上限，对话/氛围型偏下限，数量是规则自然落地的结果，不凑数不偷懒
- **最小拆解单位是"情绪节拍"**，严禁以笼统"场景"为单位，所有情绪转折点、信息爆发点、氛围转换点必须标记
- **前置建构强制要求**：开场增补5-15个无台词视觉引入镜头，第一镜定义世界观基调，默认避免平庸全景开场，优先局部特写引入再拉开全貌，最后一镜必须是强烈悬念钩子
- **动作四镜头链**：起势→过程→落点→反应，缺一不可；"落点"必须含物理反馈（灰尘/震动/碎裂）
- **对话拆镜默认规则**：单句台词 = 说话者起势→听者反应→说话者高潮→结束后反应；台词说完后留半秒再切走；听者反应优先级高于说者话语
- **衔接逻辑强制标注**：场景/情绪跳转处必须标注【主逻辑+次逻辑】复合标签；连续自然镜头允许单标签；善用J-Cut（声音先行）/L-Cut（画面先行）增强情感连接
- **视觉叙事底线**：关掉声音能理解至少60%的故事和情感（动作/视觉叙事型目标80%，对话驱动型目标60%）
- **角色塑造铁律**：无台词无动作时角色必须处于"平行反应链"中，严禁呆立；每个主要角色至少一个眼神特写镜头；角色镜头语言设计从"这个角色是谁"出发，而非从性别/外貌出发
- **终极戒律**：一个镜头没有同时服务于"叙事、角色、情绪"中至少两者，即为废片；先做减法直到减去任何一个镜头都损害故事，再做加法为人物增加一个"灵魂镜头"

### 🎯 关键洞察

**加载方式三选一，按场景选择**：
- 日常生成 → 只加载基础版，单次推理执行率最高
- 精修阶段 → 先跑基础版，再追加进阶版做第二轮注入

**执行流程（7步）**：

2. **前置建构** → 增补5-15个无台词视觉引入镜头，第一镜定基调，最后一镜必须是悬念钩子

6. **节奏控制** → 快节奏=密集卡点剪辑；慢节奏=延长单画面/长镜头

### 📝 避坑指南

- ⚠️ 连续三个以上同景别镜头必须有明确节奏或情绪目的，否则视为无意识重复
- ⚠️ 硬切是合法剪辑语言，但必须有节奏或情绪动机，禁止无意识随意硬切
- ⚠️ 手持/晃动镜头必须有情绪动机，禁止无理由使用
- ⚠️ 给物件特写=宣布"它很重要"，不可滥用，滥用会稀释叙事重点
- ⚠️ 匹配转场每场戏最多用一次，过度使用会失去冲击力
- ⚠️ 垂直俯拍（上帝视角）仅在表达"宿命感"或"无力感"时使用，不可随意调用
- ⚠️ 空镜头默认控制在三秒内；氛围系/治愈系可适当延长，但画面内必须有持续视觉变化
- ⚠️ 长镜头（>10秒）结尾必须有"信息爆炸点"，否则观众注意力流失
- ⚠️ 记忆闪回必须在色调/画幅/质感上与主时空明确区分，否则观众混淆时间线
- ⚠️ 角色死亡的视觉符号设计不要只用"手中物体滑落"这一个方案，可以是光线熄灭、节奏停止、画面色彩流失等
