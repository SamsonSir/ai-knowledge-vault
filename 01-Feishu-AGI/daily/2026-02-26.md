### 📋 核心分析

**战略价值**: Selfware 协议将文件从静态数据容器升级为自包含、可进化的软件实体，彻底解决 AI agent 时代跨平台上下文流转的摩擦问题。

**核心逻辑**:

- **痛点根源**：传统文件与特定软件强绑定（PSD→Photoshop、DOCX→Word），格式兼容性差，agent 之间传递上下文时数据丢失、逻辑断裂，无法无损流转。
- **核心公理**：`a file is an APP, everything is a file`——传统软件三层结构（数据层/逻辑层/交互层）在 Selfware 中通过协议指定视图动态生成，AI 抹平视图与逻辑层差异。
- **文件结构**：`.self` 文件 = `selfware.md`（协议引用文件）+ `manifest`（定义启动、打包、安全范围、权限）+ `content/` 目录（所有写入操作只能由 agent 执行）。
- **多视图能力**：同一数据源可按需渲染为文档、演示文稿、脑图、大纲、卡片等视图，视图和内容均可随时修改，还可导出为传统软件格式或图片。
- **自进化机制**：通过 `discovery` 服务，文件携带意图和部分上下文，自动去 GitHub 等平台查找相似任务和技能，更新后询问用户是否采用，实现能力增强而无需人工干预。
- **协同与共享**：支持单文件打包分享或基于 Git 仓库协同，接收方打开后获得完整内容+操作记录，可继续迭代；多人多 agent 可基于同一 `.self` 文件并行工作，文件自动同步合并。
- **安全与权限**：协议默认排除敏感信息（环境个性化信息、Git 记录、`.ENV` 文件内容），支持文件加密，用户可手动控制打包粒度，后续将通过 hook 机制阻止 agent 读取敏感配置。
- **开放无锁定**：协议完全开放，可自定义后缀名（如 `.videoagi`、`.AJ`），所有 coding agent 均可支持，不与任何平台耦合，数据归用户所有。
- **长程运行记录**：将 agent 长程运行（如 25 小时、20 次迭代）的历史、关键决策、设计意图、技能使用记录打包进 `.self` 文件，他人接收后不仅得到软件界面，还能复刻完整开发能力。
- **经济系统重构**：Remy 提出 Web 4.0 形态——生产力单位从"人工作量"转变为"将 TOKEN 转化为价值的能力"，早期开发者通过 Selfware 技能复用，可从后续使用者产生的红利中持续获利。

### 🎯 关键洞察

**为什么 `.self` 文件比 DOCX/ZIP 更适合 agent 时代**：

DOCX 本质是 ZIP 包（改扩展名为 `.zip` 即可解压），其媒体文件存于临时目录，分享 Markdown/HTML 时图片路径极易断裂。`.self` 文件将所有相关资源自包含封装，解决了传统文件分享时的资源裂开问题，同时支持加密和权限分层管理。

**entry point 设计逻辑**：协议将入口点分为两类——`action`（交互触发）和 `script`（脚本执行），这使得带服务器依赖的复杂程序也能通过 Selfware 封装，本地运行避免了与云服务交互的障碍，依赖直接装入文件包内。

**复杂程序适配性**：少卿明确指出，带服务器的复杂程序反而是 Selfware 的最佳适用场景，因为可以将所有依赖内置，本地运行，彻底绕开云服务配置障碍。

### 🛠️ 操作流程

**AJ 实践 Selfware 协议的完整步骤（基于 Claude Code）**：

1. **安装阶段**: 使用 Claude Code 与 AI 对话，执行安装指令，创建 Selfware 实例，了解项目用法（通过 `manifest` 和 `selfware.md` 引导 agent 理解项目结构）。

2. **创建内容**: 让 agent 写文章、创建日记系统，系统以 `.self` 文件形式存在，本地运行后通过 `localhost` 访问，支持 Markdown 格式，随时记录保存。

3. **进化迭代**: 让日记系统自我进化——添加游戏化系统、时光胶囊功能；让 agent 自己写日记，实现人与 agent 共创内容。

4. **分享流程**: 将 `.self` 文件打包（可选择是否包含内容），发送给他人；接收方下载解压，通过 agent 运行（运行失败直接让 agent 继续尝试，无需手动排查）。

5. **协同更新**: 若基于 Git 仓库共享，文件变更自动同步，接收方可在原有基础上继续迭代，操作记录完整保留。

**少卿 Demo 操作流程**：

1. 本地运行 `.selfware` 文件
2. 浏览器打开视图层（`localhost` 端口）
3. 直接修改文档内容 或 指令 agent 更新文档/软件（如插入图片、新增主题）
4. 切换视图（文档→演示文稿→脑图）
5. 打包分享：选择单文件模式或 Git 仓库模式

### 📝 避坑指南

- ⚠️ **API key 暴露风险**：部署涉及第三方服务（如小龙虾）时，协议默认去除敏感信息，但当前版本需手动检查 `.ENV` 文件是否被打包，后续版本将用 hook 强制阻止。
- ⚠️ **文件打不开/运行失败**：不要手动排查，直接让 agent 运行文件，agent 能力足够强，运行失败可指令其继续尝试，通常可自行修复。
- ⚠️ **协议禁止自动更新**：discovery 服务找到新技能后必须询问用户确认，协议明确禁止文件自动更新，防止未经授权的能力注入。
- ⚠️ **视图质量依赖 agent 能力**：同一 `.self` 文件在不同 agent 上的渲染效果不同，每个人的使用体验取决于其所用 agent 的能力上限。
- ⚠️ **Git 协同需双方互信**：基于 Git 仓库的自动同步模式要求协作双方建立信任关系，否则建议使用单文件打包分享模式。
- ⚠️ **content 目录权限**：所有写入操作必须通过 agent 执行，用户不得直接操作 `content/` 目录，否则破坏文件自包含结构。

### 📝 避坑指南

- ⚠️ **传统贸易1.0思维必须抛弃**：带着产品找客户、做供应商的路子在中东已死，印度人几十年前就做完了进口生意，利润率极低且竞争激烈。必须做品牌出海+本地深耕。
- ⚠️ **阿拉伯语乱码是致命失误**：谷歌翻译直接出图、设计软件未调RTL设置，输出乱码=直接告诉用户"我在敷衍你"，比没有本地化更糟糕。
- ⚠️ **斋月不是停工期**：很多人以为斋月业务停滞，实际上斋月是很多公司业绩冲高的节点，错过iftar营销窗口是重大失误。
- ⚠️ **绕过中间人会破坏整个合作链**：在会议上直接要求加对方微信/电话，阿拉伯合作方会当场拒绝并让你联系中间人。中间人文化（Wasta）是商业基础设施，不是可以优化掉的摩擦成本。
- ⚠️ **傲慢是最贵的出海成本**：国内成功经验在中东无法直接复用，"我要改变你的商业习惯"的心态会直接碰壁。阿拉伯人不守时、宗教文化冲击、中间人文化——这些是国情，不是需要被纠正的问题。
- ⚠️ **男性商务礼仪**：不可主动与穆斯林女性握手，手放胸口点头致意即可，除非对方主动伸手。
- ⚠️ **内容红线不可碰**：裸露、酒精、猪肉、擦边内容在沙特/埃及等市场会直接被封杀，国内认为"不擦边"的内容在中东标准下可能完全不合格。
- ⚠️ **沙特≠阿联酋**：在阿联酋靠产品力竞争，在沙特靠关系和背书进入项目。用阿联酋打法做沙特会撞墙。

### 📋 核心分析

**战略价值**: 用 n8n 自动化工作流替代 Chatbot 对话式分析，通过逐条打标（22个标签）+ 统计驱动生成，将亚马逊竞品评论分析从"60分幻觉报告"升级为"90分定量+定性深度洞察"。

**核心逻辑**:

- **Chatbot 分析的根本缺陷**：LLM 基于概率"文字接龙"，面对长文本时注意力衰减，会用幻觉填充细节。即使用 Deep Research，每次生成结论都可能不同，只能得到 60 分报告。
- **解法核心：先打标，再分析**：不直接让 AI 读原始评论生成结论，而是先让 AI 逐条提取结构化标签，用统计数据作为洞察结论的支撑，从根源上消除幻觉。
- **22个标签覆盖8大维度**：人群（性别/年龄/职业/购买角色）、场景（使用场景）、产品（功能满意度/具体功能）、质量（材质/做工/耐用性）、服务（发货/包装/客服/退换货/保修）、体验（舒适度/易用性/外观/价格感知）、市场（竞品对比/复购意愿）、情感（总体评价）。
- **逐条写回机制保证质量**：每完成一条评论的打标，立即写回 Google Sheets，再处理下一条。类比"班主任逐份批改试卷"，而非批量扫描，最大程度保证分析准确度。
- **评论价值评估节点是关键创新**：从 500 条中筛选黄金样本的方法——字数越多、有效标签越多、命中关键标签（竞品对比、复购意愿）的评论综合评分越高，分别取 TOP3 好评和差评作为报告黄金样本。
- **报告结构三段式**：① 洞察概览 + 22标签维度分布统计（定量）→ ② 用户画像/核心卖点/主要痛点/改进建议/潜在机会五大模块（定性）→ ③ 典型用户评论深度解析（引用原文）。
- **性能瓶颈与实际建议**：单条评论打标+写入耗时 8~10s，500条需 1小时以上。实际建议控制在 200~300 条；或换更快模型、减少标签数量、过滤字数过少的评论。
- **标签支持垂直类目自定义**：服装类可增加"图案/Logo偏好"标签，3C类可增加"续航/充电速度"标签，通用框架可按品类灵活替换。
- **报告到商业机会的三步判断框架**：① 差评是分散（产品价值错位）还是集中（明确可改进空间）→ ② 自己是否有能力解决这些痛点 → ③ 市场上是否已有竞品针对这些痛点做了优化、入场是否已晚。

### 🎯 关键洞察

**为什么逐条打标比整体分析更准确**：

LLM 的注意力机制在处理超长上下文时存在"中间遗忘"现象。将 500 条评论一次性投喂，AI 会对中间段落的细节产生衰减，最终用概率最高的"合理内容"填充，而非真实统计。

逐条打标的本质是：把"语义理解"和"统计分析"拆成两个独立步骤。AI 只负责单条评论的语义提取（擅长），统计工作交给 Google Sheets（精确），最后再让 AI 基于统计结果做定性解读（有数据支撑，幻觉空间极小）。

**评论价值评估的评分逻辑**（原文未给出具体公式，但可推导）：
- 字数多 → 信息密度高
- 有效标签数多 → 覆盖维度广
- 命中竞品对比/复购意愿标签 → 商业价值高

三者加权即为综合评分，TOP3 好评 + TOP3 差评 = 6条黄金样本，用于报告最终的深度解析章节。

- **功能本质**：Remote Control 不迁移代码或环境，电脑端项目文件原封不动，手机只是一个遥控器，通过二维码或链接与电脑会话实时同步。

### 📋 核心分析

**战略价值**: 将 GitHub 检索能力封装成一个可复用的 Skills，让 Codex / Claude Code 作为研究员代理执行「需求对齐 → 多轮检索 → 硬过滤 → 文档阅读 → 相关性重排」全流程，非技术用户也能零门槛调研 GitHub 项目。

**核心逻辑**:

- **痛点定位**：英文搜索能力弱 + 不懂如何评判项目质量，是非技术用户进入 GitHub 的两道墙；Skills 方案绕过这两道墙，让 Agent 代劳。
- **核心机制是 Agent 作业流，不是简单搜索**：整个流程有循环、有重试、会自我纠正，跑完一次约需 10-15 分钟，等同于一个真人研究员的调研产出。
- **需求对齐阶段**：Agent 收到需求后不会直接搜索，而是先与用户确认真实意图，避免检索方向跑偏。
- **多轮检索 + 自动优化检索词**：Agent 自动拆解出若干检索词，执行检索后评估结果质量，若质量不达标则自动优化检索词并重新检索，直到满意为止。
- **硬过滤条件筛选**：在原始检索结果中先用硬条件（如 Star 数、活跃度、语言等）过滤掉不符合的项目，缩小候选集。
- **文档阅读层**：对过滤后的候选项目，Agent 会实际阅读项目 README / 文档，而非仅看元数据，确保理解项目实际功能。
- **相关性重排，而非 Star 排序**：最终排序逻辑以「需求关联度」为第一优先级，Star 数只是参考。案例中 OpenClaw 官方主仓 Star 最高，但因用户需求是「玩法汇总」，玩法合集仓库被排在第一位。
- **自动分类输出**：工具类搜索结果会自动区分「可直接用的桌面软件」与「需要自己安装的框架层」，降低用户选择成本。
- **端到端安装能力**：找到目标项目后，可直接让 Codex 执行安装，部分项目（如 Open Video Downloader）安装后直接打开桌面版，开箱即用，无需手动折腾。
- **用户侧极简操作**：用户只需做两件事——讲出需求、看报告，中间所有检索、过滤、阅读、排序全由 Agent 自动完成。

### 🎯 关键洞察

**为什么排序不按 Star 数而按需求关联度？**

Star 数衡量的是项目的「综合受欢迎程度」，而用户的需求往往是垂直的。以 OpenClaw 案例为例：官方主仓 Star 最高，但它是工具本体，不包含玩法内容；用户真正需要的是「玩法清单合集」仓库。如果按 Star 排序，用户会先看到官方仓，然后自己再去找玩法，多走一步。Agent 通过阅读文档理解项目内容后重排，直接把最匹配需求的结果放第一位，这是纯关键词搜索做不到的。

**为什么要有「需求对齐」这一步？**

用户表达的需求往往是模糊的，比如「找 Agent 记忆模块」可能指：现成可用的 SDK、论文实现、框架插件、或完整 Agent 框架。如果跳过对齐直接搜索，检索词可能完全跑偏，浪费后续所有步骤的时间。需求对齐把模糊意图转化为精确检索方向，是整个流程质量的源头保障。
