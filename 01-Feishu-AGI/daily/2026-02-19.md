### 📋 核心分析

**战略价值**: 一个人用 Seedance 2.0 + Cursor 制作 AI 同人视频，通过「稀缺性叠加 + MVP 测试 + 迭代式脚本」方法论，两天内实现全网 2500 万曝光并登上好莱坞媒体报道的完整复盘。

**核心逻辑**:

- **稀缺性叠加是爆款的底层公式**：本次爆发同时命中三层稀缺——①工具稀缺（Seedance 2.0 当时仅中国创作者可用，外网创作者还用不上，时间窗口极短）；②技术临界点稀缺（AI 视频质量首次让普通观众产生"这不可能是 AI"的震惊感）；③内容需求稀缺（《怪奇物语》粉丝对官方结局强烈不满，存在巨大的内容缺口）。三层叠加，缺一不可。
- **先 MVP 测试，再投入长视频**：不要直接做两分半的长视频。先用一个晚上做 40 秒短片发小红书，几百上千点赞 → 验证市场需求 → 再放心投入长视频制作。抖音同步测试，3k 赞，超过作者此前所有视频。MVP 成本极低，验证速度极快。
- **脚本占 80% 精力，且必须基于市场调研**：在 Cursor 里和 AI 模型一起搜索 Reddit 上粉丝对《怪奇物语》结局的具体不满点，用市场真实痛点驱动脚本，而不是自己凭空想象。创作者是"承接载体"，不是"引领者"。
- **迭代式创作，每次只构思 15 秒**：不要一开始就写完整脚本。每次只写 15 秒，生成出来看效果，再决定下一个 15 秒怎么接。最多同时构思 30 秒。整体故事框架直到做到 80% 时才成型。AI 生成的画面会反过来给脚本新灵感，脚本和生成互相喂养。
- **每 15 秒必须有一次情绪反转**：一个 scene = 一次生成（Seedance 最大 15 秒）= 一个反转节拍。两分半视频共设计了至少 8 次反转，观众永远猜不到下一秒，完播率才能撑住。
- **前 5 秒是生死线**：2 秒内划走的人可能有一半，5 秒内划走的人可能是 80%。前 5 秒必须直接给出最强爽点（本片前 5 秒就展示 Will 觉醒超能力控制夺心魔反攻 Vecna）。后面再精彩，80% 的人根本看不到。
- **每个 scene 生成 4-6 次，混剪取最优**：不是生成一次就能用。同一个 prompt 生成 4-6 条，全部拖进时间轴混剪。原因：①视觉一致性高（同 prompt 生成的素材风格、色调、角色外观接近）；②打斗穿帮用匹配剪辑遮掉；③画面好但声音不对、或声音好但画面不对，可以互相替换音画。最终成片是从多条素材里挑最优部分拼成的。
- **多参考图锁定角色状态**：即便模型已训练了 IP 角色，依然必须用参考图。原因：①锁定年龄（《怪奇物语》跨越多季，演员从 11 岁拍到成年，不锁定会随机出现任意年龄）；②统一穿着（不同季造型完全不同）；③还原战损状态（脸上的伤、血、灰尘是训练数据里没有的）。参考图直接从原剧最后一集截图获取。一个 scene 可挂 6-8 张参考图。
- **台词设计原则：简短有力 + 戳角色最深的恐惧/创伤**：每句台词极短，但必须有内在逻辑支撑。"You sound just like Papa"（Eight 对 Vecna 说）——因为 Vecna 恨透了 Papa 的控制，而他自己正在做同样的事，这句话从同一实验室出来的 Eight 嘴里说出来杀伤力最大。"Got you"（Max 拯救小 Henry）——两个字，但承载了 Max 整个角色弧线的高光。
- **打斗升级必须分层，规格一层层递进**：Will 单挑 Vecna → Eleven + Eight 联手打 Vecna → 夺心魔加入大场面。每一层都超出观众上一层的预期，场外因素（夺心魔）是打破平衡的超预期变量。

### 🎯 关键洞察

**内容稀缺性的本质**：创作者不是在引领市场，而是在比市场预期跑快一点点。《怪奇物语》粉丝的不满早就存在，内容需求早就在那里，创作者只是第一个把它具象化的人。这意味着选题阶段就应该去 Reddit、评论区找"粉丝集体遗憾"——那些地方藏着最真实的内容需求。

**时间窗口的不可复制性**：Seedance 2.0 仅中国创作者可用的窗口期极短。这类"工具稀缺"窗口一旦关闭（外网创作者也能用了），同类内容的震惊感会大幅下降。核心策略：新工具发布 → 立刻用它做最热门 IP 的"粉丝遗憾内容" → 在窗口关闭前发出去。

**病毒传播的结构**：原帖 234 万浏览，搬运带来 1680 万。最大单条搬运（巴西博主 @nerdemcartaz）492 万，超过原帖一倍。这说明内容本身的传播力远比账号粉丝量重要。被 6 种语言搬运（英、葡、土、中、日、西）的原因：《怪奇物语》是全球 IP，粉丝不满情绪是跨语言的。

**"顿悟时刻"设计**：好的剧情反转不是技巧，是让观众突然理解了一个更大的逻辑（类比《星际穿越》黑洞场景）。Max 拯救小 Henry 这个设定，来自作者看剧时一个模糊的直觉"这个小男孩好可怜，为什么没人管他"——直觉往往先于逻辑，好的创作是把模糊感觉变成清晰作品的过程。

### 🛠️ 操作流程

**1. 准备阶段：选题 + 市场调研**
- 找"粉丝集体遗憾"IP：去 Reddit、评论区搜索目标 IP 的结局不满、角色遗憾、未完成的情节
- 在 Cursor 里让 AI 模型全网搜索具体缺憾点，整理成列表
- 确认三层稀缺是否同时命中：工具稀缺（是否有时间窗口）+ 技术稀缺（AI 效果是否能震惊普通观众）+ 内容稀缺（粉丝需求是否真实存在）
- 从原剧最后一集截取每个角色的参考图，按编号整理进 Cursor 文件夹

**2. MVP 测试：先做 40 秒短片验证**
- 用一个晚上做第一集 40 秒短片
- 发小红书/抖音，观察点赞数
- 达到几百上千点赞 → 验证通过 → 进入长视频制作
- 未达预期 → 重新审视选题或内容方向

**3. 核心执行：迭代式脚本 + 生成**
- 每次只写 15 秒脚本，格式固定：角色+参考图 / 背景介绍 / 逐镜头描述 / 风格指令
- 每个 scene 挂 6-8 张参考图（角色图 + 场景图）
- 生成 4-6 次，全部素材拖进时间轴
- 看生成结果 → 决定下一个 15 秒走向（AI 画面可能反向给脚本灵感）
- 最多同时构思 30 秒，不要提前写完整脚本
- 打斗分层升级：单打 → 联手 → 场外变量介入
- 台词原则：极短 + 戳角色最深创伤/恐惧 + 每人必须有高光

**4. 剪辑与优化**
- 用匹配剪辑处理穿帮（在穿帮点切到另一条素材的对应时刻）
- 音画互换：声音好但画面不对的，换画面；画面好但声音不对的，换声音
- 快慢节奏对比：打斗用快切，文戏对峙刻意放慢，给台词留呼吸空间
- 前 5 秒必须是全片最强爽点之一，不能留到后面

**5. 发布时机**
- 新工具发布后立刻行动，不要等
- 优先发外网（X/Twitter）触达全球粉丝群体
- 发布后监控搬运情况，搬运本身是传播放大器，不需要干预

### 📝 避坑指南

- ⚠️ **不要直接做长视频**：没有 MVP 测试直接做两分半，做出来没人看的概率极高。先 40 秒测试，验证后再投入。
- ⚠️ **不要一次性写完整脚本**：提前写完整脚本会让你忽视 AI 生成画面给你的反向灵感。每次只构思 15-30 秒，保持迭代弹性。
- ⚠️ **不要只生成一次就用**：每个 scene 至少生成 4-6 次。一次生成的素材穿帮率高，且无法做混剪优化。
- ⚠️ **不要省略参考图**：即便模型已训练了 IP 角色，不挂参考图会导致年龄随机、造型随机、战损状态无法还原。多角色同框场景不挂图必然混淆角色。
- ⚠️ **不要让素材散落各处**：所有参考图、prompt、风格指令必须集中在 Cursor 的格式化脚本文件里，否则每次生成都要东翻西找，效率极低。
- ⚠️ **不要忽视前 5 秒**：80% 的观众在前 5 秒内划走。前 5 秒不是铺垫，必须直接给最强爽点。
- ⚠️ **时间窗口稍纵即逝**：新工具发布后的独占窗口期极短，必须立刻行动。等外网创作者也能用了，同类内容的震惊感会大幅衰减。
- ⚠️ **台词不能长**：文戏台词越长越弱。每句话必须极短，且必须有具体的内在逻辑支撑（戳创伤、揭矛盾、反讽），不能为了台词而台词。

### 📋 核心分析

**战略价值**: Boris Cherny 通过 Claude Code 从 0 到 10 亿美元年化收入的完整路径，揭示了 AI 时代产品构建的三条底层原则，以及工程生产力范式转移的真实数据。

**核心逻辑**:

- **偶然起点，终端选型**：Claude Code 最初是 Boris 为学 Anthropic API 随手写的终端聊天工具，选终端唯一原因是"不用写 UI，当时就我一个人"。这个"临时方案"一直沿用至今，Boris 自己也觉得不可思议。
- **AGI 时刻触发产品方向**：Sonnet 3.5 时期，Boris 给模型加了 bash 工具，问"我在听什么音乐"，Claude 自己写了 AppleScript 调用 Mac 音乐播放器查到了答案。这个瞬间让他意识到"模型就是想用工具，它只想跟世界互动"，产品方向由此确立。
- **内部病毒式传播验证 PMF**：原型做出两天后，同事 Robert 已自发在用它写代码。2024 年 11 月/12 月上线评审时，Dario Amodei 看到 DAU 增长图几乎是垂直线，以为是强制推广，实际只是 Boris 发了一个内部帖子，靠口耳相传扩散。
- **早期用例路径**：Git 操作和容器管理自动化 → 写单元测试（风险低）→ 逐步扩展到全量编码。Boris 至今可能已忘了大部分 Git 命令。
- **CLAUDE.md 的正确用法**：Boris 本人的 CLAUDE.md 只有两行（提交 PR 后自动合并；PR 提交后发到团队频道审批），团队共享的约 2000 token。他的建议：CLAUDE.md 太复杂就直接删掉从头来，每个新模型需要的指令越来越少，只放反复重复的东西，其他直接 prompt。
- **Plan Mode 的技术本质与生命周期**：Plan Mode 的全部实现是在 prompt 里加一句"请不要写代码"，Boris 周日晚上 10 点看 GitHub issues 和 Slack 反馈，花 30 分钟写完，当晚发布，周一上线。他预测一个月内 Plan Mode 将不再需要，因为 Claude 已能自主判断何时进入规划模式。Boris 自己 80% 的 session 以 Plan Mode 开始，会同时开多个终端标签页并行跑。
- **Scaffolding 是技术债**：围绕模型构建的脚手架可提升性能 10-20%，但下一个模型出来后这些优化会被抹平。他们始终把任何 scaffolding 视为技术债，权衡标准是：加功能获得 10-20% 提升 vs. 等两个月让下一个模型原生支持。
- **Agent 拓扑与"Mama Claude"**：多 Agent 协作的核心是"不相关的上下文窗口"，各 Agent 独立上下文互不污染。Anthropic 的 Plugins 功能由一群 Agent 在周末自动协作构建完成，几乎无人工干预：一个工程师给 Claude 一个 spec，Claude 在 Asana 创建任务看板，自动生成子 Agent 领取任务执行，主 Agent（内部称"Mama Claude"）只负责发指令。Boris 估计现在大多数 Agent 启动时是由另一个 Claude 来提示的，而非人类。
- **生产力数据**：Anthropic 团队规模去年翻倍，人均 PR 数量增长约 70%；自 Claude Code 推出以来人均生产力累计增长 150%。对比：Boris 在 Meta 负责 Facebook/Instagram/WhatsApp 全代码库质量管理时，几百人工作一年才能实现 2% 的提升。
- **代码库无历史**：Claude Code 代码库没有任何部分是六个月前的，一直在被重写。80% 的代码历史可能不足几个月。

### 🎯 关键洞察

**潜在需求（Latent Demand）是最重要的产品理念**

逻辑链：人们只会做他们已经在做的事 → 你不能让人做新事情 → 只能帮他们把正在做的事做得更简单。

三个具体案例：
1. CLAUDE.md：用户自发写 markdown 文件让模型读，Boris 观察到这个行为后才产品化。
2. Plan Mode：用户在 GitHub issues 和 Slack 反馈里反复说"先想想方案别写代码"，Boris 才做了这个功能。
3. Co-Work：有人用 Claude Code 监控番茄植物、恢复婚礼照片、做财务分析，非技术用户在费力安装终端工具只为用这个功能，Boris 才做了图形界面封装。

**为未来模型构建，而非当前模型**

矛盾点：产品不能用怎么找 PMF？解法：Claude Code 正式发布时只能写 Boris 约 10% 的代码，但团队赌对了模型进步方向。如果只为当前模型优化，找到的 PMF 很快会被别人用下一代模型超越。操作建议：亲自使用模型，感受能力边界，然后为你认为半年后的那个模型做产品。

**TypeScript 设计哲学的迁移**

TypeScript 的奇怪设计（字面量类型、条件类型）根源是：观察 JavaScript 程序员怎么写代码（大量反射、mutation），然后围绕这些行为设计类型系统，而不是让程序员改变写法。Claude Code 同理：不是学术性原则驱动的产品，而是"我自己想用什么工具"——先给自己做，再给团队，再给公司，再给外部用户。

**初心者心态的具体案例**

内存泄漏排查：Boris 按老办法抓内存快照、在开发者工具里翻性能分析；同事 Chris 直接让 Claude Code 处理，Claude 自己抓快照、自己写分析工具，比 Boris 更快找到泄漏。Boris 的结论："我的大脑有时还停在六个月前。"

### 🛠️ 操作流程

**如何用 Claude Code 最大化生产力（Boris 的个人工作流）**

1. **准备阶段**：卸载 IDE，100% 切换到 Claude Code（Boris 从 Opus 4.5 开始执行）。CLAUDE.md 保持极简，只写反复重复的规则，其余直接 prompt。
2. **核心执行**：80% 的 session 以 Plan Mode 开始，在多个终端标签页同时开 Plan Mode 并行推进不同任务。方案确定后让 Claude 执行，Opus 4.5 以来基本每次都能准确执行，执行阶段不需要盯着。难任务启动 3-10 个子 Agent 并行研究，汇总结果再决策。
3. **验证与优化**：通过 PR 数量衡量产出（Boris 每天约 20 个 PR）。代码审查、安全审查、issue 标注、生产部署全部用 Claude Agents SDK 自动化。让 Claude 通过 git blame 自动识别相关工程师，在 Slack 上自动发确认问题，拿到答案后继续执行。

**如何构建 Agent 拓扑（Plugins 案例复刻步骤）**

1. 准备一份清晰的功能 spec。
2. 给主 Claude（Mama Claude）spec，指定用 Asana 看板管理任务。
3. 主 Claude 在 Asana 创建任务列表，自动生成子 Agent。
4. 子 Agent 自动领取任务并行执行，主 Claude 只负责协调指令。
5. 全程几乎无人工干预，周末可完成完整功能构建。

### 📝 避坑指南

- ⚠️ **CLAUDE.md 膨胀陷阱**：很多人把 CLAUDE.md 搞得太复杂。正确做法：删掉，从头来过。模型跑偏了再加一点回来。每个新模型需要的指令越来越少。
- ⚠️ **为当前模型优化的陷阱**：只为当前模型优化找到的 PMF，很快会被别人用下一代模型超越。要为六个月后的模型构建。
- ⚠️ **Scaffolding 技术债**：任何脚手架优化都假设为技术债。10-20% 的提升 vs. 等两个月让模型原生支持，要持续做这个权衡。
- ⚠️ **高级工程师的"强观点"陷阱**：过去被奖励"有很强的观点"，但很多观点现在已不再适用。内存泄漏案例说明：老方法不一定比让 Claude 直接处理更快。需要持续保持初心者心态。
- ⚠️ **上下文窗口污染**：多 Agent 协作时，各 Agent 必须保持独立上下文，不能让彼此的上下文或历史上下文互相污染，否则推理质量下降。
- ⚠️ **试图让用户做新事情**：产品方向上，不要试图让用户做一件完全不同的事，他们不会去做。只能帮他们把正在做的事做得更简单（潜在需求原则）。
- ⚠️ **跟模型对赌**：The Bitter Lesson 的核心教训——利用通用计算的方法最终总是胜过依赖人类知识的手工设计。永远不要跟模型对赌。

### 📋 核心分析

**战略价值**: ClawHub 5700+ 技能中仅 53 个官方内置，本文逐一拆解功能/平台/风险，给出最小安全配置路径，帮你在 ClawHavoc 事件（341 个恶意技能，占比 12%）背景下做出最安全的选择。

**核心逻辑**:

- **Tool vs Skill 本质区别**：Tool 是「器官」（read/write/exec 决定能不能做），Skill 是「教科书」（教怎么组合 Tool 完成任务）。安装 Skill 不赋予新权限，只是说明书。
- **默认全加载是风险源**：53 个官方技能默认全部自动加载，必须用 `skills.allowBundled` 白名单模式主动限制，只启用你真正需要的。
- **最小安全配置只需 9 个技能**：`gog, github, tmux, session-logs, weather, summarize, clawhub, healthcheck, skill-creator`，其余按需追加。
- **VM 部署有平台限制**：obsidian/apple-notes/bear-notes/things-mac/apple-reminders 均依赖本地 Mac 文件系统，OpenClaw 跑在 VM 上时这些技能无法使用，云端 Notion 是 VM 环境唯一无限制的笔记方案。
- **消息类技能的不可逆风险**：wacli/imsg/bird 等技能让 AI 以你名义发出的消息无法撤回，且提供完整历史消息访问权，与仅发送的 `message` 工具有本质区别。
- **1password 是全库或全无**：一旦授权，AI 可访问整个密码库所有条目，无法限制单条目。若必须用，建议创建「AI 专用密码库」隔离。
- **gog 是性价比最高的单个技能**：一个技能覆盖 Gmail + Calendar + Tasks + Drive + Docs + Sheets，支持自然语言指令（「今天有什么会议」「给 Sarah 发跟进邮件」），权限可随时在 Google 账户设置撤销。
- **summarize 是时间杠杆最高的技能**：40 分钟播客 20 秒出摘要，15 页 PDF 一段话总结，有用户报告每周节省 5 小时以上。
- **coding-agent 实现「AI 调度 AI」**：通过 Telegram 发指令「克隆这个仓库，研究它，做个 demo 网站」，它自动调用 Codex/Claude Code 等在后台执行，无需手动介入。
- **ClawHavoc 安全背景**：2026 年 2 月安全研究团队在 ClawHub 发现 341 个恶意技能（约占总数 12%），安装任何第三方技能前必须审查源码。

**来源**: 架构解析：OpenClaw 如何用一个 WebSocket 控制平面统一所有 AI 交互？ | **时间**: 2026-02-19 | **原文链接**: `https://mp.weixin.qq.com/s/g_dJK59u...`

### 📋 核心分析

**战略价值**: 用单一 WebSocket Gateway 作为控制平面，把 CLI/WebChat/移动端/浏览器自动化/消息通道统一成一个可调度、状态一致、可治理的 AI 系统，而非多套互不相认的 AI 分身。

**核心逻辑**:

- **多入口 ≠ 多套系统**：每多一个端，就多一套协议、状态、上下文同步逻辑。CLI 跑到一半的任务 Web 端看不到；手机端授权了定位桌面端还在反复询问；浏览器自动化执行到第 7 步消息通道只剩"处理中"——问题根源在架构，不在模型。
- **控制平面 vs 数据平面必须分离**：控制平面负责会话/状态管理、路由调度、权限策略、事件分发；数据平面负责模型推理、工具执行、多媒体传输。Gateway 只做调度与一致性，不承担计算与执行，避免变成塞满业务逻辑的巨无霸服务。
- **WebSocket 是控制平面的唯一合理选择**：AI 交互是全双工协作（模型持续吐 token、工具持续汇报进度、用户随时插话），HTTP+REST+轮询天然别扭。WebSocket 提供全双工、低延迟长连接、跨平台一致（CLI/浏览器/移动端/桌面端均可稳定作为 WS Client）。本地模式统一接入 `ws://127.0.0.1:18789`。
- **Hub & Spoke 星形拓扑，Gateway 是唯一事实源（Single Source of Truth）**：会话状态、任务执行进度、事件广播全部在 Gateway 收敛。客户端越做越薄，只负责输入与渲染，不各自维护复杂状态机。扩展新入口（如 Telegram bot）= 多一个节点，而不是多一套系统。
- **协议层用 JSON-RPC 2.0 + 命名空间**：结构规整、可演进，按能力域分组：`chat.*`（消息/会话/typing）、`agent.*`（编排/任务启停/工具触发）、`node.*`（注册/能力上报/心跳）、`browser.*`（自动化控制/DOM/截图回传）、`system.*`/`debug.*`（系统诊断）。
- **设备抽象为 Node + 能力声明（capabilities）**：iOS 节点声明 `["camera", "location", "notification"]`，CLI 节点声明 `["exec", "file_read"]`。Gateway 维护能力感知路由表，Agent 需要"定位"时精准路由到具备该能力的节点，而非盲目广播。
- **消息通道统一骨架，通道差异压到边缘层**：WhatsApp/Telegram/Slack 的 webhook/回调/签名/格式差异不污染核心逻辑，统一消息结构为 `{id, type, channel, payload, metadata}`，再做版本协商向后兼容、通道隔离互通、ACK+超时重试+离线持久化+历史回放。
- **浏览器自动化支持人机共存于同一 Session**：用户手动点击后 Agent 接管，或 Agent 跑流程时用户从手机发"暂停"，均通过统一状态协调器处理，避免"到底谁在控制"的混乱。
- **安全模型分层**：local-only / token / tailscale 多模式，配合细粒度权限（admin/read/write/pairing）。
- **可观测性是演进前提，不是锦上添花**：全链路 tracing、实时指标、统一日志——多端协作出问题，没有这套根本查不清。

---

### 🎯 关键洞察

**为什么"控制平面"这个抽象是关键**：

传统做法是"能力 + 入口"捆绑——每个端自己维护 Agent 逻辑、自己管状态、自己做推送。这在单端时没问题，但一旦多端协作，状态漂移是必然结果（网络切换、后台挂起、多标签页、并发操作）。

OpenClaw 的反直觉之处在于：**先不做更漂亮的 UI，先把"调度大脑"抽出来**。这个顺序决定了系统能否真正统一。

原因链：
> 控制平面统一 → 状态在 Gateway 生产与分发，客户端只是"显示器" → 扩展新端不需要重写 Agent 逻辑 → 系统整体性出现

### 🛠️ 操作流程

**案例一：CLI 发起任务，多端同步**

1. **CLI 输入 prompt** → 包装成 `chat.sendMessage` 控制消息发送至 Gateway
2. **Gateway 路由** → 转发至 Agent 或触发 Agent 编排
3. **模型流式返回** → Gateway 以事件形式把 token/进度广播到同一 Session 的所有订阅者
4. **WebChat + Mobile 同步展示** → 同一条流式回复，状态在 Gateway 统一生产，客户端只渲染

**案例二：WebChat 下发浏览器自动化**

1. **用户在 WebChat 输入**："帮我登录并截图订单页"
2. **Gateway 判断需要 `browser.*` 能力** → 调度 Browser Node（Puppeteer/Playwright）
3. **Browser Node 执行** → 持续上报步骤事件、截图、DOM 变化、失败原因
4. **Gateway 推送截图/进度** → 所有相关端（包括手机端）同步看到同一张截图、同一进度条
5. **可中断**：手机端发"停止" → Gateway 接收后下发中断指令，Browser Node 停止执行

### 📝 避坑指南

- ⚠️ **不要把控制平面做成"又一个业务服务"**：Gateway 只做调度与一致性，一旦开始在里面写业务逻辑，它就会变成新的巨无霸，失去统一控制平面的意义。
- ⚠️ **能力声明必须与实际权限对齐**：节点声明了 `location` 但系统权限未授予，Gateway 路由成功但执行失败，排查成本极高。
- ⚠️ **移动端长连接不稳定是必然，不是 bug**：Wi-Fi/4G/5G 切换、后台挂起都会断连，必须在设计阶段就加入自适应重连 + 离线消息队列，而不是上线后打补丁。
- ⚠️ **优先级队列不做会导致取消指令被堵塞**：中断/取消消息必须比普通消息有更高优先级，否则任务跑飞了取消指令还在队列里排队。
- ⚠️ **可观测性不是后期再加的事**：多端协作出问题时，没有全链路 tracing 和统一日志，根本无法定位是哪个节点、哪个事件、哪一步出了问题。
- ⚠️ **消息 id 必须全局唯一**：用于 ACK 确认和去重，重复消息在多端广播场景下会造成状态错乱。
- ⚠️ **协议命名空间要从第一天就规范**：`chat.*`/`agent.*`/`node.*` 等分组一旦混乱，后续扩展新端时会产生大量历史包袱。

### 📋 核心分析

**战略价值**: PingCAP CTO 东旭的第一手实战洞察——AI Coding 已过奇点，Clawdbot 代表的「本地全能 Agent」是下一代操作系统雏形，26 年应用层将迎来大爆发。

**核心逻辑**:

- **AI Coding 评分演进**：25 年初不及格（幻觉频发）→ 60 分（局部能干好）→ 80 分（大模块自主架构）→ 当前接近 90 分：10 万行以内项目，人类只需提需求+验证结果，AI 产出优于 Senior Engineering 团队，且快得多。东旭本人每天消耗近 Billion 级 Token，产出几万行代码，相当于几十人一个月的工作量。

  1. 长上下文信息召回率：从 GPT-5.1 Thinking 模式的 ~30%，跃升至 GPT-5.2 的接近 90%。召回率低时，三轮 Agent 交互后幻觉严重；90% 召回率使复杂项目整体准确率大幅提升。
  2. Context Engineering 最佳实践：Claude Code 主导 System Prompt + Tool Use 编排，社区验证后快速反哺新版本。

- **Skills vs MCP 的核心差异**：

| 维度 | Skills | MCP |
|------|--------|-----|
| 交互方式 | 自然语言（对 LLM 和人类最友好） | 结构化调用 |
| 组合性 | 可叠加，简单 Skills 组合出复杂 Skills | 几乎无法组合 |
| 上手难度 | 可让 Coding Agent 现场生成 | 配置门槛高 |
| 多挂载问题 | 无 | 挂太多易触发灾难性遗忘 |
| 环境依赖 | 低 | 强依赖远端服务器环境 |

- **Clawdbot 的本质**：Claude Code 级别的编程 Agent + Computer Use 能力 + 无权限限制 + 本地记忆系统。底层架构猜想：内核是编程 Agent → 中间层是 Skills → 外层是语音/Chatbot Interface，即「未来操作系统」雏形。

- **本地 vs 云端的关键差异**：核心需求是「完全可控的执行环境」。本地优势是安全（随时关机/拔网线）；云端配置门槛极高，且敏感信息暴露风险大。Clawdbot vs Manus：前者开源+本地优先+内置 Skills，极客圈传播快；后者云端+付费墙，C 端体验差异不大，但 Manus 无法直接控制本机。

- **moltbook 的涌现效应**：所有 Clawdbot 用户可将 Bot 接入，Bots 互相交流最佳实践、分享奇技淫巧。东旭的 Bot 在其睡觉时自主学习新技巧并改进工作流。Andrej Karpathy 已转发关注。这是 Multi-Agent 协作能力的早期形态。

- **Box 概念**（Skills 的基础设施延伸）：Skills 有副作用（执行后环境被污染，难以稳定复用）。Box = Skills + 完整执行环境（虚拟机+浏览器+上下文）绑定打包。每次调用拿到的是「菜谱 + 干净厨房」，副作用被隔离，原子能力可反复调用并组合。典型场景：登录 Box + 下单 Box 组合 → 在外卖 App 买咖啡，全程无需现成 API。

- **Token 消耗的经济学瓶颈**：当前 AI 面临的不是能力问题，是 ROI 问题。东旭敢每天烧 Billion 级 Token，因为产出收益远超成本。但普通用户无法将算力转化为高价值结果，导致使用意愿受限。26 年应用层爆发 + AI Coding/多模态对 Token 消耗远高于早期 Chatbot，全球算力已不够用，叙事回归「堆算力堆卡」。

### 🎯 关键洞察

**未来软件开发的两极分化**：
- 100% 人类手写（代码当艺术品）
- 100% AI 执行（人介入反而是添堵）
- 中间地带会消失

**未来组织形态**：一位 Senior Engineer 负责一个模块，下带一群 Agents。核心能力是「判断力」——在 Agent 给出的多个方向里选对路，知道何时触发重构。模块间、Agents 间最好无交互（协作引入问题，边界模糊，速度不匹配）。协作的前提是清晰边界 + 共同使命，后者在未来组织里权重极高。

**创业方向的两极**：
- 极致人味：AI 做不到的物理世界连接（山村图书馆数字借阅系统、八线城市小超市线上下单）。长尾需求 + 个体强正反馈。
- 极致效率：用 AI 把效率提升数量级（Clawdbot 路线）。
- 夹在中间的方向越来越难做。

**编程的未来赛道类比景德镇瓷器**：工业制造（AI）已能满足刚需，但人们仍会为「老师傅窑里烧出的瓷器」付溢价，因为有对美的追求。编程将分化为：AI 完成刚需 / 竞技+艺术+审美赛道 / Builders（成就感来自创造本身而非写代码）。

### 🛠️ 操作流程（Clawdbot 重度使用 SOP）

1. **环境准备**: 本地机器（Mac mini 或同等配置），给予 Clawdbot 完整电脑控制权限，配置 Telegram 作为指令下发入口。

2. **记忆系统初始化**: 建立长期记忆模块，将高频需求固化为 Routine（如邮件处理、定时任务）。

3. **Skills 构建**: 无现成 Skill 时，直接让 Coding Agent 现场生成。遇到新能力需求（如语音转文本），Bot 会自主实现并固化进工具箱。

4. **Box 封装（副作用隔离）**: 将有副作用的 Skills 封装为 Box（绑定虚拟机+浏览器+环境），确保每次调用环境干净，支持组合调用。

5. **接入外部工作流**: 将 Bot 接入券商账户、邮件系统、签证表单等，设定触发条件（如每 2 小时检查邮件）。

6. **moltbook 社区学习**: 将 Bot 接入 moltbook，让其自主学习其他 Bot 的最佳实践，在后台（包括睡眠时间）自主改进工作流。

7. **验证与迭代**: 人类只需验证结果，不介入执行过程。

**来源**: Seedance2.0 重塑AI短剧的生产范式 | **时间**: 2026-03-13 | **原文链接**: `https://www.woshipm.com/ai/6342108....`

### 📋 核心分析

**战略价值**: Seedance 2.0 以统一多模态架构终结了 AI 短剧"工业组装"时代，将原本需要 4 人团队+多平台协作的制作流程压缩为单人单平台完成，彻底重写内容生产成本结构。

**核心逻辑**:

- **传统流程人力配置不可压缩**：单集 AI 短剧最少需要导演+编剧+执行导演+动画师四个角色，规模化生产还需多个执行导演和动画师团队并行，人力与沟通成本极高。
- **剧本拆解依赖多模型横跳**：用 LLM 提取角色/场景/道具清单，测试下来 Claude Sonnet 4.5 拆解最全面、角色人设提取最完整；GPT 5.2、Gemini 3.0 Pro、Qwen 3 Max 表现也可接受。
- **视觉设定无统一平台**：Midjourney 做场景概念图更精细，即梦做人物角色更符合国人审美，nano banana 做打斗场景更稳定——三个平台各有专长，但拼接效率极低，且高度依赖主观判断。
- **角色一致性是最大技术瓶颈**：需为每个主要角色生成正面、侧面、四分之三侧面等多角度图集；追求高一致性的团队还需训练专属 LoRA 模型，门槛高、耗时长、效果不稳定。
- **视频生成是"开盲盒"环节**：三种主流方式（单图生成、多帧首尾帧/序列帧、多主体参考生成）各有缺陷——单图质量不稳定，多帧成本最高，多主体参考生成对复杂指令理解不精准，均需多次重跑。
- **后期修补涉及工具链碎片化**：AI 补帧（提升帧率）、AI 超分辨率（提升画质）、AI 对口型（修正口型）三类工具分散，无整合平台，最终还需在传统剪辑软件完成合成调色。
- **Seedance 2.0 核心突破是统一多模态架构**：同时理解文字、图片、视频、音频，用"@"符号直接引用素材，实现"所见即所得"式导演级指令，单指令可生成包含多场景、多镜头的连续视频，最长支持 15s。
- **双分支扩散变换器实现真正音画同步**：可用一段音频驱动视频节奏和情绪，画面与声音在生成阶段即完成融合，不再依赖后期对轨。
- **物理引擎升级减少人工修正量**：更准确模拟重力、碰撞、流体等物理规律，直出画面的反物理诡异感大幅降低，官方 case 均为未经剪辑的直出成片。
- **安全红线已落地**：Seedance 2.0 当前不支持上传任何含写实人脸的素材，原因是生成质量已达到难以辨别真假的程度。

### 🎯 关键洞察

**为什么传统流程必然被淘汰**：

传统 AI 短剧的核心矛盾是"分镜和故事创作是串联整个短剧的核心，但这两件事需要大量前置筹备"。具体路径是：

> 剧本拆解（LLM）→ 设定生成（多平台）→ 多角度资产图集（手动/LoRA）→ 分镜图生成（反复调整）→ 视频生成（多次重跑）→ AI 专项优化（补帧/超分/对口型）→ 剪辑合成（传统软件）

每个箭头之间都是人工衔接，每个节点都有独立的失败概率。创作者扮演的是"项目经理"而非"导演"，大部分精力消耗在技术实现和流程管控，而非创意本身。

**Seedance 2.0 的范式转移逻辑**：

原因 → 统一多模态架构让模型在生成阶段即完成多信息融合，不再需要人工在工具间传递素材。
动作 → 创作者只需在单一平台用"@引用+自然语言描述"完成全流程指令。
结果 → 4 人团队压缩为 1 人，多平台工具链压缩为单平台，制作成本结构性下降。

   - 多视频融合：写 `我要在@视频1和@视频2之间加一个场景，内容为xxx`
   - 连续动作：写 `角色从跳跃直接过渡到翻滚，保持动作连贯流畅 @图1@图2@图3`
   - 无音频素材时：可直接引用参考视频里的声音

3. **验证与优化**
   - 官方 case 均为直出未剪辑，以此为质量基准判断是否需要重跑
   - 充分利用首尾帧+视频延长组合，最大化 15s 上限的利用率
   - 当前禁止上传含写实人脸素材，设计阶段即需规避

### 📝 避坑指南

- ⚠️ **视频延长时长选择错误**：延长操作中，生成时长应填写"新增部分"的时长，而非目标总时长。延长5s就填5s。
- ⚠️ **写实人脸素材上传被拒**：Seedance 2.0 当前明确不支持上传任何含写实人脸的素材，前期素材准备阶段需完全规避，否则流程中断。
- ⚠️ **火山方舟 API 时间节点**：API 接入在 2026-02-24 18:00 后才开放，此前只能用体验中心，无法程序化调用。
- ⚠️ **传统 LoRA 方案已过时**：为角色一致性训练 LoRA 的路线，在 Seedance 2.0 的@引用多模态方案面前性价比极低，新项目不建议投入。
- ⚠️ **多主体参考生成的不稳定性**：即便在 Seedance 2.0 中，复杂多主体指令仍可能需要多次尝试，提示词需尽量明确每个@素材的具体作用。
