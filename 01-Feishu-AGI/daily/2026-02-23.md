### 📋 核心分析

**战略价值**: Alexander Embiricos（OpenAI Codex PM）系统阐述了 AI 编程从"辅助补全"到"全权委托"的拐点机制、OpenAI 的通用 Agent 终局战略，以及人类作为 AGI 瓶颈的产品化解法。

**核心逻辑**:

- **拐点已发生，不可逆**：GPT-5.2 Codex（2025年12月）是分水岭——之前是 tab 补全/配对编程，之后是"讨论 spec → 放手让 Agent 跑"。OpenAI 内部大多数人已不打开 IDE，代码绝大部分由 AI 写。2025年8月后增长20倍，12月后再翻倍。
- **瓶颈不在模型，在人机交互**：重度用户每天用 AI 30次，但 Embiricos 认为应该是"数万次"。即使他自己也"太懒"去发现所有 AI 能帮忙的场景。解法不是更强的模型，而是 AI 自动连接上下文、在合适时机主动介入，用户不需要学 prompt。
- **三阶段路线图**：① 先在编码领域做好（LLM 天然擅长）→ ② 意识到 Agent 操作计算机的最佳方式就是代码（"所有 Agent 本质上都是编码 Agent"）→ ③ 观察有效模式后做高度产品化功能。"接下来几个月内快速跑完三个阶段"。
- **委托式界面取代配对式界面**：Codex App（2026年2月2日，macOS，首周百万下载）专为"委托"设计——多 Agent 管理、变更审查、Skills 扩展，**刻意没有文本编辑功能**。Plan Mode：Agent 先只读研究代码库，提出方案，确认后再执行（类比新员工提 RFC）。
- **代码审查已自动化**：OpenAI 几乎所有代码推送到 repo 时都被 Codex 自动审查。Codex 被专门训练为**低误报率审查者**，批评大多有价值，可信任。有人用 Codex 审查其他模型的代码后，意识到"应该直接用 Codex 写"。
- **开放标准是长期生态战略**：Codex 核心 harness 开源；推动 `agents.md`（非 `codex.md`）、`agents/` 文件夹（非 `codex/`）成为跨工具标准。已有 6 万+ 开源项目采用，GitHub Copilot 支持，由 Linux 基金会旗下 Agentic AI Foundation 管理。唯一缺席：Anthropic Claude Code（坚持用 `CLAUDE.md`）。
- **通用 Agent 会打败垂直 Agent**：过去一年只有编码 Agent 真正跑通 PMF，其他领域还很初期。但终局是：企业不会配 12 个 Agent，只会配 1 个——因为员工对单一 Agent 形成肌肉记忆后，才会主动把自动化拉入工作流。ChatGPT 已是通用入口，天然有优势。类比：Dropbox 协作功能比 Slack 更高效，但人们涌向 Slack，因为引力效应更强。
- **自下而上 > 自上而下**：企业自上而下部署 AI 的结果是"严重低估 AI 潜力"——员工没用过 ChatGPT，对 AI 没直觉，面对自动化只会无力。正确路径：先让个人用上工具建立直觉，自动化自然被用户"拉入"工作流，而不是被"推给"用户。
- **编程数据够了，知识工作数据才稀缺**：GitHub 海量开源代码提供了天然训练宝库，编码数据不是瓶颈。真正缺的是知识工作数据——没人把处理邮件、准备会议、做决策的完整过程记录到网上。获取路径：① 付钱请人模拟任务并记录操作轨迹；② 收购积累了大量工作流数据的停运创业公司（如某个 Slack 类产品）。
- **SaaS 生死判断标准**：拥有"跟人类的关系"或"核心记录系统（system of record）"的 SaaS 不会死，这两样东西在 AI 时代反而更重要。危险的是"胶水层"公司——只做系统间连接和数据搬运，既不拥有客户关系也不拥有核心数据，正好是 Agent 最擅长替代的。

### 🎯 关键洞察

**为什么"所有 Agent 都是编码 Agent"这个判断是核心**

逻辑链：Agent 要有用 → Agent 需要操作计算机 → 操作计算机最可靠的方式是写代码（而非模拟鼠标点击）→ 因此编码能力是所有 Agent 的底层基础设施。

这意味着 Codex 不是"编程工具"，而是"通用 Agent 的入口"。这解释了为什么 OpenAI 在 Codex 上投入如此之重，以及为什么他们要做 Atlas 浏览器——浏览器是知识工作的另一个操作层，跟代码执行层互补。

**为什么 Dropbox 教训直接决定了 Codex 的产品策略**

Embiricos 亲历了 Dropbox 协作功能被 Slack 碾压的全过程。结论：**效率不是用户选择工具的第一标准，"喜欢用"才是**。

应用到 Codex：不做"自上而下的 workflow 自动化"（需要 Accenture 进场、部署实施工程师，慢且贵），而是做"人们真心喜欢用的工具"。用户用着用着，工具自然连接更多系统，自动化被用户自己拉进来。

**为什么 Plan Review 比 Code Review 更重要**

代码生成已经"基本上变得不值一提"，真正的质量控制窗口在执行前。Codex Plan Mode 的设计逻辑：Agent 以只读方式研究代码库 → 提出详细方案 → 人类确认 → 执行。这跟新员工提 RFC 的机制一样，在成本最低的阶段介入纠偏。

### 🛠️ 操作流程

**如何在团队中复制 OpenAI 内部的 Codex 工作方式**

1. **准备阶段**
   - 在项目根目录创建 `agents.md`，写入代码库约定、架构说明、常见任务模式
   - 在 `agents/` 文件夹下放置 Skills 脚本（常用操作的封装）
   - 安装 Codex App（macOS），配置多 Agent 并行工作环境
   - 确认所有 CI/CD 流程支持 Codex 自动审查钩子

2. **核心执行（委托式工作流）**
   - 启动任务前：用 Plan Mode，让 Agent 只读研究代码库，输出执行方案
   - 审查 plan（这是最重要的介入点），确认方向后放手执行
   - 开会时保持 Codex 在后台并行跑任务（"如果开会时它没在跑任务，就是在浪费时间"）
   - 代码推送前：触发 Codex 自动审查，处理有价值的批评（低误报，可信任）

3. **验证与优化**
   - 核心指标从 WAU 转向 DAU（Embiricos 当场认可了这个修正）
   - 目标状态：团队成员"有任何事都先问 Agent"，形成肌肉记忆
   - 定期办内部黑客马拉松，探索 Agent 新用法并在团队间分享
   - 逐步让 Agent 连接外部系统（Sentry、Google Docs 等），粘性会随之大幅增加

**Dropbox vs Slack 协作功能之争（直接影响 Codex 产品策略）**

Dropbox 协作功能：可在视频精确时间戳留言、在文档特定段落标注，信息与内容绑定，上下文完整。
Slack：只能 @ 人，信息脱离内容，效率更低。
结果：用户全部涌向 Slack。
原因：Slack 是交流重心，引力效应碾压效率优势。
Codex 应用：不做"推给用户的自动化"，做"用户喜欢用的工具"，让自动化被用户自己拉进来。

### 📝 避坑指南

- ⚠️ **不能让东西免费太久**：Codex Cloud 曾提供无限使用（仅需 ChatGPT 订阅），后改为有限额度，引发激烈用户反弹。社交媒体噪音影响了所有渠道。教训：定价策略要从一开始就设计好，不要用"无限"来获客再收紧。
- ⚠️ **自上而下部署 AI 会严重低估 AI 潜力**：如果员工从未用过 ChatGPT，直接给他们一套自动化系统，他们对 AI 没有直觉，只会感到无力。正确顺序：先让个人用上工具，再谈工作流自动化。
- ⚠️ **云端产品时机不对会搭空中楼阁**：Codex 最初做云端 Agent（给 Agent 自己的云端电脑），但用户还没熟悉工具就直接做工作流自动化，效果很差。要先建立用户基础，再整合云端能力。
- ⚠️ **企业侧不能只说"我们来了，随便用吧"**：需要大量教育、配置支持、跟开发者体验负责人对接。这是 Embiricos 承认的教训。
- ⚠️ **垂直 Agent 的粘性陷阱**：企业配 12 个垂直 Agent，员工对任何一个都不会达到"熟练"状态，不熟练就不会主动把自动化拉入工作流。应该押注通用 Agent 而非垂直 Agent。
- ⚠️ **多模态 Agent 进展比预期慢**：Embiricos 加入 OpenAI 时预期很快会有视频/音频多模态 Agent，结果进展慢于预期。通过代码操作计算机才是当前正道，不要过度押注多模态交互。
- ⚠️ **代码审查是被低估的瓶颈**：代码生成已经"不值一提"，但代码质量和方向判断才是真正难题。不要只关注生成速度，要同等重视审查和质量控制流程。

### 📋 核心分析

**战略价值**: 通过解构 Awesome OpenClaw Skills 的筛选逻辑与 28 类生态分布，提供一份可直接用于 Agent 工具选型、生态布局判断的导航地图。

**核心逻辑**:

- **筛选基数与排除率**: ClawHub 原始 5705 个 Skills → 精选 3002 个，排除 2748 个（排除率 48%），主动筛选而非放任自流，质量优先于数量。
- **垃圾内容是最大噪音源**: 1180 个（占排除总量 43%）为批量账户测试 Skills、未发布开发代码、功能重复的反复提交版本，是任何开源生态的核心噪音。
- **加密/金融类被整体清除**: 672 个（占排除量 24%），包括虚拟货币、区块链、金融交易、投资工具，原因不是技术问题，而是 AI Agent 可自主执行操作时金融工具天然带有更高法律与道德责任风险，策略是规避而非管理。
- **重复 Skills 合并保留最优版**: 492 个（18%）被合并或淘汰，用户无需在十个 GitHub 集成工具中自行判断，最优版本已被标识。
- **恶意代码经研究人员验证后排除**: 396 个（14%）通过安全审计发现含恶意代码或后门，OpenClaw 与 VirusTotal 有官方合作，每个 Skill 页面可查安全报告，排除依据是研究人员验证结果而非自动化扫描。
- **AI & LLMs 是规模最大单一类别**: 287 个，比第二大类多 100+，内部细分为模型集成、推理增强、多模型路由、记忆系统、Agent 编排、自进化引擎六大方向，反映 AI 工程正快速分化为多个专业子领域。
- **自进化系统是最激进方向**: `evolver`（自进化引擎）、`ralph-evolver`（递归自改进）、`ralph-mode`（自主开发循环，带反压力门）——"带反压力门"细节说明开发者已意识到无限制自进化风险并在设计安全护栏。
- **Moltbook 构建的是 Agent 虚拟社会而非工具**: `moltbook`（社交网络）→ `moltland`（像素 Metaverse，3x3 地块所有权）→ `moltpet`（宠物养成）→ `molt-trust`（信誉分析）→ `moltoverflow`（Agent 版 Stack Overflow），这是完整虚拟经济体系，不是比喻。
- **安全生态已成体系**: `flaw0`（OpenClaw 代码/插件/Skills 漏洞扫描）、`openguardrails`（检测长文本中的提示注入攻击）、`clawsec-suite`（ClawSec 浏览/设置）、`secure-install`（通过 ClawDex API 扫描 Skills），说明社区已将安全防御内化为基础设施。
- **分类系统按用户心智模型而非技术实现设计**: 28 个类别对应"我需要解决什么问题"而非"这个工具用了什么技术"，降低发现成本。

### 🎯 关键洞察

**双轨制演化是核心战略结构**

生态系统沿两条轨道并行演化，互补而非竞争：

- 实用工具轨（GitHub 集成、云部署、数据库管理、浏览器自动化）→ 短期价值 + 现金流
- 虚拟社会轨（Moltbook 社交、Agent 约会、虚拟宠物、数字身份）→ 长期护城河 + 生态锁定

**智能路由将成必需基础设施**

`smart-model-switching`（按成本自动选最便宜 Claude 模型）、`smart-router`（按语义领域评分选专业模型）、`relayplane`（智能模型路由代理）——当可用模型从几个增长到几十上百个时，手动选择不可行，路由系统从可选变为必需。

**Agent 身份与连续性是深层未解问题**

`agent-identity-kit`（便携式 AI Agent 身份系统）、`identity-manager`（Agent 身份映射管理）、`moltbook-registry`（官方身份注册表）、`molt-life-kernel`（管理 Agent 的"连续性和认知健康"）——这些工具背后的假设是：Agent 需要跨平台、跨时间保持的持久数字人格，而非临时会话 ID。

**开发过程可视化是 AI 时代的新需求**

传统版本控制记录代码变化，不记录思考过程。`buildlog`（回放 AI 编程会话）、`vhs-recorder`（专业终端录制）在探索：当 AI 成为开发团队成员时，如何记录和重现 AI 的推理过程与决策。

### 📋 核心分析

**战略价值**: OpenClaw 是一个开源 AI Agent 框架，通过本地部署、持久记忆、主动工作三大能力，将 AI 从"对话工具"升级为"24/7 数字员工"，AlexFinn 用它实现了月经常性收入超 1 万美元。

**核心逻辑**:

- **本地部署 > VPS 部署**：VPS 只能发挥 OpenClaw 约 20% 的能力，本地部署才能访问文件系统、剪贴板、桌面应用等高级功能，且安装只需一行命令。
- **持久记忆机制**：OpenClaw 永久记住你的偏好、目标、工作习惯。告诉它"我用 Codex 编程"或"我住加州"，此后所有决策都基于这些上下文。

- **自我改进能力**：当 AlexFinn 发现 OpenClaw 遗忘细节时，直接问"怎样确保你不再忘记任何事"，OpenClaw 随即为自己构建了全新记忆系统。
- **大脑+肌肉架构**：Claude Opus 4.6 作为"大脑"负责决策，Codex 负责编程、XAI(Grok) 负责抓取实时推文趋势、Brave API 负责网络搜索，分工降低 token 成本。

- **安全三原则**：OpenClaw 拥有管理员权限，可访问电脑上所有登录账户；不要暴露给外界（防提示注入攻击）；每条指令执行前深思后果，可要求它先提交分步计划再行动。

### 🎯 关键洞察

**为什么 99% 的人用错了 OpenClaw**

根本原因是认知框架错误：把 OpenClaw 当 ChatGPT 用（被动响应），而不是当员工培养（主动执行）。

逻辑链：
- 错误路径：用户输入指令 → AI 响应 → 用户再输入 → 循环往复（本质还是工具）
- 正确路径：一次性信息倾倒（背景+偏好+目标）→ OpenClaw 建立持久上下文 → 主动执行+自我改进 → 用户只需审核结果

**为什么本地部署远优于 VPS**

VPS 部署的本质是把 OpenClaw 关在一个沙盒里，它无法触及你真实的工作环境。本地部署让它直接嵌入你的操作系统，才能实现"眼睛和手"的完整能力。

**反向提示的力量**

传统提示是"命令执行"，反向提示是"目标委托"。当你问"基于我们的目标，你会构建什么工具"，OpenClaw 会调用它对你的全部了解来设计方案，而不是机械执行单一指令。这是从"工具使用者"到"团队管理者"的思维跃迁。

### 📋 核心分析

**战略价值**: XSCT Bench 用「场景化评测 + 多维度评分 + 难度分层」替代传统综合榜单，帮助开发者从「谁最强」转向「谁最适合我的场景」，核心是找到性价比最优解。

**核心逻辑**:

- **榜单 Gap 本质**：MMLU/HumanEval 等标准测试集给出的是抽象能力综合分，无法映射到具体业务场景。模型 A 综合 92 分但数学强、创意弱，模型 B 综合 88 分但创意更好且便宜一半——榜单无法告诉你该选哪个。
- **场景化组织替代能力维度**：直接按「营销文案」「客服对话」「代码生成」等应用场景检索，而非按模型抽象能力分类。同一排行榜切换「综合」和「基础」维度，排名会发生变化（如 Claude 综合第一，但 Qwen3.5-plus 在基础场景反超，成本仅为 Claude 的 1/20）。
- **多 Judge 联合评分抵消偏见**：单一评审模型存在系统性偏好（如 Gemini 评分偏松，动辄 90+）。采用三模型加权：Claude（anthropic/claude-sonnet-4.6）权重 50%、Gemini（google/gemini-3-Pro-preview）权重 30%、Kimi（moonshot-kimi-K2.5）权重 20%。某 Judge 失败时自动重新归一化剩余权重。
- **多维度独立评分替代总分**：AI 评审员只对每个维度独立打 0-100 分，不给总分；总分由系统按预设权重加权计算。同样是 75 分，「样样及格」和「某维度满分但另一维度极差」在明细中一目了然。LLM-Rubric 研究证明此方法可将预测误差降低 2 倍以上。
- **证据锚定防止幻觉评分**：评分 Prompt 强制要求引用被评内容中的具体文本作为依据，每个维度扣分必须指出具体缺陷位置，形成可追溯证据链。研究发现无约束评审 LLM 在超过半数情况下无法正确识别质量下降。
- **难度三分层暴露真实天花板**：Basic（宽松约束，建立基线）→ Medium（增加约束复杂度，拉开差距）→ Hard（针对已知弱点极限约束，暴露天花板）。Arena-Hard 研究表明精心设计的高难度测试集可提供 3 倍于传统基准的模型区分度。实测「写诗」场景：Basic 档所有模型 90 分上下无法区分；Hard 档（每行正好十个汉字 + 回文写作 + 倒序输出）只有头部模型维持高分。
- **评分与被测分离防止应试行为**：被测模型只接收 system_prompt + user_prompt，不含任何评分维度和权重信息；评分标准只传递给 AI 评审模型，避免模型针对评分维度堆砌关键词导致分数虚高。
- **网页生成双轨评分**：代码逻辑评分 × 视觉截图评分各占 50%。流程：模型生成 HTML → 无头浏览器渲染为 960×600 截图压缩为 JPEG → 传给 AI 做多模态视觉评分，与代码评分加权合并。纯代码评分无法感知「结构完整但渲染出纯白底黑字无设计感」的页面。
- **场景推荐指数区分用户类型**：三种难度的简单平均会抹平「Basic 强但 Hard 差」的差异，因此设计三套权重方案：日常场景（Basic 60% / Medium 30% / Hard 10%）、专业场景（Basic 20% / Medium 50% / Hard 30%）、极限场景（Basic 10% / Medium 30% / Hard 60%）。同一批数据，不同场景下排名不同。
- **能力天花板判定**：某难度均分 ≥ 60 视为「通过」该难度，最高通过的难度即为能力天花板。天花板为 Medium 的模型在极限场景下质量不稳定。

### 🎯 关键洞察

**为什么 LLM-as-a-Judge 是主流但需要修正**：伯克利大学研究表明强大 LLM 评审（如 GPT-4）与人类偏好一致率可达 80% 以上，接近人类评审员之间的一致率。但原始方案存在三类失效：①同公司模型互评存在隐性偏好；②无约束评审超过半数情况下无法识别质量下降；③单一总分无法解释扣分原因。XSCT Bench 的六策略本质是在 LLM-as-a-Judge 框架上打补丁，每个策略对应一个具体失效模式。

**模型已知失效模式（Hard 难度专门针对这些设计）**：
- 长链推理衰减：多步骤推理后期准确率随推理链延长显著下降
- 自我纠错失败：被告知答案有误后难以正确识别并修正
- 复杂约束处理：多个相互作用约束同时存在时顾此失彼
- 一致性崩溃：长文本或多轮对话中早期承诺与后期输出矛盾

**区分「模型问题」vs「Prompt 问题」的方法**：设计「能力边界」用例横向对比。例如定向数数测试（图像生成精确数量物品，从 2 个数到 10 个）：基础场景（4 个红苹果 + 3 个三明治 + 2 瓶矿泉水）只有 Nano Banana 做到，其他模型普遍出错。通过横向对比可确认是模型能力天花板问题，而非 Prompt 问题，再决定是否值得继续优化 Prompt。

### 🛠️ 操作流程

**Step 1：单个测试用例评分计算**
- 各维度独立打分，按预设权重加权
- 示例：creativity 85分×40% + coherence 78分×30% + language_style 82分×30% = **82.0分**

**Step 2：计算各难度平均分**
- Basic 两用例 82.0 + 78.0，平均 **80.0**
- Medium 两用例 71.0 + 69.0，平均 **70.0**
- Hard 两用例 52.0 + 48.0，平均 **50.0**

**Step 3：计算场景推荐指数**
- 日常场景分 = 80×0.6 + 70×0.3 + 50×0.1 = **74.0**
- 专业场景分 = 80×0.2 + 70×0.5 + 50×0.3 = **66.0**
- 极限场景分 = 80×0.1 + 70×0.3 + 50×0.6 = **59.0**

**Step 4：能力天花板判定**
- Basic ✓（80 ≥ 60）
- Medium ✓（70 ≥ 60）
- Hard ✗（50 < 60）
- 天花板 = **Medium**，结论：日常场景 74 分可用，极限场景不稳定

### 📝 避坑指南

- ⚠️ **不要用综合榜单做场景选型**：综合分高的模型不一定适合你的场景，数学强的模型对营销文案场景毫无意义。
- ⚠️ **单一 Judge 评分不可信**：Gemini 等模型评分偏松，单独使用会导致分数虚高，必须多模型加权。
- ⚠️ **不给 AI 评审员约束会产生幻觉评分**：无约束评审超过半数情况下无法识别质量下降，必须要求引用具体文本作为评分依据。
- ⚠️ **被测模型不能看到评分维度**：否则模型会针对评分维度堆砌关键词，分数虚高但实际效果差。
- ⚠️ **网页评测不能只看代码**：结构完整的 HTML 可能渲染出纯白底黑字无设计感的页面，必须加入视觉截图评分。
- ⚠️ **三难度简单平均会掩盖天花板**：「Basic 强 Hard 差」的模型均分看起来还行，但放到极限场景会崩，必须用场景推荐指数区分。
- ⚠️ **先确认模型能力天花板，再优化 Prompt**：效果差先横向对比其他模型，确认是模型问题还是 Prompt 问题，避免在无法突破的天花板上浪费时间。

**范式转换的本质**：从"Software-as-a-Service"→"Service-as-Software"。

### 📋 核心分析

**战略价值**: 把 Codex CLI 接入零轮询模式，再用 GitHub Actions + Self-hosted Runner 打通「push 代码 → 自动审计 → 飞书通知」全链路，实现真正的无人值守 Agent Team。

**核心逻辑**:

- Codex CLI 没有内置 Hook，用 shell `&&` 串联 `codex exec` 和回调脚本来模拟，效果与 Claude Code 内置 Hook 完全等价
- `nohup bash -c "codex exec ... && on-codex-complete.sh" &` 是核心范式：后台启动、立即返回、完成后自动回调
- Codex 有两种模式：`--full-auto` 做审计+修复（会写文件），`--review --base main` 做只读审查（不改代码）
- Claude Code 负责复杂多文件开发，Codex CLI 负责安全审计/快速修复，两者分工明确，不要混用
- Self-hosted Runner 跑在 Mac Mini 上的好处：7x24 在线、直接访问本地 CLI 工具、不需要暴露公网
- GitHub Actions Workflow 用 `paths` 过滤器只在 `backend/` 或 `frontend/src/` 有变更时才触发，避免无效审计
- `git diff --name-only HEAD~1..HEAD` 收集变更文件列表，最多取前 50 个，作为上下文传给 OpenClaw
- OpenClaw（大副）收到 `[AUTO-AUDIT]` 消息后派发给 Forge（CTO Agent），Forge 分析变更范围决定用 review 还是 exec 模式
- Forge 的 Skill 文件 `SKILL.md` 明确写「立即结束 session，不等待」，回调由 `on-codex-complete.sh` 负责通知
- Token 消耗从手动触发的 15,000–50,000/次 降到自动化的 ~700/次，覆盖率从「想起来才审计」变成「每次 push 都审计」

---

### 🎯 关键洞察

**为什么 Codex 零轮询要用 shell wrapper 而不是其他方案？**

Codex CLI 是一个标准的命令行进程，退出码即状态。shell 的 `&&` 天然就是「前一个命令成功才执行下一个」的语义，配合 `nohup ... &` 后台化，完美复刻了 Claude Code Hook 的行为，零依赖、零配置。

**为什么 Runner 不能放在 iCloud 同步目录？**

iCloud 会在文件被写入时触发同步，Codex 大量写文件时会产生冲突和锁，导致审计失败。必须用 `~/repos/` 这类普通目录。

**三层架构的本质**：触发层（GitHub Actions）只负责感知变更，决策层（OpenClaw → Forge）只负责分析和派发，执行层（Codex CLI）只负责干活。每层职责单一，任何一层出问题都不影响其他层。

**2. 核心执行：在 Mac Mini 上注册 Self-hosted Runner**

```bash
# 克隆项目到非 iCloud 目录
mkdir -p ~/repos
cd ~/repos
git clone https://github.com/你的用户名/你的项目.git

# 创建 Runner 目录
mkdir -p ~/actions-runner && cd ~/actions-runner

# 去 GitHub 仓库 → Settings → Actions → Runners → New self-hosted runner
# 按页面指引下载 Runner 包，然后注册：

  --token YOUR_TOKEN \
  --name "mac-mini" \
  --labels "self-hosted,macOS,ARM64,audit" \
  --unattended

# 安装为系统服务（开机自启）
./svc.sh install
./svc.sh start
```

**3. 创建 GitHub Actions Workflow**

在项目根目录创建 `.github/workflows/audit.yml`：

```yaml
name: Auto Audit

on:
  push:
    branches: [main]
    paths:
      - 'backend/**'        # ← 改成你的后端路径
      - 'frontend/src/**'   # ← 改成你的前端路径

concurrency:
  group: audit-${{ github.repository }}
  cancel-in-progress: false

jobs:
  trigger-audit:
    runs-on: [self-hosted, audit]
    timeout-minutes: 5

    steps:
      - uses: actions/checkout@v4
        with:
          fetch-depth: 0

      - name: 收集变更文件
        id: changes
        run: |
          FILES=$(git diff --name-only HEAD~1..HEAD -- backend/ frontend/src/ | head -50)
          if [ -z "$FILES" ]; then
            echo "skip=true" >> $GITHUB_OUTPUT
          else
            echo "skip=false" >> $GITHUB_OUTPUT
            echo "files<<EOF" >> $GITHUB_OUTPUT
            echo "$FILES" >> $GITHUB_OUTPUT
            echo "EOF" >> $GITHUB_OUTPUT
          fi

      - name: 通知 OpenClaw 审计
        if: steps.changes.outputs.skip != 'true'
        env:
          NO_PROXY: localhost,127.0.0.1
        run: |
          openclaw agent \
            --agent main \
            --message "[AUTO-AUDIT] 代码变更，请派发审计。
          变更文件：
          ${{ steps.changes.outputs.files }}
          仓库路径：$HOME/repos/你的项目名" \
            --json
```

**4. 配置 Forge 的审计 Skill**

创建 `workspace-forge/skills/codex-audit-dispatcher/SKILL.md`：

```markdown
# Codex 审计派发器（自动模式）

收到审计任务后：
1. 分析变更文件列表
2. 决定用 review 模式还是 exec 模式
3. 调用 dispatch-codex.sh 后台执行
4. 立即结束 session，不等待

完成后 on-codex-complete.sh 会自动通知。
```

### 📋 核心分析

**战略价值**: Claudeception 是一个 Claude Code 插件，通过 Hook 机制在每次会话中自动提取有价值的工作知识，写入 Skill 文件，让 Claude Code 持续学习你的工作模式，越用越懂你。

**核心逻辑**:

- **原理**：Claude Code 启动时加载所有 Skill 的名称+描述（每个约 100 token），根据当前任务语义匹配并加载对应 Skill。Claudeception 利用这套系统的"可写"特性，把会话中产生的有价值知识自动写成新 Skill 文件。
- **触发条件（5种）**：① 完成调试发现不明显解法；② 通过反复试错找到 workaround；③ 解决根因不明的报错；④ 摸索出项目特有配置方式；⑤ 完成任何需要"真正发现"才能解决的任务。
- **质量门控（4条，必须同时满足）**：① 需要实际探索才能发现（查文档就能知道的不算）；② 对未来任务有帮助（一次性特殊情况不算）；③ 有明确触发条件（能精准匹配类似场景）；④ 已验证有效（猜测方案不算）。作者原话：如果这个知识对六个月后遇到同样问题的人没帮助，就不提取。
- **description 字段是匹配精度的核心**：写 `Helps with database problems` 什么都匹配不上；写 `Fix for PrismaClientKnownRequestError in serverless` 才能精准命中。Claudeception 自动生成的描述按此标准执行。
- **学习内容不只是偏好，而是完整工作模式**：包括写作流程（选题→大纲→正文）、文件命名规范（如 `2026-02-16_AI工具_常青_Claude_标题.md`）、问题排查顺序（搜索报错→检查配置→问AI）、文件目录结构（drafts/images/data）。
- **慢热机制**：前两周几乎无感知，因为它在积累有价值的知识而非记录所有操作。第三周起开始出现"它怎么知道我要什么"的体验。
- **学术支撑**：Voyager（2023，AI游戏中自动积累技能库）、CASCADE（2024，"元技能"概念）、SEAgent（2025，AI通过试错学习软件环境）、Reflexion（2023，自我反思提升AI表现）。
- **Claudeception 本质是"元技能"**：它不只是一个 Skill，而是一个能创造新 Skill 的 Skill，对应 CASCADE 论文中"学习如何学习"的概念。
- **适用人群**：Claude Code 每天使用超过 2 小时的重度用户、有固定工作流程的人、需要统一团队风格的负责人。偶尔使用者体感不明显。
- **GitHub 信息**：作者 blader（同时是 Humanizer 作者），两个项目合计 7800+ Star，Claudeception 本身 1660 Star。

### 📝 避坑指南

- ⚠️ **装上就想看效果**：前两周完全无感知是正常现象，它在积累知识而非立即响应。至少坚持 2-3 周。
- ⚠️ **低估学习深度**：它学的不是"文件放哪个目录"这种简单偏好，而是完整工作模式（流程、命名规范、排查顺序、目录结构）。
- ⚠️ **多项目混用**：同时做多个性质不同的项目会导致模式混乱，建议为每个项目创建独立工作区，使用项目级安装。
- ⚠️ **Windows 脚本不兼容**：自带 `.sh` 脚本仅支持 Mac/Linux，Windows 需将脚本逻辑转换为 Node.js 版本，并修改 settings.json 中的 command 字段。
- ⚠️ **description 写模糊**：`Helps with database problems` 这类描述无法被精准匹配，必须写具体的报错名称、环境、触发条件。
- ⚠️ **已有 settings.json 被覆盖**：只合并 `hooks` 字段，不要整个文件替换。
- ⚠️ **工作流程频繁变动**：Claudeception 需要稳定的模式才能有效学习，工作流程经常变化会降低学习质量。
