> 文档 ID: `K577wG9OliqzMwkhDjAc8N2jnge`

**来源**: Claude Code 的"懒加载"更新：AI 终于学会了"随叫随到" | **时间**: 2026-01-15 | **原文链接**: https://mp.weixin.qq.com/s/NkUapSg4...

### 📋 核心分析

**战略价值**: Claude Code 通过 Tool Search「懒加载」机制，将 MCP 工具的上下文占用从 15 万 token 压缩至 2000 token（节省 98.7%），彻底解除大规模 MCP 工具集成的上下文瓶颈。

**核心逻辑**:

- Claude 上下文窗口约 20 万 token，相当于一本 300 页书的信息量，是 AI 单次对话的硬性上限
- 工具说明书（tool definitions）是上下文的主要消耗源之一，7 个以上 MCP 服务器的工具定义可占用 67,000+ token，即上下文的 1/3
- MCP（Model Context Protocol）是 Anthropic 推出的统一工具接入标准，类比 AI 的 App Store，社区已开发数千个 MCP 服务器
- 旧机制是「全量预加载」：每次对话开始前，所有工具定义全部注入上下文，无论本次对话是否用到
- Tool Search 触发条件：当所有 MCP 工具说明书总量**超过上下文窗口的 10%** 时，自动切换为懒加载模式
- 懒加载机制：工具不预先注入，而是维护一份「工具通讯录」，AI 按需搜索并调用，中间结果不占用主上下文
- 实测效果：同等工具集场景下，上下文占用从 150,000 token → 2,000 token，**节省 98.7%**
- 对用户无感：工具数量未超过 10% 阈值时，行为与旧版完全一致，无需任何配置
- MCP 开发者需重点维护 `server instructions` 字段，这是 AI 在懒加载模式下判断「何时调用该工具」的核心依据，以前可忽略，现在是关键
- Cloudflare 提出的「Code Mode」更激进：AI 直接写脚本调用工具（如从 Google Drive 取数据直存 Salesforce），中间结果完全绕过模型上下文，效率提升更显著

---

### 🎯 关键洞察

**为什么 10% 是触发阈值？**
原因：20 万 token 的 10% = 2 万 token，这是 Anthropic 判断工具定义开始「显著挤压」有效对话空间的临界点。低于此值，预加载的即时性优于懒加载的搜索开销；超过此值，懒加载的空间收益远大于搜索延迟成本。

**懒加载 vs 预加载的本质差异**：

| 维度 | 预加载（旧） | 懒加载 Tool Search（新） |
|------|------------|------------------------|
| 工具注入时机 | 对话开始前全量注入 | 按需搜索后单次注入 |
| 上下文占用 | 全部工具定义（如 67,000 token） | 仅当前调用工具定义（约 2,000 token） |
| 适用场景 | 工具少（< 上下文 10%） | 工具多（> 上下文 10%） |
| 用户感知 | 无需操作 | 无需操作（自动切换） |
| 开发者影响 | `server instructions` 可忽略 | `server instructions` 必须认真写 |

**Code Mode（Cloudflare 方案）的逻辑**：
传统接力赛路径：`工具A → 模型 → 工具B → 模型`（每步结果都过模型，占上下文）
Code Mode 路径：`模型写脚本 → 脚本直接执行 A→B`（中间结果完全不经过模型）
本质是把「模型作为数据中转站」变成「模型作为代码生成器」，上下文只存代码，不存数据。

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| Tool Search 自动触发 | 无需配置，系统自动检测 | 工具定义 > 上下文 10% 时自动启用 | 无法手动强制开启或关闭 |
| MCP server instructions 字段 | 在 MCP 服务器配置中完善 `server instructions` 描述 | AI 能准确判断何时调用该工具 | 描述模糊会导致工具被漏调用 |
| 多 MCP 服务器配置 | 可放心配置 7 个以上 MCP 服务器 | 不再因工具过多导致上下文爆满 | 超过阈值后搜索有微小延迟 |
| Code Mode（Cloudflare） | 让 AI 生成脚本直接串联工具 | 中间数据不过模型，上下文极度节省 | 需要工具支持脚本直接调用 |

### 🛠️ 操作流程

1. **准备阶段（MCP 开发者）**: 检查并完善每个 MCP 服务器的 `server instructions` 字段，明确描述「该工具适用于什么场景、能做什么操作」，这是懒加载模式下 AI 检索工具的唯一线索。

2. **核心执行（Claude Code 用户）**: 正常配置所有需要的 MCP 服务器，无需任何额外操作。当工具总定义量超过上下文 10%（约 2 万 token），Claude Code 自动切换 Tool Search 模式，工具按需调用。

3. **验证与优化**: 观察长对话中 AI 是否出现「忘记前文」的情况——若上下文空间被有效释放，长对话稳定性应显著提升。如发现某工具未被正确调用，优先检查该工具的 `server instructions` 描述是否足够清晰。

### 💡 具体案例/数据

- GitHub 用户实测：7 个以上 MCP 服务器，工具定义占用 **67,000+ token**，约为 20 万 token 上下文的 **1/3**
- Anthropic 官方数据：同等工具集场景，Tool Search 模式将上下文占用从 **150,000 token → 2,000 token**，节省 **98.7%**
- Cloudflare Code Mode 实测：效率提升「非常明显」（原文未给出具体数字，但方向一致）

---

### 📝 避坑指南

- ⚠️ MCP 开发者若不更新 `server instructions` 字段，在懒加载模式下工具可能被 AI 忽略，导致功能失效，这是最高优先级的适配项
- ⚠️ Tool Search 是自动触发机制，不存在手动配置入口，不要试图在配置文件里找开关
- ⚠️ 懒加载模式下，AI 每次调用工具前有一次搜索开销，对响应速度有微小影响，工具数量极多时更明显
- ⚠️ Code Mode 目前是 Cloudflare 的实践方案，并非 Claude Code 原生功能，需要工具链支持脚本直接执行

## 📗 文章 2

> 文档 ID: `FI3TwUSyDiF7kYkTqoZc79KHnRd`

**来源**: 成峰：太香了！我用Claude Skills做了个会学习的剪辑Agent，10分钟自动剪视频！ | **时间**: 2026-01-15 | **原文链接**: https://mp.weixin.qq.com/s/n2Zs-mVe...

### 📋 核心分析

**战略价值**: 用 Claude Code Skills 构建一个「自更新剪辑 Agent」，替代剪映的手工操作，实现 10 分钟自动剪 30 分钟口播视频，且越用越懂你。

**核心逻辑**:

- 剪映的根本缺陷是「无法理解语义」，导致重复片段漏识别、字幕错别字多，手工剪 20-30 分钟视频极耗时
- Agent 与剪映的本质区别：剪映 = 固定工具 + 手工操作；Agent = 自适应系统 + 自动学习，不是用更好算法替换，而是用「会自我完善的系统」替换
- Skills 拆分原则：不做一个大 skill 靠指令区分任务，而是把 5 个核心任务拆成 5 个独立 skill 放入 `.claude/skills/` 目录，原因是人需要在阶段性产出处做检查（如审查稿确认后再执行剪辑）
- 输入 `/v` 即可触发 Claude Code 列出全部 5 个可用技能菜单，选一个执行一个，无需重复说明方法论
- 自更新机制是核心竞争力：每次任务后给反馈，AI 把反馈永久写入 Skills 文件，用 10 次覆盖 80% 个人习惯，用 50 次完全符合个人需求
- 双模型分工：FunASR 负责字级别剪辑（识别口误、语气词），Whisper 负责字幕生成，两者各有优势不可互换
- 自定义词典是字幕准确率的关键杠杆：预置词典文件路径 `.claude/skills/videocut:字幕/词典.txt`，录入专有名词后 Whisper 转录时自动纠错，第一次就准确
- 审查稿格式有明确语义符号：⏸️ 表示静音片段（需删除），🔴～～ 表示重复片段（需删除），格式可按需让 AI 修改
- 整个工作流 4 步闭环：下载 Skills → 安装环境 → 剪口播（含审查+自更新） → 加字幕（含词典纠错+烧录）
- 最终效果与剪映手动剪辑结果基本一致，但全程自动化

---

### 🎯 关键洞察

**为什么「自更新」比「更好的算法」更有价值**：

传统工具的上限是固定的，再好的算法也只能逼近「通用最优」，无法逼近「你的最优」。自更新机制的逻辑链是：反馈 → 写入 Skills → 下次执行时直接应用 → 再反馈 → 再写入。每一次迭代都在缩小「通用规则」和「个人习惯」之间的 gap。这是一个复利结构，而不是线性改进。

**为什么要拆成多个独立 Skills 而不是一个大 Skill**：

人机协作的关键节点是「人的确认」。一个大 skill 会把所有步骤串成黑盒，人无法在中间介入。拆成独立 skill 后，每个 skill 是一个检查点，人可以在审查稿阶段发现问题并修正，避免错误传递到最终剪辑。

---

### 📦 配置/工具详表

| 模块/功能 | 关键设置/代码 | 预期效果 | 注意事项/坑 |
|----------|-------------|---------|-----------|
| Skills 存放路径 | `.claude/skills/` | Claude Code 重启后自动识别 | 克隆时不要创建子文件夹，直接克隆到此目录 |
| Skills 仓库地址 | `https://github.com/Ceeon/videocut-skills` | 获取全部 5 个 skill 文件 | 需要新建专用文件夹作为剪辑工作空间 |

| 自更新触发 | `/v` → `videocut:自更新` | 将本次反馈永久写入 Skills | 每次剪辑后都应执行，是积累个性化的关键 |

### 🛠️ 操作流程

**第一步：下载 Skills**

新建一个专用文件夹（作为剪辑工作空间），在 Claude Code 中输入：

```
帮我下载剪辑 Skills：
- 地址：https://github.com/Ceeon/videocut-skills
- 存放路径：.claude/skills/（直接克隆到此目录，不要创建子文件夹）
```

完成后重启 Claude Code，输入 `/v` 确认 5 个 skill 已出现。

**第二步：安装环境**

输入 `/v` → 选择 `videocut:安装`

AI 自动安装依赖并下载两个模型（FunASR + Whisper，合计约 5GB），等待完成即可。

**第三步：剪口播**

1. 输入 `/v` → 选择 `videocut:剪口播`
2. 告诉 AI 视频文件路径
3. AI 自动执行：转录视频 → 识别口误（逐句检查） → 识别语气词（嗯/哎/诶等） → 识别静音（≥1s） → 生成审查稿
4. 审查稿符号说明：⏸️ = 静音片段（删除），🔴～～ = 重复片段（删除）
5. 人工检查审查稿，如有不满意（如「保留更多的嗯」、「某词识别错了」）继续与 AI 沟通直到满意
6. 输入 `/v` → 选择 `videocut:自更新`，AI 将本次调整写入 Skills
7. 确认后 AI 自动执行剪辑

**第四步：加字幕**

1. 打开 `.claude/skills/videocut:字幕/词典.txt`，添加专有名词（公司名、产品名、频道名等），例如：
   ```
   成峰的频道
   AI产品自由
   我的品牌名
   ```
2. 输入 `/v` → 选择 `videocut:字幕`
3. AI 自动执行：Whisper 转录 → 词典纠正错别字 → 生成字幕文件
4. 确认字幕正确后，AI 自动烧录到视频
5. 如发现词典有遗漏，随时 `/v` → `videocut:自更新` 补充

---

### 📝 避坑指南

- ⚠️ Skills 克隆时必须直接放到 `.claude/skills/` 目录，不要创建子文件夹，否则 Claude Code 无法识别
- ⚠️ 安装环境需下载约 5GB 模型，提前确认磁盘空间充足
- ⚠️ 审查稿确认步骤不能跳过，这是人工介入纠错的唯一窗口，跳过会导致错误传递到最终视频
- ⚠️ 自更新必须主动触发（`videocut:自更新`），不会自动执行，不用就等于放弃了 Agent 的核心价值
- ⚠️ 词典文件要在第一次加字幕前录入，否则第一次转录就会出现专有名词错误

**来源**: Claude Cowork 完全指南：从看懂到会用（上）| **时间**: 2026-01-15 | **原文链接**: `https://mp.weixin.qq.com/s/IIoL5kmH...`

### 📋 核心分析

**战略价值**: Cowork 是 Claude Code 的图形化封装，用"下载即用"的傻瓜壳保留了满血 Agent 能力，首次让非程序员能直接调用本地文件系统级 AI 自动化。

**核心逻辑**:

- **AI 工具进化路径清晰分四阶段**：网页大模型（只能处理喂给它的内容）→ 云端智能体 Manus（有云端虚拟机但碰不到本地文件）→ 本地 IDE 辅助 Cursor/Windsurf（能读写本地文件但主要是辅助）→ 命令行全权代理 Claude Code（能读写任意本地文件、执行命令、联网、调用工具）
- **Claude Code 能力天花板有实证**：Anthropic 官方称 90%+ 代码由 Claude Code 自己写，Cowork 本身就是 Claude Code 花 10 天写出来的；有程序员同时开十几个项目实现百倍提效
- **Claude Code 门槛也是天花板**：需要安装 CLI、配置 API Key（国内用户有额外障碍）、纯命令行界面无按钮、非程序员不知道 terminal 和 npm 是什么
- **Cowork 的核心动作是"降门槛不降能力"**：官方明确 Cowork 与 Claude Code 使用同一套 Agent SDK（让 AI 能自主规划、执行任务的底层技术），不是阉割版
- **底层能力模块三件套**：Connectors（连接器，让 AI 读取 Gmail、Notion、Google Drive 等外部服务）+ Skills（技能包，针对文档/PPT/表格场景优化，用户可自行添加，开放扩展）+ Claude in Chrome（浏览器插件，让 Cowork 能操作浏览器完成网页任务）
- **并行任务队列是核心效率点**：可同时派多个任务并行处理，不需要等一个完成再启动下一个，是"给同事留便签"式协作而非"盯着它干活"
- **当前准入门槛三重限制**：① 需要 Anthropic 官方账号（国内注册有障碍）② 需要 $100/月 Claude Max 会员 ③ 目前仍是 research preview（研究预览版，功能随时可能调整）
- **安全风险是真实存在的硬伤**：Cowork 需要文件系统权限，若指令理解偏差或遭遇 prompt injection 攻击（黑客通过恶意内容让 AI 执行非预期操作），后果不可逆；官方承认 agent 安全"是行业内仍在积极开发的领域"
- **当前仅支持 macOS**，Windows 和跨设备同步在路上，现在入局属于早期红利窗口
- **工具越傻瓜，对人的需求表达能力要求反而越高**：你必须能清晰说出自己要什么，模糊需求会导致 AI 执行偏差

---

### 🎯 关键洞察

**为什么 Cowork 是真正的分水岭而不是炒作**：

前三个阶段的 AI 工具，能力边界都有一个共同限制——要么碰不到本地文件（网页端、云端智能体），要么门槛太高普通人用不了（Claude Code）。Cowork 是第一个同时满足"满血本地文件操作能力 + 普通人可上手"的产品。

逻辑链：Claude Code 能力已被验证（Anthropic 自己用它写代码、Cowork 本身就是它写的）→ Cowork 用同一套 Agent SDK 但换了图形界面 → 门槛从"程序员级别"降到"会说话会传文件" → 本地文件系统自动化首次对非技术用户开放。

**为什么"AI 越傻瓜，对人要求越高"这个判断很重要**：

工具门槛降低后，瓶颈从"会不会用工具"转移到"知不知道自己要什么"。官方演示里文件整齐、需求明确，但真实场景是桌面 1000 个命名混乱的文件——你能不能把需求说清楚，决定了 AI 能不能帮到你。认知停留在"AI = 聊天机器人"层面的人，即使用上 Cowork 也会觉得没用。

**来源**: 用 IDE 方式接入 Agent Skills，实操带你对 Skills 祛魅 | **时间**: 2026-01-15 | **原文链接**: https://mp.weixin.qq.com/s/GUtYULF6...

### 📋 核心分析

**战略价值**: 用 Trae IDE 可视化接入 Anthropic 官方 Agent Skills + OpenSkills 工具链，完整演示从环境搭建、技能调用、自建技能到多技能组合的全流程，彻底拆解其"按需加载"底层机制。

**核心逻辑**:
- Agent Skills 本质是三文件标准：`SKILL.md`（必须）+ 实现代码（可选）+ 依赖文件（可选），不是黑科技
- OpenSkills 是配套 CLI 管理工具，两者关系：Agent Skills = 书，OpenSkills = 图书管理员，AGENTS.md = 目录，AI = 读者
- 核心机制是"渐进式披露"：Stage 1 只读 AGENTS.md（每技能约 50 tokens），Stage 2 用户触发后才调用 `openskills read skill_name` 注入完整 SKILL.md（数千 tokens），实现 100 个技能不撑爆 Context Window
- AGENTS.md 放在项目根目录后，Trae 3.5.20+ 可在设置中配置为"上下文文件"自动加载；旧版本也可能通过约定俗成机制生效
- AI 不一定真正调用了技能，可能用通用能力模拟执行——需主动追问确认，看到 AI 发起 `openskills read` Tool Call 才算真调用
- `skill-creator` 是官方提供的"创建技能的技能"，SKILL.md 的 `description` 字段是触发机制核心，必须写清楚功能、触发场景、使用条件
- 技能创建后必须执行 `openskills sync` 同步到 AGENTS.md，否则 AI 不知道该技能存在
- 多技能组合时，在 prompt 中明确指定技能名称（如"必须至少使用 content-refiner 和 docx 两个技能"），AI 会依次读取两份 SKILL.md 并协同执行
- 技能质量需迭代：一次生成不代表好用，需在真实任务中测试，根据反馈修改 SKILL.md 的 SOP 描述
- SkillsMP 技能市场（https://skillsmp.com/zh）已有 6 万+ 技能，可搜索现成技能直接用或二次改造

### 🎯 关键洞察

**为什么"渐进式披露"是核心设计哲学**：
传统做法把所有技能文档一次性塞入 Context → 爆炸。OpenSkills 的两阶段策略：先给 50 tokens 的摘要目录，只有 AI 判断需要某技能时才注入完整文档。这使得安装 100 个技能的 token 消耗等同于只安装 1 个，且 AI 执行时专注度更高（上下文无噪音）。

**为什么 description 字段决定技能能否被触发**：
AI 在 Stage 1 只能看到 `name` 和 `description`，这是唯一的匹配依据。description 写得模糊 → AI 不知道何时调用 → 技能形同虚设。官方建议格式：`"Use when Claude needs to... for: (1)... (2)..."`，把所有触发场景都写进 description，正文只写操作指南。

### 📋 核心分析

**战略价值**: 蚂蚁集团灵光 App 的完整技术架构拆解——从流式 HTML 全模态输出、组件化渲染、闪应用 iframe 沙箱，到万能自然语言数据查询接口，提供一套可复刻的 Agent 时代生成式 UI 工程范式。

**核心逻辑**:

- **整体流程**：用户提问 → 网络搜索收集信息 → 模型流式生成全模态 HTML 回复 → 按需触发 `generate_flash_app` 工具生成闪应用 → 前端以 iframe 加载渲染
- **全模态输出选 HTML 直出**：模型直接输出 HTML+CSS+JS，配合内置组件库保持样式统一，天然支持交互；代价是需要额外处理安全风险（XSS/注入），而非选择扩展 Markdown 方案
- **流式 HTML 数据结构**：每个流式片段携带 `templateId: "html"`、`templateType: "NATIVE"`，`templateData.text` 字段承载 HTML 片段，前端持续接收并触发增量渲染
- **组件化渲染机制**：HTML 中用 `component-type` 属性标识组件类型（`image`/`echarts`/`video`），JS 运行时内置 `ComponentRegistry` 注册组件状态，`componentMonitor` 上报 `init/succ/error` 三态
- **闪应用触发机制（推测）**：模型调用 `generate_flash_app` 工具，输入为自然语言需求描述，工具立即返回 `artifact_id`，后台另起 Agent 异步生成；主回复 HTML 中先渲染占位容器，前端轮询状态，完成后以 iframe 嵌入
- **闪应用双层架构**：内层是模型按需生成的应用代码，外层是通用基础层（`base.js` SDK + React 运行时 + Tailwind 样式），整体嵌入 iframe
- **万能查询接口 `lingguang.data.fetch(query, schema)`**：用自然语言描述数据需求 + JSON Schema 定义返回结构，后台 Agent 驱动搜索+爬取+LLM 提炼，直接返回结构化数据——彻底解耦"先备数据再生成应用"的串行依赖
- **跨端通信用 CandyJar 协议**：类 JsBridge 机制，闪应用（iframe）与宿主通信，支持千问 App 和 Web 端灵光统一调用，屏蔽平台差异
- **安全策略三层叠加**：CSP 白名单限制资源域 + `form-action: none` 禁止表单提交 + iframe `sandbox` 属性精细授权，模型输出视为不可信内容
- **样式方案**：Tailwind 提供样式变量，模型可直接引用变量或自行编写 CSS，复杂度自适应

### 🎯 关键洞察

**万能查询接口的范式价值**

传统方案：先准备数据结构 → 再让模型生成应用（串行，数据结构未必满足需求）

灵光方案：模型生成应用时直接声明数据需求 → 后台 Agent 实时获取并结构化返回（并行，按需索取）

这个接口本质上是把"数据层"变成了一个可被自然语言驱动的动态服务，应用生成和数据获取解耦，是 Agent 时代平台接口的核心范式转变。

**安全模型的设计逻辑**

模型输出 = 不可信代码。灵光的应对策略是：不阻止模型输出 JS（因为需要交互能力），而是通过 CSP + sandbox + form-action 三层围栏限制其能触达的资源范围，把攻击面收窄到白名单域名内。

### 🛠️ 操作流程

1. **准备阶段（复刻全模态输出）**
   - 搭建流式 SSE/WebSocket 接口，返回结构固定为 `{templateId, templateType, templateData: {text}}`
   - 引入组件库（ECharts、Lottie、视频播放器），注册 `ComponentRegistry`，实现 `componentMonitor(id, status, type, payload)` 上报函数
   - 配置 Tailwind CSS 作为样式基础层

2. **核心执行（闪应用生成链路）**
   - 主 Agent 回复时调用 `generate_flash_app` 工具，传入自然语言需求，立即拿到 `artifact_id`
   - 主回复 HTML 中插入占位 `<div artifact_id="flashapp-xxx" data-async-render-node="params">`，内嵌 `triggerMiniApp(JSON)` 脚本和 `handleArtifactTimeout(120)` 超时控制
   - 后台独立 Agent 异步生成闪应用代码（内层：模型生成；外层：base.js SDK + React + Tailwind 打包）
   - 前端轮询生成状态，完成后以 iframe 嵌入，sandbox 属性设为 `allow-scripts allow-same-origin allow-popups allow-modals`

3. **验证与优化（安全 + 数据）**
   - 在 iframe 宿主页写入完整 CSP meta 标签，白名单域名按实际业务配置（参考上方表格）
   - 实现 `lingguang.data.fetch`：接收 `query`（自然语言）+ `schema`（JSON Schema），后台走 GeneralSearch 工具检索 → Fetch 爬取渲染 JS → LLM 提炼 → 按 schema 结构化返回
   - 存储 API 优先对接服务端，降级逻辑写 `localStorage` fallback
   - 跨端通信实现 CandyJar 协议层（类 JsBridge），统一 Web 端和 App 端的 `lingguang.*` API 调用

### 📝 避坑指南

- ⚠️ **`unsafe-inline` + `unsafe-eval` 已开放**：CSP 中这两项是为了支持模型生成的内联 JS，安全边界完全依赖域名白名单，白名单配置错误会直接导致安全漏洞
- ⚠️ **iframe sandbox 缺 `allow-same-origin` 会导致 base.js SDK 无法通信**：`allow-same-origin` 是 CandyJar 协议跨 iframe 通信的前提，不能删
- ⚠️ **`handleArtifactTimeout(120)` 是硬超时**：闪应用生成超过 120 秒直接失败，后台 Agent 生成复杂应用时需控制在此范围内
- ⚠️ **`lingguang.storage` 降级为 `localStorage` 时数据不跨设备**：服务端存储不可用时静默降级，开发时需测试降级路径
- ⚠️ **视频组件白名单只有 `['mp4', 'webm', 'ogg']`**：模型生成视频组件时若输出其他格式（如 `mov`、`avi`）直接报错，需在 Prompt 中约束格式
- ⚠️ **`form-action: none` 会拦截所有表单**：闪应用内如有登录/搜索表单需求，必须改用 JS fetch 替代原生 form submit
